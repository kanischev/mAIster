## Problem Statement

Solo-technical CEO/CIO/staff-engineers who run several AI coding agents in parallel currently manage them through multiple terminals, ad-hoc tmux sessions, and homegrown shell wrappers (openclaw via ACP, superset.sh, SDD pipelines). They lose context between sessions, can't see project state in one place, can't reliably control what comes out of agent work, and ship drift into PROD because agent output is hard to inspect end-to-end. They need a control plane that runs the spine `Task → Workspace → Flow → HITL → Diff → Merge` end-to-end with a Web UI, leveraging existing CLI Flow frameworks instead of replacing them.

## Demand Evidence

- **No paying customers yet** (honest baseline).
- **Revealed preference**: 2-3 known solo-technical operators are actively building partial homegrown solutions (multi-terminal Claude Code, ACP+openclaw integration, superset.sh, deterministic SDD pipeline attempts) and remain unsatisfied. Building crutches = real demand signal.
- **Validation cohort identified**: user (Albert) + 2 friends (solo-tech CEO/CIO archetype). User is their own first dogfooder.
- **Status quo gaps named explicitly**: superset.sh is single-user local; existing Flow frameworks (aif, spec-kit, superpowers) are single-process; openclaw/Claude Code assume one workspace at a time. No tool covers multi-workspace + HITL in a Web surface.

## Status Quo

Today's stack for the target user:
- **Coding agents**: Claude Code, Codex, opencode, openclaw, Aider, OpenHands.
- **Orchestration**: multiple terminal windows, tmux, ad-hoc scripts.
- **Pipelines**: spec-kit, aif, superpowers (opinionated CLI Flow frameworks, single-workspace).
- **Local automation**: superset.sh — single-user, not team-ready, missing pieces.
- **Task tracking**: YouGile / Jira / Linear or Markdown TODO files; disconnected from agent runs.
- **Review**: manual diff inspection, ad-hoc test runs, no AI-Judge integration.
- **Project memory**: Obsidian / Markdown notes; no automatic lesson capture.

Net cost to the user: cognitive overhead of supervising N agent terminals, copy-paste workflows, lost context across sessions, drift in PROD output, no portfolio view.

## Approaches Considered

### Approach A: Python FastAPI + React (Codex stack)
Backend = FastAPI + SQLModel + SQLite + asyncio subprocess + Git worktree. Frontend = Vite/React + Tailwind + shadcn-lite. SSE for live logs. HITL file-based.
- Effort: M (~1 week solo + AI)
- Risk: Med
- Pros: native fit with aif/spec-kit (in-process Python); Codex spec pre-detailed; mature Python async/subprocess ecosystem.
- Cons: two stacks (Py BE + JS FE); more maintenance for solo; async Python subprocess has edge cases.
- Reuses: aif/spec-kit Python integration in-process; Git worktree primitives; shadcn-ui components.

### Approach B: Next.js full-stack monolith (chosen)
Next.js App Router (server actions + RSC) + Drizzle + **Postgres** (primary; SQLite fallback via dialect switch) + `child_process.spawn` for flow runners + Tailwind + shadcn-ui. SSE through Route Handlers. Python tooling (aif) invoked as subprocess via `uv` from Node.
- Effort: M-L (~1-1.5 weeks solo + AI)
- Risk: Med
- Pros: unified TS stack — one package manager, one build, better DX with Claude Code; rich shadcn/Tailwind ecosystem for HITL surface and Stage 2 polish; user is comfortable with TS and JVM (closer than Python-deep).
- Cons: aif Python → Node bridge via subprocess (overhead); `child_process` + Git worktree less idiomatic than asyncio subprocess.
- Reuses: shadcn-ui ecosystem; Next.js patterns (well-trained in Claude Code); Drizzle's SQL-flavored API close to JOOQ/QueryDSL mental model.

### Approach C: Tauri desktop + embedded backend (lateral)
Tauri shell (Rust) + Python/Node sidecar backend + React frontend. Native dialogs for HITL, single-binary distribution for friends.
- Effort: M-L (~1.5-2 weeks)
- Risk: High
- Pros: single-binary install — friends don't need docker compose; native HITL dialogs richer than web modals.
- Cons: 3+ stacks (Rust shell + JS/TS frontend + Python flow runners); Tauri = niche if user has no prior fluency.
- Reuses: shadcn-ui frontend; Rust subprocess primitives.

## Recommended Approach

**Approach B — Next.js full-stack monolith + SQLite + Drizzle**, chosen by user.

**Rationale (from user)**: JVM-first background, comfortable with TS and Python. NodeJS monolith is the simplest/fastest start. Python tooling (`uv`) is installable and callable from Node via subprocess when aif integration is needed.

**Concrete stack**:
- **Framework**: Next.js 15+ (App Router, server actions, RSC where it fits).
- **DB**: Postgres 16 (primary, in docker compose) via Drizzle ORM with `pg` driver. SQLite kept as optional fallback for ultra-light dev via Drizzle's dialect switch (`DB_URL=file:./dev.db`).
- **UI**: Tailwind + shadcn/ui components.
- **Subprocess / Git**: Node `child_process.spawn` + simple Git worktree wrapper.
- **Live updates**: SSE via Route Handlers (`/api/runs/[id]/stream`), one SSE message per stdout line, `lastEventId`-based reconnect. **Stdout streamed to disk in parallel** (`.maister/<project-slug>/runs/<run-id>/<block-id>.log` via `fs.createWriteStream`) — SSE read-side tails the file, so server doesn't OOM on >10MB block output (folded from eng-review §4 P2).
- **Python bridge**: `child_process.spawn` invokes `uv run <flow-cmd> [--resume <block-id>]` inside the same container (single image with Node 20 + Python 3.12 + uv). Each block = one spawn, runs to natural exit. No long-running process across HITL waits.

- **Error taxonomy** (`lib/errors.ts`): typed `MaisterError extends Error` with discriminated `code` field — `PRECONDITION | SPAWN | NEEDS_INPUT | HITL_TIMEOUT | CRASH | CONFLICT | CONFIG`. UI handlers branch on `code`, not string matching. Folded from eng-review §2 C1.
- **HITL protocol (block-based, revised post-eng-review)**:
  - Flow execution model: each Flow = sequence of **blocks**. A "block" = one subprocess invocation that runs to natural completion (no long-running process kept alive while waiting on user).
  - Block exit conventions:
    - **Exit code 0, no artifact** → block complete, advance to next block (or run → `Done`).
    - **Exit code 0 + `.maister/<project-slug>/runs/<run-id>/needs-input.json` artifact** → block produced a structured question. Run → `Needs input` state.
    - **Exit code ≠ 0** → block failed. Run → `Failed`, error surfaced in UI.
  - `needs-input.json` schema: `{block_id, question, context, response_schema}` (zod-validated).
  - On `Needs input`: UI surfaces request panel with form derived from `response_schema`. User submits answer; maister writes `.maister/<project-slug>/runs/<run-id>/input-<block-id>.json` atomically (tmp + rename helper `atomicWriteJson`) and re-invokes Flow with `--resume <block-id>` (or whatever the Flow framework's resume convention is).
  - On resume: Flow reads input file, advances, continues to next block(s).
  - TTL: 24h in `Needs input` → `Abandoned`. **No zombie subprocess to SIGTERM** — block-based model has no long-running process during the wait.
  - No `chokidar`, no `fs.watch`, no polling needed — block transitions are driven by subprocess exit codes (kernel events, not polling).
  - **Alternative (deferred to Phase 2)**: ACP-style live notification protocol where workspace stays "open" and receives push events. Adds complexity; block-based covers POC needs.
- **Run/Task state in Postgres**: `projects`, `tasks`, `runs`, `workspaces`, `hitl_requests`. No event log table — run stdout/stderr persisted as flat file `.maister/<project-slug>/runs/<run-id>/<block-id>.log`, UI tails the file. Task status is the board axis (`Backlog | InFlight | Done | Abandoned`); run status is the execution axis (`Pending | Running | NeedsInput | Review | Done | Abandoned | Crashed | Failed`). **task ↔ run is 1:N** with monotonic `attempt_number` per task — a terminated-without-merge run sends the task back to `Backlog`, so the user can fire another attempt via the same task card. `tasks.latest_run_id` points at the current attempt.
- **Distribution**: `docker compose up` brings up two services (Next.js + Python + uv app; Postgres 16 container with named volume), host repos bind-mounted under `/repos`. Friends are docker-compose-fluent — onboarding cost accepted.

### `maister.yaml` minimal v1 schema

```yaml
schemaVersion: 1
project:
  name: myapp                    # human label; slug derived (kebab-case)
  repo_path: /repos/myapp        # absolute path to the parent repo
  main_branch: main
  branch_prefix: maister/
flows:                           # list of Flows the project supports
  - id: bugfix
    name: Bugfix
    command: uv run aif run --task '{prompt}' --flow bugfix --workspace '{workspace_path}'
  - id: feature
    name: Small feature
    command: uv run aif run --task '{prompt}' --flow feature --workspace '{workspace_path}'
```

Refuse to register on: `schemaVersion` mismatch, duplicate `flows[].id`
within the file, a Flow command missing the `{prompt}` or `{workspace_path}`
placeholder, slug collision with an existing project, or `repo_path`
collision with an existing project (one repo = one project). User picks
one Flow from `flows[]` at task-creation time. Resist growth — no per-Flow
secrets, no inline scripts, no `if`-blocks.

### Concrete first vertical slice (~1-1.5 weeks target)

1. **Multi-project registry**: register N projects via UI form (paste path to
   directory containing `maister.yaml`) or `MAISTER_PROJECTS_DIR` env
   **recursive** auto-discovery on startup. `maister.yaml` v1 schema
   (above) — `project` block + `flows[]`. Slug derived from `project.name`
   (kebab-case); both slug and `repo_path` are unique across projects (one
   repo = one project). Collisions reject the registration with a clear
   `MaisterError({code: 'CONFLICT'})`.
2. **Task creation UI** (per-project): title + prompt + Flow dropdown
   (populated from current project's `flows[]`). New task lands in
   `Backlog` column.
3. **Per-project board** (2 columns: `Backlog | In Flight`): click **Launch**
   on a Backlog task card → preconditions check → `git worktree add` to
   isolated branch → spawn `uv run <flow-command> ...` subprocess → task →
   `InFlight`. In Flight bucket holds Run statuses `Running | NeedsInput |
   Review | Crashed`. **task ↔ run is 1:N**: a failed/abandoned run sends the
   task back to `Backlog` (Launch button reappears, next click = attempt
   N+1 against the same task). Successful merge → task → `Done`; explicit
   "Discard task" → task → `Abandoned` (terminal).
4. **Portfolio home** (superset.sh-style grid): all active workspaces across
   all projects. Card = project · branch · status · last activity · quick
   actions. Filters by project + status.
5. Live log stream (SSE) in run-detail view; readline-line-split stdout →
   SSE messages; tails `.maister/<project-slug>/runs/<run-id>/<block-id>.log`.
6. HITL (block-based per protocol above): subprocess exits with
   `needs-input.json` artifact in the project-scoped subtree → run →
   `NeedsInput` → UI form from `response_schema` → user submits → atomic
   write of `input-<block-id>.json` → Flow re-invoked with
   `--resume <block-id>` → continues.
7. Diff view: `git diff` raw output rendered in `<pre>` (no syntax
   highlighting in v1).
8. Buttons: mark ready / merge to local main (`git merge --no-ff` on parent
   repo's `main_branch`; on conflict, abort + surface "resolve manually") /
   abandon (cleanup worktree, task → `Abandoned`).
9. Workspace lifecycle: on Next.js startup, reconcile DB `runs` against
   `git worktree list` **per project**. `Running` runs with no live PID →
   `Crashed`, prompt to recover or discard. Worktrees in `Abandoned/Done`
   older than 7d → GC by a cron route across all projects.
10. **Concurrency scheduler**: enforce global `MAISTER_MAX_CONCURRENT_RUNS=3`
    cap. Runs above the cap → `Pending`; auto-promote on slot free; queue
    position shown in UI.
11. Skills connector (read-only): `SKILLS_REPO_PATH` env var; UI shows
    `ls <path>/skills/*/SKILL.md`; no invocation, no injection in POC.

## Success Criteria

**POC complete (T+1 to T+1.5 weeks)**:
- ≥2 projects registered via `maister.yaml` v1 (each with ≥2 Flows in `flows[]`).
- Portfolio home shows active workspaces across all projects.
- Per-project board renders Backlog | In Flight columns.
- Task created on a project board → Launch click → workspace created automatically (git worktree) with precondition checks; task → `InFlight`.
- Headless Claude Code (via aif) runs, logs streamed to UI from the project-scoped log file.
- At least one HITL file-based round-trip works (Flow writes request, user answers via Web UI, Flow resumes).
- Diff visible in UI, merge-to-main button works for clean-merge case; task → `Done`.
- Runs persisted in Postgres, surviving restart (with `Crashed` reconciliation, per-project).
- Global concurrency cap honored: 3 `Running` across projects, 4th queues with position badge; auto-promotes on slot free.
- **Retry loop (ralph-loop friendly)**: abandon attempt N on a task → task auto-returns to `Backlog` → click Launch → attempt N+1 spawns against the same task with a fresh worktree.
- **Registration collisions rejected**: registering a second `maister.yaml` with the same slug or the same `repo_path` returns `MaisterError({code: 'CONFLICT'})`.

**Dogfood (T+1.5 to T+2 weeks)**: run one aif Flow against the maister repo itself, produce a non-trivial PR-sized diff, manually merge. Full self-driven feature ship is a stretch goal, not a gate.

**External validation (T+3 weeks)**: at least 1 of 2 friends installs maister on their machine, configures one project, ships at least one PR end-to-end through a maister Flow on their own repo. If 0/2 — thesis not validated; reassess wedge.
