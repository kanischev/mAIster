### Sweep sibling fields when fixing a boundary-validation bug
**Source**: 2026-05-26-12.45.md, 2026-05-26-12.53.md
**Rule**: When fixing a single Zod field whose value flows into `path.resolve` / `path.join` / `fs.*` / `child_process.spawn` argv / SQL / HTML, audit every sibling field in the same schema in the same pass. Apply the equivalent regex / `.refine()` / length cap to every sibling whose value reaches the same class of sink. This does NOT violate the general "minimal changes / no scope creep" rule — siblings that touch the same sink are part of the same security-fix scope, not "while you're there" cleanup. Reason: in patch 12.45 a regex was added to `runId` / `stepId` but the equally-affected `worktreePath` (consumed by `child_process.spawn` `cwd`) was missed; a follow-up round (patch 12.53) had to land the same fix on the sibling.

**Generalize the sweep far beyond Zod fields.** When fixing ONE instance of a defect class, grep and fix EVERY sibling in the SAME pass: every reader of a leaked field (board / inbox / run-detail all read permission `schema`), every recorder of a value (runner ×2 / takeover / `recordDefaultArtifacts` — the highest-volume recorder is the one usually missed), every route using a shared wrapper, every OpenAPI operation, secret-ref validators (`headers` vs `signing_secret_ref`), and token-revocation sites (detach vs `updateAgentLink({enabled:false})`). Fix the **class**, not the cited line, and RECORD the grep in the fix close-out — a prior patch prescribed exactly this and the next patch records it still didn't happen. A `// mirrors X:line` / `// Task N reconciles both` comment is the author confessing known duplication that will drift — treat it as a must-fix, not a note.

### Renumbering a shared ADR / migration namespace is grep-to-zero, never a global find-replace
**Source**: 2026-06-10-23.57.md, 2026-06-11-12.51.md, 2026-06-11-14.45.md, 2026-06-09-14.23.md, 2026-06-09-18.47.md
**Rule**: When a sibling branch must renumber its ADR/migration because main already claimed the number, NEVER run a global `sed s/ADR-071/ADR-072/`. Two sibling features each own an adjacent number and each mentions the OTHER's number in prose, so a global replace over-reaches (rewrites a legitimate sibling ref like `[ADR-072](#adr-071-…)`) and the separately-scoped ADR vs migration passes desync (under-reach). Procedure:
1. Enumerate every occurrence and classify each by the feature it names; only the renumbered feature's own references change.
2. Run **ONE** pass covering BOTH the ADR number and the migration number together.
3. Finish with **grep-to-zero** across ALL forms: `ADR-NNN`, the lowercase `adr-nnn` **anchor slug as a SECOND sed pass**, `NNNN_name`, AND bare `` `NNNN` ``. `CLAUDE.md` and `.ai-factory/ROADMAP.md` ARE part of the surface (not just `docs/`); exclude `.ai-factory/plans` / `patches` archives and generated `web/.next`.
4. Leave opaque PERSISTED id-segment literals frozen (`runs.…:adr071:rework-comments` is never parsed back to a number; "correcting" it orphans existing rows) — add a one-line WHY comment so a future reader doesn't "fix" it.
5. Verify the migration number against `migrations/meta/_journal.json`, not the prose (`0029` was documented in 5 places but shipped as `0030`).

Do the renumber/re-port as its OWN focused session, AFTER rebasing/merging onto the integration target — never reconcile docs against a branch baseline that is behind main (editing a stale pre-feature copy re-derives corrections main already shipped). macOS BSD-grep gotcha: for glob-unsafe paths like `app/api/[slug]/…`, `grep -rlZE … | xargs -0 perl` malfunctions — use `grep -rlE … | while IFS= read -r f; do perl -pi -e '…' "$f"; done` (quoted `"$f"` prevents `[slug]` shell-globbing; one perl per file).
