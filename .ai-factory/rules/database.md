# Database Rules

> Area-specific conventions for persistence: Drizzle ORM, Postgres 16 (and
> SQLite via dialect switch for ultra-light dev), schema, migrations,
> queries. Loaded after `rules/base.md`. Authoritative source: root
> `CLAUDE.md` §Stack and design doc.

## Rules

- ORM is Drizzle. Do not introduce Prisma, TypeORM, Knex, or hand-rolled query builders. SQL-flavored, JOOQ-like mental model.
- Primary database is Postgres 16 in docker-compose with a named volume. SQLite is supported only via Drizzle dialect switch (`DB_URL=file:./dev.db`) for ultra-light dev — never as a production target.
- Schema lives in `web/lib/db/schema.ts`. Core tables: `projects`, `executors`, `flows`, `tasks`, `runs`, `workspaces`, `step_runs`, `hitl_requests`. Add new tables only when an entity has independent lifecycle — do not split for cosmetic normalization.
- Database client lives in `web/lib/db/client.ts`. Import `db` from there exclusively. Do not instantiate connections elsewhere.
- Migrations are generated by `drizzle-kit` into `web/lib/db/migrations/`. Never hand-edit committed migration files. To change a migration, generate a new one.
- Always run a migration on startup (or via a deploy step) before the app accepts traffic. Do not rely on auto-migrate at first-query time.
- IDs are server-generated. Use deterministic ULIDs/UUIDs or DB-side identity columns — never client-supplied IDs for new rows.
- All timestamps are stored in UTC. Convert in the UI layer, not in queries.
- Cross-table reads use Drizzle joins; do not N+1. If you find yourself looping queries, model the join.
- Query helpers live next to the schema (`web/lib/db/queries/<entity>.ts` is acceptable) and are imported by `web/lib/*` orchestration modules — never directly by Route Handlers or Client Components.
- Never expose the Drizzle client or schema types to the browser. `web/lib/db/*` is server-only.
- Treat the `runs` table as the source of truth for run state. The filesystem (`.maister/<project-slug>/runs/<run-id>/`) holds artifacts; the DB holds the state machine. The `tasks` table is the source of truth for board status (`Backlog | InFlight | Done | Abandoned`).
- **`projects` table**: `slug` and `repo_path` are both UNIQUE columns. One repo = one project. Archival is soft via `archived_at`; never hard-delete a project row in the current target.
- **`tasks` table**: `task ↔ run is 1:N` (one task can spawn many runs over its lifetime). Current schema uses `tasks.attempt_number` as a mutable high-water mark; designed run-attempt schema moves immutable attempts to `runs`.
- State transitions for `runs.status` are explicit: `Pending → Running → NeedsInput → Running → Review → Done | Failed | Abandoned | Crashed`, with designed `NeedsInputIdle` checkpoint/resume. Reconcile on startup if the process state disagrees with the DB.
- State transitions for `tasks.status` (driven by the **latest** run):
  - new task → `Backlog`.
  - Launch click → task → `InFlight`. Stays `InFlight` while the latest run is `Pending | Running | NeedsInput | Review | Crashed`.
  - Latest run → `Done` (merged) → task → `Done` (terminal).
  - Latest run → `Failed | Abandoned` → task → `Backlog` (retry-eligible; next Launch spawns attempt N+1).
  - Explicit "Discard task" action → task → `Abandoned` (terminal).
- Indexes on hot paths include `runs.status`, `runs.project_id`, `runs.task_id`, `tasks.project_id`, `tasks.status`, `workspaces.run_id`, `step_runs.run_id`, `hitl_requests.run_id`, `projects.slug` (unique), `projects.repo_path` (unique).
- Never store secrets or API keys in the database. They live in `.env` server-side.
- Postgres connection string and SQLite path are read from `DB_URL` only. Do not split into separate vars or hardcode hosts.
- For development convenience, prefer Postgres in docker-compose over SQLite so determinism issues are caught against the production engine.
- Indexes go in migrations, not in ad-hoc `CREATE INDEX` statements. Index any column used in `WHERE`, `ORDER BY`, or join conditions on the hot paths.
- For destructive operations (drop column, drop table, type change) generate a new migration that runs in two steps: deploy backwards-compatible schema first, then remove the old shape after the app rolls.
- Wrap multi-statement updates in a Drizzle transaction (`db.transaction(async (tx) => ...)`). Long-running operations (worktree, subprocess) stay outside the transaction — only the DB writes go inside.
