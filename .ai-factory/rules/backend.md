# Backend Rules

> Area-specific conventions for the server-side runtime: Route Handlers,
> Server Actions, `web/lib/*` modules, the `supervisor/` daemon, ACP
> sessions, SSE, worktree management. Loaded after `rules/base.md`.
> Authoritative sources: root `CLAUDE.md` (locked architectural
> decisions), `web/CLAUDE.md`, and `.ai-factory/ARCHITECTURE.md`.

## Rules

- Agent processes (`claude-agent-acp`, `codex-acp`) are spawned and owned by `supervisor/`, NEVER by Next.js. Permission HITL stays live through supervisor deferreds; checkpoint/idle resume is implemented.
- The web tier reaches the supervisor through `web/lib/supervisor-client.ts` ONLY. No other `web/lib/*` module spawns agents, and no `web/*` module imports from `supervisor/*`. Wire contract: HTTP + SSE.
- Drive run-state transitions from **ACP notifications** (live path: supervisor SSE `session.update` / `session.permission_request` / `session.exited` / `session.crashed`) plus durable HITL artifacts (`input-<step-id>.json`). Never use `fs.watch`, `chokidar`, or polling.
- Supervisor exit conventions: `session.exited` (clean exit code 0 OR intentional shutdown via `DELETE /sessions/:id`) â†’ block done, advance; `session.crashed` (non-zero exit / killing signal / heartbeat orphan) â†’ run state `Crashed`. Structured HITL artifacts on disk drive `NeedsInput` / `NeedsInputIdle` transitions.
- Supervisor pipes child stdout to `.maister/<project-slug>/runs/<run-id>/<step-id>.log` via `fs.createWriteStream` (`flags: "a"`) in parallel with the SSE bridge. Server must not buffer >10MB step output in memory; per-session in-memory ring buffer is capped at 1000 events for `lastEventId` replay.
- One SSE message per stdout line. Include a monotonically increasing `id` so reconnects with `lastEventId` resume cleanly.
- Throw `MaisterError` with a discriminated `code` (`PRECONDITION | SPAWN | NEEDS_INPUT | HITL_TIMEOUT | CRASH | CONFLICT | CONFIG | EXECUTOR_UNAVAILABLE | FLOW_INSTALL | ACP_PROTOCOL | CHECKPOINT`) for known domain failures inside `web/`. The `supervisor/` daemon uses an internal `SupervisorError` (`PRECONDITION | SPAWN | EXECUTOR_UNAVAILABLE | ACP_PROTOCOL | CHECKPOINT | CRASH`) translated to JSON `{ code, message }` at the HTTP boundary; `supervisor-client.ts` re-throws as `MaisterError`. Never throw plain `Error` for domain errors. Never wrap third-party errors to "look typed" â€” extend the taxonomy if a new failure mode is real.
- All writes under `.maister/<project-slug>/runs/<run-id>/` go through `atomicWriteJson` (tmp + rename via `lib/atomic.ts`). The Flow may read these files mid-write otherwise.
- HITL handoff sequence: runner inserts `hitl_requests` before `NeedsInput`; user submits through `POST /api/runs/{runId}/hitl/{hitlRequestId}/respond`; permission responses resolve `/sessions/:id/input`; form/human responses write `input-<step-id>.json` via `atomicWriteJson`; runner owns `NeedsInput -> Running`.
- TTL: a run sitting in `NeedsInputIdle` for 24 hours transitions to `Abandoned` through the keep-alive sweeper.
- Run creation preconditions (enforced in `POST /api/runs` before spawn): project exists & not archived, parent repo clean, target branch free, worktree path free, global concurrency cap not hit. Reject with `MaisterError('PRECONDITION', â€¦)` if any check fails â€” never spawn against a dirty repo.
- **task â†” run is 1:N**: a task may spawn many runs over its lifetime (retry / ralph-loop). Current schema uses `tasks.attempt_number` as the high-water mark; designed run-attempt schema moves the immutable attempt stamp to `runs`.
- **Task lifecycle**: new task â†’ `Backlog`. Launch click â†’ preconditions â†’ run spawned â†’ task â†’ `InFlight`. Latest run merged â†’ task â†’ `Done` (terminal). Latest run `Failed | Abandoned` â†’ task auto-returns to `Backlog` (Launch reappears). Only an explicit "Discard task" UI action sets task â†’ `Abandoned` (terminal). Run-level abandon is **not** task-level abandon.
- **Board interaction**: a Backlog task card carries a **Launch** button; click triggers `POST /api/runs` with `taskId`.
- Concurrency cap: `MAISTER_MAX_CONCURRENT_RUNS=3` (env-configurable, **global** across projects). Runs above the cap enter `Pending` and auto-promote when a slot frees. The UI surfaces queue position. No per-project sub-cap.
- Multi-executor via ACP: both `claude` (`claude-agent-acp` binary) and `codex` (`codex-acp` binary) are required. The supervisor dispatches on `executor.agent`; no bespoke `ExecutorAdapter` interface â€” ACP is the executor contract. Per-step executor override resolution: run launcher override â†’ task override â†’ project per-flow `executor_override` â†’ project `default_executor` â†’ flow's `recommended_executor` (highest priority wins). New executors land by adding to the `executors[]` table â€” no code change in call sites.
- Crash recovery on Next.js + supervisor startup: reconcile the `runs` table vs `git worktree list` **per registered project** AND vs the supervisor's live `GET /sessions` set. `Running` rows with no live ACP session and no checkpoint become `Crashed`; `NeedsInputIdle` rows with a valid `acp_session_id` checkpoint stay valid for resume. The UI surfaces "Recover or discard" on the relevant project board and the portfolio card.
- Use `git merge --no-ff` for run merges. On conflict, abort the merge cleanly and surface `MaisterError('CONFLICT', â€¦)`; the run stays in `Review` and the UI tells the user to resolve manually. No auto-resolve.
- GC: a cron Route Handler (`/api/cron/gc`) removes `Abandoned/Done` worktrees older than 7 days across all projects. Triggered externally (cron job or scheduled fetch), not via setInterval.
- `maister.yaml` v2: `project` block + `executors[]` (with optional `env` for env-router and `router: ccr` for Claude Code Router) + `flows[]` (each tag-pinned via `version`). Validate `schemaVersion: 2` strictly and refuse to register on mismatch, on duplicate `executors[].id` or `flows[].id`, on unknown `default_executor` / `flows[].executor_override` references, or on a malformed Flow manifest (`flow.yaml`). Do not silently migrate.
- Project registration: register N projects via `POST /api/projects` (path to dir containing `maister.yaml`) or `MAISTER_PROJECTS_DIR` env **recursive** auto-discovery on startup. Slug derived from `project.name` (kebab-case). Both `slug` and `repo_path` are unique across registered projects (one repo = one project). Collisions on either column reject the registration with `MaisterError({code: 'CONFLICT'})`. Auto-discovery skips collisions with a logged warning (no hard crash). Project archival is soft (`archived_at`); no hard delete in the current target; archived `repo_path` stays reserved.
- Server-only secrets: read API keys and tokens from `.env` server-side only (web AND supervisor). Never log them, never include them in SSE payloads, never expose them in subprocess argv visible to the frontend. Per-executor `env` (`ANTHROPIC_BASE_URL`, `ANTHROPIC_AUTH_TOKEN`) is injected into the spawned child env by `supervisor/src/spawn.ts`; spawn logs report `hasEnv: true|false` only, never the values. The lifecycle integration test enforces this with a sentinel-in-log assertion.
- Suggested layering inside `web/lib/`, lowest first: `errors` â†’ `atomic` â†’ `config` â†’ `db` â†’ `executors` â†’ `flows` â†’ `projects` â†’ `worktree` â†’ `supervisor-client` â†’ `scheduler` â†’ `reconcile`. A lower module never imports a higher module. There is NO `lib/runner.ts` â€” agent spawn lives in `supervisor/`, not Next.js.
- `web/lib/*` modules are server-only. Add a top-level `import 'server-only'` to any module that must never be bundled for the browser.
- Route Handlers stay thin: validate input, call into `web/lib/*`, format the response. Do not put orchestration logic, subprocess spawning, or `execSync` calls in `app/api/*/route.ts`.
- If two Route Handlers need shared logic, extract it into `web/lib/`. Do not call one Route Handler from another.
- **Process-lifecycle escalation (SIGTERM â†’ exit-event â†’ SIGKILL)**: Node's `subprocess.killed === true` after `subprocess.kill(sig)` successfully *sends* the signal â€” NOT after the child exits. Shutdown / kill-escalation logic MUST gate on the observed `exit` (or `close`) event, never on `subprocess.killed`. Required pattern: send SIGTERM â†’ race the exit promise against a grace timer â†’ if the timer wins, send SIGKILL and await the exit event before declaring the process gone. Tests for any process supervisor MUST include a SIGTERM-resistant fixture (a child that traps and ignores SIGTERM, or hangs) and assert the port is freed / pid is gone after `shutdown()` returns. Cooperative-exit happy-path tests do not exercise this branch.
- **Managed-sidecar health probes MUST validate target identity, not just liveness**: a readiness probe for a supervisor-managed sidecar (CCR, future routers, future proxies) MUST identify the *intended* service, not merely "something answered on host:port". Accept ONE of: (a) a target-specific endpoint that returns a known body shape (e.g. `GET /health` returning `{"service":"ccr",â€¦}`), (b) a recognizable response header set by the target, or (c) a response-body schema check. Generic `GET /` returning any non-5xx is insufficient â€” `EADDRINUSE` on the spawned child can leave the port owned by an unrelated process that happens to answer 200/404, and the manager will transition to ready against the wrong process. Additionally, register a `child.once("exit", ...)` listener BEFORE starting the probe loop; if the target child exits before becoming ready, abort the probe immediately and surface the failure with the exit code/signal in the error message. Mandatory test shape: another HTTP server bound to the configured port BEFORE the manager spawns the target child â€” the manager must fail fast, not transition to ready.
- Scope labels are planning labels, not blockers. Large features such as AI-Judge, background agents, Telegram, A/B runs, custom ACP extensions, guard enforcement, plugin sandboxing, extra executors, durable orchestration, auth/RBAC, Flow designer UI, event log table, and external issue-tracker sync need an explicit plan and contract updates before implementation.
- **Cross-resource identifiers come from server state, never the body**: when an HTTP route operates on a server-held resource (live ACP session in the supervisor registry, run row, project row, workspace) and the same resource record holds an identifier (project slug, run id, step id, workspace path), the handler MUST derive that identifier from the server-state lookup â€” never accept it as a request-body field. Even when both sides "should" agree, a body-supplied locator opens a cross-resource injection vector: an attacker (or buggy caller) with a valid sessionId can use it to write under a different runId's directory tree, fetch another tenant's row, or mutate a sibling resource. Drop the redundant body field; if the same field is genuinely needed cross-tenant (multi-tenant + un-authenticated), add an explicit equality check against the server-state value with a stated 409 mismatch response. Regex constraints on a body field protect syntactic shape only, NOT trust-boundary integrity.
- **Two-phase commit for routes whose terminal response depends on a downstream side-effect**: any state-changing route whose successful response depends on an external side-effect outside the route's own DB (HTTP call to supervisor, file write under `.maister/...`, queue publish, RPC) MUST split the DB commit into two phases. Phase 1 (inside a transaction with `SELECT FOR UPDATE` + terminal-status guard + idempotency-marker null check) commits "user intent" â€” typically the request body or chosen option â€” WITHOUT the idempotency marker (`respondedAt` / `completedAt` / `processedAt`). The side-effect runs OUTSIDE any transaction. Phase 2 marks the idempotency column only after the side-effect acks. Failure classification is mandatory: downstream 404 (deferred / target lost) â†’ terminal in the current milestone, transition the parent resource to `Failed` / `Crashed` AND mark the row terminal-failed (return 410); downstream 5xx / network â†’ retryable, leave the marker null and return 503 so the caller can replay with the latest user intent. Returning success-like statuses (2xx) BEFORE the side-effect is acked is a defect; deferring the terminal transition to a future reconciler milestone is also a defect â€” surface it in the current milestone.
- **No hidden deferreds**: any catch / error-handler in code that observes a remote-deferred-creation signal (SSE event named `*.request` / `*.permission_request` / `*.awaiting_*`, a queue message of type "request", a long-poll callback registration on another process) MUST explicitly release the deferred (resolve / cancel / reject â€” pick the right semantic for the failure cause) before the handler returns. "Log the error and continue" is wrong: the deferred-creating side (typically the supervisor or a sibling service) is now leaked until its own keep-alive timeout fires, which is invisible to operators (no error log on this side) and presents to users as "the agent silently hangs". Regression tests MUST include a case that simulates the failure in the releasing-side code AND asserts the release API was invoked exactly once on the creating side (via a spy / mock). Without that test, the contract is fragile.
- **Flow bundles are content-addressed per run**: Flow runtime code MUST derive the flow bundle path via `systemCachePath(flowRefId, runs.flowRevision)`. Reading `flows.installed_path` at run execution time is forbidden â€” that column is the project's mutable current-installation pointer (re-installs overwrite it), not the per-run truth. The per-run truth is the git SHA snapshotted into `runs.flow_revision` at launch, which addresses the immutable `~/.maister/flows/<flow_ref_id>@<short_sha>/` directory. Resumes, re-spawns, and recovery all resolve the same bundle bytes regardless of intervening flow upgrades.
