# M38 decide routing — validate the routing SOURCE's producer at compile, not just the consumer wiring

**Date:** 2026-06-22 19:56
**Files:** web/lib/flows/graph/compile.ts, web/lib/flows/graph/__tests__/compile-decide.test.ts, web/lib/flows/edge-style.ts(+test), web/components/flows/node-form/node-side-form.tsx, web/lib/flows/node-side-form-labels.ts, web/lib/flows/editor/editor-labels.ts, web/lib/flows/graph/__tests__/run-context.test.ts, scripts/verify-m38-surface.mjs, docs/decisions.md, docs/system-analytics/flow-graph.md, docs/flow-dsl.md, .ai-factory/specs/feature-m26-structured-output-run-context.md, .ai-factory/requests/2026-06-22-g4-guardrail-hooks.md
**Severity:** medium

## Problem

`/aif-review` of the M38 flow-routing branch surfaced silent logical holes in
the new `decide` table, plus doc drift and DRY/dead-code:

- A `decide:{from:verdict}` node with **no** verdict-producing gate (or >1)
  compiled fine but always fell through to `default` at runtime — no signal.
- A `decide:{from:output.<path>}` node with **no** `output.result` compiled fine
  but `vars` was always empty → routing always terminated.
- `output.result.on_mismatch` without `rework.commentsVar` compiled fine but the
  "error-feedback injection" the feature is named for silently did nothing.
- Spec/ADR said the run-context pointer is appended in "both session modes" and
  that `from:verdict` routes by the "calibrated" verdict — both contradict the
  code (only `new-session` graph dispatch exists; calibration is bypassed under
  `decide`). A near-trivial test asserted "no SECRET" with no secret ever
  injected. The Studio label bundle was duplicated verbatim across two builders.

## Root Cause

The feature validated the **consumer** side of routing thoroughly (every
producible `case.target` ⊆ `transitions`, `when` parses, allow-list guard at
runtime) but never validated that the chosen routing **source** has its data
**producer**. `from: X` is a source selector; the engine resolved it lazily and
fell back gracefully (default / terminal) when the producer was absent — correct
fail-safe behavior, but a silent footgun for a flow author. Same class for
`on_mismatch`: it required a `rework` block but not the one sub-field
(`commentsVar`) the rework actually consumes.

## Solution

Co-located three compile-time guards in `verifyDecideAndOnMismatch` (compile.ts),
where the existing decide cross-ref checks already live, all `CONFIG`:
- `from: verdict` → exactly one `ai_judgment`/`skill_check` gate (covers the
  zero-gate silent-default AND the >1-gate last-wins ambiguity).
- `from: output.<path>` → `output.result` must be declared.
- `on_mismatch` → `rework.commentsVar` must be set.

These guards only fire when `decide`/`on_mismatch` is present (a brand-new M38
surface), so no existing flow is affected; misconfigured authored flows are now
rejected at install (`compileManifest` runs in the authored-flow validate path)
rather than misbehaving at launch. Also: collapsed the duplicated `NodeSideForm`
label bundle (editor-labels now calls `buildNodeSideFormLabels`), removed dead
`isConditionalOutcome`/`blankDecide` duplication, made the trivial run-context
test assert the closed key-set, hardened the verify-m38 gate (real AsyncAPI +
branch-scoped migration checks; correct env paths), and reconciled the
spec/ADR/analytics prose with the as-built behavior.

## Prevention

- When a feature adds a **source selector** (`from:`, `mode:`, `kind:` that picks
  where a value comes from), add a compile-time check that the selected source's
  **producer exists** — do not rely on graceful runtime fallback, which reads as
  "works" while silently routing to the default path.
- A compile guard that requires a parent block (`rework`) must also require the
  specific **sub-field the runtime consumes** (`commentsVar`), or the feature
  no-ops silently.
- Adding stricter validation inside `compileManifest`/`verifyDecideAndOnMismatch`
  is safe ONLY after grepping **every** fixture that compiles a manifest — unit
  (`*-decide.test.ts`), integration (`*.integration.test.ts`), AND the e2e seed
  (`seed-e2e.ts`) — because `compileManifest` runs on read paths (run-detail,
  packages viewer, inbox, topology), so a non-compliant fixture turns a read view
  into a throw, not just a test failure.
- A unit test whose negative assertion (`not.toContain("SECRET")`) can never fail
  because the trigger is never injected proves nothing — assert the closed
  projection (exact key-set) instead.
