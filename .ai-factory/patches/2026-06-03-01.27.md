# External-gate report: run-row lock didn't protect the gate row it mutates

**Date:** 2026-06-03 01:27
**Files:** web/lib/flows/graph/gate-store.ts, web/app/api/v1/ext/runs/[runId]/gates/[gateId]/report/__tests__/route.integration.test.ts
**Severity:** high

## Problem

A follow-up adversarial Codex review of the M16 external-gate report path found
that the previous concurrency fix protected the wrong row. `reportExternalGate`
SELECTs the latest live `external_check` `gate_results` row, then later UPDATEs
that row **by id only**. The route wraps this in a transaction whose only lock is
`SELECT id FROM runs WHERE id = $1 FOR UPDATE` — a lock on `runs`, not on
`gate_results`.

That run-row lock serializes report-vs-report (two CI webhooks racing), but it
does **not** serialize gate-state writers that mutate the gate row without taking
the run lock:

- **HITL override** (`markGateOverridden` → status `overridden`): a sealed gate.
- **Rework staleness** (`markDownstreamStale`, called from `runner-graph` rework
  and takeover-return → status `stale`): invalidated evidence.

If either committed after the report's live-row SELECT but before its UPDATE, the
report wrote the row back to `passed`/`failed` and recorded a `test_report`
artifact — reopening a sealed gate or restoring invalidated evidence, hiding a
real review blocker.

## Root Cause

The same TOCTOU-class mistake as the prior patch, one level down. The earlier
fix correctly serialized report-vs-report by locking the parent `runs` row and
re-reading `runs.status` under it — but the *gate row* mutation has its own
check-then-act (SELECT live row → UPDATE that row by id) and its own set of
concurrent writers that don't participate in the run lock. A lock only protects
the rows it covers; the protected invariant lived on `runs`, while the data being
mutated lived on `gate_results`. The by-id UPDATE was unconditional, so whatever
status a concurrent writer had committed was silently overwritten.

## Solution

Compare-and-swap the gate row on its **exact observed status** inside
`reportExternalGate`, instead of an unconditional by-id UPDATE. New private
`casLiveTransition(d, id, fromStatus, toStatus, extra)` runs
`UPDATE gate_results SET status=…, ended_at=now(), … WHERE id = $1 AND status =
$fromStatus RETURNING id` and returns whether a row matched. Both mutation sites
use it:

- non-supersede: `transition(live.id, status)` → `casLiveTransition(…, live.status,
  args.status, { verdict })`.
- supersede: `markGateStale(live.id)` → `casLiveTransition(…, live.status, "stale",
  {})` before appending the fresh row.

When the CAS matches 0 rows (a concurrent override/rework changed the status out
from under the report), it throws `GateNotReportableError`, which the route
already maps to 404 — the report is rejected, the transaction rolls back, no gate
or artifact mutation. The CAS UPDATE also acquires the gate row lock when it runs,
so an in-flight writer is waited out and then re-evaluated post-commit.

Why CAS and not a gate-row `FOR UPDATE`:
- DB-dialect-agnostic — `gate-store.ts` runs on both Postgres and SQLite; CAS
  needs no `isPostgres()` branch (SQLite rejects `FOR UPDATE`).
- Lock-free and fully contained to the single consumer (`reportExternalGate` is
  called only by the report route — verified).
- The other writers (`markGateOverridden`, `markDownstreamStale`) stay
  unconditional/authoritative; the report yields to them, which is the correct
  asymmetry (a human override or a rework must win over a late external webhook).

The run-row `FOR UPDATE` is **kept** — it is what makes report-vs-report correct
(the second report reads the first's committed row under the lock, sees the same
commit, and updates in place instead of appending a duplicate). Removing it would
break the same-commit concurrency test; the gate CAS does not subsume it.

Exact-status CAS (`status = live.status`) — not set-membership
(`status IN LIVE_EXTERNAL_STATUSES`) — is required: `stale` is a live, re-reportable
status, so a set-based CAS would still let a report racing with rework restore a
just-staled gate. Exact-match rejects the report whenever the row changed at all
during the race, while still allowing legitimate sequential re-reports on an
already-stale gate.

Added a deterministic two-racer regression (`it.each` over `overridden` / `stale`):
a blocker holds the conflicting gate write uncommitted on a second pooled
connection (so the report's SELECT still sees `pending`), the test waits via
`pg_stat_activity` until the report is parked on the gate-row lock at its CAS
write, then commits the blocker — landing the change squarely in the SELECT→write
window. Asserts the report is rejected (404/409), the gate stays
`overridden`/`stale`, and no fresh passed/failed row or `test_report` artifact is
written. Red/green verified: with the status guard dropped to a by-id UPDATE both
racers return 200 and reopen the gate (tests fail); restored, all 16 pass.

## Prevention

- A row lock protects only the rows it covers. When a transaction mutates table B
  (here `gate_results`) but locks a row in table A (here `runs`), any writer that
  touches B without going through A's lock is unserialized. Put the concurrency
  control on the row whose invariant you are protecting.
- For an unconditional `UPDATE … WHERE id = $1` that follows a `SELECT` of that
  row, ask "who else writes this row, and do they share my lock?" If not, make the
  UPDATE a CAS on the expected state and check the affected-row count — fail closed
  when it's 0.
- CAS on the EXACT observed value, not a permissive set, when the requirement is
  "reject if it changed under me." Set-membership CAS silently permits
  within-set transitions.
- To test a check-then-act window deterministically without sleeps: hold the
  conflicting write uncommitted on a second connection (forces the racer to block
  on the row lock at exactly its write), wait on `pg_stat_activity`
  (`wait_event_type='Lock'`) until it is parked, then commit. Lock ordering — not
  timing — sequences the interleave.
