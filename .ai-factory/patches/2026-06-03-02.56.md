# M16 external-operations: /aif-verify findings (spec contract, trust-boundary, DRY, test gaps)

**Date:** 2026-06-03 02:56
**Files:** web/lib/flows/graph/external-gate-readiness.ts (new), web/lib/flows/graph/{evidence-readiness,gate-store}.ts, web/lib/queries/{board,portfolio,readiness}.ts, web/lib/tokens/ext-handler.ts, web/app/api/v1/ext/runs/route.ts, web/app/api/v1/ext/projects/[slug]/tasks/route.ts, web/app/api/v1/ext/projects/[slug]/tasks/[taskId]/route.ts, docs/api/external/operations.openapi.yaml, docs/api/web.openapi.yaml, docs/error-taxonomy.md, docs/db/integrations-domain.md, + tests (external-gate-readiness.test, ext-runs, gate-store-external, tokens, report route)
**Severity:** high

## Problem

A deep `/aif-verify` pass on the M16 branch (after the suite was already green) found defects the mechanical gates can't see: 2 HIGH (a published HTTP status contract that contradicts the code; a cross-project existence/state oracle), 4 MEDIUM (quadruplicated readiness logic with drift; two spec overstatements; an untested config path), and several LOW (a copy-pasted error mapper that disagreed across siblings; doc/test nits).

## Root Cause

- **H1 (OpenAPI ↔ code):** each ext write route owned a private `httpStatusForCode`; the OpenAPI was authored omitting the `409` the routes return for the *dominant* `PRECONDITION` failures, and `/runs` documented a `422` it never emits (its `CONFIG` mapped to 400). The spec drifted from code AND from `error-taxonomy.md`.
- **H2 (trust-boundary):** `launchRun` (shared with the internal route) resolves the task project-unscoped and throws `PRECONDITION` ("not found" vs "not in Backlog" → 409) BEFORE `ctx.authorize`. A project-A token + a project-B task UUID could distinguish existence/backlog-state — the existence-hide invariant the rest of M16 upholds was missing on this one route. The existing trust-boundary test only covered the exists+Backlog→404 case, so the not-Backlog 409 leak went uncaught.
- **M1 (DRY):** the external_check readiness-collapse (live-attempt filter → latest-per-gate by createdAt/id → `{passed,overridden}` allow-list) was hand-rewritten in 4 readers with `// mirrors X:line` comments, and had already drifted (single-run `gateId` key vs multi-run `${runId}:${gateId}` key).
- **M4 (test gap):** `resolveGateExternalConfig` (the manifest→config resolver) had zero test call sites; every report-route test seeded an empty manifest → always hit the default-`true`, so the manifest `staleOnNewCommit:false` opt-out was never proven end-to-end.

## Solution

- **H1/L1:** one shared `httpStatusForExtCode` in `ext-handler.ts` (`CONFIG→422` canonical across the surface; `PRECONDITION`/`CONFLICT`→409), replacing the 3 private copies; `/runs` body-parse fixed 400→422. OpenAPI gained `409` on the three write routes (referencing a broadened `Conflict` response); `error-taxonomy.md` now states the canonical 422 (was a "422/400" hedge) + a 422 table row.
- **H2:** fixed **route-local** in the ext `/runs` route — pre-resolve the task scoped to `ctx.projectId` and return a uniform 404 (existence-hide) BEFORE `launchRun`. The shared service stays byte-unchanged (internal route unaffected). Mirrors the task GET/PATCH + gate-report routes.
- **M1:** extracted `web/lib/flows/graph/external-gate-readiness.ts` — `collapseLatestExternalPerGate(rows, keyOf)` + `EXTERNAL_GATE_READY_STATUSES` + `isExternalGateReady`. The `keyOf` param preserves the intentional single-run vs multi-run keying difference. **Crucially, the plan's suggestion to also reuse `latestAttemptByNode` in board/portfolio was rejected after verifying that helper keys by `nodeId` only (single-run); reusing it would have collapsed live attempts across runs — a multi-run bug.** The live-attempt computation stayed caller-side.
- **M2:** dropped the top-level `overridden` readiness verdict (the rollup never produces it — overridden collapses to `ready`) from the TS union, the OpenAPI enum, and a route test's accepted-set.
- **M3:** marked `RunDTO.taskId/flowId/startedAt` `nullable: true` to match the projection.
- **M4:** new test seeds a real flow manifest with `gates[].external.staleOnNewCommit:false`, reports a new commit, and asserts NO supersede (1 row, none staled) — proving the value flows manifest → `resolveGateExternalConfig` → route.
- **L5/L6:** asserted same-commit re-report keeps exactly one gate row (artifacts append); added a `verifyToken` revoked-over-expired precedence test. **L3/L4:** Designed→Implemented tag; tightened a 403 description to the real `editSettings` guard.

Red/green proven: H2 (no pre-check → 409), M4 (resolver forced `true` → supersede 2 rows). Full suite: unit 1189/1189 + integration 489/489 green; typecheck/lint 0; `validate:docs:all` 104/104; both OpenAPI specs lint-clean.

## Prevention

- **A spec is code's contract — re-derive the status codes from the handler, not from the spec's first draft.** For every route mapping `MaisterError` → HTTP, enumerate the codes the service actually throws and confirm each appears in the OpenAPI. A 409 that's the *dominant* failure but undocumented is worse than an obscure one.
- **Existence-hide must hold on EVERY token-scoped route.** A shared service that resolves a resource project-unscoped will leak across tenants unless the *route* scopes-then-404s before calling it. Don't push the existence-hide into a service shared with a session-auth caller; fix it route-local.
- **When extracting a shared helper, verify the helper you're told to reuse actually fits.** `latestAttemptByNode` is single-run-keyed; board/portfolio are multi-run. Share only the genuinely-identical core (the collapse + the allow-list); leave per-surface differences (the key, the live-attempt filter) parameterized or caller-side.
- **A `// mirrors X:line` comment is a confession that an abstraction is missing** — and the copies will drift (they had).
- **Config that gates production behavior needs a SET test through the real parse/resolve path**, not just a direct-param test that bypasses the resolver.
