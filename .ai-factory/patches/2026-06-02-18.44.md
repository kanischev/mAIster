# M16 external-gate report: idempotency, terminal-run guard, read-model/contract alignment

**Date:** 2026-06-02 18:44
**Files:** web/lib/flows/graph/gate-store.ts, web/app/api/v1/ext/runs/[runId]/gates/[gateId]/report/route.ts, web/lib/queries/board.ts, web/lib/queries/portfolio.ts, web/lib/queries/readiness.ts, web/app/api/v1/ext/runs/route.ts, web/app/api/v1/ext/projects/[slug]/tasks/route.ts, web/app/api/v1/ext/projects/[slug]/tasks/[taskId]/route.ts, web/app/api/v1/ext/runs/[runId]/gates/[gateId]/report/__tests__/route.integration.test.ts, web/lib/queries/__tests__/board.integration.test.ts, web/lib/queries/__tests__/portfolio.integration.test.ts, docs/api/external/operations.openapi.yaml, docs/error-taxonomy.md, docs/system-analytics/external-operations.md
**Severity:** medium

## Problem

A `/aif-review` of the M16 external-operations branch surfaced five non-blocking
findings on the token-authed `/api/v1/ext/*` surface:

1. The external-gate report endpoint was **non-idempotent**: `reportExternalGate`
   did SELECT-then-write inside its transaction with no row lock, so two
   concurrent CI webhook deliveries for the same `commitSha` both took the
   supersede branch and appended two fresh superseding `external_check` rows. A
   concurrent HITL override landing between the route's reportability pre-check
   and the transaction made the function throw a bare `Error` → surfaced as 500.
2. `board.ts` / `portfolio.ts` / `getRunReadiness` treated a blocking
   `external_check` in status `skipped` as **ready** (`READY = {passed,
   overridden, skipped}`), diverging from the authoritative `assertEvidenceReady`
   (allow-list `passed|overridden`) and the analytics doc.
3. The report route returned `code: "VALIDATION"` for a bad body while the other
   ext routes (and `error-taxonomy.md`) use `CONFIG`.
4. `POST /api/v1/ext/runs`, `POST .../tasks`, `PATCH .../tasks/[taskId]` parsed
   and zod-validated the body **before** token verification (the report route
   already parsed after auth).
5. The report body's `payload` (and sibling `summary`/`externalRunUrl`/
   `commitSha`) had no size cap, despite being serialized verbatim into the
   inline `test_report` artifact + gate verdict jsonb.

## Root Cause

- **#1 (the substantive one):** the report path conflated "wrap writes in one
  transaction" with "serialize concurrent reports". A transaction gives
  atomicity, NOT mutual exclusion — without a row lock, two transactions each
  read the same `passed` row and each append a fresh supersede row (phantom-write
  on an append-based supersede; a row `FOR UPDATE` on the gate rows would not
  help because the second row does not exist yet at select time). The bare
  `throw new Error(...)` for the empty-live-row case predated the typed-error
  taxonomy, so a legitimate race produced an untyped 500.
- **#2:** the read models were authored with a `skipped`-inclusive ready set and
  their tests codified that set, so the read model and its tests agreed with each
  other while both diverged from `assertEvidenceReady` + the analytics contract.
  Latent today (no code path sets an `external_check` gate to `skipped`), but a
  real future-drift trap.
- **#3/#4/#5:** ordinary inconsistencies from routes authored at slightly
  different times against an evolving shared `handleExt`.

## Solution

- **#1:** serialize reports per-run with a Postgres `SELECT id FROM runs WHERE
  id = $1 FOR UPDATE` as the first statement inside the report transaction
  (guarded by `isPostgres()`, matching the `scratch-runs`/`runner-graph` idiom).
  The run-row lock is a stable mutex — it avoids the append-phantom problem and
  forces the second report to re-read the first's fresh row, where the
  same-`commitSha` supersede condition is now false → in-place update, one row.
  Introduced `GateNotReportableError` (exported from `gate-store.ts`); the route
  catches it around the transaction and returns the same 404 the pre-check would
  have, never 500. Also added a terminal-run guard: a report on a run in
  `Done|Abandoned|Crashed|Failed` returns 409 `CONFLICT`.
- **#2:** dropped `skipped` from the `READY` set in `board.ts`/`portfolio.ts` and
  added a `skipped`-external_check → `blocked` bucket to the `getRunReadiness`
  rollup, aligning all three read models with `assertEvidenceReady`. Updated the
  two read-model tests that asserted the old (divergent) `skipped → not-pending`.
- **#3:** report bad-body code `VALIDATION` → `CONFIG`.
- **#4:** moved each ext write route's body parse inside the `handleExt`
  callback (authenticate-then-validate).
- **#5:** capped `summary`/`externalRunUrl`/`commitSha` via `.max()` and
  `payload` via a serialized-size `.refine()` (sibling-field sweep — all four
  reach the same inline-artifact sink).

Added a real two-racer concurrency test (the project rule mandates two racers for
contention claims) + a terminal-run 409 test. Reconciled `operations.openapi.yaml`
(new `Conflict` 409 response), `error-taxonomy.md`, and
`external-operations.md` (Expectations + Edge cases) in the same change.

## Prevention

- "Wrap in a transaction" ≠ "serialize". For any read-then-write that can run
  concurrently, ask whether a row lock (or unique constraint) is needed. For
  **append-based** supersede/versioning, lock a STABLE parent row (the run), not
  the rows being read — the new row does not exist yet, so `FOR UPDATE` on the
  read set cannot serialize it.
- A Postgres-only branch (`isPostgres()` + `FOR UPDATE`) is invisible in tests
  unless `process.env.DB_URL` is pointed at the testcontainer in `beforeAll`
  (and deleted in `afterAll`) — the established pattern. A concurrency test that
  silently skips the lock branch is a false-green.
- When a read model classifies a domain enum (ready/blocked/pending), assert it
  against the ONE authoritative classifier (`assertEvidenceReady`) — and when
  tests and the read model agree but the authoritative gate disagrees, the tests
  are codifying the bug.
- Sibling-field sweep applies to size caps, not just regex/path validators: if
  one field of a schema flows into a sink (here an inline DB artifact), cap every
  sibling that reaches the same sink in the same pass.
