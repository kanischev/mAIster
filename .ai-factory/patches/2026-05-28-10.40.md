# M6 adversarial-review pass-through — 5 defects + 4 instructional rules

**Date:** 2026-05-28 10:40
**Files:** `web/lib/executors.ts`, `web/lib/__tests__/executors.integration.test.ts`, `supervisor/src/ccr-manager.ts`, `supervisor/src/__tests__/_fixtures/mock-ccr.mjs`, `supervisor/src/__tests__/ccr-manager.test.ts`, `supervisor/src/__tests__/ccr-manager.integration.test.ts`, `compose.yml`, `.env.example`, `docs/getting-started.md`, `docs/configuration.md`, `docs/supervisor.md`, `.ai-factory/skill-context/aif-implement/SKILL.md`, `.ai-factory/skill-context/aif-verify/SKILL.md` (new), `.ai-factory/skill-context/aif-plan/SKILL.md` (new), `.ai-factory/rules/backend.md`
**Severity:** high

## Problem

Codex adversarial review of `feature/m6-executor-registry` (2026-05-28) surfaced 5 defects that passed M6's verify gate cleanly:

1. **Stale per-flow override** — `upsertExecutorsFromConfig()` skipped flows without `executor_override`, so `flows.executor_override_id` persisted after the operator removed the YAML field. `POST /api/runs` then kept using the stale executor. The integration test asserted the stale behavior as "documented".
2. **Docker runtime can't run `router=ccr`** — supervisor compose block never wired `MAISTER_CCR_AUTH_TOKEN` or mounted `~/.claude-code-router/config.json`. The shipped runtime could not exercise the feature.
3. **SIGTERM-resistant child stays alive** — `shutdown()` predicate `!exitedInTime && !proc.killed`. `proc.killed` becomes true after a successful `kill()` *signal send*, NOT after the child exits. SIGKILL never fired against a child that ignored SIGTERM; the manager reported `idle` while the daemon was still alive with the port bound.
4. **Health probe accepts wrong process** — `probeHealth()` did `GET / → status < 500`. On `EADDRINUSE`, the spawned CCR exits and an unrelated process answering the port can make the manager transition to `ready`.
5. **`docs/supervisor.md` stale** — `router` still documented as "reserved", 503 as "not used in M3", env merge as `{...process.env, ...executor.env}`. M6 made all three load-bearing without touching the spec.

## Root Cause

Five defects, three deeper causes:

1. **Asymmetry between write path and read path** (defects 1) — `upsertExecutorsFromConfig` cared only about "what to write when the field is present"; `resolveExecutor` cared about "what's in the column". Nobody specified the contract: "absent in YAML ⇒ null in DB". The integration test then enshrined the asymmetry as "documented behavior" — the worst possible outcome.
2. **Runtime/deployment scope was never enumerated** (defect 2) — the M6 plan listed code + docs tasks but never traced "new env var + new config file + new sidecar process" into compose. Unit/integration tests bypass the container entirely, so the test suite passing was not evidence of deployment readiness.
3. **Node lifecycle traps tested only on the happy path** (defects 3, 4) — `subprocess.killed` semantics and "any-200-is-ready" are well-known traps. The tests used a cooperative-exit mock and an empty-port assumption; by construction neither bug could surface.
4. **Contract-surface enumeration trusted the plan's docs list** (defect 5) — the plan listed 5 doc files. M6 changed the `POST /sessions` contract (`router=ccr` load-bearing, 503 real) but `docs/supervisor.md` was not on that list because nobody re-derived "which contract surfaces did the diff touch?" from the diff itself.

## Solution

**Two-pass fix.** Pass 1 landed instructional updates so future plans/implementations/verifies catch this class of issue *before* it ships. Pass 2 (this patch) is the worked-example fix for each of the 5 defects, each guarded by a regression test on the previously-broken branch.

### Pass 1 — Instructional updates (preventative)

- `.ai-factory/skill-context/aif-implement/SKILL.md` — appended 4 curated rules: A1 bidirectional config→DB persistence, A2 runtime contract symmetry, A3 never trust `subprocess.killed` for escalation, A4 health probes must validate target identity.
- `.ai-factory/skill-context/aif-verify/SKILL.md` (new) — 3 rules: B1 re-derive contract surfaces from the diff, B2 runtime parity gate, B3 config↔DB round-trip gate.
- `.ai-factory/skill-context/aif-plan/SKILL.md` (new) — 3 rules: C1 enumerate deployment touchpoints in the plan, C2 trace every contract surface to its spec file, C3 call out config-state symmetry in acceptance criteria.
- `.ai-factory/rules/backend.md` — 2 rules: SIGTERM→exit→SIGKILL escalation, sidecar identity-validating health probes.

### Pass 2 — Code fixes (this patch)

- **Defect 1** — `upsertExecutorsFromConfig()` now iterates every `config.flows[]` entry and writes `executorOverrideId = resolved ?? null` in the same transaction (no `continue` on absent). Flipped the misleading test assertion to `.toBeNull()`. New integration case: removing the override + calling `resolveExecutor()` returns `tier: "projectDefault"`.
- **Defect 3** — Replaced `!exitedInTime && !proc.killed` with `!exitedInTime` alone (escalation gates on the observed `exit` event). After SIGKILL we `await exited` so the call doesn't return until the process is actually gone. New mock-ccr `--ignore-sigterm` flag traps SIGTERM. New integration test asserts the call completes within the grace + SIGKILL window AND the port is freed afterwards.
- **Defect 4** — Probe URL changed from `/` to `/health` (CCR exposes it, confirmed by grep on installed `dist/cli.js`). Predicate tightened to `status === 200` only; 404 surfaces as a target-identifying "identity check failed" error. New child-exit listener attached BEFORE the probe loop; an early exit aborts the probe and surfaces an identity-aware error. mock-ccr now returns `{service: "mock-ccr"}` on every path. New integration test binds the port with a non-CCR HTTP server BEFORE the manager starts → manager fails fast.
- **Defect 2** — `compose.yml` supervisor service now passes `MAISTER_CCR_AUTH_TOKEN` and sets `MAISTER_CCR_CONFIG_PATH=/app/.ccr/config.json`, with a read-only bind mount from `${MAISTER_CCR_CONFIG_HOST_PATH:-${HOME}/.claude-code-router}` to `/app/.ccr`. Verified via `docker compose config` that base + prod overlay both inherit the wiring. `.env.example` + docs updated.
- **Defect 5** — `docs/supervisor.md` updated: `router: ccr` documented as load-bearing, 503 cases enumerated (config missing / malformed / not ready / identity mismatch / token missing), env merge updated to `{...process.env, ...ccrLayer, ...executor.env}` with three-layer explanation, new "CCR lifecycle" subsection cross-links to `executors.md#ccr-setup`.

## Prevention

The instructional updates from Pass 1 are the prevention. In the next AI Factory run:

- `/aif-plan` will refuse to land a plan that adds new env vars or sidecar processes without a "Deployment wiring" task (rule C1).
- `/aif-plan` will require every contract surface change to name its spec file (rule C2).
- `/aif-plan` will reject SET-only acceptance criteria for YAML→DB persistence tasks (rule C3).
- `/aif-implement` will catch asymmetric write loops at code-write time (rule A1) and bare `proc.killed` predicates at review time (rule A3); will refuse generic `GET /` health probes (rule A4); will block landing if new runtime deps aren't wired into compose (rule A2).
- `/aif-verify` will re-derive contract surfaces from the diff instead of trusting the plan (rule B1); will audit compose for new deps (rule B2); will block on missing CLEAR-half tests (rule B3).
- `.ai-factory/rules/backend.md` rules apply at code-review time even when the skill-context isn't loaded.

The fix itself is the first worked example for those rules. Every regression test added exercises the previously-broken branch, which is the new rule's mandatory test shape.
