# error-taxonomy drift: PRECONDITION reused by registration but not documented

**Date:** 2026-05-31 14.26
**Files:** docs/error-taxonomy.md
**Severity:** low

## Problem

(`/aif-review` analytics-surface reconciliation.) The ADR-025 repo-onboarding
route (`POST /api/projects`) began throwing `PRECONDITION` (409) on clone
failure / missing target dir / clone-target collision, but the
`docs/error-taxonomy.md` `PRECONDITION` row still listed only
`POST /api/runs validation before spawn` in its "Where thrown" column. The
canonical error catalog under-described where the code is raised. (The OpenAPI
spec, by contrast, was already complete — 201/401/403/409/422/502 all
machine-defined, including the 502 `FLOW_INSTALL` body — so this was the lone
spec drift.)

## Root Cause

When a new route reuses an EXISTING `MaisterError` code, there's no diff signal
on `web/lib/errors.ts` (the union is unchanged), so the error-taxonomy table is
easy to miss — the surface to update is keyed by *thrown-at* sites, not by the
code list. The new throw site simply wasn't traced back to the catalog.

## Solution

Extended the `PRECONDITION` row's "Meaning" (added the repo-onboarding
preconditions) and "Where thrown" (`POST /api/projects` source resolution via
`resolveProjectSource`) columns. No code change — this is a documentation
completeness sync per `docs/CLAUDE.md` "code wins".

## Prevention

- When a route throws an EXISTING error code (not just a new one), update the
  code's `docs/error-taxonomy.md` "Where thrown" / "Meaning" columns in the same
  change — reusing a code is still an error-surface change.
- Matches the `/aif-review` skill-context rule: re-derive every analytics/spec
  surface (error-taxonomy included) from the diff and treat drift as same-review
  must-fix, never trusting that an unchanged `errors.ts` means no doc impact.
