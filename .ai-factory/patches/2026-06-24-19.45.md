# Orchestrator + form node types missing from the Flow Studio editor

**Date:** 2026-06-24 19:45
**Files:** web/lib/flows/editor/node-form.ts, web/lib/flows/node-visuals.ts,
web/components/board/flow-graph-view.tsx, web/lib/flows/graph/topology.ts,
web/lib/flows/editor/editor-labels.ts, web/lib/flows/node-side-form-labels.ts,
web/components/flows/node-form/node-side-form.tsx, web/messages/{en,ru}.json,
+ 4 test files
**Severity:** medium

## Problem

The Flow Studio graph editor (route `/flows/{slug}/{capId}`) offered only 5 of
the engine's 7 node types in its add-node palette (ai_coding, cli, check, judge,
human). The `orchestrator` (M37/ADR-098) and `form` (T4) node types could only
be authored by hand-editing `flow.yaml` or importing a package, and when present
they rendered as a featureless gray dot on the canvas — in both the editor and
the read-only board viewer.

## Root Cause

A UI-side enumeration constant drifted from the schema it mirrors. The engine's
source of truth is the `nodeSchemaBase` discriminated union in
`web/lib/config.schema.ts` (7 members). The editor re-declares the set in a
hand-maintained `NODE_TYPES` array (`node-form.ts`), and the editor's `NodeType`
union is `(typeof NODE_TYPES)[number]`. When `orchestrator` and `form` landed in
the schema, nobody widened `NODE_TYPES`, so every editor-side exhaustive map
(`NODE_VISUALS`, the `blankNode` switch, the i18n `nodeType` label map) silently
covered only 5 types. The read-only viewer's `nodeVisual(type)` falls back to a
neutral gray dot for any type absent from `NODE_VISUALS`, so the omission also
produced the gray-dot rendering rather than a throw.

Two duplicated enumerations of the same closed set, with no compile-time link
forcing them to agree.

## Solution

Widened the single union source (`NODE_TYPES`) to include `orchestrator` and
`form`, then treated the resulting TypeScript compile errors as the authoritative
checklist of every exhaustive map to update (the task brief said exactly this):

- `blankNode` switch → schema-valid skeletons. `action.prompt` is `min(1)`, so
  orchestrator seeds `prompt: "TODO"` (NOT the literal `""` the brief suggested,
  which fails validation); `form` seeds the REQUIRED `settings.form_schema:
  "TODO"`.
- `NODE_VISUALS` → reused the already-defined, light/dark-tuned `cv-teal`
  (orchestrator) and `cv-rose` (form) tokens — distinct from the 5 existing node
  hues — to avoid editing `globals.css` (Turbopack serves a stale compiled
  `globals.css` after a new `--cv-*` token, a known trap). Added two SVG glyphs
  (`sitemap`, `form`) to the shared `NODE_ICON_PATHS` registry, which fixes the
  gray dot in BOTH the editor and the read-only viewer at once.
- i18n `toolbar.nodeType` + new `nodeForm` keys (delegation/maxFanout/maxDepth/
  formSchema) in EN and RU, keeping the enforced key-tree parity.
- Sidebar form: `isPromptType` + the ai_coding settings block + workspaceAccess
  now include orchestrator; added an orchestrator-only `settings.delegation`
  sub-panel (max_fanout/max_depth, sparse like the hooks block) and a form-only
  `form_schema` field.
- `topology.ts` `nodeRoleForType`: mapped `orchestrator → "agent"` (one-line
  fall-through) so the read-only viewer labels it "Agent" not "Other", with no
  `GraphNodeRole` union or role-label-map churn.

The `compat.engine_min >= 1.6.0` orchestrator floor needed no editor change: it
is already enforced server-side at save (`compileManifest`/`validateGraphManifest`
→ `ORCHESTRATOR_ENGINE_MIN` in `config.ts`), matching how the hooks (1.8.0),
decide (1.7.0), and agent-binding (1.5.0) floors are deferred to compile rather
than surfaced in the editor's inline validation or auto-set.

## Prevention

- When a discriminated-union schema and a UI-side mirror constant both enumerate
  the same closed set, **derive the UI list from the schema**, or add a
  type-level exhaustiveness assertion (e.g. a `satisfies Record<SchemaType, …>`
  or a `const _exhaustive: NodeType = schemaType`) that fails to compile when
  they drift. Two independent enumerations of one set will diverge.
- **The compiler is the authoritative checklist, not grep.** A pre-pass
  `grep` for hand-built `NodeSideFormLabels` literals MISSED one
  (`node-side-form.test.ts`) because it builds the object typed by inference
  (`NodeSideFormProps["labels"]`) and never names the type — only `tsc` caught
  it. Widen the union first, then let `tsc --noEmit` enumerate the edit set.
- A map whose miss degrades to a silent fallback (here `nodeVisual` → gray dot)
  hides drift instead of failing loudly. Prefer a `Record<Union, …>` (compile
  error on a missing key) over a `Record<string, …>` + runtime default for any
  map keyed by a closed union.
- Fresh git worktrees have no `node_modules` — the LSP reports a blizzard of
  "Cannot find module 'react'" / JSX-intrinsics errors that are pure env noise.
  Run `pnpm install --frozen-lockfile` and trust real `tsc`/`eslint`/`vitest`
  over the inline diagnostics.
