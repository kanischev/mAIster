# Run-schedules review-findings batch: transition-vs-value re-arm, terminal-create symmetry, CAS fencing token

**Date:** 2026-06-10 20:48
**Files:** web/lib/run-schedules/service.ts, web/lib/run-schedules/dispatch.ts, web/lib/api/project-route-helpers.ts (new), web/lib/api-error.ts (new), web/app/api/projects/[slug]/schedules/* (3 routes), web/components/schedules/{schedule-edit-modal,schedules-panel}.tsx, web/messages/{en,ru}.json, docs/system-analytics/run-schedules.md, docs/api/web.openapi.yaml
**Severity:** medium

## Problem

Branch review of M28 (`/aif-review` on `feature/user-facing-cron-schedule`)
surfaced five non-blocking defects that all passed the original test suite:

1. `PATCH {enabled: true}` on an *already-enabled* schedule recomputed
   `next_fire_at` from now, silently swallowing a due fire when the flag was
   bundled with an unrelated field (rename).
2. `createSchedule` rejected `Abandoned` tasks but accepted `Done` tasks —
   both are `target_terminal`, so the accepted schedule could never fire
   (eternal `skipped_target_terminal`); the modal picker also offered
   terminal tasks.
3. The tx2 final-outcome CAS keyed only on `last_fire_outcome='dispatching'`
   — a launch hung past the scheduler attempt timeout could overwrite a
   *newer* reclaim's `dispatching` marker and cross-attribute its run id.
4. API error codes other than `CONFIG` reached the UI as untranslated
   English fallbacks (copied from the `user-edit-modal` reference pattern).
5. `httpStatusForCode` / `resolveProject` / `errorResponse` were pasted into
   all three new route files, continuing an existing divergence pattern
   (sibling tasks route already drifted to `CONFIG→422`).

## Root Cause

- **Transition vs value semantics (1):** the spec's state diagram models
  re-arm as the `Paused→Active` *transition*, but the code keyed on the
  *patch value* (`patch.enabled === true`) without consulting
  `current.enabled`. Value-keyed guards on idempotent PATCH fields are a
  recurring trap: the UI always sends real transitions, so tests written
  from the UI's perspective never exercise the redundant-value case.
- **Asymmetric enum handling (2):** the create-precondition was written
  against one named status (`Abandoned`) instead of the *classification*
  the launch gate actually uses (`target_terminal`). When a guard and a
  classifier encode the same business rule independently, they drift.
- **CAS without a fence (3):** a CAS that matches only on a *state label*
  cannot distinguish two attempts that both legitimately hold that label at
  different times. Timeout-based reclaim makes "two owners of the same
  marker" reachable; only a per-attempt stamp (fencing token) makes the
  stale owner's write miss.
- **(4)/(5)** convention debt copied from the reference implementation —
  each new surface that clones it widens the gap.

## Solution

- Re-arm condition now requires the actual transition:
  `patch.enabled === true && !current.enabled` (service.ts), with an
  integration test pinning that a redundant `enabled:true` + rename leaves a
  due `next_fire_at` untouched.
- Create now rejects both terminal statuses with one PRECONDITION message
  (`Cannot schedule a terminal task (<status>)`); the modal filters
  `Done`/`Abandoned` out of the picker. Spec + OpenAPI updated in the same
  pass (code-wins rule).
- `LaunchIntent` carries `stagedAt` (the `last_fired_at` value tx1 stamped);
  tx2's CAS adds `eq(lastFiredAt, stagedAt)` as the fencing token. New
  integration test simulates the W1 reclaim mid-launch and asserts the hung
  launch's result is dropped.
- New client-safe `web/lib/api-error.ts` (`readApiError`) translates known
  codes through a shared `apiErrors` message namespace (EN+RU), keeping the
  server detail behind the translated label; both schedule components
  adopted it.
- New `web/lib/api/project-route-helpers.ts` consolidates
  `httpStatusForCode` / `errorResponse` / `notFoundResponse` /
  `resolveProject`; the three schedule routes now share one copy
  (CONFIG→400 per the M11c decision).

## Prevention

- When a PATCH field can arrive redundantly, guard side-effects on the
  **transition** (`patch.x !== current.x`), not the patched value; add a
  redundant-value test case whenever a PATCH field triggers recomputation.
- Preconditions that mirror a classifier (here: launchability) must be
  expressed in the classifier's terms (reject the *classification*, not one
  enum member) — or call the classifier directly.
- Any CAS whose guard state can be re-entered by a timeout/reclaim path
  needs a fencing token (per-attempt stamp in the WHERE), not just the state
  label. Review cue: "who else can legitimately hold this marker while I'm
  still running?"
- New UI surfaces must not clone the untranslated-error fallback; use
  `readApiError` + the `apiErrors` namespace.
- New project-scoped routes must import
  `@/lib/api/project-route-helpers` instead of pasting their own
  `httpStatusForCode` / `resolveProject` copies.
