# Package management: trust authz scope, capability-record ownership, registration attach parity

**Date:** 2026-06-12 12:56
**Files:** web/lib/packages/attach.ts, web/app/api/projects/[slug]/flow-packages/_lib.ts, web/app/api/projects/[slug]/packages/[attachmentId]/trust/route.ts, web/app/(app)/projects/[slug]/page.tsx, web/components/board/panels/project-packages-section.tsx, web/app/api/projects/route.ts, docs/{system-analytics/packages.md, configuration.md, decisions.md, api/web.openapi.yaml}
**Severity:** critical

## Problem

Adversarial review (Codex) of the M33 package-management branch found three defects:

1. **(critical)** `POST .../packages/{attachmentId}/trust` was gated on project-scoped
   `managePackages` (project admin), but `trustPackageRevision` fans trust + runs
   `setup.sh` for EVERY project attached to the same platform `package_installs`
   row — a Project-A admin could trust executable content for Project B.
2. **(high)** Package-shipped MCP/restriction records upserted on
   `(projectId, source, kind, capabilityRefId)` let a second attached package
   silently overwrite the first's record; detach deleted by refId without kind or
   ownership, removing other packages' live records.
3. **(high)** Registration `packages[]` used legacy `installPackage`, creating no
   `package_installs`/`project_package_attachments` rows, no `package_install_id`
   FKs, and silently dropping `mcps[]`/`restrictions[]` ingestion — bootstrapped
   packages were invisible on the packages tab and unmanageable (detach/upgrade/
   trust unreachable).

## Root Cause

1. ADR-088 specifies trust as "one operator decision per package revision" (a
   platform-level decision), but the route was placed under the project URL tree
   and inherited the project-scoped authorize helper of its siblings. Nobody
   reconciled the authz scope against the blast radius of the operation. The
   review's own proposed fix (project-scope the fan-out) was WRONG — the fan-out
   is the documented design and `flow_revisions` has no projectId; the gate was
   the bug (validate review findings against design docs before implementing).
2. The unique index made same-(kind,id) collisions structurally invisible:
   `onConflictDoUpdate` treats "row exists" as "mine to refresh". Ownership lived
   only in `material.packageInstallId` jsonb, which neither the upsert nor the
   delete consulted.
3. The registration path was written before the attach pipeline existed in final
   form and documented itself as a "one-shot bootstrap"; configuration.md was
   written to match the code, so docs-vs-code consistency checks could not catch
   that the bootstrap produced a degraded state vs the UI path.

## Solution

1. New `authorizePackageTrust(slug)` in flow-packages/_lib.ts: `requireGlobalRole("admin")`
   + shared `resolveProjectBySlug`; trust route swapped to it. UI: board page passes
   `canTrust = user.role === "admin"` (global, not project role) and the section hides
   the Trust button otherwise. Fan-out itself unchanged (design-correct); a
   multi-project integration test now PINS the cross-project fan-out as intended
   behavior, and a route test pins 403 for non-global admins.
2. `writeIngestionRecords` pre-checks existing `flow-package` rows for matching
   (kind, refId) pairs owned by a different `material.packageInstallId` → throws
   `MaisterError("CONFLICT")` inside the attach/upgrade tx (no partial group).
   `deleteIngestionRecords` adds `material ->> 'packageInstallId' = install.id`
   so detach/upgrade removes only owned rows; same-id different-kind coexists.
3. Registration `packages[]` loop now runs `installPackageRevision` (trust via
   `resolveTrust(source)`, matching the admin install route) + `attachPackage`
   (extended with `roleRefs` pass-through; returns `memberFlows: InstallResult[]`
   so the runner-reconfiguration sync keeps working); `capabilityDerived`
   agent_definition entries are rebuilt from the install manifest for the
   existing SET/CLEAR ingest. New real-fixture integration test registers a
   project with packages[] and asserts install + attachment + FK wiring +
   ingestion rows + GET /packages visibility.

Docs synced in the same pass: packages.md (trust gate, ownership-scoped detach,
registration parity, new edge case), configuration.md bootstrap semantics,
ADR-088 trust/attach/packages[] bullets, OpenAPI descriptions (trust = global
admin; attach 409 collision cause; detach ownership).

## Prevention

- An endpoint's authz scope must match the WRITE SCOPE of what it mutates: if a
  project-URL route updates platform-scoped rows (no projectId in the WHERE),
  that's a red flag — either constrain the writes or elevate the gate.
- Any upsert with `onConflictDoUpdate` whose conflict target does NOT include
  the owning entity needs an ownership pre-check (or the owner in the key);
  any bulk delete keyed on author-chosen ids needs an ownership predicate.
- When two code paths produce "the same" domain state (bootstrap vs UI attach),
  route them through the same service functions; spy-only tests on the
  duplicated path hide divergence — at least one test must assert the resulting
  DB rows, not the call.
- When fixing review findings, validate each against the design docs first: one
  of three Codex recommendations here contradicted ADR-088 and would have
  broken `flow_revisions` semantics.
