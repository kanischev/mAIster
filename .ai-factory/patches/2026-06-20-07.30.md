# Scheduler target validation dropped fields after typed editor rollout

**Date:** 2026-06-20 07:30
**Files:** web/lib/scheduler/job-targets.ts, web/lib/scheduler/job-admin.ts, web/components/admin/scheduler-job-edit-modal.tsx, web/components/admin/scheduler-run-schedules-overview.tsx, docs/screens/admin-scheduler.md, docs/system-analytics/scheduler.md, docs/system-analytics/run-schedules.md
**Severity:** high

## Problem

The scheduler admin API and editor accepted target payloads that did not match
the exact OpenAPI shapes. Unknown keys were dropped while known keys were
extracted, and `timeoutMs: "5s"` became `5` because timeout parsing used
`Number.parseFloat`. The page also documented a last-run status/link for task
schedules but only rendered the status text.

## Root Cause

The typed target model normalized values but did not own the full target
contract. It treated shape validation and field extraction as separate concerns,
which let extra keys survive the API parse and disappear before persistence.
The editor also kept passing placeholder fields for inactive target kinds, so
the old permissive parser masked UI assumptions.

## Solution

Moved exact key validation into `normalizeSchedulerTargetDraft` for every job
kind and command sub-kind, then switched timeout string parsing to full-string
numeric parsing with `Number(...)`. The editor now builds only the active target
shape. The server admin service logs rejected target shapes with structured
`[FIX:scheduler-target-validation]` fields and still returns `CONFIG`/422.

The task schedules overview now links a present `lastRunId` to `/runs/{id}`,
with matching EN/RU i18n and docs status updates from Designed to Implemented.

## Prevention

- When OpenAPI says `additionalProperties: false`, the parser must reject
  unknown keys before extracting known fields.
- Boundary validators should test typo keys, sibling keys from another union
  member, and partial numeric strings such as `"5s"`.
- UI form builders must emit only the active schema branch. Empty placeholders
  belong in component state, not in the submitted object.
- SDD-heavy features need a docs status grep before merge: implemented screen
  pieces should not remain tagged Designed in related analytics docs.
