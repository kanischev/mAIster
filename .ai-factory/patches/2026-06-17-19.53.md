# Running-scratch composer corrupted codex built-in commands; missing OpenAPI path; unguarded stream parse

**Date:** 2026-06-17 19:53
**Files:** web/lib/capabilities/running-catalog.ts, web/lib/capabilities/__tests__/running-catalog.test.ts, web/lib/runs/launch-progress.ts, docs/api/web.openapi.yaml
**Severity:** medium

## Problem

Three findings from `/aif-review` of `feature/unified-capability-composer`:

1. **(correctness)** The running-scratch composer's autocomplete catalog
   (`buildRunningCommandCatalog`) converted EVERY live ACP `availableCommand`
   (except `mcp:`) into an `@skill:<slug>` chip and re-derived the wire form via
   `surfaceFormForSkill(slug, agent)`. On **codex**, skills are emitted as
   `$slug` but native built-ins as `/status`, `/skills`, etc. Stripping the sigil
   and re-applying codex's `$` turned the built-in `/status` into `$status` — a
   command codex does not recognize. Selecting a built-in from autocomplete on a
   codex session silently produced a no-op. claude was unaffected (built-ins and
   skills share the `/` sigil, so it round-tripped). The unit test only exercised
   the **claude** built-in case, so the bug was invisible.
2. **(contract drift)** The new route `GET /api/projects/{slug}/capability-catalog`
   (consumed by `scratch-dialog.tsx` + the scratch launcher) was absent from
   `docs/api/web.openapi.yaml`, while its sibling `…/commands` was documented —
   violating the project's "trace every contract surface" rule + docs R3.
3. **(robustness)** `readLaunchStream`'s `JSON.parse` of a `data:` frame was
   unguarded — one malformed/partial frame would reject the whole launch read.

## Root Cause

The composer's three-layer token model (display chip / canonical storage /
runner wire form) is only lossless for commands that have a **canonical
identity** — i.e. project skills, which exist in the static catalog under a
slug. Native/built-in commands have NO canonical identity: their only correct
representation IS the emitted wire string. The code treated the live stream as
"all skills," so it minted a canonical `@skill:` token for every command and
re-derived the wire form from the sigil map. For any adapter whose built-ins and
skills use **different sigils** (codex: `/` built-ins vs `$` skills), re-deriving
corrupts the built-in. This also contradicted locked decision D8 ("native
commands are typed raw, NOT prompt artifacts").

The OpenAPI gap was the usual spec-after-code drift: the route was added in a
code phase but the Phase-0 contract-surface table only listed `…/commands`, so
the new aggregator route never got a path.

## Solution

1. `buildRunningCommandCatalog`: only chipify a live command when its slug
   **matches a static-catalog skill** (`staticSkillBySlug.get(slug)`); otherwise
   `continue` (skip). Built-ins are no longer chips — they stay typeable as raw
   text the agent resolves natively, and the codex `/status`→`$status` re-sigil
   can no longer happen. Removed the now-unreachable `?? slug` fallbacks. Updated
   the affected tests: the "native/global → chip" test became "native/built-in →
   excluded," ADDED a codex `/status` regression test (asserts no `status` entry
   and no `$status` surfaceForm while a real `$aif-plan` skill still surfaces),
   and the dedup test switched from the non-catalog slug `review` to the catalog
   skill `aif-plan`.
2. Added the `GET /api/projects/{slug}/capability-catalog` path (tagged
   Implemented, `?agent=` query, 200/401/403/404) + a `ProjectCapabilityCatalogEntry`
   response schema to `web.openapi.yaml`. redocly valid, zero new warnings.
3. Wrapped `readLaunchStream`'s `JSON.parse` in try/catch → skip a malformed
   frame and keep reading.

No `[FIX]` console logging was added: the two TS files are pure / client-safe
modules whose contract is "no logging deps here — callers log," and the project
`no-console` lint + logger-boundary rule forbid `console.log`. Skill logging
default was consciously overridden per the instruction-priority hierarchy.

## Prevention

- **Cross-runner wire-form round-trip rule:** a value may only be re-derived
  from a canonical token if it HAS a canonical identity (is in the catalog).
  For anything sourced from a live/native stream, preserve the emitted form
  verbatim or exclude it — never re-apply a per-adapter sigil to a string you
  didn't mint. Especially when adapters disagree on sigils (codex `/` vs `$`).
- **Test the divergent adapter, not just the convenient one.** The bug hid
  because the test used claude (where `/`-built-ins round-trip). Any cross-runner
  normalization test MUST include the adapter where the sigils differ (codex).
- **Contract-surface trace must include READ aggregator routes.** The Phase-0
  contract table listed the obvious `…/commands` route but missed the sibling
  catalog GET. When a code phase adds an HTTP route, grep the route tree against
  the OpenAPI paths before exit (the review's analytics gate would have caught
  it earlier).
- **Guard every `JSON.parse` of a streamed/partial frame.** SSE/NDJSON readers
  must tolerate a malformed line without aborting the whole read.
