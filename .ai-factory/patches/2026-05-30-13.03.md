# M10 setup-before-trust RCE, non-atomic revision remove, and setup-failure state drift

**Date:** 2026-05-30 13:03
**Files:** web/lib/flows.ts, web/lib/flows/lifecycle.ts, web/app/api/runs/route.ts, docs/system-analytics/flow-packages.md
**Severity:** critical

## Problem

A second Codex adversarial review of M10 found three holes the first fix round
missed:

1. (critical) `installRevision` ran `runSetupSh` (i.e. `bash setup.sh` from the
   cloned/copied package) during install, *before* trust was resolved. For an
   untrusted git source the explicit trust/enable lifecycle was already too
   late — arbitrary package code executed on the web host. Supply-chain RCE for
   anyone allowed to manage packages.
2. (high) `removeRevision` did check-then-act (referencing-run + enabled-anywhere
   checks, then a separate mark-Removed + cache delete) with no transaction or
   lock. A concurrent `enableRevision`/rollback could read the revision as
   `Installed`, pass validation, and race the deletion — leaving a project
   enabled to a `Removed` revision whose files were gone. Launch also never
   checked `packageStatus === 'Installed'`, so the broken pointer reached runner
   startup.
3. (medium) A failed `setup.sh` left the revision at `packageStatus='Installed'`
   with only `setupStatus='failed'`, contradicting the documented
   `Installing -> Failed` state machine and inflating installed-package counts.

## Root Cause

The first M10 pass treated "install" as one monolithic step that also ran setup,
so the security-critical ordering (trust → setup → enable) collapsed: setup ran
at the earliest possible moment instead of the latest safe one. The lifecycle
operations were written as straight-line `read; check; write` sequences under
the implicit assumption of a single actor — correct for the happy path, but with
no concurrency control on a globally-shared `flow_revisions` table where remove
and enable mutate overlapping state. The setup-failure status was modeled on two
axes (`packageStatus` + `setupStatus`) without deciding which is authoritative
for "is this revision usable," so a failure surfaced on the non-authoritative
axis only.

## Solution

- **Setup after trust.** `installRevision` no longer runs setup; it records
  `setupStatus = 'pending'` (setup.sh present) or `'not_required'`. A new
  exported `runRevisionSetup` executes setup and is called only from
  trust-established contexts: `installFlowPlugin`'s trusted-by-policy auto-enable
  branch, and `enableRevision` (which runs the deferred setup once
  `assertEnableable` confirms the flow is not untrusted). Untrusted sources never
  reach a setup execution until an operator explicitly trusts + enables.
- **Atomic enable/remove.** Both wrap their mutation in `db.transaction` and
  `SELECT ... FOR UPDATE` the `flow_revisions` row (the codebase's existing
  Postgres-lock / SQLite-serialized pattern, per the HITL respond route).
  `removeRevision` re-checks the referencing-run and enabled-anywhere guards
  under the lock and CAS-updates `package_status` to `Removed`;
  `enableRevision` re-verifies `package_status === 'Installed'` under the lock
  before moving the pointer. Launch refuses any enabled revision whose
  `package_status !== 'Installed'`.
- **Setup failure ⇒ Failed.** `runRevisionSetup` sets `package_status='Failed'`
  on non-zero exit; `packageStatus` is the authoritative usability axis.

## Prevention

- Any code that fetches and then **executes** third-party content must order
  the steps fetch → verify/trust → execute, and the execution call must be
  physically separate from the fetch/install function so it cannot be invoked
  pre-trust. Review checklist: "where is setup.sh / postinstall / hooks run, and
  what trust gate precedes that exact line?"
- Multi-step mutations on shared rows are not atomic just because each statement
  is. Use a transaction + row lock (`FOR UPDATE`) or a conditional/CAS update
  whenever two operations can mutate overlapping state (here: remove vs enable).
  Add a launch/use-time precondition that re-validates the referenced row's
  status, so a lost race degrades to a clean refusal, not a broken pointer.
- When a status is modeled on two axes, pick ONE as authoritative for the gating
  decision and make every failure path write it. Keep the diagram, code, and
  query in sync on which axis gates.
- Regression coverage (Docker-gated integration): install untrusted with a
  setup.sh → assert setup.sh did NOT run (no side effect / sentinel absent)
  until enable; concurrent remove-vs-enable on the same revision → exactly one
  wins, no enabled-to-Removed pointer; setup failure → `package_status='Failed'`
  and enable/launch refuse.
