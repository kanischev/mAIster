# Codex findings: consensus grammar output gap + over-broad manifest normalizer

**Date:** 2026-06-26 12:43
**Files:** web/lib/flows/flow-dsl-grammar.ts, web/lib/config.ts, web/lib/flows/__tests__/flow-dsl-grammar.test.ts, web/lib/flows/editor/manifest-normalize.ts, web/lib/flows/editor/__tests__/manifest-normalize.test.ts
**Severity:** high

## Problem

Two defects surfaced by an adversarial Codex review of the branch:

1. **[high] Assistant grammar omitted the compile-required consensus outputs.**
   The always-injected Flow DSL grammar (`buildFlowDslGrammar`) documented the
   consensus node's *inputs* (`prompt`, `participants`, `synthesizer`,
   `material_axes`, `workspace`, `rounds`, `on_no_consensus`) but never its
   mandatory `output.produces` contract. The runtime loader
   (`validateConsensusOutputs`) hard-requires EXACTLY `consensus_plan`
   (`kind: plan`, `current: true`) and `debate_log` (`kind: human_note`,
   `current: true`). A consensus node authored from the "authoritative" grammar
   would pass the grammar guidance and still fail compile/launch with `CONFIG`.

2. **[medium] The canvas→YAML normalizer pruned arbitrary passthrough string
   arrays.** `pruneEmptyListEntries` trimmed/filtered EVERY non-empty string
   array on EVERY object. But `flowYamlV1Schema` and every node schema are
   `.passthrough()`, and `x-*` extension arrays are a supported, tested feature
   (`manifest-io.test.ts` asserts they survive parsing). Any canvas-originated
   save therefore silently trimmed whitespace-significant values or dropped
   entries from author extension data — manifest corruption outside the stated
   blank-row fix.

## Root Cause

1. **Drift guard checked Zod shape, not compile-time invariants.** The grammar's
   drift guard (`flow-dsl-grammar.test.ts`) introspects `config.schema.ts` for
   node types / settings keys / enum values. The consensus output contract is a
   `superRefine` rule in `config.ts` (`CONSENSUS_REQUIRED_OUTPUTS`), not a Zod
   key/enum — structurally invisible to the guard. So the grammar could omit a
   real compile-blocking contract and no test noticed.

2. **The normalizer's predicate was "is a string array," not "is a known DSL
   list field."** The blank-row problem it solves comes from exactly one source:
   the free-text `StringListField` control (its `add()` appends `""`).
   `MultiSelectField` already trims and rejects empties at add-time, so it never
   leaks a blank. The fix-author generalized to "all string arrays" instead of
   scoping to the handful of fields that actually need it, and the regression
   test only exercised in-scope fields — so the over-reach onto passthrough
   extension keys was never observed.

## Solution

1. **Grammar contract + real drift guard.** Documented the consensus
   `output.produces` block inline in the grammar's consensus bullet ("MUST also
   declare `output.produces` with EXACTLY these two artifacts or compile fails
   (CONFIG): `{ id: consensus_plan, kind: plan, current: true }` and
   `{ id: debate_log, kind: human_note, current: true }`"). Exported
   `CONSENSUS_REQUIRED_OUTPUTS` from `config.ts` and added a grammar test that
   iterates it, asserting the grammar text contains every required id + kind and
   `current: true`. Now a change to the runtime contract fails the grammar test
   until the grammar is updated — a guard over the compile invariant, not just
   the Zod shape.

2. **Scoped the normalizer to an allowlist.** Added `PRUNED_LIST_FIELDS` (the
   seven free-text `z.string().min(1)` DSL list fields the `StringListField`
   control edits: `restrictions`, `decisions`, `roles`, `assignees`,
   `allowedTargets`, `allowedPaths`, `material_axes`) and gated the prune on
   `PRUNED_LIST_FIELDS.has(key)`. Any other string array — extension `x-*` keys
   included — now recurses untouched and survives verbatim. Added a regression
   proving an `x-flow-tags` / `x-lanes` extension array survives unchanged while
   a targeted `restrictions` field is still cleaned.

Verification: `tsc --noEmit` clean; eslint clean on the 5 changed files;
manifest-normalize (6) + flow-dsl-grammar (4) + config-consensus (12) +
flow-assistant context (1) all green.

No `[FIX]` runtime logging was added: both changes are pure/compile-time
(a serialize-boundary pure function, grammar text, and a const export) and are
covered by deterministic regression tests, not log inspection. The project
convention (`web/CLAUDE.md`) forbids `console.log` in committed code.

## Prevention

- When a "reference"/SSOT artifact (grammar, prompt, generated doc) is meant to
  be authoritative, its drift guard must cover compile-time invariants
  (`superRefine`/loader rules), not only the declarative Zod shape. Export the
  invariant constant and assert the reference mirrors it.
- A normalizer/sanitizer that walks a `.passthrough()` schema must operate on a
  KNOWN-key allowlist, never a structural predicate ("is a string array"). The
  presence of `.passthrough()` + tested `x-*` extension fields is the tell that
  a blanket transform will corrupt author data.
- When fixing a bug whose trigger comes from ONE UI control, scope the fix to
  that control's exact target fields and make the regression test cover an
  out-of-scope sibling that must be left alone — not only the in-scope cases.
