# Rework-comment template crash in authored flow graphs + per-file materialization merge

**Date:** 2026-06-09 19:23
**Files:** plugins/aif/flows/dev/flow.yaml, plugins/aif/flows/bugfix/flow.yaml, web/lib/flows/graph/runner-graph.ts, web/lib/flows/graph/__tests__/rework-comments.test.ts, web/lib/capabilities/materialize-bundle.ts (+test), docs/{flow-dsl,database-schema,system-analytics/flow-graph,system-analytics/hitl,system-analytics/reconciliation-gc,db/erd}.md
**Severity:** critical

## Problem

The shipped `aif-dev` flow crashed the run on the FIRST `rework` decision: the
`fix` node prompt read `{{ steps.review.vars.review_comments }}`, but on a rework
jump `steps.review.vars` is `{}`, and strict templating (`renderStrict`) THROWS
`MaisterError("CONFIG", "undefined template var: ‚Ä¶")` on a missing deep leaf ‚Üí
node Failed ‚Üí run Failed. `aif-bugfix` had the sibling defect: its `fix` prompt
read `{{ task.prompt }}` only, so a rework silently re-ran against the original
bug text, discarding the reviewer's feedback (the declared `commentsVar` was
wired to nothing). Neither was caught: the only flow test (`aif-flows.test.ts`)
is a STATIC manifest load ‚Äî it never renders a rework-target prompt, and the
runtime e2e (T9) is deferred.

Separately, capability materialization (`copyBundleArtifactsToWorktree`) merged
bundle files into a repo-local skill dir at PER-FILE granularity (`cp force:false`),
so a partial repo-local skill (`SKILL.md` only) + the bundle's `references/*`
produced a Frankenstein skill.

## Root Cause

The reviewer's rework comments are NOT persisted on the reworked node attempt
(`markNodeReworked` writes status/decision only, no `vars`) and are NOT under
`steps.<reviewNode>.vars.*`. They are injected as a TOP-LEVEL context var named
by the node's `commentsVar` (`pendingInjectedVars`), i.e. `{{ review_comments }}`.
The convention was implicit and unenforced ‚Äî a flow author could (and did) write
the wrong path. Worse, `pendingInjectedVars` is set ONLY on a rework jump, so a
node that references `{{ review_comments }}` on its INITIAL visit (aif-bugfix
`fix` is both entry node AND rework target) or after a rework with no comments
would throw on the missing top-level var. So the naive "just append
`{{ review_comments }}`" fix would have replaced a rework crash with an ENTRY
crash.

## Solution

Root-cause fix at the engine, not just the YAML: `runGraph` now seeds EVERY
`commentsVar` declared anywhere in the graph to `""` once per run
(`collectDeclaredCommentsVars`), overlaid by the real comments on the rework
attempt: `extraVars: { ...declaredCommentsVars, ...(pendingInjectedVars ?? {}) }`.
A declared `commentsVar` is now always renderable (empty outside a rework), so a
reference can't crash on entry or on an empty-comment rework. Then the YAML: dev
`fix` ‚Üí `{{ review_comments }}`; bugfix `fix` ‚Üí `/aif-fix {{ task.prompt }}` +
`{{ review_comments }}`. Also fixed `aif-dev` `takeover: commit` ‚Üí `takeover:
checks` (restores the M11b re-validation of a human's local edits before commit).
Materialization switched to PER-ENTRY precedence (skip a whole existing skill
dir / agent file, never merge into it). Added a runtime regression test that
loads the real flows and renders the rework-target prompts through the real
`buildContext` + `renderStrict` (RED before the fix, GREEN after), and a
Frankenstein-skill test. Analytics docs reconciled for the new `form` node type
and the `live-session-by-step` reconcile reason.

## Prevention

- A rework-target prompt MUST read the reviewer's feedback via the TOP-LEVEL
  `{{ <commentsVar> }}`, NEVER `{{ steps.<reviewNode>.vars.<commentsVar> }}`
  (the reworked attempt persists no vars). The engine now seeds declared
  commentsVars so the reference is safe even on a non-rework visit.
- A new authored flow graph needs a RUNTIME test that renders its rework-target
  prompts, not just a static manifest-load test ‚Äî `commentsVar` wiring is a
  templating-convention contract the schema validator cannot catch.
- When a node can be reached on BOTH its initial visit and a rework jump (entry
  == rework target), every var its prompt references must be defined on both
  paths.
- Repo-local precedence over a bundled multi-file artifact (skill dir) must be
  per-ENTRY, not per-file ‚Äî a per-file union yields a Frankenstein artifact.
- Adding a graph node type / reconcile reason / enum value is not done until
  every analytics surface (flow-dsl, flow-graph, reconciliation-gc, db schema +
  ERD) that enumerates it is updated in the same change.
