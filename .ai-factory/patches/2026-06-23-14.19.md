# Codex adversarial-review findings: run.json final-write leak gate + honest hook-arming logs

**Date:** 2026-06-23 14:19
**Files:** web/lib/flows/graph/runner-graph.ts, web/lib/agents/launch.ts, web/lib/flows/graph/__tests__/run-context.integration.test.ts
**Severity:** high

## Problem

A Codex adversarial review of branch `feature/guardrail-hook-engine` (M40 guardrail
hooks + the M38 P7 run-context work it sits on) raised two `high` findings:

1. **run.json git-leak via the unguarded final write.** The P7 (ADR-103) run-context
   loop in `runGraph` writes `<worktree>/.maister/run.json` (intent/task-prompt + every
   node's vars + gate verdicts) on each iteration ONLY when `runContextWriteSafe` is true
   (the git-leak gate). But the **final post-loop rewrite** — the one that captures the
   last node's terminal transition — was **unconditional**. On a normal terminal
   completion in a git worktree where `.maister/run.json` is NOT git-ignored
   (`runContextWriteSafe === false`), the loop writes were skipped but the final write
   still ran, leaking run.json into `git status`, the agent's `git add -A`, and the
   promoted diff — exactly the failure mode the gate exists to prevent.

2. **Guardrail hooks logged as "armed" while nothing enforces them.** The in-progress
   review-fix pass added two `log.debug(..., "guardrail hooks armed")` lines (agent
   launch + orchestrator node) with an `armed: [...]` field. Phase-2 supervisor
   enforcement is not built yet (this branch is Phase 1 of a 6-phase plan), so "armed"
   overstated reality: the rule set is resolved and sent, but the supervisor
   accept-and-ignores it. A developer reading the DEBUG log would believe the guardrail
   was operational.

## Root Cause

1. **Two write sites, one gate, applied to only one.** The git-leak gate
   (`runContextWriteSafe`, computed once at run start via `ensureRunContextExcluded`)
   was wired to the in-loop write but the author treated the trailing "capture the last
   transition" write as a separate concern and forgot it reads the same unsafe worktree.
   Classic "fixed the cited line, missed the sibling recorder" — the highest-risk
   recorder (the terminal one that always runs on success) was the unguarded one.

2. **Wording drift between phases.** The verb "arm" is correct *inside the resolver*
   (a rule is "armed" = present in the resolved config that gets sent). The DEBUG log
   reused "armed" to describe the run, where it reads as "enforced". Resolved-and-sent
   was conflated with enforced because the emitter (Phase 1) and the enforcer (Phase 2)
   land in different commits.

## Solution

1. Wrapped the final `writeRunContext` call (runner-graph.ts ~3601) in
   `if (runContextWriteSafe) { ... }`, mirroring the in-loop write verbatim. Minimal /
   surgical — did NOT centralize writes behind a helper (the safety flag is computed
   once and gates each site; matching the existing inline pattern is the consistent
   choice, and a re-check per write would re-run two `git` subprocesses).
2. Reworded both DEBUG logs: message `"guardrail hooks armed"` →
   `"guardrail hooks resolved (enforcement: Phase 2)"`; field `armed:` → `rules:`; added
   a `// NOTE: resolved-and-sent ≠ enforced … Phase 2 … accept-and-ignore` comment at
   each site. Left the supervisor accept-and-ignore schema, the resolver, and the
   Phase-1 threading **intact** — they are deliberate and load-bearing (the supervisor
   `StartSessionRequestSchema` is `.strict()`; without the acceptor the web emitter's
   `hooksConfig` field fails the parse and runs can't spawn). Docs already say
   "Designed — not yet coded", so no doc change was needed.

**Class-sweep (recorded per the boundary-fix rule):**
- `grep -rn writeRunContext web/lib supervisor/src` → exactly 2 call sites
  (runner-graph.ts:1975 already-gated, :3595 now-gated). Class fully closed.
- `grep -rn "guardrail hooks armed"` + `armed:` over web/supervisor → only the 2 logs;
  no test asserted on the wording, so the reword is safe.

**Regression test** (RED-before-GREEN proven): a new
`run-context.integration.test.ts` case drives `runFlow` through a git worktree where
`.maister/run.json` is **tracked** (`git add`) — `ensureWorktreeGitExclude` still adds
the `.maister/` exclude, but `git check-ignore` consults the index, so a tracked path is
reported NOT-ignored → `isRunContextWriteSafe === false`. A `{"sentinel":true}` body
proves the final write is skipped (sentinel intact). With the gate defeated the test
fails with `run.json` overwritten by `{"intent":"fix the bug",...}` — the leak itself.

## Prevention

- **When a value has a safety/validity gate, gate EVERY write/emit of it — sweep the
  whole class in one pass, not just the cited line.** Grep the writer function name to
  enumerate sites; the trailing "capture the final state" write that runs on the happy
  path is the one most likely to be forgotten and the most damaging (it always fires).
- **A "best-effort, correctness-never-depends-on-it" helper is exactly where a leak
  hides** — "best-effort" justifies skipping on failure, not skipping the *safety check*.
  Consider making the gate intrinsic to the writer when there are ≥2 call sites and the
  check is cheap/cacheable; here it stays inline by design (two git subprocesses, run
  once).
- **Don't let log/comment vocabulary outrun the implementation across a phased feature.**
  In Phase N, words like "armed"/"enforced"/"active" must describe what Phase N actually
  does. "resolved (enforcement: Phase 2)" states the real state and points at the gap.
- **A reviewer flagging "false safety boundary" on a deliberately-phased branch is a
  signal to make the intermediate state HONEST, not necessarily to build the next phase
  early.** Verify what is load-bearing (the accept-and-ignore acceptor IS — removing it
  breaks launch) before "removing the half-ship".
