# Inbox-context route + service shipped untested under an inaccurate "covered by X" justification

**Date:** 2026-06-18 23:21
**Files:** web/app/api/runs/[runId]/inbox-context/__tests__/route.test.ts (new), web/lib/queries/__tests__/inbox-context.integration.test.ts (new), web/lib/queries/__tests__/hitl-stage.test.ts, web/lib/queries/__tests__/inbox-context.test.ts, docs/api/web.openapi.yaml
**Severity:** medium

## Problem

`/aif-verify` on the inbox-card-redesign branch surfaced three warnings:

- **F1:** The new `GET /api/runs/[runId]/inbox-context` route and its service
  `getInboxCardContext` had NO test. Only the pure helper
  `extractLastAgentMessage` was unit-tested. Plan T1.2 explicitly promised
  integration tests (foreign-run 403, 200 shape, gates+progress mapping,
  missing-events→null). T4.2 re-labeled the gap as "covered by the
  portfolio-inbox integration run" — but that suite exercises
  `getCrossProjectHitlInbox`/`getHitlInbox`, NEVER `getInboxCardContext` or the
  route. So the security boundary (readBoard on the server-derived projectId)
  and the "never 500" degradation contract were unverified.
- **F2:** Two new test files (`hitl-stage.test.ts`, `inbox-context.test.ts`)
  shipped with 7 `prettier/prettier` warnings — the formatter was never run on
  them, adding new lint warnings (against the "no new warnings" rule).
- **F3:** The OpenAPI summary for the new path still read
  `Designed (inbox card redesign)` though the feature shipped. The T5.1
  status-flip task named only `inbox.md`/`hitl.md`, missing the OpenAPI summary.

## Root Cause

The unifying root cause is **a test/doc deliverable promised in the plan was
trimmed during implementation and the gap was papered over with a
plausible-but-false "already covered" justification.** The portfolio-inbox
integration test sits next to the new code and tests adjacent read models, so
"covered by portfolio-inbox" reads as true at a glance — but the two suites
call different functions. Nobody re-checked what portfolio-inbox actually
exercises. F2/F3 are the same shape at smaller scale: the formatter and the
OpenAPI status tag were out of the implementer's checklist because the plan's
task text (T5.1) under-enumerated the surfaces (named two markdown docs, not
the OpenAPI summary).

## Solution

- **F1:** Added two tests mirroring established harnesses:
  - `inbox-context/__tests__/route.test.ts` (unit) — copies the diff route's
    fakeDb + mocked-authz pattern; asserts 200 + `readBoard` on the
    server-derived projectId, 404 (no run, authz/peek untouched), 403 (authz
    rejection), 401 (no session), and that the run-scoped fields are derived
    from the run row, not a body.
  - `inbox-context.integration.test.ts` (real PG, mirrors
    run-node-status.integration) — asserts gate mapping from the current node
    attempt, progress `{done,total}` over the compiled 2-node manifest,
    last-agent-message coalesced from a real `run.events.jsonl` (via
    `MAISTER_RUNTIME_ROOT` pointed at a temp dir, captured+restored), and
    null degradation when the events file / workspace are absent (never 500).
- **F2:** `eslint --fix` scoped to the two files only (not repo-wide — that
  reformats ~60 unrelated files).
- **F3:** Flipped the OpenAPI summary to `Implemented (inbox card redesign)`.

No runtime `[FIX]` logging was added: none of the three findings is a runtime
bug — they are test-coverage, lint, and doc-status gaps. The service already
degrades every field to null under `try/catch` (verified by reading).

## Prevention

- **A "covered by X" claim is a verification target, not a free pass.** When an
  implementer says a new SUT is "covered by" an existing suite, open that suite
  and confirm it actually imports/calls the SUT. Adjacency in the directory
  tree is not coverage. Re-derive the promised test assertions from the plan,
  not from the implementer's summary.
- **A read-only GET still has a security boundary.** "No body → no injection"
  satisfies the trust-boundary class, but authorization on the server-derived
  resource (foreign-run → 403) is a distinct concern that needs its own test.
- **Run the formatter on every new test file before committing.** `pnpm
  test:unit` passing does not imply `prettier/prettier` is clean; lint and test
  are separate gates.
- **A status-flip task must enumerate EVERY surface that carries the tag** —
  screens doc + analytics doc + the OpenAPI `summary`/`description`. Grep the
  feature name across `docs/` (incl. `*.openapi.yaml`) for the old status word
  before declaring the flip done.
- **ESLint bracket-path gotcha:** `app/api/runs/[runId]/...` is glob-hostile.
  `--fix` with the literal path or `\[`-escaping fails ("no files matching");
  use a `*` wildcard for the dynamic segment
  (`app/api/runs/*/inbox-context/__tests__/route.test.ts`).
