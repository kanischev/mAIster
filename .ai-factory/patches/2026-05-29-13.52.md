# Adversarial-review round 2: API password gate, run-detail HITL, real run ids, stale doc

**Date:** 2026-05-29 13:52
**Files:** web/lib/authz.ts, web/lib/errors.ts, web/app/api/{runs,projects,projects/[slug]/tasks,runs/[runId]/hitl/[hitlRequestId]/respond}/route.ts, web/app/(app)/runs/[runId]/page.tsx, web/components/board/run-hitl-response.tsx, web/lib/queries/{run,hitl,portfolio}.ts, web/app/(app)/page.tsx, web/messages/{en,ru}.json, docs/{error-taxonomy,configuration}.md, docs/system-analytics/projects.md, web/lib/__tests__/authz-db-authoritative.integration.test.ts, web/lib/queries/__tests__/portfolio.integration.test.ts
**Severity:** critical

## Problem

Second Codex adversarial pass found four issues after the first fix round:

1. **(critical)** Forced password change was page-only. `getSessionUser()` carried
   `mustChangePassword`, but `requireSession()`/`requireGlobalRole()` returned the
   user without enforcing it, so the seeded `admin@maister.local` / `maister-admin`
   could call privileged APIs (`POST /api/projects`, etc.) before changing the
   well-known default password — contradicting the migration/doc contract.
2. **(high)** Form/human HITL "Review" linked to `/runs/${runId}`, a route that did
   not exist — a form/human pause left the run unreachable/stuck in `NeedsInput`.
3. **(medium)** The portfolio Needs-You CTA built `runId` from the project slug, so
   the link pointed at `/runs/<slug>`, never the paused run.
4. **(medium)** `docs/system-analytics/projects.md` still described registration as
   "Designed", body `{ maisterYamlPath }`, and `400 BAD_CONFIG` — none matching the
   implemented `{ dir }` / 422 / 409 / 502 route.

## Root Cause

1. The authz layer split "is authenticated" from "is authorized" but had no third
   gate for "account state requires action before any use". The page redirect was
   mistaken for a complete control; APIs bypass the page entirely.
2. The board UI was built against the planned Run-detail surface (web/CLAUDE.md) that
   M9 never shipped — a dangling forward reference.
3. The Needs-You query (`getPortfolio`) never selected `runs.id`, so the call site
   substituted the only id it had (the slug) — a placeholder that became wrong data.
4. Doc drift: the contract doc predated the live route and was never re-derived from
   the implementation.

## Solution

1. Added `requireActiveSession()` = `requireSession` + reject `mustChangePassword`
   with a new `PASSWORD_CHANGE_REQUIRED` (403). `requireGlobalRole`/`requireProjectRole`
   now call it, so EVERY role-gated API fails closed; `requireSession`/`getSessionUser`
   stay permissive so the change-password flow itself works. Mapped 403 in
   `httpStatusForAuthz` + each route switch. Taxonomy + configuration docs updated.
2. Built a minimal protected `app/(app)/runs/[runId]/page.tsx` (RBAC viewer; notFound
   for non-members) + `RunHitlResponse` client (permission buttons / form-human JSON
   response → existing respond route) + `getRunDetail` query. The board link resolves.
3. `getPortfolio` now selects `runs.id`; `PortfolioNeed.runId` is required and threaded
   to the Needs-You href.
4. Rewrote the stale `projects.md` sections to the implemented contract.

## Prevention

- Authorization has THREE gates, not two: authenticated, account-active (no forced
  action pending), authorized (role). Enforce the middle one in the shared API path,
  never only at the page layer — APIs do not run page middleware/layouts.
- A UI link to a route is a contract: the route must exist (or the link must be
  conditional). Grep app-tree for `href="/x/..."` targets without a matching
  `page.tsx` during review.
- Never substitute a "close enough" id (slug) for a real foreign key at a call site;
  select the real id in the query and make the type require it.
- `/aif-verify` and review must re-derive doc contracts from the implementation, not
  trust the doc's own status tag.
- Tests added: forced-change user blocked on a role-gated path then allowed after
  clearing; `getPortfolio().need.runId === runs.id` (and `!== slug`).
