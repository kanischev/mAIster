# Codex adversarial-review findings on the platform-agents/subagents split

**Date:** 2026-06-22 09:56
**Files:** web/lib/packages/__tests__/attach.integration.test.ts, web/app/api/__tests__/projects-register.integration.test.ts, web/components/flows/artifact-editors/frontmatter-artifact-editor.tsx, web/lib/flows/editor/editor-labels.ts, web/components/flows/__tests__/{frontmatter-artifact-editor,package-files-editor}.test.ts, web/messages/{en,ru}.json, docs/system-analytics/agents.md
**Severity:** high

## Problem

A Codex adversarial review of the 3 studio commits surfaced a no-ship:

1. **[high] Integration test break.** `collectInventory` now writes
   `inventory.platformAgents` on every install, but
   `attach.integration.test.ts` still asserted the OLD exact shape
   `toEqual({ skills, agents })`. The package-attach integration path would
   fail the moment the changed installer ran.
2. **[medium] Structural editor missing a declared field.** The plan promised
   `maister-agents/*.md` gets a structural editor for the FULL platform
   frontmatter incl. `capability_profile`, but the agent form exposed every
   field except `capability_profile` (no control + no raw editor for parseable
   frontmatter ‚Üí the field was uneditable through the promised path).
3. **[medium] Stale analytics doc.** `docs/system-analytics/agents.md` still
   said platform agents ship as `agents/<stem>.md` projected into the catalog,
   contradicting the new package-viewer `maister-agents/` convention.

## Root Cause

1. The change was verified with `pnpm test:unit` only; the broken assertion
   lives in the **integration** project (testcontainers), which was never run
   in the loop. A persisted-shape change (`manifest.inventory`) has assertions
   OUTSIDE the unit run. `tsc` did not catch it either: the sibling stale
   literals (`projects-register` mocks) are loosely typed / `as never`, so an
   exact-shape `toEqual` only fails at runtime, not at compile time.
2. The structural editor reused the pre-existing `FrontmatterArtifactEditor`
   `agent_definition` branch, which predated the "full schema" requirement and
   never had a `capability_profile` control. The acceptance criterion ("full
   platform frontmatter") was asserted in the spec but never checked field-by-
   field against the rendered form.
3. `package-viewer.md` was updated for the new convention but the durable
   catalog doc (`agents.md`) was not reconciled in the same pass ‚Äî two
   conventions left ambiguous for the next reader.

## Solution

1. Updated the `toEqual` to include `platformAgents: ["triager"]` and seeded a
   `maister-agents/triager.md` in the fixture so the new inventory member is
   covered. Swept siblings per the project rule: added `platformAgents: []` to
   the two `projects-register` mock manifests (they don't break ‚Äî consumed via
   `?? []` ‚Äî but carried the stale shape). Ran the **integration** project
   (attach 11/11, projects-register 17/17) against real Postgres to confirm.
2. Added a `CapabilityProfileField` (raw-JSON object editor): a local draft so
   intermediate invalid text survives typing; a valid object commits to
   frontmatter, an invalid one shows a notice and is NOT saved (editable, never
   blocks ‚Äî same "warn, don't block" contract as the schema badge). Wired the
   `agentCapabilityProfile` / `agentCapabilityProfileInvalid` labels (en+ru) and
   added render + round-trip tests.
3. Added a boundary note to `agents.md`: the standalone `/agents` catalog reads
   the flow's `agents/<stem>.md` (`<flowRefId>:<stem>`, unchanged); the new
   package-viewer split (`maister-agents/` platform-agents vs
   `capability/**/agents/` subagents) is package-viewer-only and does NOT feed
   the catalog (Phase 2). Cross-links package-viewer.md (docs R7).

## Prevention

- When changing a **persisted shape** (manifest/inventory/jsonb), run BOTH
  `pnpm test:unit` AND `pnpm test:integration` (testcontainers) before claiming
  green ‚Äî exact-match (`toEqual`) assertions on the shape live in the
  integration project, outside the unit run.
- When a TS type gains a required field, grep EVERY literal constructor AND
  every `toEqual`/`toMatchObject` of the old shape; `tsc` alone misses
  loosely-typed / `as never` / `any` fixtures (they fail only at runtime).
- When a spec lists the fields a structural editor must cover, check each field
  has a rendered control before claiming the editor implements the schema ‚Äî
  reusing a pre-existing editor does not imply full coverage.
- When introducing a new convention that overlaps an existing documented one,
  reconcile the existing canonical doc in the SAME change, not only the new
  screen doc.
