# M12 evidence-contract integrity — declared-vs-produced, history-blocking readiness, mutable git refs

**Date:** 2026-06-02 03:12
**Files:** web/lib/flows/graph/runner-graph.ts, artifact-store.ts, evidence-readiness.ts, default-artifacts.ts, web/lib/queries/board.ts, web/lib/worktree.ts, web/app/api/runs/[runId]/artifacts/[artifactId]/payload/route.ts, web/app/api/runs/[runId]/takeover/return/route.ts
**Severity:** high

## Problem

A whole-feature adversarial review (Codex) of the completed M12 typed-artifacts
branch surfaced three defects in the **evidence contract** — the feature's core
promise ("an artifact is evidence a run is ready or not ready"). All three were
invisible to the per-task TDD + per-task review used to build M12, because each
task passed in isolation; the gaps were in the contract's *end-to-end*
invariants.

- **F1** A node could declare `output.produces: [{id, kind, requiredFor:[review]}]`,
  never produce it, and the run still reached `Review`. The runner only recorded
  `diff`/`commit_set`/`path` kinds; every other declared kind (`lint_report`,
  `ai_judgment`, `human_note`, …) was silently skipped with no record and no
  failure.
- **F2** After a normal rework, evidence stayed blocked forever. `supersedePrior`
  and `markArtifactsStale` only touched `validity='current'` rows, so re-producing
  a required artifact left the old `stale` row orphaned; readiness + the board
  badge blocked on *any* non-current `requiredFor` row, so the orphan blocked
  permanently.
- **F3** Git artifact payloads were not immutable. `headRef` was stored as a live
  branch NAME and the payload route rendered against the live branch, so once the
  branch advanced, an old artifact rendered the wrong diff.

## Root Cause

- **F1 — declared contract with no enforcement that the producer covers it.** The
  manifest (`output.produces`) declared evidence aspirationally; the runner-inline
  producer only implemented a subset of kinds and had no "declared ⇒ produced or
  fail" backstop. The migrated `aif` manifest declared 3 artifacts the runner
  literally could not produce — masked because the e2e *seeded* artifacts instead
  of running the real runner.
- **F2 — readiness evaluated over append-only history instead of current state.**
  The model keeps every artifact row (supersession/staleness only mutate
  `validity`). Readiness asked "do any non-current `requiredFor` rows exist?" —
  which is permanently true after the first rework. The correct question is
  per-entity-current: "for each required def, does a `current` row exist?"
- **F3 — a mutable identifier used where an immutable one was required.** A branch
  name is a moving pointer; audit evidence needs the commit SHA pinned at record
  time. And the recording happened at **multiple sites** (runner ×2, takeover,
  default-artifacts) — a classic sibling-sweep surface.

## Solution

- **F1:** added a catch-all inline producer (declared non-git/no-path output →
  recorded under its declared id+kind from the node's captured stdout, file
  locator to `<nodeId>.log` when present, else 1 MB-capped inline) + a §3.6
  backstop (after the produces loop, any declared id lacking a `current` artifact
  → `markNodeFailed(PRECONDITION)`). Now every declared output is produced or the
  node fails.
- **F2:** `supersedePrior` retires ALL prior rows of the `(run,node,def)` (not
  just current); `assertEvidenceReady` + board `mergeBlocked` evaluate
  per-def-current (a def is satisfied iff a `current` row exists), ignoring
  stale/superseded history.
- **F3:** new `resolveRefSha` (git `rev-parse --verify <ref>^{commit}`, schema-
  hardened, no shell); all four git-record sites store an immutable SHA in
  `headRef` (branch-name fallback now WARN-logged); the payload route renders
  git-range AND git-log against the stored `locator.headRef`.

## Prevention

- **After building a multi-phase feature around a central contract, run a
  whole-feature adversarial review of the contract's END-TO-END invariants — not
  just per-task correctness.** Per-task TDD verifies each task in isolation and
  systematically misses cross-cutting contract gaps (declared-vs-produced,
  history-vs-current, multi-site record sweeps). The three highest-severity M12
  bugs all survived green per-task suites and per-task reviews.
- **Declared-contract enforcement:** whenever a manifest/schema *declares* outputs
  that a separate component *produces*, enforce "declared ⇒ produced or fail" at
  runtime AND verify the producer actually covers every declarable kind. A
  declaration the producer can't satisfy is a latent silent-skip.
- **Append-only / soft-delete state must be queried by current-per-entity, never
  "no non-current rows exist".** Any readiness/availability check over a table
  where history rows are retained (validity/status FSMs, supersession chains) must
  group by the entity and check its *latest/current* state. "Block on any
  non-current row" is a permanent-block bug waiting for the second write.
- **Sibling-sweep extends to artifact/record sites, not just validation fields.**
  When a value (a ref, a path, a sink-bound field) is written at multiple
  recorders, fixing one demands grepping and sweeping ALL of them — the
  highest-volume site (`recordDefaultArtifacts`, one row per node finish) was the
  one missed in the first F3 pass. Grep every recorder before claiming a sweep
  closed.
- **Don't let seeded e2e fixtures stand in for real-runner execution of a new
  contract.** The aif manifest "passed" e2e while being unproducible through the
  real runner, because the e2e seeded the artifacts. A contract that ships should
  be exercised through its real producer at least once.
