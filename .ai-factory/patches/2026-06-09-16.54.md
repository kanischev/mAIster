# Patch: workbench lifecycle review-fix discipline

## Context

Claude review of the M27 workbench lifecycle implementation found several
valid gaps where the plan and implementation could drift: export vs handoff
semantics, lifecycle claim testing, stop atomicity, and UI error detail.

## Improvement

- For lifecycle plans, separate policy-level read-model actions from UI/API
  sub-actions. `snapshotCommit` and `handoffBranch` should be documented as
  Export dialog sub-actions unless they are actually emitted by the shared
  lifecycle policy DTO.
- For git-writing UI, include the exact route each primary button calls in the
  acceptance criteria. "Export" must mean pushing the existing run branch;
  creating a separate handoff branch must be a secondary action.
- For lifecycle claim/CAS code, require one real Postgres integration test in
  addition to mocked race tests, because `SELECT ... FOR UPDATE` and stale
  reclaim semantics cannot be proven with pure mocks.
- For stop/slot-release flows, plan the state mutation transaction separately
  from non-fatal queue promotion, and require structured logging for any
  swallowed scheduler failure.
