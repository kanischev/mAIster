# Scratch workspace dialog verification fixes

**Date:** 2026-06-01 15:31
**Files:** `web/lib/scratch-runs/types.ts`, `web/lib/scratch-runs/attachments.ts`, `web/lib/scratch-runs/service.ts`, `web/lib/worktree.ts`, scratch route tests, scratch UI components, messages, E2E smoke
**Severity:** high

## Problem

The scratch workspace dialog branch had verification blockers after the first implementation: empty branch names were documented as fallback but rejected by validation, uploaded filenames did not reject Windows-style traversal, failed launches removed worktrees but left created branches behind, new UI copy bypassed i18n, and multipart scratch launch/message coverage was incomplete.

## Root Cause

The implementation crossed several boundaries at once: API parsing, git worktree creation, artifact storage, DB persistence, and client UI. Some contracts were updated in docs but not normalized at the first boundary, and the upload filename helper assumed POSIX path semantics even though browser filenames can contain backslashes. Compensation also cleaned the worktree path but not the branch ref created by `git worktree add -b`.

## Solution

Normalized blank optional branch names in the request schema, tightened upload filename checks against both POSIX and Windows path separators, added branch deletion to failed-launch compensation, routed new launcher/dialog/rail labels through translations, and expanded unit/smoke coverage for multipart launch/message, upload limits, safe detail serialization, branch fallback, duplicate ACP profiles, and rail project launch.

## Prevention

When a feature adds a new boundary format, cover both JSON and multipart paths in route tests. When a helper protects filesystem paths, test POSIX and Windows-looking separators regardless of host OS. When creating git refs as part of a multi-step operation, compensation must clean every side effect created before the failing step.
