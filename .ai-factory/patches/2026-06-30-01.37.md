# Codex adversarial-review fixes: onExhaustion terminal evidence bypass + scratch retryable clobber

**Date:** 2026-06-30 01:37
**Files:** web/lib/flows/graph/runner-graph.ts, web/lib/scratch-runs/service.ts, web/lib/flows/graph/__tests__/runner-graph-rework-reset.integration.test.ts, web/lib/scratch-runs/__tests__/mark-scratch-prompt-retryable.integration.test.ts, docs/system-analytics/flow-graph.md
**Severity:** high

## Problem

Two [high] findings from a Codex adversarial review of `feature/flow-rework-reset-loop`:

1. **onExhaustion bypasses the terminal review-evidence gate.** The runner's
   review-readiness guard (`assertEvidenceReady("review")`) runs only on the
   `!isRework && terminal` path. The ADR-118 `onExhaustion` branch runs while
   `isRework` is true and lets `resolveTransition(node, onExhaustion)` return
   `null` (a terminal target — the compile loop-back guard allows `exhausted →
   done` because `done` is not an `allowedTarget`). In that config an exhausted
   loop marked the node Succeeded, set `currentNodeId = null`, and the run exited
   to `Review` + `deliverRunIfAutoReady` WITHOUT the evidence check — so stale /
   missing `requiredFor:[review]` evidence or a failed blocking gate could reach
   Review (and possibly auto-delivery).

2. **`markScratchPromptRetryable` clobbers a moved run.** The initial-launch
   prompt-failure handler wrote `scratch_runs.dialogStatus = WaitingForUser` +
   `runs.status = Running` unconditionally — no row lock, no status guard.
   `session_ready` is emitted BEFORE `POST /prompt`, so a concurrent
   discard/stop/recover or a live supervisor event can move the run to a terminal
   / NeedsInput / Review state while the prompt is in flight; a late
   `EXECUTOR_UNAVAILABLE` then resurrected/overwrote that newer state.

## Root Cause

1. The evidence gate's guard condition (`!isRework && terminal`) was written
   before `onExhaustion` existed. `onExhaustion` introduced a SECOND way to reach
   a terminal transition (`isRework && exhaustion → terminal`) that the original
   guard's `!isRework` predicate structurally excludes. The invariant "any
   terminal route to Review is evidence-gated" was encoded as a property of the
   non-rework branch instead of a property of *reaching terminal*.
2. The retryable helper was the lone scratch state-writer without the
   lock-then-read-then-conditional-write fence its siblings already use
   (`completeScratchPromptTurn`, `markScratchCrashed`). A transaction gives
   atomicity, not serialization — without a guard re-read under the lock it is a
   last-writer-wins blind UPDATE.

## Solution

1. In the `onExhaustion` branch, compute `next = resolveTransition(...)` BEFORE
   marking the node, and when `next === null` (terminal) run the SAME
   `assertEvidenceReady(runId, "review", db)` the `!isRework` path runs; on
   not-ready, `markNodeFailed(PRECONDITION)` + fail the run instead of routing to
   Review. A non-terminal `onExhaustion` (the typical human-node route) is gated
   at that target's own terminal transition, so it is unchanged. Integration
   tests: `onExhaustion → done` with a STALE `requiredFor:[review]` def → Failed
   (AC-11); with a CURRENT def → Review (AC-12).
2. Fence `markScratchPromptRetryable` with `lockRunRows` + re-read
   `dialogStatus` under the lock; only apply the retryable transition when the
   dialog is still IN-FLIGHT (`Starting`/`Running` — the launch sets `Running`
   for an initial prompt). Any other state (terminal / NeedsInput / Review /
   WaitingForUser) is a no-op that preserves the newer state and logs a skip.
   Integration tests cover both in-flight states (→ retryable) and all six
   moved states (→ preserved, not resurrected).

## Prevention

- A safety gate keyed on `requiredFor:[review]` evidence must fire on EVERY
  terminal route to Review, not on a single decision branch. When adding a new
  way to reach a terminal/promotion state, grep for the existing terminal
  guards (`assertEvidenceReady`, readiness checks) and replicate them on the new
  path — a guard predicate like `!isRework` silently excludes new branches.
- Every state-writer that flips `runs.status` / a dialog FSM must follow the
  module's established lock-then-guard pattern; a new writer that does an
  unconditional UPDATE is a clobber. When fixing one, sweep the siblings (here
  `markScratchCrashed` was already guarded — confirming the gap was isolated).
- Both fixes need a real DB integration test (state-moved for the CAS/fence,
  stale-evidence for the gate); a mocked unit test would not have caught either.
