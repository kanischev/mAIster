# Project-registration saga: all-or-nothing on flow-install failure (finding 3)

**Date:** 2026-05-29 12:33
**Files:** web/app/api/projects/route.ts, web/app/api/__tests__/projects-register.integration.test.ts, docs/api/web.openapi.yaml, docs/system-analytics/projects.md
**Severity:** high

## Problem

`POST /api/projects` wrote the project graph (project + executors + owner
membership) and *then* installed Flow plugins. Three issues:

1. **(headline)** On a flow-install failure the catch deleted only `flows`
   rows; the `projects` row (+ executors + owner membership) **remained**. The
   slug/repo_path uniqueness was now occupied, so retrying the same
   `maister.yaml` returned **409 forever**, and a flow-less, launch-broken
   project showed up in the portfolio. No archive route existed to recover.
2. **(latent)** The phase was documented as "inside a transaction" but was four
   separate autocommit writes — a crash between the project insert and the
   owner-membership insert produced an **owner-less project** (invisible to its
   creator, since visibility is membership-based).
3. **(concurrency)** The collision check was a read-then-write TOCTOU; two
   admins registering the same dir both passed it, and the second insert tripped
   the unique constraint as a raw error → **500**, not a clean 409.

## Root Cause

A state-changing route mixed a DB write with a non-transactional external side
effect (git clone + symlink) and committed the DB part *before* the side effect
— with only partial, one-table compensation. There was no atomic unit and no
full rollback, so any side-effect failure left durable, conflicting state. The
"is this slug taken?" guard relied on an application read instead of the DB's
own unique constraint.

## Solution

Option 2 (compensate-on-failure saga), plus the two correctness fixes:

- **Atomic phase (c):** project + executors + default-executor + owner
  membership now run in ONE `db.transaction`. `upsertExecutorsFromConfig`
  calls `.transaction()` on the db it is handed, so passing the outer `tx`
  nests via a Postgres savepoint and joins the unit. (Fixes #2.)
- **Full compensation on install failure:** delete the `projects` row — FK
  `ON DELETE CASCADE` removes executors, flows, and the owner membership — then
  `rm -rf` the slug-scoped `.maister/<slug>` artifact subtree (matching
  `installFlowPlugin`'s default `workspaceRoot = process.cwd()`). DB rollback
  runs first (frees the unique slug/repo so a retry is not 409-blocked); disk
  cleanup is best-effort (leftover symlinks re-resolve to the cache on retry).
  The shared, content-addressed system cache (`~/.maister/flows/<id>@<sha>`) is
  intentionally retained. (Fixes #1.)
- **Unique-violation → 409:** a concurrent duplicate that beats the read-time
  collision check fails the unique constraint inside the tx; `isUniqueViolation`
  (pg `23505`) translates it to `CONFLICT` (409). Nothing committed → no
  compensation needed. (Fixes #3.)

## Prevention

- A route that commits DB state and then performs a non-transactional external
  side effect MUST be all-or-nothing: either stage the side effect before the
  commit, or fully compensate (prefer deleting the aggregate root and relying on
  FK cascade over hand-deleting each child table). Partial, single-table
  compensation is a defect.
- Never claim "inside a transaction" without an actual `db.transaction` wrapper;
  prove atomicity with a test that asserts no partial rows survive a failure.
- Uniqueness is the DB's job: back app-level collision checks with the unique
  constraint and translate `23505` to a domain 409, instead of trusting a
  read-then-write window.
- Regression test added: a failing flow install leaves zero rows AND the
  identical request then succeeds (proves both rollback and retry-ability).
