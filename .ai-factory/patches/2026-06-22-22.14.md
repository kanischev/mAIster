# M38 verdict-routing fail-open + P7 run.json git-leak (Codex adversarial review)

**Date:** 2026-06-22 22:14
**Files:** web/lib/flows/graph/gates-exec.ts, web/lib/flows/graph/run-context.ts, web/lib/flows/graph/runner-graph.ts, web/lib/flows/graph/__tests__/runner-graph-decide-routing.integration.test.ts, web/lib/flows/graph/__tests__/run-context.integration.test.ts
**Severity:** high (Finding 1 silent fail-open onto approve/promote) + medium (Finding 2 diff/commit pollution)

## Problem

Two holes surfaced by an adversarial Codex review of the unmerged M38
(`flow-routing-runcontext`) branch:

1. **Advisory verdict-routing gate fails OPEN.** A `decide:{from:verdict}` node
   whose sole routing gate is `mode: advisory` (or a `blocking` `skill_check`
   downgraded/skipped by execution-policy `checks: advisory|skip`) that produced
   NO parseable verdict returned `{ok: true, verdict: undefined}`, so
   `computeDecideOutcome` fell through to the decide `default` branch вЂ” often an
   approve/promote. A broken or flaky judge silently drove business routing.

2. **`run.json` written even when `.maister/` git-exclusion is not guaranteed.**
   The P7 run-context blackboard (`<worktree>/.maister/run.json`, carrying task
   prompt + node vars + gate verdicts) was written unconditionally after a
   best-effort `ensureWorktreeGitExclude` that swallows git errors and no-ops on a
   non-git path. If the exclude silently failed in a git worktree, the agent's
   `git add -A` could commit run.json into the delivered branch.

## Root Cause

1. The per-gate fail-closed guard in `runNodeGates` was gated on
   `isEffectivelyBlockingGate`, which returns `false` for any non-`blocking`
   mode AND downgrades a `blocking` non-review check (incl. `skill_check`) under
   relaxed check-strictness. `compile.ts verifyDecideAndOnMismatch` enforces
   *exactly one* `ai_judgment/skill_check` gate on a `from:verdict` node but never
   constrains its `mode`. So the "no verdict в‡’ fail closed" intent (stated in the
   code's own comment + the existing `mode:"blocking"` regression) had a hole for
   every routing gate that isn't effectively-blocking at runtime.

2. P7 treated the `.maister/` exclude as fire-and-forget and the write as
   unconditional. The exclude is genuinely best-effort by design
   (`run-context.ts`: "correctness never depends on it вЂ¦ a write failure never
   fails the run"), but nothing verified the *outcome* before writing вЂ” and the
   whole codebase architecturally assumes `.maister/` is ignored (`discardWorktree`
   uses `git clean -fd`, not `-fdx`, precisely so an ignored `.maister/` survives).

## Solution

1. **Single post-loop invariant in `runNodeGates`** (gates-exec.ts): after the
   gate loop, `verdictRouting && routedVerdict === undefined && !blockingFailedGateId`
   в‡’ set `blockingFailedGateId` to the verdict gate's id в†’ node Fails closed
   (PRECONDITION), identical to the blocking path. One change closes *every*
   no-verdict vector (advisory, policy-downgraded `skill_check`, skipped gate) вЂ”
   strictly better than Codex's compile-time-only suggestion, which cannot see the
   runtime execution-policy vectors. The legitimate low-confidence/"fail" verdict
   path is untouched (a parsed verdict sets `routedVerdict`, so the guard skips).

2. **Confirm-then-gate** (run-context.ts + runner-graph.ts): new
   `isRunContextWriteSafe(worktreePath)` uses `git check-ignore` (ground truth вЂ”
   honors info/exclude AND the repo's own .gitignore) gated behind
   `rev-parse --is-inside-work-tree`. Safe вџє not a git tree (no leak surface) OR
   the path is confirmed ignored. `ensureRunContextExcluded` now returns this
   boolean; the runner skips `writeRunContext` when unsafe. **Rejected Codex's
   "fail the run with PRECONDITION/CONFIG"** вЂ” it contradicts the documented
   best-effort invariant; the fix preserves "never fail the run" by degrading to
   no-blackboard instead.

Regression tests: an `mode:"advisory"` no-verdict node asserts run Failed + the
default branch never ran; three `isRunContextWriteSafe` cases (non-git в†’ safe,
git+unignored в†’ unsafe, git+excluded в†’ safe, with a hermetic
`core.excludesFile=/dev/null`).

## Prevention

- **A fail-closed guard must not itself be gated on a relaxable predicate.** When
  "absence of X в‡’ fail closed", put the check on the *direct* signal (`X === undefined`)
  at the decision point, not behind `isEffectivelyBlockingGate` / mode / policy that
  can be relaxed. Default-route is a business decision вЂ” never reach it on a producer
  that yielded nothing.
- **For a remediation that "fails the run", first grep the target's own doc comment
  for a best-effort/never-fail invariant.** A reviewer's "fail closed в†’ throw" can be
  the wrong trade when the artifact is explicitly best-effort; "skip silently" closes
  the leak without a robustness regression.
- **`git check-ignore` is the ground-truth ignore test** (info/exclude + .gitignore +
  global), but distinguish "not a git tree" (no leak surface в†’ write) from "git tree,
  not ignored" (leak в†’ skip) via `rev-parse --is-inside-work-tree`, else you break the
  legitimate non-git path (an existing test wrote run.json into a plain `mkdtemp`).
- **Sibling-writer grep (recorded per skill-context rule):** capability
  materialization also writes under `<worktree>/.maister/capabilities/...` relying on
  the same best-effort exclude (pre-existing M14/ADR-043, NOT this finding). Left
  untouched (surgical scope + ADR-041 sensitivity) but flagged here for a deliberate
  future pass rather than a silent in-scope edit.
