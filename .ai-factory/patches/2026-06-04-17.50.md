# M15 readiness: fail-open calibration on out-of-range confidence + read-model phase-scope drift

**Date:** 2026-06-04 17:50
**Files:** web/lib/flows/graph/gates-exec.ts, web/lib/db/schema.ts, docs/system-analytics/readiness.md, web/lib/flows/graph/__tests__/calibrate-verdict.test.ts, web/lib/flows/graph/__tests__/calibrate-verdict-exec.integration.test.ts, web/lib/queries/__tests__/readiness-ssot-consistency.integration.test.ts
**Severity:** high

## Problem

Two findings from a Codex adversarial review of the M15 branch:

1. **[high] Out-of-range confidence passes a calibrated blocking gate.**
   `calibrateVerdict` gated only on `typeof parsed.confidence === "number"`, then
   compared `confidence >= confidence_min`. A malformed agent verdict
   `{"verdict":"pass","confidence":2}` cleared the threshold (`2 >= 0.8`) and was
   persisted `passed`/`above_threshold`. `Infinity` passed too. Because readiness
   trusts only `gate_results.status`, this fail-OPENED a promotion-relevant gate.

2. **[medium] Read-model badge vs enforcer phase-scope divergence.**
   `assertEvidenceReady(runId, phase)` filters artifacts `requiredFor` the phase,
   while `getRunReadiness` / `computeReadinessByRun` filter `requiredFor` non-empty
   (any phase). The readiness.md Purpose/flowchart said "that phase" while the
   normative State-machine section said "non-empty" — internal doc drift on top
   of the code split. Untested gap: no test seeded a `requiredFor:["merge"]`-only
   artifact.

## Root Cause

1. The confidence domain (`0..1`) was enforced on the **config** side
   (`gateCalibrationSchema` = `z.number().min(0).max(1)`) but NOT on the
   **untrusted agent-output** side. `parseVerdict` accepted any JSON number, and
   `calibrateVerdict` never range-checked before the `>=` comparison. The mental
   model assumed confidence ∈ [0,1] without validating the assumption at the
   trust boundary — classic "validated the schema, not the wire".

2. The any-phase read-model was a deliberate fail-closed superset (surface a
   later-phase blocker on the badge early), but that intent was never written
   down; the doc's looser prose said "that phase", making it read like a bug.

## Solution

1. Range-guard inside `calibrateVerdict`'s `typeof === "number"` branch:
   `!Number.isFinite(c) || c < 0 || c > 1` → fail-closed with a NEW frozen
   outcome `invalid_confidence`. Placed in the *present* branch (not the absent
   branch) so `allow_missing_confidence` can never rescue an out-of-range value.
   Added the variant to the `GateVerdict.calibration.outcome` union, a WARN log
   at the call site, and unit + executor-integration coverage (confidence 2, 1.5,
   -0.5, ±Infinity, NaN; boundary `1` stays in-domain). Updated readiness.md
   flowchart, truth table, and the calibration expectation.

2. Chose the **any-phase + document-the-intent** direction (user-confirmed) over
   changing read-model behavior — it matches the system's fail-closed posture and
   the existing SSOT test. Documented the intentional phase-scope split in
   readiness.md's classifier section and pinned it with a regression test asserting
   read-models `blocked`, `assertEvidenceReady("review")` ready, `("merge")` blocked.

## Prevention

- **Validate at the trust boundary, not just the schema.** When a numeric range
  is enforced by zod on the config/producer side, the same range MUST be
  re-checked wherever the value arrives from an UNtrusted producer (agent stdout,
  external report, user body). A `z.number().min().max()` on one side is not a
  guarantee on the other.
- **Out-of-range ≠ missing.** A present-but-invalid value must fail-closed on its
  own path and must never fall through to a "missing / allow-default" branch.
- **When the enforcer and a read-model deliberately diverge, write the invariant
  down AND pin it with a test** (superset direction: badge may be stricter, never
  laxer). Latent divergences with no test read as bugs to the next reviewer.
- Review checklist: every `typeof x === "number"` before a comparison that gates
  promotion needs a `Number.isFinite` + domain check in the same expression.
