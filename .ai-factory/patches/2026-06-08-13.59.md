# Relative Authored Bridge Source Dirs Treated As Git Sources

Date: 2026-06-08

## Files

- `web/lib/flows.ts`
- `web/lib/flows/__tests__/package-authoring.test.ts`

## Severity

High

## Problem

`install-authored-flow-package --source-dir ./exported` validated the authored
package directory successfully, then passed the original relative path into the
generic Flow installer. The generic installer only classified absolute paths and
`file://` URLs as local directories, so relative authored package sources could
fall through to git-source handling instead of installing the already validated
local bytes.

## Root Cause

The authored bridge and generic installer resolved the same source at different
trust boundaries. The bridge validated package contents from a relative
directory, but the generic installer received the unresolved string and applied
its own source-kind detection.

## Solution

Resolve the authored bridge source directory once before validation, logging, and
installation, then pass that absolute path into the generic installer with the
authored-package untrusted trust override. Added a regression test that changes
the process cwd, installs from a relative package directory, and asserts the
installer copies local package bytes instead of treating the input as git.

## Prevention

When a local package reader and a generic installer share one source argument,
canonicalize the path before crossing the boundary. Regression coverage should
exercise both absolute and relative authored package directories.
