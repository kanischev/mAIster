# Patch: Admin User Contract Fix

**Date:** 2026-05-30 13:38
**Skill:** `/aif-fix`

## Problem

Verification found contract drift in admin user management:

- `POST /api/auth/register` returned `{ id, email, status }`, while the plan and public API contract require `{ status: "pending" }`.
- OpenAPI omitted `422 CONFIG` responses for admin status and role mutation routes, even though the handlers return them for invalid JSON or body validation failures.

## Root Cause

Runtime and OpenAPI were updated independently during the larger user-management implementation. The registration module kept useful internal return fields, and the route accidentally exposed them instead of projecting the public response shape.

## Fix

- Projected the register route success response to `{ status }` only.
- Updated `RegisterUserResponse` in `docs/api/web.openapi.yaml` to status-only.
- Added documented `422 CONFIG` responses for `PATCH /api/admin/users/{userId}/status` and `/role`.
- Added focused integration coverage for the public register API response shape, status metadata updates, disabled-to-active re-enable, role changes, and last-active-admin protection.

## Verification

- `pnpm --filter maister-web typecheck`
- `pnpm --dir web exec vitest run --project integration 'app/(auth)/__tests__/register.integration.test.ts' lib/__tests__/users.integration.test.ts`
- `pnpm --filter maister-web lint`
- `pnpm validate:docs`
- `git --no-pager diff --check`

## Prevention

For public API route handlers, keep service return types richer when useful, but explicitly project response DTOs at the route boundary and mirror that DTO in OpenAPI tests or focused integration coverage.
