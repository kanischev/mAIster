# Authored package import copied unvalidated non-regular entries

**Date:** 2026-06-08 14:41
**Files:** web/lib/flows/package-authoring.ts, web/lib/flows/__tests__/package-authoring.test.ts
**Severity:** high

## Problem

Authored Flow package imports validated only regular package files, but the
authored bridge installer later copied the original source directory into the
installed revision cache. A package with safe visible files and a symlinked
`setup.sh` could pass validation, enter the cache as untrusted bytes, and later
become executable after an explicit trust decision.

The package reader also treated `flow.yaml.name` as the package slug even when
the manifest name was a human display label such as `Bugfix`, making otherwise
portable packages fail import.

## Root Cause

The validation boundary and the install boundary used different byte sets:
`readPackageFiles()` silently ignored non-regular directory entries while the
generic local installer copied the full source tree. The code assumed skipped
entries were harmless, but skipped entries can still be copied and observed by
later setup detection and execution.

The slug derivation bug came from overloading one manifest field for two
different concepts: display name and filesystem-safe package identifier.

## Solution

The package reader now fails fast when it sees a non-regular entry, including
symlinks, and logs the refused package path with a `[FIX]` marker. This keeps
the validated package surface aligned with what the installer is allowed to
copy.

Package slug resolution now uses `flow.yaml.name` only when it already satisfies
the authored package slug contract. Human display labels fall back to the safe
source directory slug while remaining available as package metadata name.

Regression coverage now verifies both cases: display-label fallback and
symlinked package entries being rejected before the authored bridge install can
reach the generic copy path.

## Prevention

For package, archive, or directory import code, validation must cover every
entry that can later be copied, installed, executed, or exported. Skipping a
filesystem entry is not sufficient unless the later install/export path also
uses the normalized validated representation.

During review, compare "validated bytes" to "persisted/copied bytes" for every
path that crosses a trust boundary. Also separate display labels from slugs or
other sink-safe identifiers instead of reusing one field for both meanings.
