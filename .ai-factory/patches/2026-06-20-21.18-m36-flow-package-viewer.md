**Severity:** critical + high

## Problem

A Codex review of the M36 local-package + project-less assistant feature surfaced
5 issues:

1. **[critical] Symlinked missing parents bypass working-dir confinement.**
   `resolveWithinWorkingDir` realpath'd only the immediate parent and fell back to
   the LEXICAL path when it didn't exist yet. A package containing `link -> /outside`
   let a write to `link/newdir/file` pass — then `mkdir(...,{recursive:true})`
   followed the symlink and created/wrote outside the root.
2. **[high] Assistant follow-up turns not bound to the lock holder.** Project-less
   assistant message turns only called `requireActiveSession()` — any active user
   with a run id could inject prompts into another holder's live session and write
   into the locked working dir.
3. **[high] Archive import decompresses before enforcing uncompressed limits.** The
   pre-parse cap only bounded the COMPRESSED blob; `readZipEntries`/`readTarGzEntries`
   fully materialized the uncompressed content before `planImport` enforced per-file/
   total caps → a small zip/tar.gz bomb could exhaust memory.
4. **[high] Project-less assistant recovery could not work.** The shared recover
   route required both a workspace row AND a project row; a local-package assistant
   run intentionally has neither, so recovery threw "workspace/project not found".
5. **[high] Local-package mutations used session-only auth.** Studio routes gated
   destructive ops with `requireSession()`, which does NOT enforce `accountStatus ===
   active` / `mustChangePassword === false` — a disabled or password-gated session
   could still rename/archive/delete/edit/import/commit/cut.

## Root Cause

1. The confinement guard checked only ONE level (the immediate parent) and trusted a
   lexical fallback for not-yet-existing dirs. The actual escape vector is a symlink
   in any **already-existing ANCESTOR** segment, because `mkdir -p` follows symlinks
   in existing segments (leaf dirs it creates fresh are safe).
2. The project-less branch was a placeholder ("any active user, per ADR-095") that
   never bound the run to its launching user or the editor lock the launch asserted.
   The launch route gated correctly (`assertHoldsLock`), but the follow-up message /
   recover paths did not carry the lock and never re-checked ownership.
3. Cap enforcement was placed AFTER full collection (`planImport`) instead of DURING
   parse. The compressed-blob pre-cap is not a proxy for uncompressed size.
4. `loadScratchRecoveryRows` was written before the project-less assistant run kind
   existed and hard-required the project/workspace rows + `requireProjectAction(null)`.
5. `requireSession()` is intentionally permissive (it must let the change-password
   page clear the gate). It is the wrong helper for any state-changing route.

## Solution

1. Walk up to the NEAREST EXISTING ancestor, `realpath` IT, and verify containment;
   reject if it resolves outside the root. Leaf segments that don't exist yet are
   created fresh, never followed. (`paths.ts`)
2. Added `assertUserHoldsLock` (user-scoped live-lock check) in `lock.ts` and
   `assertLocalPackageAssistantActor(run, userId, {requireLock})` in `service.ts`:
   READ requires `created_by_user_id === caller`; DRIVE (message/recover) also
   requires the caller to hold a live working-dir lock. Wired into the message-append
   else-branch, the GET read route, and the recover route. A lock takeover by another
   user flips `locked_by_user_id` and revokes the launcher's write ability.
3. Thread the uncompressed caps into `readZipEntries` (declared `header.size`
   pre-check + running ACTUAL-bytes counter as each entry inflates) and
   `readTarGzEntries` (per-entry + cumulative running counter with an abort flag that
   stops accumulation on the first breach). Peak memory bounded to ~maxBytes.
4. `loadScratchRecoveryRows` branches on `run.localPackageId`: loads the local
   package as a workspace-/project-SHAPED view (worktreePath = slug-derived working
   dir) and carries `confineRoot` through to `createSession`, re-asserting the
   assistant's confinement on resume. Auth branches to the actor check.
5. `requireSession()` → `requireGlobalRole("member")` on every MUTATION
   (POST/PUT/PATCH/DELETE) and `requireActiveSession()` on every READ (GET) across all
   studio local-package routes — matching the documented member-level authoring intent
   (`manageLocalPackages: "member"`). Also added `ACCOUNT_INACTIVE → 403` to the shared
   `project-route-helpers.httpStatusForCode` (it mapped UNAUTHORIZED/PASSWORD_CHANGE_
   REQUIRED but omitted ACCOUNT_INACTIVE → a disabled session surfaced as 500).

## Prevention

- **Path confinement must check the realpath of the nearest EXISTING ancestor, not
  just the immediate parent**, because `mkdir -p` / git ops follow symlinks in any
  existing segment. A lexical fallback for a missing parent is always unsafe.
- **Decompression/expansion limits must be enforced DURING the streaming/parse step
  with a running counter**, never after full materialization. A compressed-size cap
  is not an uncompressed-size cap (zip/tar bombs).
- **A new run kind that drops a backing row (no project, no workspace) fans out to
  EVERY shared route** (read, message, recover, stop, discard, HITL). Bind it to a
  durable owner field (`created_by_user_id`) and re-check on each access; do not leave
  a placeholder `requireActiveSession()` as the only gate.
- **`requireSession()` is for the change-password flow ONLY. Every state-changing or
  role-scoped route must use `requireActiveSession()` (or `requireGlobalRole`).** Grep
  for `requireSession(` in route handlers during review.
- When fixing one auth-gate finding, **sweep every sibling route sharing the wrapper**
  and confirm the error taxonomy maps the auth codes (UNAUTHENTICATED/UNAUTHORIZED/
  ACCOUNT_INACTIVE/PASSWORD_CHANGE_REQUIRED) to 401/403, not the 500 default.
