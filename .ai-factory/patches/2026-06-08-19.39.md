# User-owned token actor uniqueness collision

**Date:** 2026-06-08 19:39
**Files:** `web/lib/db/schema.ts`, `web/lib/assignments/service.ts`, `web/lib/db/migrations/0031_token_actor_scope_support.sql`, `web/lib/db/migrations/meta/0031_snapshot.json`, `web/app/api/v1/ext/runs/[runId]/hitl/__tests__/route.integration.test.ts`, `docs/database-schema.md`, `docs/db/assignments-domain.md`, `docs/db/erd.md`
**Severity:** high

## Problem

User-owned API tokens stored `ownerUserId` on their `api_token` actor rows for
HITL attribution, but `actor_identities_project_user_uq` was still a table-wide
unique constraint on `(project_id, user_id)`. A valid personal token could fail
with a database uniqueness violation when the owner already had a human actor in
the project, or when two different personal tokens shared the same owner.

The current `0031_snapshot.json` also omitted the existing
`actor_identities_project_token_uq` partial unique index, so the Drizzle
metadata did not describe the live schema.

## Root Cause

The feature crossed two identity dimensions: human ownership and token identity.
The implementation correctly added `ownerUserId` to API-token actors for audit
joins, but the older user uniqueness invariant was modeled as a global unique
constraint instead of a `kind = 'user'` partial unique index. The API-token
upsert targeted `(project_id, token_id) WHERE kind = 'api_token'`, so it could
not handle conflicts raised by the unrelated `(project_id, user_id)` constraint.

The snapshot drift came from reconciling only the new token columns and owner
index, without preserving the partial actor token index already introduced by
migration `0026`.

## Solution

Converted `actor_identities_project_user_uq` to a partial unique index where
`kind = 'user'`, and updated `ensureUserActor` to use a matching
`targetWhere` conflict target. Kept `ownerUserId` on `api_token` actors so
personal token attribution remains available.

Updated migration `0031` to drop the old unique constraint and create the
partial user index. Updated `0031_snapshot.json` to include both partial actor
indexes. Added debug pino logs with `[FIX:token-actor-uniqueness]` around the
user/API-token actor ensure paths.

Added integration regressions proving a user-owned token actor can coexist with
the owner human actor, two personal tokens for the same owner receive distinct
API-token actor rows, and the migrated schema exposes partial user/token actor
indexes rather than the old user constraint.

## Prevention

- When an upsert uses a partial unique index, verify every other unique
  constraint over the inserted columns cannot fire first.
- If a row has both identity and ownership fields, keep uniqueness scoped to
  the identity kind that actually owns the invariant.
- During migration metadata fixes, verify existing partial indexes remain in
  the latest snapshot, not only newly added columns and indexes.
- Add regression coverage at the DB invariant level when the bug is a unique
  constraint or `ON CONFLICT` target mismatch.
