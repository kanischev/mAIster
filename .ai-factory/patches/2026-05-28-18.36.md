# M7 adversarial review — resume-ordering, SSE event-log ownership, spec drift

**Date:** 2026-05-28 18:36
**Files:** `web/app/api/runs/[runId]/hitl/[hitlRequestId]/respond/route.ts`, `supervisor/src/spawn.ts`, `web/app/api/runs/[runId]/stream/route.ts`, `docs/api/supervisor.openapi.yaml`, `docs/api/async/supervisor-sse.asyncapi.yaml`, tests
**Severity:** high

## Problem

Codex adversarial review on the M7 branch returned `needs-attention` with three
findings:

1. **Critical:** the HITL response route for form/human kinds set
   `runs.status = 'Running'` inside the same transaction that recorded the
   user's reply, then fired `runFlow(runId)` asynchronously. `runFlow`
   detects resume via `loaded.run.status === 'NeedsInput' && currentStepId
   !== null`. Because the status was already `'Running'` by the time
   `queueMicrotask` fired, the resume guard never matched. The runner
   restarted at step 0 and re-executed every prior CLI/agent step (with
   their real side effects — git commits, subprocess invocations, supervisor
   spawns).
2. **High:** the supervisor opened `<stepId>.events.jsonl` per spawn and the
   web SSE bridge tailed `<currentStepId>.events.jsonl`. In
   `slash-in-existing` mode the supervisor reuses ONE session across many
   flow steps, so the writer keeps appending to `<first-stepId>.events.jsonl`
   even after the run's `currentStepId` advances. The bridge then opened a
   non-existent file for later steps, silently dropping live observability
   and breaking `Last-Event-ID` resume.
3. **High:** the supervisor OpenAPI still documented `POST /sessions/{id}/input`
   as the M3 501 stub, and the AsyncAPI still emitted `session.permission_auto`.
   Generated clients and contract tests would have pointed at the wrong wire
   shape.

## Root Cause

All three failures share a pattern: a contract was implemented in code but the
**ordering invariants** that the contract depended on were never centralized in
one place.

- **Resume ordering:** the runner-side rule "I only resume when status is
  NeedsInput" was implicit. The HTTP route was written from a UX angle ("I'm
  giving the user a 200 — the run is conceptually Running now") without
  consulting the runner's resume gate. The DB column `runs.status` was used
  for two unrelated purposes — UX hint and resume signal — and the route only
  thought about the former.
- **Event log ownership:** the path was chosen at spawn time using
  `request.stepId`, the field that happened to be in scope. Nothing in the
  spec or code said "this path MUST be stable across the lifetime of the run
  the web bridge thinks it is observing." The contract between supervisor and
  bridge was never explicit.
- **Doc drift:** Tasks 18–21 of the original plan were marked DEFERRED. The
  ROADMAP entry called out the deferral, but the deferred state was treated
  as "follow-up" rather than "blocker." Codex flagged this immediately.

## Solution

1. **Resume ordering:** the form/human path of
   `POST /api/runs/[runId]/hitl/[hitlRequestId]/respond` now writes only
   `hitl_requests.response` + `respondedAt` and leaves `runs.status` at
   `NeedsInput`. The runner is the single owner of the `NeedsInput → Running`
   transition (it already runs the transition on entry via `loaded.run.status
   = "Running"`). The response body returns `runStatus: 'NeedsInput'` to match
   the row state at response time. If `runFlow` fails to start, the run stays
   resumable — the input artifact is on disk and `runner-human.ts`'s
   resume-from-existing-input branch picks it up on the next call.
2. **Event log ownership:** events.jsonl path is now per-RUN
   (`<runId>/run.events.jsonl`). Multiple spawns for the same run append via
   `fs.createWriteStream` with `flags: "a"` — Linux gives atomic per-write
   append, and the supervisor's writes are one syscall per line. The web SSE
   bridge tails the single file unconditionally, with `Last-Event-ID` resume
   working across step boundaries. The supervisor's `eventsLogPath` return
   value is now stable across consecutive sessions of the same run.
3. **Spec sync:** `supervisor.openapi.yaml` 0.5.0 → 0.7.0 with the real
   `InputBody` schema (discriminated on `action`), `200 / 404 / 409` responses,
   and `NEEDS_INPUT` + `HITL_TIMEOUT` added to `SupervisorErrorCode`.
   `supervisor-sse.asyncapi.yaml` 0.5.0 → 0.7.0 with `session.permission_auto`
   removed and `session.permission_request` ({requestId, options, toolCall})
   added. Prose updated everywhere `permission_auto` appeared.

Tests touched:
- `web/app/api/runs/[runId]/hitl/[hitlRequestId]/respond/__tests__/route.test.ts`:
  asserts `runs.status === 'NeedsInput'` after form happy path with an inline
  comment explaining the contract.
- `supervisor/src/__tests__/spawn.test.ts`: new regression case that spawns
  two sessions for the same `runId` (one with `stepId=stepA`, one with
  `stepId=stepB`) and asserts both write to the same `run.events.jsonl` file
  with their respective `sessionId` field intact.

## Prevention

- **Single ownership for state transitions.** When a status field doubles as a
  resume signal, ONE component owns each transition. Other components signal
  their work via different fields (here: `hitl_requests.respondedAt`, the
  input artifact on disk). Cross-component coupling on the same column is a
  smell. Document the ownership map in `system-analytics/<domain>.md`
  Expectations section.
- **Contracts between long-lived processes must be path-stable.** When two
  processes share a file by path, the path's variables MUST be stable for the
  duration of the read side's consumer. `stepId` is not stable for
  `slash-in-existing`; `runId` is. Lint rule worth considering: any path
  computation in supervisor that includes `stepId` should justify in a
  comment why per-step is correct (vs. per-run / per-session).
- **Deferred docs are blockers, not follow-ups, when the surface is
  externally observed.** OpenAPI/AsyncAPI specs are not internal notes — they
  feed generated clients, mocks, and contract tests. If a milestone changes a
  wire surface, the spec rewrite ships in the same PR or the milestone is not
  shippable. Add a pre-merge gate that diffs implementation route names /
  Zod schemas vs spec keys.
- **Adversarial review IS the gate for "is the design right."** Unit tests
  passing means the code does what it was written to do; they do not catch
  whether the code's contract is the right contract. Run an adversarial pass
  (Codex / external reviewer) AFTER the green test suite, BEFORE the
  feature-branch merge.
