# M11c settings: overstated visibility, mutable audit, and sparse-map overreporting

**Date:** 2026-06-01 13:16
**Files:** web/lib/config.schema.ts, web/lib/flows/graph/ledger.ts, web/components/board/panels/flow-settings-panel.tsx, web/app/(app)/runs/[runId]/page.tsx, web/messages/en.json, web/messages/ru.json, web/lib/__tests__/config.schema.test.ts, web/lib/flows/__tests__/enforcement.test.ts, web/lib/db/__tests__/schema.integration.test.ts, web/components/board/__tests__/flow-settings-panel.test.ts, web/lib/__tests__/i18n-settings-keys.test.ts
**Severity:** high

## Problem

A Codex adversarial review of the M11c branch found three real defects in the
node-settings enforcement surface:

1. **(high) `instructed` settings are reported but never delivered to the agent.**
   The graph runner builds an `ai_coding`/`judge` agent step from
   `def.action.prompt` only; no `settings` field or `instruct` verdict reaches
   the prompt, executor input, capability profile, or adapter launch. The
   run-detail panel labels capability classes `Instructed`, which reads as "an
   instruction was delivered" — a false compliance signal.
2. **(medium) The enforcement snapshot is mutable despite being documented as an
   append-only audit.** `setEnforcementSnapshot` unconditionally updated
   `node_attempts.enforcement_snapshot` by attempt id. A `NeedsInput` resume
   reuses the same attempt row and re-runs the gate, so the verdicts could be
   recomputed and overwritten — losing the first-attempt audit (and able to
   rewrite history across a deploy that flips the static table).
3. **(medium) Sparse `enforcement` maps expand into undeclared classes.** Each
   key of `enforcementMapSchema` had `.default("instruct")`, so a manifest
   declaring only `enforcement: { mcps: "strict" }` parsed into all six classes.
   The evaluator treats key presence as "declared", so the snapshot/UI reported
   five classes the author never set. The existing unit test missed this because
   it passed a hand-built object literal, bypassing the Zod parse that injects
   the defaults — a false green for the production path.

## Root Cause

- **#1** is the documented M11c↔M14 carve (M14 owns *materialization* / delivery)
  — NOT a runtime bug — but the UI wording did not communicate that, so the
  honest scope ("declared, recorded for visibility, not yet delivered") leaked
  as an implied-delivery signal.
- **#2** — "set" semantics on a column whose contract is "write-once audit",
  combined with a runner that legitimately re-enters the same attempt on resume.
  No write guard encoded the contract.
- **#3** — defaults were modeled at the wrong layer: as per-key parse defaults
  instead of an evaluation-time fallback. Parse-time defaults make a sparse
  author intent indistinguishable from an explicit all-classes declaration. The
  unit test asserted the right behavior on an object that never went through the
  schema, so it could not catch the parse-time expansion.

## Solution

- **#1 (respecting the carve — no materialization, M14 owns that):** added an
  honest caption to the run-detail settings panel (EN + RU
  `run.settingsDeclaredIntentNote`): verdicts are the flow author's *declared
  intent*, recorded for visibility, *not yet enforced or delivered to the agent*
  in this build (materialized in M14). No settings are synthesized into the
  prompt (that is M14's scope).
- **#2:** made `setEnforcementSnapshot` write-once —
  `UPDATE … WHERE id = ? AND enforcement_snapshot IS NULL` (+ `.returning` to log
  whether the write landed). A resume re-eval is now a no-op; the first-attempt
  verdicts are preserved.
- **#3:** removed the per-key `.default("instruct")` from `enforcementMapSchema`
  (keys are now `.optional()`), so a parsed manifest keeps the map sparse. The
  `instruct` default already lives at evaluation
  (`evaluateNodeEnforcement`: `declared = enforcement?.[cls] ?? "instruct"`), so
  a class declared by a data field still resolves correctly.
- Tests: reframed the schema test to assert sparse parse; added end-to-end
  tests that parse through `flowYamlV1Schema` before `evaluateNodeEnforcement`
  (the regression a bare object literal cannot catch); added a write-once
  resume regression to the schema integration suite; added the panel caption
  label to the panel test + the i18n key-parity contract.

## Prevention

- A "visibility/observability" surface MUST state what it does NOT do. If a value
  is recorded-but-not-acted-on, the label/UI/docs must say so — an unqualified
  status word (`Instructed`, `Enforced`) implies action was taken.
- A column documented as append-only/write-once MUST encode that as a write
  guard (`WHERE col IS NULL`), not rely on callers writing it once; verify with a
  resume/re-entry regression test, since reuse paths are exactly where it breaks.
- Apply defaults at the layer that does NOT erase author intent. A sparse config
  map should stay sparse through parse; fold the default in at evaluation. A
  parse-time default destroys the "was this explicitly declared?" signal.
- A validation/derivation test that constructs its input by hand (object
  literal) does NOT exercise the parse layer. For schema-driven behavior, the
  test MUST parse real input through the production schema, or it is a false
  green for the deployed path.
