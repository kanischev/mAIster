# Scheduler review follow-up hardened display, validation, and run lists

**Date:** 2026-06-20 21:18
**Files:** web/lib/scheduler/job-targets.ts, web/components/admin/scheduler-jobs-table.tsx, web/lib/scheduler/job-admin-schema.ts, web/app/api/admin/scheduler-jobs/__tests__/route.test.ts, web/lib/queries/runs-list.ts, web/app/(app)/runs/page.tsx, web/lib/queries/scheduler.ts, web/components/admin/scheduler-run-schedules-overview.tsx, web/types/scheduler.ts, web/lib/scheduler/jobs.ts, docs/screens/README.md, docs/system-analytics/run-schedules.md
**Severity:** critical

## Problem

The admin scheduler table reused strict target summarization during render, so a
legacy malformed scheduler job target could crash `/admin/scheduler` before an
operator could inspect or repair it. Review also found smaller drift and
consistency issues: duplicated run rows from direct workspace joins, a hardcoded
runner filter, duplicated schedule overview DTOs, duplicate scheduler kind
constants, POST target normalization split between route schema and service,
and docs still marking shipped scheduler pieces as Designed.

## Root Cause

Write-boundary validation and read-path display used the same strict helper.
That is correct for API persistence but too brittle for legacy database rows.
The surrounding scheduler management work also added parallel local constants
and DTOs while the page was evolving, leaving the implementation easier to drift
than the SDD contract allowed.

## Solution

Added a tolerant display wrapper for scheduler target summaries while preserving
strict `summarizeSchedulerTarget` and `normalizeSchedulerTargetDraft` behavior
for tests and write paths. The jobs table now renders an "Invalid target"
fallback with the validation message in the title instead of throwing.

Moved create-route target semantics back to the scheduler admin service so
structured rejection logging is reachable on POST and PATCH. The edit modal now
omits `target` for no-target job updates. The run ledger now selects one
workspace branch with a lateral subquery, the runner filter maps over
`ADAPTER_IDS`, schedule overview row types share a single DTO shape, and the
scheduler job kind list reuses the catalog source.

Updated shipped screen and analytics docs from Designed to Implemented.

## Prevention

- Strict validators belong at write boundaries; render paths need tolerant
  summaries for persisted legacy data.
- Shared UI/server row contracts should live in one client-safe type module.
- Query tests should assert security and fan-out boundaries for generated SQL
  when full database integration is not part of the change.
- Status-doc grep belongs in the review-fix checklist for SDD-heavy branches.
