# Fix Patch: Platform-Agent Analytics Drift

## Problem

Platform-agent analytics still contained stale lifecycle/status wording after
the package-source rework:

- `agents.md` described a usage-guarded catalog delete and "admin agents CRUD",
  but the implemented admin surface is package-owned definitions plus
  list/read/toggle/un-quarantine/resync.
- `domain-events.md` still called the `agent_triggers` consumer future and
  marked the `task.triage_requeued` emitter as Designed.
- `db/agents-domain.md` described route-layer catalog deletes that no longer
  exist.

## Root Cause

The implementation moved from host-local/editable agent definitions to
package-sourced catalog projection, but some analytics phrases were carried
forward from the older Stage-3 draft.

## Solution

- Updated the agent catalog state machine to show missing package/file rows
  disabling on resync instead of deleting.
- Reworded linked HTTP artifacts to match the implemented admin API.
- Marked `agent_triggers` and the `task.triage_requeued` emitter as
  implemented in the domain-event analytics.
- Clarified that DB cascade behavior is now an FK backstop, not an admin
  delete lifecycle.

## Prevention

- When a feature reworks ownership from editable local resources to package
  projection, grep analytics and ERD docs for stale CRUD/delete lifecycle
  claims.
- Status tags in Purpose, Expectations, and Linked artifacts must agree with
  each other and with the shipped route methods.
