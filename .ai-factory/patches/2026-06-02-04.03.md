# M19 crash-recover: real runner resume mode for session-less nodes

**Date:** 2026-06-02 04:03
**Files:** web/lib/config.schema.ts, web/lib/flows/graph/compile.ts, web/lib/db/schema.ts, web/lib/db/migrations/0016_peaceful_shape.sql, web/lib/runs/state-transitions.ts, web/lib/flows/graph/current-node-kind.ts, web/lib/runs/recover-classify.ts, web/lib/queries/run.ts, web/lib/runs/recover.ts, web/lib/flows/graph/runner-core.ts, web/lib/flows/runner.ts, web/lib/flows/graph/runner-graph.ts, web/lib/runs/__tests__/recover-classify.test.ts, web/lib/queries/__tests__/run-recoverable.test.ts, web/lib/runs/__tests__/recover.integration.test.ts, web/lib/flows/__tests__/crash-resume.integration.test.ts (new), web/e2e/_seed/seed-e2e.ts, docs/* (recover matrix, ADR-034, reconciliation-gc, flow-dsl, configuration, runs ERD)
**Severity:** high

## Problem

Codex round-3 review found that the round-2 fix (exposing the Recover button for
session-less crashed nodes) drove a **broken** backend path. Verified against the
runners:
- **Graph flows** — `driveResume` flipped Crashed→Running and called
  `runFlow`→`runGraph`, but `runGraph` treats `Running` + existing node attempts
  + no recognized resume mode as "already owned" and returns without executing
  (`runner-graph.ts` no-op guard). Recover reported `{state:"redispatched"}`
  while doing **nothing**; the run then re-crashed.
- **Linear flows** — `Running` made `isResume=false` → restarted from step 0,
  **duplicating** earlier steps' side effects.
- The recover integration test asserted only that a **mocked** `runFlow` was
  called, so the no-op/duplicate was invisible.
- Deeper: a real reconcile crash (`crashRunningRun`) **nulls `current_step_id`**,
  so there was no retained node to resume to at all — recovery was broken for
  agent runs too (the e2e seed had hand-set `current_step_id`, masking it).

## Root Cause

Two layered causes:
1. **A capability was advertised without verifying the runner implemented it.**
   `classifyRecover` returned `redispatch` for session-less nodes and the UI
   exposed it, but neither runner had a crash-recover resume mode — the
   "redispatch" plan had no working executor behind it. The test mocked the exact
   seam (`runFlow`) that would have exposed the gap, so CI stayed green over a
   no-op.
2. **The crash transition discarded the data the recovery needed.**
   `crashRunningRun` nulled `current_step_id` for a "clean terminal" read, but
   recovery needs the crashed node id. Nothing retained it, so even a correct
   runner had no target.

## Solution

Built a real crash-recover resume mode (user-chosen over narrowing):
- **Retain the node id**: new nullable `runs.resume_target_step_id` (migration
  0016). `crashRunningRun` copies `current_step_id → resume_target_step_id` in
  one UPDATE (right-hand sides evaluate pre-update), still nulling
  `current_step_id`. Fixes recovery for agent AND session-less crashes.
- **Per-node `retry_safe` opt-in** (manifest graph nodes + linear steps, default
  false): a session-less crashed node is redispatch-recoverable ONLY when the
  Flow author marks it `retry_safe` (re-running repeats side effects —
  accepted-risk). `classifyRecover(run, nodeKind, retrySafe)`: agent+session →
  resume-agent; session-less+retry_safe → redispatch; else discard-only.
- **Runner crash-resume mode**: `driveResume` passes
  `runFlow(id, {crashResume:{targetStepId}})`. Both `runGraph` and `runFlow`
  recognize it as a 3rd resume mode (alongside NeedsInput- and takeover-resume),
  resuming FROM the target node (fresh attempt) instead of no-op'ing / restarting
  from 0. Single-winner claim = atomic CAS-clear of `resume_started_at`
  (`WHERE resume_started_at IS NOT NULL`); the loser bails.
- **Tests against the REAL runners** (Codex's demand): new
  `crash-resume.integration.test.ts` drives `runGraph`/`runFlow` for a crashed
  `retry_safe` check (graph) and cli (linear), proving the node re-runs (not a
  no-op), upstream is not re-run, the linear path resumes from the crashed step,
  and two concurrent dispatches yield exactly one execution.

## Prevention

- A recovery/branch decision that returns an "action" (redispatch, resume,
  retry) MUST have an integration test that drives the REAL executor for that
  action and asserts an OBSERVABLE effect (a new attempt ran, a row advanced) —
  never just "the dispatcher was called" against a mock. Mocking the executor
  seam hides whether the action does anything.
- Before exposing a UI affordance for a backend capability, trace the capability
  to a working executor end-to-end; "the classifier returns X" is not proof "X
  works".
- When a state transition nulls a field for cosmetic/terminal reasons, check
  whether any RECOVERY path needs that field; retain it in a dedicated column
  rather than overloading the live cursor.
- A "clean terminal" simplification (nulling `current_step_id`) can silently
  break a feature that reads it; hand-set fixtures that re-populate it mask the
  break — seed fixtures should mirror the REAL transition's output.
