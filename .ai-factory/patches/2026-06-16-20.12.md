# OpenAPI RunDiffResponse drifted when the scratch /diff branch gained the prepared shape (M35 review fix batch)

**Date:** 2026-06-16 20:12
**Files:** docs/api/web.openapi.yaml, docs/system-analytics/workbench.md, web/components/scratch/scratch-composer.tsx, web/components/scratch/scratch-permission-panel.tsx, web/components/scratch/scratch-inspector-actions.tsx, web/components/runs/run-inspector.tsx, web/lib/scratch-runs/__tests__/session-summary.test.ts, web/e2e/scratch-detail.spec.ts, web/messages/en.json, web/messages/ru.json
**Severity:** medium

## Problem

`/aif-review` of the M35 run-detail branch flagged one merge-blocker plus
non-blocking polish. The blocker: `GET /api/runs/{runId}/diff` was changed so
the **scratch** branch now returns the same prepared shape as the flow branch
(`files`, `perFile`, `scope`, `scopes`, `renderUnavailableReason`; T3.3), but the
shared `RunDiffResponse` schema in `docs/api/web.openapi.yaml` still documented
`files`/`perFile` as **"Present for flow runs"** (now false) and never documented
`scope`/`scopes`/`renderUnavailableReason` at all. An old route test even asserted
scratch had *no* `files` — it had been updated, the contract had not. The branch
plan's own Phase-1 exit ("`web.openapi.yaml` is updated in the same commit") was
unmet.

## Root Cause

Classic code-lands-docs-lag on a **shared response schema with per-variant
behavior**. When the scratch branch was brought to parity with the flow branch,
the author updated the code and the route tests but not the OpenAPI schema. Two
distinct sub-faults:

1. A **field-level "present for X" note** (`files`/`perFile` "Present for flow
   runs") became stale the moment a sibling variant (scratch) started emitting
   the field — no one swept the note when the second variant gained it.
2. `scope`/`scopes`/`renderUnavailableReason` were **pre-existing M30/ADR-082
   drift** (the flow branch already emitted them undocumented); the M35 branch
   propagated the same undocumented fields to the scratch path, widening an
   existing gap rather than creating it.

## Solution

- `RunDiffResponse`: added `scope` (enum) and `scopes` (per-scope availability
  map, reusing the existing `RunChangeSummaryScopeAvailability` schema, `required:
  [run]` since scratch reports only `run`), added nullable `renderUnavailableReason`,
  promoted `scope`/`scopes` into `required`, and rewrote the `files`/`perFile`
  notes to "flow and scratch runs (M35, T3.3)". Validated clean with
  `npx @redocly/cli lint` (only pre-existing warnings remain).
- Flipped `system-analytics/workbench.md`'s "M35 target hierarchy (Planned)"
  callout to Implemented with the scratch Files/Diff + copy-control + Markdown
  facts.
- Batched the non-blocking review findings (user opted into all): deduped
  attachment normalization to a single site (the conversation), added missing
  `aria-label`s on the scratch composer/permission form controls (+5 EN/RU i18n
  keys), swapped the run-inspector danger palette from raw Tailwind `red-*` to the
  codebase `#d9534f` convention, added scratch-promote success feedback +
  `router.refresh()` (a web-tier merge emits no SSE), added a `getDb`-mocked
  `getScratchSessionSummary` unit test (the `?? 0` null-profile fallback), and
  extended the scratch e2e to open a file and assert the copy control.

## Prevention

- When a **route branch's response shape changes**, re-derive the OpenAPI schema
  for **every** response variant the route can emit (the RULES.md "re-derive every
  analytics/spec surface" rule applies to the wire, not just state machines).
- **Sweep field-level "present for `<variant>`" notes** whenever a sibling variant
  gains that field — the same instinct as the "sweep sibling Zod fields" rule, but
  for documentation. A shared schema with `(Present for flow runs)` prose is a
  drift trap the moment a second variant is added.
- A route test that asserts a field is **absent** (`expect(body.files).toBeUndefined()`)
  is a tripwire: when it flips to asserting presence, treat the matching OpenAPI
  schema as a required edit in the same commit.
- Run `npx @redocly/cli lint <spec>` after any hand-edit to the OpenAPI; it
  catches structural errors but **not** semantic drift — that needs the review-exit
  re-derivation pass.
