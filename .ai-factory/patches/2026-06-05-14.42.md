# M17 review remediation — migration-number collision, linear-runner reconcile gap, sanitized-fixture leak

**Date:** 2026-06-05 14:42
**Files:** web/lib/db/migrations/{0024,0025}_*.sql + meta/{_journal,0024_snapshot,0025_snapshot}.json, web/lib/reconcile.ts, web/lib/flows/graph/current-node-kind.ts, web/lib/runs/state-transitions.ts, web/lib/flows/runner.ts, web/lib/services/hitl.ts, web/app/api/v1/ext/runs/[runId]/hitl/route.ts (+ tests), docs/api/external/operations.openapi.yaml, docs/system-analytics/hitl.md, docs/decisions.md
**Severity:** critical

## Problem

A `/aif-review` of the M17 branch surfaced four merge-blocking/important defects, three of which are recurring classes worth recording:

1. **Migration-number collision.** The branch forked from `main` at `603dcae9`, before `main` merged `0022_platform_acp_runners` / `0023_drop_legacy_executors`. M17 then numbered its own migrations `0022`/`0023`. On merge, `_journal.json` hard-conflicts at idx 22/23 and two of the four same-named snapshot files silently win — a naïve merge drops migrations, so `criticality`/`human_confidence` columns and the `actor_identities` partial-unique index never apply in prod (`ensureApiTokenActor`'s `onConflict` then throws at runtime).
2. **Drizzle snapshot-chain corruption.** `0022_snapshot.id` was byte-identical to `0021`'s and `0023.prevId` was an orphan — the snapshots had been hand-edited, not `drizzle-kit generate`d, so the next generate would diff against the wrong baseline.
3. **Reconcile mis-handled a new "Running mid-flow" linear state.** M17's flat-runner `on_reject` repark created a *new* state (a `Running` linear run whose `currentStepId` points mid-flow at a goto target). The unchanged `reconcile.ts` classified a session-less gate/human node as `redispatch → bare runFlow`, which for a LINEAR run restarts from step 0 and re-runs prior agent/cli side-effects. The in-code comment and ADR-052 *claimed* "reconcile → crashResume," which the code did not deliver.
4. **Response leak hidden by a sanitized test fixture.** The ext `GET …/hitl` DTO passed `row.schema` verbatim; for `permission` rows that schema carries `supervisorSessionId`/`requestId`/`toolCall`. The integration fixture seeded a *sanitized* permission schema, so the leak shipped green against its own test.

## Root Cause

1. Migrations were numbered sequentially from the **fork point**, not from `main`'s tip — concurrent migrations on `main` were invisible to the branch's numbering.
2. Snapshot metadata was edited by hand; the `id`/`prevId` linked-list invariant (`snapshot[n].prevId == snapshot[n-1].id`, every `id` unique) was not maintained because the tool that enforces it was bypassed.
3. A new run state was introduced without auditing **every** reconcile/recover classifier branch for the linear-vs-graph distinction. The gate-redispatch branch implicitly assumed graph semantics (graph runners no-op to their current node; linear runners restart at 0).
4. The DTO projected the row shallowly (`schema: row.schema ?? null`) instead of per-`kind`, and the test used non-production fixture data, so the assertion couldn't catch the real payload.

## Solution

1. Renumbered M17 migrations to `0024`/`0025` (after `main`'s tip) + fixed the journal idx/tags + rebuilt the snapshot `id`/`prevId` chain to descend from `0021`. (User chose in-place renumber over a rebase; final re-chain happens at their rebase.)
2. Added a `linear-gate-orphan` reconcile reason + an `isLinearFlow` classifier input resolved via a new `resolveCurrentNodeContext` (reads `manifest.steps[]` vs `nodes[]` in one load). A session-less linear gate/human orphan now classifies `crash` → `crashRunningRun` retains the goto target in `resume_target_step_id` → operator Recover (`resumeCrashedRun` → `driveResume`) threads `crashResume` and resumes from the goto target. Graph gate-redispatch is provably unchanged (flag is optional/defaults false). Corrected the false comment + ADR-052 window-(c) wording. Added a pure-classifier test + an end-to-end `runReconcileSweep` integration test.
3. Projected `schema → null` for `kind === "permission"` in the ext DTO, and reseeded the test with the production schema shape + an assertion that `supervisorSessionId` never appears in the body.
4. Also: extracted-service two-phase fixes (idempotent scratch-status re-assert on the already-delivered branch; a real two-racer `Promise.allSettled` CAS test asserting one 200 + one typed CONFLICT); a run-wide `on_reject` repark cap (not just per-step); unconditional `rework-comments-<gotoStepId>.json` (over)write to prevent stale re-injection; ext-respond OpenAPI `410`/`503`; ADR index rows; criticality on run-detail; client-side confidence clamp.

## Prevention

- **When a feature branch adds DB migrations, run `git log HEAD..<base>` for `migrations/` before merge.** If the base advanced past the fork with its own migrations, renumber/rebase so numbers and `_journal.json` idx don't collide. Never assume the fork-point's "next number" is free at merge time.
- **Never hand-edit Drizzle snapshots.** Regenerate via `drizzle-kit generate` so the `id`/`prevId` chain stays consistent; a duplicated `id` or orphan `prevId` is a red flag a reviewer can grep for (`grep -H '"id"\|"prevId"' meta/*_snapshot.json`).
- **Introducing a new run/lifecycle state ⇒ fan it out to EVERY reconcile/recover/scheduler/read-model branch (skill-context "fan-out" rule), and explicitly distinguish linear (`steps[]`) from graph (`nodes[]`) wherever a branch assumes one execution model.** A `redispatch` that means "resume in place" for graph means "restart from 0" for linear.
- **A claim in a code comment / ADR about a recovery path is not evidence it exists.** Verify the claimed machinery end-to-end (drive the real sweep + recover), not via a hand-injected shortcut in the test.
- **Response DTOs must project per-discriminant (per `kind`), and security/leak tests must seed PRODUCTION-shaped fixtures** — a sanitized fixture lets a leak pass its own assertion.
