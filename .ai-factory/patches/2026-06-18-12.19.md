# Runner-rail follow-ups: locked-decision drift accepted + popover-clip verified by layout math

**Date:** 2026-06-18 12.19
**Files:** .ai-factory/plans/runner-readiness-rail-popover-settings.md (annotation only — no code change)
**Severity:** low

## Problem

`/aif-verify` surfaced two non-blocking notes on the runner-readiness-rail work:
1. **Locked-decision drift.** Decision #2 (locked with the user) said each popover
   row shows `capabilityAgent · model · providerKind` as text. The shipped row
   renders `model · providerKind` only and moved adapter identity to the popover
   header — a silent deviation from a user-locked decision. Component + test +
   docs all agreed with the shipped form, so only the plan text was stale.
2. **Popover clip risk.** The plan's resolved-Q #1 said "lock overflow-visible +
   high z-index"; the shipped popover instead anchors to a `relative` chips
   container, leaving the question of whether it clips against the expanded
   rail's `overflow-x-hidden`.

## Root Cause

1. The implementer reorganized the popover (adapter in a new header, dropping the
   per-row `capabilityAgent`) without flagging that this contradicts a locked
   decision. The deviation is reasonable (`capability_agent == adapter` for all
   five ready families, so per-row it duplicates the header; `model` is the
   distinguishing identity), but a locked decision changing silently is the
   defect — it leaves the plan steering future implementers to "re-add
   capabilityAgent."
2. No clipping bug actually exists. Layout math: the rail (`RailCollapse` aside)
   is `w-[260px]` with `overflow-x-hidden` (content box ≈ 232px); the popover is a
   fixed `w-[220px]` panel anchored `left-0` to the chips container and opens
   upward (`bottom-full`), and it lives *outside* the scrolling `<section>`. 220 <
   232, so it fits horizontally and is not clipped; it opens into the full-height
   rail above the launch footer.

## Solution

Owner chose to keep the shipped form. Added one prominent **"Accepted deviation"**
note inside the locked Decision #2 that (a) records capabilityAgent is dropped
from the row (shown once in the header; still a `RailRunnerDTO` field server-side)
and (b) reconciles the clipping resolution (z-[140] + relative-container anchor,
fits the 260px rail). The note explicitly names every stale spot it supersedes
(Decision #2 bullet, Task 2, Task 4 case (1), Resolved-Q #2) so a single edit
covers the sibling sweep without mangling historical task text. No code changed.

## Prevention

- An implementer deviating from a **locked** plan decision must call it out and
  get sign-off, or conform — never ship the deviation silently with the plan left
  contradicting the code. `/aif-verify` should treat any locked-decision wording
  that the diff contradicts as a finding, not a silent pass.
- When accepting a deviation, rewrite the stale plan/spec bullet in the SAME pass
  (verify skill-context: documented bullets are the most dangerous drift). One
  superseding note that names all stale siblings satisfies the sweep.
- A CSS overflow/clip concern can often be settled by layout math (container
  width vs fixed popover width + positioning context) without spinning up the dev
  server — cheaper and deterministic. Reserve a live pixel check for genuinely
  ambiguous cases.
