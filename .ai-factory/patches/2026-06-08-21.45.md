# Atomic token success audits for mutating external routes

**Date:** 2026-06-08 21:45
**Files:** web/lib/tokens/ext-handler.ts, web/app/api/v1/ext/projects/[slug]/tasks/route.ts, web/app/api/v1/ext/projects/[slug]/tasks/[taskId]/route.ts, web/app/api/v1/ext/runs/route.ts, web/app/api/v1/ext/runs/[runId]/hitl/[hitlRequestId]/respond/route.ts, web/lib/services/runs.ts, web/lib/services/hitl.ts
**Severity:** high

## Problem

Mutating external API routes could persist successful business side effects before
the success token audit row was written. If `recordTokenAudit()` failed after
the route work completed, the HTTP request failed closed but task/run/HITL state
could already be committed without the required `token_audit_log` row.

## Root Cause

`handleExt()` wrote the default success audit after `work()` returned. That is
safe for read-only routes, but not for mutating routes whose work commits DB
state internally. The gate-report route already had the correct pattern
(`successAuditInWork: true` plus an in-transaction success audit), but sibling
mutating routes had not adopted it.

## Solution

Move success audit writes for mutating external routes into the same work path
that commits their success state:

- task create/update wrap the service write and success audit in one route-level
  transaction;
- run launch records the audit inside the existing launch transaction, so audit
  failure also triggers the existing worktree compensation path;
- HITL respond records the success audit in the delivered-marker transaction or
  before returning other success statuses, preserving the existing retryable
  intent-claim flow;
- `recordRequiredTokenAudit()` is reused by in-work audit writers so audit
  failures remain structured and fail closed.

## Prevention

During review of any new mutating `/api/v1/ext/*` route:

- require `successAuditInWork: true` when the route returns 2xx after durable
  state changes;
- write the success audit in the same transaction as the route's success marker
  whenever the marker is in the DB;
- add a regression that forces `recordTokenAudit()` to throw and asserts no
  successful business state is committed without an audit row;
- for multi-store flows, document the retryable crash window and ensure the
  next retry can finish the audited success marker.
