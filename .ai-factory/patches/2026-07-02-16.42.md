## Problem

`/aif-review` (same day, this branch) found that the response contract of
`GET /api/v1/ext/hitl` was fictional at HEAD — on an operation the branch's
own diff had just touched (added `count`, rewrote the example):

1. The route returned the web-internal `CrossProjectHitlItem[]` VERBATIM —
   ~15 undocumented fields crossed the external boundary (`assigneeUserId`
   internal user UUID, `assignmentStaleEvidenceSummary` jsonb, assignment
   internals, `agent`, `stage`, `branch`, `projectName`, non-permission
   `schema`).
2. The spec's required `title`/`createdAt` did not exist in the payload at
   all (actual analogues: `prompt`/`taskTitle` + a RELATIVE `time` string
   like "5m") — every real response violated the documented required set.
3. The spec's `kind` enum documented 3 of the 6 real values (the query has
   no kind filter; `permission`/`form`/`hook_trip` flow through).
4. Secondary (same review): the new `tool-contract.test.ts` guard did not
   lock numeric/string bounds, `dispatchTool`'s triage casts lied about
   null, and a missing spec path crashed test collection with a raw
   TypeError instead of a clean assertion.

## Root Cause

The route pre-dated the "explicit response-DTO projection at the boundary"
rule and shipped `inbox.items` as-is; the spec's `ExtHitlInboxItem` was
authored as the INTENDED projection that never got implemented — and the
route's integration tests used only `objectContaining`/`toMatchObject`
(partial matchers), so no test could ever see the gap. The same-day input
sweep (patch 2026-07-02-14.14) explicitly scoped response DTOs out, but the
review rule operates at the touched-SURFACE level: editing an operation's
response schema puts that whole response in scope. The permission-`schema`
supervisor-handle leak was ALREADY nulled by the earlier fix — this was
contract-lie + over-exposure, not a secrets leak.

## Solution

Owner decisions (asked, not assumed): fix now; kind-policy = code-wins
(enum extended to all 6 kinds — nothing hidden from ext clients; the
alternative, filtering to the documented 3, would have been a behavioral
narrowing).

- `web/lib/queries/hitl.ts`: `HitlItem` gains `createdAt` (ISO 8601,
  absolute counterpart of the relative `time`; the row already carried it —
  the mapper just dropped it). Additive for all web consumers.
- `web/app/api/v1/ext/hitl/route.ts`: explicit projection to exactly the 7
  documented fields (`title` ← `prompt`); `count` invariant unchanged.
- Spec: `ExtHitlInboxItem.kind` enum → all 6 values + descriptions state
  the exact-projection guarantee. Redocly: valid, 0 warnings.
- Exact-shape integration test (`toEqual` — extra key on either side
  fails) + ISO round-trip + `count` assertion, seeded via the existing
  fixtures; green on real PG (4/4 route, 7/7 portfolio-inbox).
- `tool-contract.test.ts`: `MIRRORED_BOUNDS = [minimum, maximum,
  minLength]` asserted wherever the spec declares them (generic loop + the
  run_plan nested items); collection-time `?.` guard + per-tool existence
  test replaces the TypeError crash path. RED run enumerated 10 drifted
  tools — ALL `minLength` (no minimum/maximum drift existed) — mirrored
  into `TOOL_SPECS` in one pass (+ `hitl_respond.confidence` 0..1). The
  run_delegate `target` inner shape stays name-only by its documented depth
  limit. mcp suite 168/168 (was 142), tsc 0, scoped eslint 0.
- `dispatchTool` triage casts made null-honest (`string | null` /
  `number | null`) — no runtime change (`!== undefined` already forwarded
  null).
- Tact0 plan: pinned `since/status/kind` query semantics (FR-B2 + T0.5
  pointer), TB.6 test must STUB `Intl` TZ (UTC CI flake), TC.4 must add the
  `TOOL_OP` entry for the contract guard.

No `[FIX]` runtime logging: repo is no-console/pino-only, and the changes
are a deterministic boundary projection + declarative schema/test
artifacts, all locked by tests (same rationale as patch 2026-07-02-14.14).

## Prevention

- A spec response schema is only trustworthy when SOME test asserts the
  exact shape (`toEqual`, both-direction key equality). Partial matchers
  (`objectContaining`/`toMatchObject`) on a route response are a review
  smell — they verify presence, never absence, and absence is where
  contract lies live.
- When a diff touches ANY part of an operation (even one response field or
  its example), reconcile the operation's WHOLE request+response surface
  against code at HEAD — "I only touched `count`" does not scope the review
  to `count`.
- A route returning a shared web-internal read-model must project at the
  boundary; the internal type gaining a field must never implicitly widen
  an external contract.
- The tool↔spec guard now locks props/required/types/enums AND
  minimum/maximum/minLength both ways. Remaining unlocked (recorded):
  `format`, `pattern`, nested `run_delegate.target` inner types/bounds, and
  the full response-DTO ↔ spec audit for the other 24 operations.
