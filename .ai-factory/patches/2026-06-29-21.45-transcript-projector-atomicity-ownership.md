# Flow transcript projector: partial-write stranding + cross-run attribution

**Date:** 2026-06-29 21:45
**Files:** web/lib/runs/run-transcript-projector.ts, web/lib/runs/__tests__/run-transcript-projector.integration.test.ts
**Severity:** high

## Problem

Two adversarial-review findings in the reconcile-on-read flow transcript
projector (T-B2):

1. The per-attempt `run_messages` upserts ran outside a transaction while the
   resume cursor was `max(supervisor_event_id)` across the run's rows. A
   mid-batch failure (DB error / request timeout) after a high-`monotonicId`
   row committed but before the rest were written advanced the cursor; the next
   call short-circuited on `logMax <= dbMax` and permanently skipped the
   unwritten attempts even though `run.events.jsonl` still had the data.
2. `projectRunTranscript` inserted rows using the log-stamped `nodeAttemptId`
   without verifying it belonged to the run, and `getRunNodeTranscript` read by
   `nodeAttemptId` alone. The schema has independent FKs for `run_id` and
   `node_attempt_id`, so a mis-attributed line could create a row whose
   `run_id`/`node_attempt_id` point at different runs, and the run-scoped
   transcript route (authorized on the requested run) could surface it —
   crossing run/project boundaries.

## Root Cause

1. The cursor optimization assumed "rows present up to `dbMax`" implies "a
   complete projection up to `dbMax`". That only holds if writes are atomic.
   Non-transactional row-at-a-time writes broke the invariant: the cursor is
   derived from the same rows it is meant to gate, so a partial write
   self-certifies as complete.
2. The projector trusted an external, supervisor-supplied identifier
   (`nodeAttemptId`) as an ownership token without validating it against the
   run, and the read path relied on `node_attempt_id` uniqueness for tenancy
   that the schema does not actually enforce across runs.

## Solution

1. Wrapped the entire projection (all attempts/messages) in
   `client.transaction(...)` — all-or-nothing. The cursor can now only advance
   after a COMPLETE commit; a failure rolls back and the next call re-derives
   the full transcript idempotently (`ON CONFLICT DO UPDATE`).
2. Added an ownership guard: fetch `node_attempts.id WHERE run_id = runId` into
   a set and skip (WARN) any log line whose `nodeAttemptId` is not owned by the
   run. The read query (`getRunNodeTranscript`) now filters by BOTH
   `run_messages.run_id` AND `run_messages.node_attempt_id` (defense-in-depth).

Regression tests (real PG): a cross-run mis-stamped line creates no foreign row
and cannot be read under the wrong run; an injected mid-transaction failure
rolls back (0 rows) and a clean re-projection repairs the full transcript.

## Prevention

- A reconcile-on-read cursor MUST be derived from a durably-committed boundary,
  not from the same rows it gates — make the write atomic, or advance an
  explicit cursor only after the batch commits.
- Never treat an externally-supplied id as a tenancy/ownership token; validate
  it against the owning aggregate before write, and filter reads by the
  owner key in addition to the child key when the schema lacks a composite
  ownership constraint.
- For projector/read-model pairs, add tests for partial-failure repair and
  cross-tenant attribution, not just the happy path.
