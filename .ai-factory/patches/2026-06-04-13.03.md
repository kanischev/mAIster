# M18 review fixes: glab host-probe drift + PR-adapter branch-ref hygiene

**Date:** 2026-06-04 13:03
**Files:** web/lib/instance-config.ts, web/lib/worktree.ts, web/lib/runs/pr-adapter.ts, web/lib/__tests__/instance-config.test.ts, web/lib/runs/__tests__/pr-adapter.test.ts
**Severity:** medium

## Problem

Two findings from the M18 `/aif-review`:

1. **Doc drift (C2).** `docs/system-analytics/instance-config.md` asserts — tagged
   "(Implemented, M18)" — that `hostToolStatus()` probes `glab` and returns
   `[{git…},{gh…},{glab…}]`, rendered in `/settings`. The code returned only
   `[git, gh]` (`instance-config.ts`). An "Implemented" doc that the code does
   not implement.
2. **Input hygiene (MEDIUM).** The PR adapters (`gh`/`glab` CLI + Gitea/GitVerse
   REST) interpolate `sourceBranch`/`targetBranch` into an argv / REST body, but
   never validated the ref format. On the `allowTargetDrift:true` promote path
   the local-merge regex (`promoteLocalMerge` → `branchNameSchema`) is bypassed,
   so a request-derived `targetBranch` reached `gh pr create --base` / the Gitea
   `base` field with no format guard. Not shell-injection (`execFile` array, no
   shell), but a missing boundary check on a body-derived value reaching an
   external CLI/REST call.

## Root Cause

1. The doc was written to the *intended* M18 behavior (all CLI PR prerequisites
   surfaced in `/settings`) but the `glab` probe line was never added to
   `hostToolStatus()` — classic "doc describes the design, code shipped a subset"
   drift. Per `docs/CLAUDE.md` "code wins", but here the design (probe all three
   CLI prereqs, graceful-absence) was the correct target, so the fix is to make
   the code match the doc, not delete the doc claim. Product steer: support for
   the console tools is connected "as needed" (not must-have), but the mechanics
   must be consistent across all three PR providers in both docs and code.
2. Validation was correctly centralized for the *local-merge* git path
   (`worktree.ts` schemas) but the *PR* path is a separate sink that grew its own
   adapter module; the branch refs crossing that second boundary were never run
   through the same `branchNameSchema`. The drift gate (`allowTargetDrift`)
   created a path where even the local-merge regex no longer covered the value
   before it hit the external call.

## Solution

1. `hostToolStatus()` now returns `[git, gh, glab]` (graceful-absence preserved —
   `probeTool` degrades a missing binary to `{ available:false }`, never throws).
   The `/settings` page already `.map()`s the array, so the new row renders with
   no UI change. Gitea/GitVerse stay token-based (env, documented in prose) — a
   deliberately different mechanism from the CLI probe. Code now matches the
   already-correct doc; no doc edit required.
2. Exported `branchNameSchema` from `worktree.ts` (single source of truth — it
   allows `/`, unlike pr-adapter's `SAFE_SEGMENT`) and added a shared
   `assertSafeBranchRefs(args)` guard in `pr-adapter.ts`, called as the first
   statement of all three `createOrUpdatePr` methods (gh, glab, Gitea). BOTH
   `sourceBranch` and `targetBranch` are validated (sibling-field sweep rule —
   sourceBranch is server-generated but swept for consistency; targetBranch is
   the request-derived one). Unsafe refs throw `PRECONDITION` before any provider
   call.

Tests: `instance-config.test.ts` asserts the exact `[git, gh, glab]` set;
`pr-adapter.test.ts` adds a parametrized block proving each of the three adapters
rejects an unsafe `targetBranch` (and `sourceBranch`) with `PRECONDITION` and
spawns NO CLI / makes NO fetch. Full suite: 163 promote-related tests green,
typecheck clean.

No `[FIX]` console logging was added: it would violate the project's `no-console`
lint rule (web/CLAUDE.md "use a logger boundary"), and a validation guard that
throws a typed `PRECONDITION` (surfaced as HTTP 409) is already observable — a
log line per rejected branch would be noise. Project CLAUDE.md overrides the
skill's generic logging default.

## Prevention

- When a value already validated for one sink (git CLI) grows a SECOND sink
  (external PR CLI/REST adapter), apply the SAME schema at the new boundary —
  don't assume the upstream regex still covers it, especially when a bypass flag
  (`allowTargetDrift`) routes around the original check.
- When a doc tags a behavior "Implemented, M<n>", a review must reconcile it
  against the actual function body, not the doc's own status tag (the M18 review
  rule: "re-derive every analytics/spec surface from the diff; never trust the
  doc's status tag").
- Multi-provider adapters: validate shared inputs ONCE in a helper called by
  every provider implementation, so the rule stays consistent across all of them
  rather than living (or being forgotten) per-adapter.
