# CCR admin start passes a literal `~`, diagnostics reads the wrong instance, PATCH skips configPath guard

**Date:** 2026-06-18 22:18
**Files:** supervisor/src/ccr-manager.ts, supervisor/src/http-api.ts, supervisor/src/__tests__/ccr-manager.test.ts, supervisor/src/__tests__/sidecars-route.test.ts, supervisor/src/__tests__/main-wiring.test.ts, web/lib/acp-runners/sidecar-schema.ts, web/app/api/admin/router-sidecars/route.ts, web/app/api/admin/router-sidecars/[sidecarId]/route.ts, web/app/api/admin/router-sidecars/__tests__/route.test.ts
**Severity:** high

## Problem

Three defects in the ADR-094 CCR admin start/stop feature, all surfaced by an
adversarial code-reading review (the test gate was green). Two would make the
feature dead-on-arrival in production despite passing tests:

1. **Literal `~` configPath (high).** The seeded `ccr-default` sidecar stores
   `configPath: "~/.claude-code-router/config.json"`. The web start route
   forwards it verbatim and the supervisor `CcrManager.preflight()` calls
   `fs.access`/`readFile` on the string directly — no tilde expansion. In a
   real install the Start button looks for a literal `~` directory under the
   supervisor cwd and returns `EXECUTOR_UNAVAILABLE` even when the config
   exists in `$HOME`. Tests missed it: the supervisor route test used `/c.json`
   and the web/e2e paths stub the supervisor.
2. **Diagnostics reads the wrong CCR instance (high).** `GET /diagnostics`
   reported a hard-coded `ccr-default` using `ccr.getState()` (no id). For the
   keyed manager that reads the anonymous *default* manager — but admin Start
   (`/sidecars/:id/start`) and session-spawn-with-sidecar both populate the
   *keyed* `ccr-default` instance. So a successful Start left diagnostics at
   `idle`; reconcile persists that, and launch resolution rejects runners whose
   stored readiness is not Ready → a started sidecar reconciles back to
   NotReady. Arbitrary admin-created sidecar ids were never represented at all.
3. **PATCH bypasses the configPath traversal guard (medium).** The prior sweep
   added the `..`-rejection regex to POST create and the supervisor start body,
   but the `PATCH /api/admin/router-sidecars/[sidecarId]` update body left
   `configPath` an open `z.string().min(1)`. An admin could PATCH a sidecar to
   `../../etc/passwd`; the value persists and is later forwarded to the fs sink.

## Root Cause

1. `~` is shell/OS sugar that `fs.*` never expands. The seed used the
   human-portable home-relative form but no layer expanded it. It cannot be
   expanded web-side: per CLAUDE.md the supervisor may run on a different
   host/user than the web tier, so the web tier's `os.homedir()` is the wrong
   home. The expansion belongs at the consuming fs boundary (the supervisor).
2. `createKeyedCcrManager.getState(undefined)` intentionally returns the
   default manager's state; the diagnostics route called the no-id overload and
   hard-coded a single `ccr-default` row, so it could neither see keyed state
   nor enumerate other instances. The supervisor had no surface to enumerate
   its keyed instances.
3. Classic missed-sibling: a boundary-validation fix landed on the create and
   supervisor boundaries but not the update boundary that writes the same field
   into the same fs sink.

## Solution

- **Finding 1 — `ccr-manager.ts`:** added `expandHome(p)` (handles `~` and
  `~/…` via `os.homedir()` + `path.join`) and applied it once at config-path
  resolution: `const configPath = expandHome(opts.configPath ?? defaultConfigPath())`.
  Idempotent on absolute paths; also covers a `~`-containing
  `MAISTER_CCR_CONFIG_PATH`. The web tier keeps forwarding the stored value
  verbatim (correct — expansion is host-local to the supervisor).
- **Finding 2 — `ccr-manager.ts` + `http-api.ts`:** added
  `listStates(): ReadonlyArray<{ id; state }>` to the `CcrManager` interface.
  The keyed facade enumerates its `managers` map (id → `getState()`); the
  anonymous single/external managers return `[]` (they own no catalog id). The
  diagnostics route now seeds `ccr-default` as `idle` (so readiness sees a state
  rather than "diagnostics missing") and overlays every started instance from
  `listStates()` — fixing the seeded sidecar AND representing admin-created ids.
  Item shape `{id, kind:"ccr", state}` is unchanged, so the supervisor OpenAPI
  `sidecars` array contract (already unbounded) still holds.
- **Finding 3 — new `web/lib/acp-runners/sidecar-schema.ts`:** extracted one
  shared `sidecarConfigPathSchema` (min-1 + `..`-rejection regex) and imported
  it in BOTH the POST create route and the PATCH update route, replacing each
  inline definition. One definition → the two boundaries cannot drift.

## Sibling sweep (recorded per project rule)

`configPath` → fs-sink class, re-audited end to end this pass:
- supervisor `SidecarStartBodySchema.configPath` — already `..`-rejecting (prior
  patch); now also tilde-expanded at the `CcrManager` consume site. **covered.**
- web POST create (`route.ts`) — switched to shared `sidecarConfigPathSchema`. **fixed (no drift).**
- web PATCH update (`[sidecarId]/route.ts`) — switched to shared schema. **fixed (was the gap).**
- web start route — forwards the stored value, no body schema; guarded at
  create/PATCH (producing) + supervisor (consuming). No change.
- web `SidecarInstanceConfig` (client type) — compile-time only. No change.

`getState`/diagnostics-visibility class: the only consumer of CCR instance
state for readiness is `evaluateRunnerReadiness` / `evaluateSidecarReadiness`
(match by id) fed from `GET /diagnostics`. That single producer is fixed; no
other diagnostics emitter exists.

## Logging note (deviation from the skill's generic `[FIX]` rule)

No `[FIX]` `console.log` was added. Project rules (skill-context precedence +
`web/CLAUDE.md` `no-console`) mandate the pino logger boundary, and the
supervisor already logs every CCR state transition structurally. The debugging
surface the rule exists to provide is already present: the `EXECUTOR_UNAVAILABLE`
preflight error now embeds the *expanded* absolute path, and `/diagnostics`
exposes real keyed states. Adding console logging would break lint and violate
surgical-changes; the logger boundary already covers it.

## Prevention

- A "config path" or any home-relative value that reaches `fs.*` MUST be
  tilde-expanded at the consuming boundary, never the storing one, when the
  producer and consumer can live on different hosts/users.
- When a keyed/multi-instance manager exposes a no-arg `getState()` that aliases
  a default instance, any *diagnostics/observability* reader of "the instances"
  must enumerate the keyed registry — a no-arg call silently reports the wrong
  (default) instance. Give the manager an explicit `listStates()` surface.
- Boundary-validation fixes sweep BOTH the producing (create) AND mutating
  (update/PATCH) boundaries that write the same field to the same sink, in the
  same pass — and share ONE schema so they cannot drift (don't mirror a regex
  with a `// matches X` comment).
- A green suite over mocked/stubbed dependencies certifies the handler, not the
  end-to-end contract. For an injected dependency with host-local behavior
  (`fs`, `~`, process state), add at least one test against the real component.
