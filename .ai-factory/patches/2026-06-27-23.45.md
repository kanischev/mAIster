# Project-less Studio-assistant scratch runs leaked concurrency slots forever

**Date:** 2026-06-27 23:45
**Files:** web/lib/scheduler.ts, web/lib/runs/recover.ts, web/lib/reconcile.ts,
web/lib/scratch-runs/service.ts, web/app/api/scratch-runs/[runId]/discard/route.ts,
web/components/studio/studio-ai-tab.tsx, web/lib/studio/flow-assistant/context.ts,
.env.example, docs/system-analytics/scratch-runs.md (+ 3 test files)
**Severity:** high

## Problem

The live DB held 6 scratch runs stuck at `runs.status='Running'` /
`scratch_runs.dialog_status='WaitingForUser'`, `project_id = NULL`,
`local_package_id` set (Studio local-package AI assistant runs), with dead
supervisor sessions since the last restart. They permanently filled the global
flow/scratch concurrency pool (cap=6) → no delivery run could start. Dropping
them from the Studio AI panel did nothing.

## Root Cause

Three interlocking defects, all rooted in M36 making `runs.project_id` nullable
for the project-less Studio assistant WITHOUT teaching the slot/lifecycle
machinery about the new null case (every consumer that branches on a project
silently excluded — or mis-handled — the project-less variant):

1. **Reconcile structurally could not see them.** `loadCandidates` iterates
   *registered projects* and filters `runs.project_id = project.id`. A
   `project_id IS NULL` run is never a candidate, so reconcile never crashed its
   dead session. No other sweep covers it (keep-alive/resume act only on
   `NeedsInput*`). A `Running` run with a dead session and no worktree is stuck
   forever.
2. **The cap counted them against the shared delivery pool.**
   `launchLocalPackageAssistant` called `assertScratchCapacity*`, and
   `countLiveRuns` / the scratch cap-check / the recover cap-check counted
   `runKind IN (flow,scratch)` with no local-package filter — so a dev-time
   authoring assistant competed with real delivery + user scratch runs.
3. **No working manual drop.** The generic discard route gated on
   `requireProjectAction(run.projectId, …)`; with `projectId = NULL` that throws,
   so the route could never terminate an assistant run. The Studio AI panel had
   no drop affordance at all.

## Solution

- **Separate slot pool (Fix 3):** new `MAISTER_MAX_CONCURRENT_ASSISTANTS`
  (default 5) + `assertAssistantCapacity*` counting `local_package_id IS NOT
  NULL`. Excluded assistant runs from **all three** flow-pool sibling counters
  (`countLiveRuns`, `assertScratchCapacityAvailableInTransaction`,
  `resumeCrashedRun`) with `isNull(runs.localPackageId)`. Pointed the assistant
  launch at the new cap. This alone stops delivery starvation immediately.
- **Reconcile auto-crash (Fix 1):** added a second `loadCandidates` query for
  `project_id IS NULL` runs (repo_path/worktree null). Keyed the load-bearing
  `worktreeExists` and grace-anchor branches on `cand.projectId == null` (NOT a
  broad `runKind === 'scratch'`), so behavior is identical for every
  project-bound candidate and only the new project-less rows change: a null
  worktree reads as present (not `worktree-gone`, which precedes the live-session
  skip and would crash a LIVE chat); a live session → skip; a dead session past
  grace → `markScratchCrashed`.
- **Manual drop (Fix 2):** instead of extracting a new service primitive + a new
  studio route (the plan's approach), added the dual-authz branch to the EXISTING
  discard route — `if (run.projectId) requireProjectAction(...) else
  assertLocalPackageAssistantActor(run, userId, {requireLock:false})` — mirroring
  the GET/recover/stream routes. The route body already no-ops worktree removal
  when there is no workspace. Wired a trash-icon drop button into the Studio AI
  panel.
- **Capability pointer (Fix 4):** a "Capability sources" block in the assistant
  context prompt pointing at package-local `skills/`, `agents/`, `mcps/`,
  `rules/`, `schemas/` for on-demand read (no inlined descriptions).

## Prevention

- **When a column becomes nullable, grep every consumer that branches on it.**
  A new null case (here `runs.project_id`) fans out to EVERY slot counter, every
  reconcile/sweep candidate query, and every route authz gate — not just the
  obvious call site. The class-sweep rule applies to "new null value" exactly as
  to "new enum value".
- **Sibling concurrency counters must apply the SAME predicate.** There were
  three flow-pool counters; missing one would let a launch admitted by one path
  be rejected by another (pool drift). Grep all cap-count sites
  (`countLiveRuns` / `assertScratchCapacity*` / `resumeCrashedRun`) in one pass.
- **A terminal/`worktree-gone` check that precedes the live-session skip is
  load-bearing.** Any run that legitimately has no worktree (agent
  none/repo_read, project-less assistant) MUST resolve `worktreeExists=true` or a
  LIVE session gets crashed. Cover with a "live session ⇒ skip" test.
- **Real-PG integration is mandatory for null-column / SQL-predicate changes.**
  tsc + mocked unit tests are blind to a candidate query that silently excludes a
  row. The reproductions (dead project-less run → Crashed; live → skip; assistant
  excluded from scratch cap) only have teeth on a real Postgres testcontainer.
- **Audit the whole route class, record the grep.** Only `/discard` was the
  user-facing gap; `/stop`, rename, `/commands` not supporting assistant runs is
  intentional (not assistant operations). Recording the audit prevents a future
  "fix" from adding redundant paths.

Open follow-up (runtime, not code): confirm the read-only assistant ACP session
can actually read `skills/*/SKILL.md` from its cwd on demand (Fix 4 step 10) — if
confinement blocks it, the pointer needs a read allowance.
