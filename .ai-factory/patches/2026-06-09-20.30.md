# Materialized worktree content leaks via `git add -A`; `.git/info/exclude` is worktree-shared

**Date:** 2026-06-09 20:30
**Files:** web/lib/capabilities/materialize-bundle.ts, web/lib/services/runs.ts, plugins/aif/flows/{dev,bugfix,evolve,init,roadmap}/flow.yaml, docs/{flow-dsl,configuration}.md, docs/system-analytics/{flow-graph,hitl}.md, docs/plans/2026-06-08-aif-flow-package-design.md
**Severity:** medium

## Problem

`/aif-review` of `feature/aif-flow-package` flagged that the per-run
`.ai-factory/config.yaml` git-ownership override (and, after the decision to
materialize the bundle's `scripts/`, the `commit-gate.sh`) could be **swept into
a commit and promoted into the user's repo**. The existing guard
(`git update-index --skip-worktree`) only covers the override when the file is
already **tracked** (true for the MAIster dogfood, false for a fresh consumer).
Separately, the review documents drifted: `docs/flow-dsl.md` stated `settings`
is "OPTIONAL on every node type" — false after the branch added the `form` node
(where `settings.form_schema` is required) — and the `form` runtime/UI was
undocumented in `flow-graph.md` / `hitl.md`.

## Root Cause

Two non-obvious mechanics:

1. **`skip-worktree` only hides changes to TRACKED files.** A freshly-materialized
   untracked file cannot be skip-worktree'd; staging-then-skip leaks it via the
   commit (the index entry is still committed). So skip-worktree alone can never
   hide untracked materialized content.
2. **In a linked git worktree, `.git/info/exclude` resolves to the SHARED common
   dir** (`git rev-parse --git-path info/exclude` → `<main>/.git/info/exclude`).
   Writing it would leak the ignore rule to every worktree + the main checkout —
   so it is NOT a per-worktree mechanism. The only per-worktree, `git add -A`-robust
   ignore is a worktree `.gitignore` (git honors it even when untracked), with the
   `.gitignore` edit itself `--skip-worktree`'d so it never shows in the run diff
   or trips the commit gate's `--untracked-files=no` clean-tree check.

The docs drift root cause is the recurring "code lands, docs lie" pattern: adding
an enum/node type updates the discriminated schema but not the prose invariants
("optional on every node type") nor the traversal/HITL diagrams that enumerate
node types.

## Solution

- Added `ensureWorktreeGitignore(worktreePath)` to `materialize-bundle.ts`:
  appends `/.gitignore` + `/.ai-factory/config.yaml` to the worktree `.gitignore`
  (idempotent) and best-effort `--skip-worktree`s the `.gitignore`. Wired into
  `launchRun` after the config override — the override is the one materialized file
  that MUST sit at a fixed worktree path. Repo-local `skills/`/`agents/` are NOT
  ignored (a consumer may track its own `.claude/`; in the dogfood they are
  repo-local, so nothing is materialized).
- The commit gate stays an **inline `command_check`** in the 5 flows (clean tree +
  Conventional Commits subject). A `commit-gate.sh` helper was briefly materialized
  to DRY it, but it was MAIster-authored ORPHAN code (not upstream AIF, never
  called by `/aif-commit`) and `.claude/scripts/` littered the worktree — so it was
  deleted and the gate left inline.
- Docs: qualified the `settings`-optional claim (except `form`), added the `form`
  node to `flow-dsl.md` + the `flow-graph.md` traversal + `hitl.md` (three-kinds
  table, decision tree, UI/output-vars prose); added an "Implementation deltas"
  section to the design doc (takeover→checks, flat templating vars,
  materialize-bundle site, no-rules, `maintainer` role caveat, packaging-model
  deferral pointer); documented the `metadata` block in `configuration.md`.

## Prevention

- **When materializing files into a run worktree, ignore them via a worktree
  `.gitignore` (+ skip-worktree the `.gitignore`), never `.git/info/exclude`** —
  the latter is common-dir-shared in worktrees. Verify with
  `git rev-parse --git-path info/exclude`.
- **`skip-worktree` is for tracked files only.** For untracked materialized
  content the ONLY leak-safe mechanism is gitignore.
- **Before materializing an artifact into the worktree, ask whether it NEEDS to be
  there.** A `command_check` script runs with CWD=worktree, so its logic can stay
  inline in the manifest — no file to materialize or leak. Only discovery-bound
  content (skills/agents under `.claude/`) and fixed-path config genuinely require
  materialization. Don't ship a separate script just to DRY a one-line check.
- **Review checklist for a new node-type / enum value:** grep the universal prose
  invariants (`flow-dsl.md` "optional on every node type"), the `system-analytics`
  diagrams that enumerate the type set (traversal action list, HITL decision
  tree), and the canonical `configuration.md` schema — a list addition is not
  enough if a sibling sentence now contradicts it.
- A `type:"string"` form field with `options` is intentionally NOT `type:"enum"`
  here: the form-HITL UI keeps a free-text input alongside option buttons, so
  `enum` (which rejects values outside `options`) would break "pick or type".
