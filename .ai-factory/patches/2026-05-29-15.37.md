## Problem

Codex adversarial review surfaced three M8 lifecycle defects:

1. **(critical) Checkpoint cancellation can be interpreted as normal
   permission completion.** When the supervisor checkpoints a session
   mid-permission (sweeper-driven or via `POST /sessions/:id/checkpoint`),
   it cancels every pending permission with reason `"checkpoint"` and
   propagates the reason onto `session.exited.reason`. The web runner-agent's
   `SupervisorEvent.session.exited` type stripped the optional `reason`
   field, AND the consumer loop broke on `session.exited | session.crashed`
   without inspecting it. The adapter returns `stopReason: "end_turn"`
   after a journaled-cancelled permission (the cancellation is queued for
   `--resume` replay), so the runner accepted that as success. Result:
   steps marked succeeded with unapproved permissions, racing the sweeper's
   `markCheckpointed` transition.

2. **(critical) Accepted idle responses depend on a volatile
   queueMicrotask driver.** The `/respond` idle branch committed the
   operator's intent (`hitl_requests.response` set, `respondedAt` null),
   transitioned `NeedsInputIdle → NeedsInput`, spawned a fresh supervisor
   session, and returned 202 — but `scheduleResumedSessionDrive` was just
   `queueMicrotask`. A web-process restart between the 202 and the
   microtask attaching left the run claimed but undelivered with no
   consumer. A same-payload retry returned another 202 without
   re-scheduling the driver. Operator intent could strand indefinitely.

3. **(high) Resumed last-step completion bypasses scheduler promotion.**
   The resume-driver's `completeResumedStepAndHandoff` wrote
   `runs.status = 'Review'` for the last-step case and returned without
   calling `promoteNextPending`. Same omission in `failResumedRun` and
   `crashResumedRun` invocations. Capacity freed by a resume terminal
   stayed effectively locked until some other run hit a terminal, so
   Pending runs starved.

## Root Cause

1. The supervisor wire contract was correct; the web type was the wrong
   side of the bug. The `SupervisorEvent` union deliberately mirrors
   `supervisor/src/types.ts`, but the optional `reason` field was added
   in the supervisor's M8 work without lockstep update on the web side.
   Once the type didn't carry `reason`, no consumer could inspect it,
   and the runner naturally defaulted to "trust `stopReason`". The mock
   adapter's behavior (return `end_turn` after journaling a cancelled
   permission) matched real-adapter behavior and made the contract
   leak invisible at the unit level — the M8 spike test asserted
   supervisor-side journaling but never the runner's interpretation of
   the prompt return value.

2. The `/respond` idle branch was correct at the durable-write layer
   (claim is persistent, `runs.status` is persistent), but treated the
   in-process driver attachment as a fire-and-forget side effect. The
   microtask is a delivery optimization, not a durability mechanism.
   The DB already carries the recovery signal
   (`response IS NOT NULL AND respondedAt IS NULL` against a
   `NeedsInput` row) — what was missing was a boot-time consumer that
   reattaches the driver or rolls back.

3. The resume-driver's terminal-write paths were added in milestone
   feature work that focused on correctness of the `NeedsInputIdle ↔
   NeedsInput` and stored-intent flows. The scheduler-promotion
   contract is documented in `runFlow` (`web/lib/flows/runner.ts:586`)
   but wasn't mirrored. The scheduler cap math
   (`count(status IN ('Running', 'NeedsInput'))`) means a `Review`/
   `Failed`/`Crashed` transition genuinely frees a slot — so the
   omission was a real starvation bug, not just sloppy code.

## Solution

### Code changes

- **Web type sync.** Added optional
  `reason?: "checkpoint" | "intentional"` to `SupervisorEvent`
  `session.exited` variant in `web/lib/supervisor-client.ts`. Mirrors
  `supervisor/src/types.ts` and the AsyncAPI spec.

- **New error code.** Added `STEP_CHECKPOINTED` to `MaisterErrorCode`
  in `web/lib/errors.ts`. Distinct from `CHECKPOINT` (terminal resume
  failure) — `STEP_CHECKPOINTED` is paused, not failed. Documented in
  `docs/error-taxonomy.md` and updated the exhaustiveness assertion
  in `web/lib/__tests__/errors.test.ts`.

- **New state transition.** Added `markCheckpointedFromExit` in
  `web/lib/runs/state-transitions.ts` — identical SQL to
  `markCheckpointed`, distinct log marker
  (`trigger: "exit"`) so the audit trail discriminates sweeper-driven
  from runner-observed checkpoints.

- **Runner-agent suppresses success on checkpoint exit.** In
  `web/lib/flows/runner-agent.ts`:
  - `EventConsumer` interface gains `checkpointReasonObserved()`.
  - The event-loop sets `checkpointObserved = true` when
    `ev.type === "session.exited" && ev.reason === "checkpoint"`.
  - `runNewSession` and `runSlashInExisting` branch BEFORE the
    `stopReason` check: if checkpointed, call
    `markCheckpointedFromExit` and return `errorCode:
    "STEP_CHECKPOINTED"` with `ok: false`.

- **Runner handles STEP_CHECKPOINTED.** In `web/lib/flows/runner.ts`:
  - New local `checkpointed: boolean` flag.
  - New branch in the step loop after `needsInput` and before
    `!result.ok`: mark `step_runs` row `NeedsInput`, persist
    `acpSessionId`, set `checkpointed = true`, break.
  - New post-loop branch BEFORE the existing `if (needsInput)`: skip
    terminal status write (markCheckpointedFromExit already moved the
    row to NeedsInputIdle), `cleanupSlashSession`, then
    `promoteNextPending` (slot is free since `NeedsInputIdle` doesn't
    count against the cap).

- **Resume-driver promotes on terminal.** In
  `web/lib/runs/resume-driver.ts`:
  - New helper `promoteAfterResumeTerminal(db, runId, kind)` —
    dynamic-imports `promoteNextPending` + `runFlow`, wraps in
    non-fatal try/catch with structured logging. Mirrors the
    pattern at `runner.ts:586`.
  - Called after the last-step Review write in
    `completeResumedStepAndHandoff` (status-guarded by checking the
    returning rows count).
  - Called after `failResumedRun`, `crashResumedRun` (×2 sites) in
    `runResumedSession` — only when the status-guard write succeeded
    (`r.ok`).
  - Belt-and-suspenders: `scheduleResumedSessionDrive`'s `.catch`
    block now funnels any uncaught throw through `crashResumedRun`
    + `promoteAfterResumeTerminal`, so a live-process driver bug
    cannot strand a row between recovery boots.

- **New startup recovery sweep.** Created
  `web/lib/runs/resume-recovery.ts:runResumeRecoverySweep`. Selects
  `runs.status='NeedsInput' AND acpSessionId IS NOT NULL` joined to
  the latest `hitl_requests` row with `response IS NOT NULL AND
  respondedAt IS NULL`. For each candidate, consults
  `listSessions()`: live session → re-call
  `scheduleResumedSessionDrive`; session gone → atomic
  `rollbackResumedRun` (status-guarded) preserving the intent; 5xx
  → skip-this-boot. Bounded concurrency = 4. Always-on, no flag.

- **Wired into instrumentation.** `web/instrumentation.ts` awaits
  `runResumeRecoverySweep()` before `startKeepaliveSweeper()`.
  try/catch keeps boot non-fatal.

## Prevention

- **When supervisor wire types change, mirror them on the web side in
  the same PR.** `web/lib/supervisor-client.ts` deliberately duplicates
  `supervisor/src/types.ts`. Without lockstep updates, the web tier
  silently strips fields it can't see. Future supervisor type PRs
  should grep for the matching variant in
  `web/lib/supervisor-client.ts` before merging.

- **Treat queueMicrotask as a delivery optimization, not durability.**
  Any DB row whose progress depends on an in-process scheduler call
  needs a startup-recovery sweep that re-derives the next action from
  durable state. The pattern: "if the row says intent claimed but not
  delivered AND status says we're committed to delivering, recover by
  reattaching OR rolling back". Apply this to any future
  202-returns-with-async-work routes.

- **Scheduler-promotion contract: every terminal write owes a
  promote.** `runFlow` has it (`runner.ts:586`); resume-driver now
  has it. Any new code path that writes `runs.status` to a terminal
  value (`Review` / `Failed` / `Crashed` / `Done` / `Abandoned`) MUST
  call `promoteNextPending` afterward (wrapped in non-fatal try/catch),
  OR document explicitly why the slot is not being freed (e.g.
  Abandoned via TTL sweep — covered by the sweeper's own
  `releaseSlotOnIdle` call).

- **Codex adversarial reviews are load-bearing.** This pass surfaced
  defects no unit/integration test caught because they live at the
  contract seam between supervisor and web. Schedule
  `/codex:adversarial-review` (or `/ultrareview`) as the C5/C9
  gate-pass for every milestone feature branch.
