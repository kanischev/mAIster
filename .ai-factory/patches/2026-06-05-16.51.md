# M17 HITL adversarial-review remediation (C1 deferred-leak, C2 response-leak, guard allow-list, route mappers, badge SSOT, i18n, analytics drift) + aborted stale-branch rebase

**Date:** 2026-06-05 16:51
**Files:** web/lib/services/hitl.ts, web/lib/queries/run.ts, web/lib/queries/portfolio.ts, web/app/(app)/page.tsx, web/app/api/runs/[runId]/hitl/[hitlRequestId]/respond/route.ts, web/app/api/v1/ext/runs/[runId]/hitl/[hitlRequestId]/respond/route.ts, web/lib/tokens/ext-handler.ts, web/components/board/run-hitl-response.tsx, web/messages/{en,ru}.json, docs/system-analytics/{flows,reconciliation-gc}.md, plus tests
**Severity:** high

## Problem

A `/aif-review` pass on the M17 HITL hybrid surface produced 3 blocking findings + several suggestions:

- **C1 (#hidden-deferred):** the permission-delivery `catch` re-threw on any non-`HITL_TIMEOUT`/non-`EXECUTOR_UNAVAILABLE` error (notably `ACP_PROTOCOL` — a supervisor 409 that REJECTS the delivery while the deferred is still live) WITHOUT releasing the deferred. `cancelPermission` existed in supervisor-client but was never imported by the HITL service. The supervisor-side permission deferred leaked until its keep-alive timeout (~30 min), blocking the agent invisibly. No regression test.
- **C2 (#response-leak):** `getRunDetail` returned the permission `schema` (carrying `requestId` / `supervisorSessionId` / `toolCall`) verbatim for ALL kinds, serializing supervisor handles into the run-detail RSC payload. The two sibling readers (board card, cross-project inbox) already redacted it; the run-detail path was the lone gap.
- **#guard-denylist:** the permission fresh-claim path gated only on a `!TERMINAL` deny-list, silently admitting any future non-terminal status (e.g. `HumanWorking`) into delivery.
- Route status mappers: `ACCOUNT_INACTIVE` → 500 (should be 403); `ACP_PROTOCOL` unmapped → 500; ext route forced every `PRECONDITION` → terminal 404 (wrong for transient race/inconsistency).
- Badge SSOT drift: home "needs you" used `inbox.count` (HITL rows, NeedsInput/Idle) while per-project badges used assignment-based `totalNeeds` (incl. HumanWorking/Review) — the two could disagree.
- Analytics drift: flows.md + reconciliation-gc.md still described `on_reject.goto_step` as "designed/unexecuted" though M17 implemented the linear repark (ADR-052).

## Root Cause

- C1: the two-phase-commit catch enumerated only the two "expected" supervisor failures and let everything else fall through to a bare `throw`. The codebase's own RULES.md flags exactly this — a catch on a deferred-creating path that does not release the deferred. The fix needs a `delivered` flag because the SAME catch also covers post-delivery DB failures, where the deferred is already resolved and cancelling would be a double-resolve bug.
- C2: M17 added redaction to the new board/inbox readers but the pre-existing run-detail loader (`schema: pending.rawSchema`) was never updated — a partial-coverage wave.
- #guard-denylist: a deny-list guard always rots when a new non-terminal status lands.

## Solution

- C1: import `cancelPermission`; set `let delivered = false` before `deliverPermission`, `delivered = true` immediately after it returns; in the catch fall-through, `if (!delivered)` best-effort `cancelPermission(...)` then re-throw. Regression test mocks `deliverPermission` to throw `ACP_PROTOCOL` and asserts `cancelPermission` called exactly once. Verified the EXECUTOR_UNAVAILABLE path keeps the deferred (correct — retry) and the HITL_TIMEOUT path needs no cancel (404 = deferred already gone).
- C2: `schema: pending.kind === "permission" ? null : pending.rawSchema` + a testcontainers integration test asserting the handles never reach the payload.
- #guard-denylist: introduced shared `PENDING_HITL_RUN_STATUS = {NeedsInput, NeedsInputIdle}` (renamed from `PENDING_FORM_RUN_STATUS`) and gated the permission fresh-claim on it. Confirmed via `runner-agent.ts:269` that a pending permission is always `NeedsInput`; two ext tests that seeded an unrealistic `"Running"` permission run were corrected.
- Mappers: `ACCOUNT_INACTIVE` → 403, `ACP_PROTOCOL` → 502 (web + shared ext mapper); dropped the ext route's `PRECONDITION` → 404 override (genuine-absence is existence-hidden upstream; the residual PRECONDITION is the transient flavor → 409).
- Badge SSOT: home headline now uses `portfolio.totalNeeds`; inbox block keeps its own HITL count.
- i18n: typed codes → `run.error.<CODE>` messages (EN+RU, full parity verified).
- Atomicity: permission Phase-2 `respondedAt` + scratch dialog flip wrapped in one transaction (assignment completion stays outside — `completeAssignment` opens its own tx and SQLite forbids nesting).

## Prevention

- When hardening a leak/redaction in one reader, grep EVERY sibling reader of the same field in the same pass (board + inbox + run-detail were three readers of permission `schema`; only two got the M17 fix).
- A `catch` on a path that observes a remote deferred-creation signal MUST release the deferred; use a `delivered`/`committed` flag to distinguish pre- vs post-side-effect failure so the release is only attempted when the deferred is still live.
- Prefer an explicit allow-list of valid states over a `!terminal` deny-list for any state-gated mutation.
- Test fixtures must reflect the real lifecycle status (a pending permission is `NeedsInput`, never `Running`); a loose guard can hide an unrealistic fixture for a long time.

## Stale-branch rebase finding (C3 — deferred, NOT applied)

The branch is 3 commits behind `main`. A pre-merge rebase was attempted (`m17-pre-fix-backup`, `m17-phaseA-tip` tags set as safety nets) and **aborted** after discovering it is a **semantic re-port, not a conflict resolution**:

- main's `feat(acp): add platform runner configuration` **dropped the `executors` table**, added `platform_acp_runners`, and re-wrote every query/service onto a new runner model. M17 built its HITL work on the OLD executor model, so a rebase breaks every `executors` / `executorId` / `defaultExecutorId` / `ensureApiTokenActor` / `resolveConfidence` reference — each must be re-ported onto main's new model.
- Migration metadata: branch `0024/0025` drizzle snapshots were baselined on `0021` (skipping main's `0022/0023`), and **main itself is missing its `0022/0023` snapshot files** — so a clean `drizzle-kit generate` regeneration is not mechanical until main's snapshot chain is repaired.

Recommendation: do C3 as its own focused session (re-port HITL onto the platform-runner model + repair migrations + full typecheck/test), not bundled with code-finding remediation.
