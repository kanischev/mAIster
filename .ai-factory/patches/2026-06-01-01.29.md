# Scratch run contract verification fixes

**Date:** 2026-06-01 01:29
**Files:** docs/system-analytics/workspaces.md, docs/system-analytics/runs.md, docs/architecture.md, docs/system-analytics/scratch-runs.md, docs/api/web.openapi.yaml, AGENTS.md, .env.example, docs/decisions.md, web/app/api/scratch-runs/[runId]/route.ts, web/components/scratch/scratch-dialog.tsx, web/messages/en.json, web/messages/ru.json, web/lib/capabilities/__tests__/catalog.test.ts
**Severity:** high

## Problem

The scratch-runs branch passed code gates but still failed verification because
public docs and OpenAPI overclaimed implemented behavior. Workspace docs still
called implemented branch targeting planned, promotion docs described
pull-request promotion as implemented, Web OpenAPI had duplicate 409 responses
and a capability-profile response shape that did not match the route, scratch
status labels were rendered as raw enum values, and the capability catalog test
did not directly prove all source scopes were cleared.

## Root Cause

The feature changed multiple contract surfaces after the first implementation
pass. Follow-up fixes corrected runtime behavior, but not every narrative doc,
OpenAPI schema, UI label, and test assertion was reconciled to the as-built
state. The capability catalog test also asserted only a loose minimum update
count, which allowed a missing source scope to look covered.

## Solution

Updated docs to describe the current state: local-merge promotion is
implemented, pull-request promotion is designed and returns `CONFIG`, scratch
recovery is implemented through the explicit route, and Docker Compose
containerizes Postgres only. Merged duplicate OpenAPI 409 entries and aligned
`ScratchRunDetail.capabilityProfile` with the route response. Localized scratch
dialog status labels in EN/RU. Strengthened the capability catalog test to
cover platform, project, and flow-package records, exact clear-scope count, and
conflict upsert re-enabling through `disabledAt: null`.

## Prevention

When fixing verification findings, treat docs, OpenAPI, UI message catalogs,
and tests as one contract set. For source-sweep tests, assert the exact
configured scope count and include at least one fixture per supported source so
future source additions cannot silently drop from CLEAR handling.
