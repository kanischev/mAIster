# Path traversal via resumeSessionId + 3 length caps in StartSessionRequestSchema

**Date:** 2026-05-26 11:13
**Files:** supervisor/src/types.ts, supervisor/src/__tests__/types.test.ts
**Severity:** critical (resumeSessionId) + low (3 length caps)

## Problem

`/aif-review supervisor/src/types.ts` (driven by the freshly-evolved skill-context rules
`aif-review#1` and `aif-review#3`) surfaced that `StartSessionRequestSchema` had:

- **CRITICAL `#path-traversal`** — `resumeSessionId: z.string().min(1).optional()` flows into
  `child_process.spawn` argv at `supervisor/src/spawn.ts:62-63`:
  ```ts
  if (request.resumeSessionId) {
    args.push("--resume", request.resumeSessionId);
  }
  ```
  The child process (`claude-agent-acp` / `codex-acp`) uses this value as a key into the
  Claude Code session store at `~/.claude/projects/<cwd-encoded>/<uuid>.jsonl` (per `CLAUDE.md`
  §"M0 spike findings #2"). An HTTP caller could pass
  `resumeSessionId: "../../../tmp/attacker"` to make the child read attacker-controlled or
  other-user-controlled files on the supervisor host. Supervisor is internal-network with no
  auth in POC topology (per `CLAUDE.md` §1), which mitigates blast radius — but defense-in-depth
  matters and the assumption rots over time.

- **Three sibling fields without length caps** (defense-in-depth + DoS posture):
  - `projectSlug` — kebab-case regex but no `.max()`; siblings `runId`/`stepId` had `.max(128)`
  - `worktreePath` — absolute-path `.refine()` but no `.max()`; OS `PATH_MAX` is 4096 on Linux
  - `prompt` — bare `z.string()`; flows into agent ACP/stdin without a memory bound

## Root Cause

**Partial-coverage waves across two prior fix rounds.**

- Patch `2026-05-26-12.45.md` added `SAFE_PATH_SEGMENT` regex to `runId` and `stepId` — the two
  fields named in the original CRITICAL finding from a prior `/aif-review`.
- Patch `2026-05-26-12.53.md` was the round-2 cleanup that addressed 8 of 11 review findings,
  including the sibling `worktreePath` (`.refine` for absolute + no `..`).
- **Neither round visited `resumeSessionId`** — even though it sits in the same schema and
  feeds the same `spawn()` call, and the prevention text from patch 12.45 literally said
  "trace every `child_process.spawn` argv back to the validator. If the input is `z.string()`
  or any 'open' type, that's the flag."

The same pattern repeated for the three length-cap suggestions: each one was a one-line
`.max(N)` addition to a sibling of a field that already had a cap (or had a constraint that
made an explicit cap implicit).

This is **textbook for the skill-context rules just landed by `/aif-evolve`**:
- `aif-review#1` (path-operation audit: trace every spawn argv back to validator) found it.
- `aif-review#3` (when diff adds regex to one field, check siblings) explains why it was missed.
- `aif-fix#1` (sweep sibling fields when fixing a boundary-validation bug) drove the
  4-field-in-one-commit fix shape.
- `aif-fix#2` (when fixing review findings, default to ALL unless scoped) drove the
  AskUserQuestion that confirmed the user wanted CRITICAL + length caps in one pass instead
  of just the CRITICAL.

## Solution

**`supervisor/src/types.ts` — four schema-field changes:**

1. `resumeSessionId` — reuse the existing `SAFE_PATH_SEGMENT` const and the sibling
   `.min(1).max(128).regex(...)` shape from `runId` / `stepId`:
   ```ts
   resumeSessionId: z
     .string()
     .min(1)
     .max(128)
     .regex(SAFE_PATH_SEGMENT, "resumeSessionId must match /^[A-Za-z0-9._-]+$/")
     .optional(),
   ```
   `SAFE_PATH_SEGMENT` over `z.string().uuid()` — keeps the existing test fixture
   `"uuid-abc"` valid, doesn't over-constrain future agents whose session-id format may
   differ from RFC 4122, and matches the sibling defense pattern exactly (the principle the
   skill-context rule wants).

2. `projectSlug` — added `.max(64)` between `.min(1)` and the kebab-case regex.
3. `worktreePath` — added `.max(4096)` to `worktreePathSchema` (matches Linux `PATH_MAX`).
4. `prompt` — `z.string().max(1_000_000)` (1 MB matches the `MAX_LINE_BYTES` chosen in
   patch 12.53 for the supervisor's line-buffer cap — keeps "bound external input at 1 MB"
   consistent across the codebase).

**`supervisor/src/__tests__/types.test.ts` — four new tests:**

- `rejects resumeSessionId with path traversal segment` (`"../../../tmp/foo"` → fail on `["resumeSessionId"]` path)
- `rejects resumeSessionId longer than 128 chars` (`"a".repeat(129)`)
- `rejects projectSlug longer than 64 chars` (`"a".repeat(65)`)
- `rejects prompt longer than 1_000_000 chars` (`"a".repeat(1_000_001)`)

No happy-case test added for `resumeSessionId` — the existing
`"accepts executor.env and router=ccr"` (line 26) already exercises `resumeSessionId: "uuid-abc"`
which passes `SAFE_PATH_SEGMENT`. No churn.

**No changes to** `supervisor/src/spawn.ts`, `supervisor/src/cost.ts`, `supervisor/src/http-api.ts`,
or any `web/` file. The fix is purely at the validator — sinks already trust the validator,
which is the correct boundary.

**No `[FIX]` runtime logging** — this is a deviation from the standard /aif-fix logging rule,
documented in the FIX_PLAN.md. The fix surface is a declarative Zod schema with no executable
hook; Fastify's existing Zod-error mapper in `supervisor/src/http-api.ts` already surfaces
rejected requests as `409 PRECONDITION` with structured `{ code, message }` bodies, which is
the observability path here.

**Verification:**
- `pnpm test` → 44 supervisor unit tests pass (20 existing types + 4 new = 24 types-suite,
  + 7 cost + 8 registry + 5 spawn) + 9 integration tests pass. No fixture rewrites needed.
- `pnpm lint` → clean.
- `pnpm typecheck` → clean.

## Prevention

This fix is itself the proof of the just-landed skill-context rules. The patterns to keep
applying:

- **When `/aif-review` flags a Zod field for boundary validation, sweep ALL siblings in the
  same schema in the same fix.** Do not ship round 1 with one field fixed and round 2 with
  another. Two prior rounds (patches 12.45 and 12.53) addressed the same schema; `resumeSessionId`
  fell through both because the sweep wasn't exhaustive. The audit checklist for the next
  reviewer: list every field in the schema, list every sink each one reaches, verify the
  constraint matches the sink's invariant.

- **`resumeSessionId` (and any session-id-like field flowing into spawn argv) MUST use
  `SAFE_PATH_SEGMENT.max(128)` or stricter.** Even when the value is "obviously a UUID" by
  convention, the validator must enforce it — the child process treats the value as a
  filename, and we cannot trust the child to sanitize.

- **Length caps belong on every external string in a boundary schema, not just the ones
  whose absence created an incident.** Defense-in-depth + DoS posture is part of the
  shipping bar; pick caps from the realistic ceiling (PATH_MAX for paths, kebab realistic
  length for slugs, MAX_LINE_BYTES for prompts/blobs) and stay consistent within the codebase.

- **A future `/aif-evolve` run reading this patch can strengthen `aif-fix#1` (sibling sweep)
  with the explicit checklist:** "list every field × every sink before declaring the sweep
  done". This patch is direct evidence that text wording wasn't strong enough on its own —
  it took an AskUserQuestion in `/aif-fix` to actually surface the omission.
