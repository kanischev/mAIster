# Flow Studio assistant repeated its first answer on every follow-up message

**Date:** 2026-06-28 19:44
**Files:** web/lib/scratch-runs/service.ts
**Severity:** high

## Problem

In the Flow Studio AI assistant, every follow-up message got the SAME answer as
the first one — verbatim, including the greeting. Reproduced from the live DB
transcript: turn 1 "из чего состоит сценарий" → a full flow description; turns 2
& 3 ("крутяк", "а еще че расскажешь") → byte-for-byte the same description.

## Root Cause

`sendLocalPackageAssistantMessage` (the follow-up path) rebuilt the FULL
`buildFlowAssistantContext` grounding block (~28k tokens: DSL grammar + editing
contract + file inventory + entire flow YAML + graph dump) and prepended it to
the user's message on EVERY turn, via `groundedFlowAssistantPrompt`. But the
assistant runs in ONE live ACP session that already holds that grounding in its
history from turn 1. Re-sending the massive "here is the flow, describe/edit it"
block dwarfed a short follow-up ("крутяк") so the model re-anchored on the
original framing and re-emitted its first answer, ignoring the actual message.
The heavy context is correct for turn 1 (launch) and wrong for every turn after.

## Solution

Follow-up turns now send a SLIM prompt — just the user's text, with a one-line
`(editor focus: …)` hint only when a file/node is focused (focus can change
since launch). New helper `followUpFlowAssistantPrompt`; `sendLocalPackageAssistantMessage`
no longer calls `buildFlowAssistantContext`/`groundedFlowAssistantPrompt`. The
action-JSON contract + grammar remain in session history from turn 1, so edits
still work. Launch (turn 1) is unchanged — it still sends the full grounding.
The per-turn log now reads focus from the request body instead of the removed
context object.

## Prevention

- A persistent chat session that retains history must be grounded ONCE (first
  turn), not re-grounded every message. Re-sending a large system/context block
  each turn both wastes tokens and makes the model re-anchor on it, drowning the
  actual user message.
- When a follow-up shares a live session with an initial heavy-context turn,
  the follow-up payload should be minimal (the delta), trusting session memory.
- Symptom heuristic: "assistant repeats its first reply verbatim regardless of
  input" ⇒ look for per-turn context re-injection dominating the user message,
  not a model fault.
