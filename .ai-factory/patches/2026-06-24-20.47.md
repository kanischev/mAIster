# M39 /aif-review fixes: trigger-task idempotency, catalog re-assert, classifier SSOT, ADR-106 doc drift

**Date:** 2026-06-24 20.47
**Files:** web/lib/db/schema.ts, web/lib/db/migrations/0069_m39_task_trigger_idempotency.sql (+snapshot/journal), web/lib/services/tasks.ts, web/lib/agents/launch.ts, web/lib/packages/attach.ts, web/lib/local-packages/service.ts, web/lib/agents/project-links.ts, web/app/api/studio/local-packages/[id]/route.ts, web/instrumentation.ts, 2 new two-racer tests, ~11 docs
**Severity:** high

## Problem

A milestone `/aif-review` of the M39 stream (Stream-A studio authoring + ADR-106
package-based agents) surfaced a blocking documentation-drift cluster plus
several real correctness/test gaps:

1. **Concurrent trigger redelivery leaked an orphan auto-task.** The dedup
   pre-check in `launchAgentRun` converged the SEQUENTIAL redelivery, but two
   CONCURRENT redeliveries both passed it, both `createTask`'d, and one run won
   while the loser's task was orphaned — breaking the "one task, one run" intent.
2. **`installPackageRevision` regressed the catalog projection.** It registered
   the just-installed (possibly non-newest) revision directly; installing an
   OLDER revision silently regressed the advertised `agents` rows until the next
   boot/admin resync.
3. **Two divergent path→kind classifiers** (`inferFileKind` vs
   `classifyPackageFilePath`) already disagreed on the new `subagent` kind.
4. **Systematic ADR-106 doc drift** (deferred task T7.3): migration cited as
   `0062` in 7 doc sites (the real migration is `0068`; `0062` is a *different*
   migration), 23 `(Designed — ADR-106)` tags on fully-implemented code, phantom
   component paths in a screen doc, stale `Designed` status headers.
5. **Test gaps:** no real two-racer test for the trigger redelivery or for
   `ensureDefaultLocalPackage`'s concurrent first-fork (its own comment names a
   data-loss hazard); a `DELETE` route returned 204 for a missing package; the
   `available` agents list didn't exclude disabled catalog rows.

## Root Cause

- **A dedup pre-check only protects the path it precedes.** The run-level claim
  (`runs(agent_id, trigger_event_id)` partial-unique) was the only idempotency
  arbiter, and it sits AFTER `createTask` — so a concurrent redelivery
  double-created the task before the run claim fired.
- **Making the task idempotent introduced a NEW collision** discovered only by
  writing the two-racer test: a shared `taskId` ⇒ same `task-<id>/attempt-N`
  branch ⇒ the loser failed at `addWorktree` (a `PRECONDITION`) *before* the
  run-claim, which a `CONFLICT`-only catch could not converge. The clean
  resolution is to never let the loser launch at all.
- **A "convenient" projection write** (`upsertAgentRow` writes the processed
  install's `versionLabel` unconditionally) diverged from the newest-per-name
  invariant that only `resyncAgents` enforces.
- **Doc drift = a deferred renumber/flip task** (T7.3) that never ran after the
  rebase moved the migration 0062 → 0068.

## Solution

1. **Idempotent auto-task (migration 0069):** add `tasks.agent_id` +
   `tasks.trigger_event_id` + partial-unique `tasks_agent_trigger_event_uq`
   (mirrors the `runs` claim). `createTask` inserts with
   `onConflictDoNothing().returning()`; on 0 rows it REUSES the winner's task,
   skips activity/event/subscribe, and returns `deduped: true`.
2. **Converge the concurrent loser, don't collide:** `launchAgentDrivenFlowRun`
   reads `created.deduped` and, instead of launching (which would collide on the
   shared branch), calls `convergeToTriggerWinner` — a one-shot BOUNDED reselect
   of the winner's run (≤1s, NOT state-polling), falling back to a `deduped`
   marker if the winner fails in the narrow pre-insert window. Only the task
   claim WINNER reaches `launchRun`, so the run insert never hot-contends.
3. **Re-assert newest-per-name:** `installPackageRevision` calls
   `resyncAgents(db)` right after the per-install `registerPackageAgents`.
4. **One classifier:** delete `inferFileKind`; `LocalPackageFileMeta.kind` is now
   `ReturnType<typeof classifyPackageFilePath>` (the client-safe SSOT shared with
   the editor + commit gate).
5. **Docs:** flip every `(Designed — ADR-106)` → `(Implemented — ADR-106)`,
   `0062` → `0068` across agents.md / agents-domain.md / database-schema.md /
   flow-dsl.md / decisions.md / ROADMAP / web.openapi.yaml, rewrite the
   "Status convention" paragraph, fix phantom component paths.
6. **Tests/nits:** two real two-racer tests (concurrent trigger redelivery → 1
   task + 1 run; concurrent `ensureDefaultLocalPackage` → 1 default + winner's
   repo survives); `DELETE` 404s a missing package; `available` excludes
   `enabled === false`; stale `0062` test names/comments → `0068`.

## Prevention

- **A two-racer test is not optional for a dedup contract — it finds the
  SECOND-order race.** The idempotent-task fix looked correct until the
  `Promise.all` test exposed the shared-branch collision. Single-threaded
  redelivery tests would have shipped the bug.
- **When you add an idempotency key that makes two callers share a downstream
  identity, re-derive every identity downstream of it** (here: the branch name
  `task-<id>/attempt-N`). Sharing the task shared the branch.
- **A projection write must re-assert its global invariant** (newest-per-name),
  not just write the row it happens to hold.
- **A `// mirrors`/parallel-classifier smell is a real DRY defect** — collapse to
  the SSOT the validation gate already uses.
- **A deferred doc-renumber task (T7.3) is a build-break risk:** a migration
  number that disagrees with `_journal.json` points readers at the wrong DDL.
  `pnpm db:generate` (verify "No schema changes" after) + the ADR-anchor/mermaid
  doc gates catch these.
