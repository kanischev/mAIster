# M15 readiness doc drift: stale "enforcer-only re-eval" edge case + missing source links

**Date:** 2026-06-04 16:25
**Files:** docs/system-analytics/readiness.md
**Severity:** medium

## Problem

A Codex adversarial review of the M15 branch flagged that
`docs/system-analytics/readiness.md` still documented the OLD read-model/merge-guard
divergence as a live edge case (lines 211-216, "Enforcer-only re-evaluation of
`artifact_required` gates"). The prose claimed the read-models (`getRunReadiness`, board,
portfolio) "report the recorded `gate_results.status` (`failed`) without re-evaluation"
and that "the enforcer is authoritative" — i.e. that a card could read `failed` for a run
the engine would merge.

The code on this same branch had already removed that divergence: the shared
`blockingGateContribution` in `readiness-core.ts` re-evaluates a `failed`
`artifact_required` gate against current artifacts, and `assertEvidenceReady`,
`getRunReadiness`, and the board/portfolio/project batch classifier all route through it
(verified at `readiness-core.ts:111`, `evidence-readiness.ts:187`, `readiness.ts:318`,
`readiness-batch.ts:142`). The edge case therefore contradicted the file's own Expectation
("no surface re-derives the verdict inline", line 187) and State-machine note (lines 79-80),
and pointed future work back toward the exact bug the branch had just closed.

Second drift, not flagged by Codex, found while verifying: the new shared classifier
`web/lib/queries/readiness-batch.ts` and `web/lib/queries/project.ts` were absent from the
doc's "Source (read-model + surfaces)" linked-artifacts list.

## Root Cause

The companion code patch (`2026-06-04-12.33.md`) consolidated three divergent surfaces into
one SSOT and updated the doc's *flowchart* and *enum*, but did not sweep the **Edge cases**
and **Linked artifacts** sections of the same domain doc. An edge case exists to name a
*deviation from the contract*; once the code change made the surfaces converge, the deviation
ceased to exist, but the bullet that described it survived as prose residue. A documented
edge case is the most dangerous kind of drift: unlike a stale comment, it reads as an
intentional, reviewed invariant and actively steers future implementers back to the old shape.

## Solution

- Rewrote the edge case to state the current uniform invariant: a `failed`
  `artifact_required` gate whose `inputArtifactRefs` are all current again contributes
  `clear` via `blockingGateContribution` and reads ready on the enforcer AND all three
  read-models alike — `gate_results.status` stays `failed` but the live artifact state
  overrides it on every surface, so no read-model diverges from the merge guard. It blocks
  only while some ref is still non-current or `inputArtifactRefs` is empty (matches the
  code's `refs.length > 0 && refs.every(...)` clear condition).
- Added `web/lib/queries/readiness-batch.ts` and `web/lib/queries/project.ts` to the
  "Source (read-model + surfaces)" list.
- Verified the rewritten invariant is already test-backed by
  `readiness-ssot-consistency.integration.test.ts:347` (re-eval ready) and `:373`
  (re-eval blocked). `pnpm validate:docs` passes (prose-only; 3/3 Mermaid blocks).

## Prevention

- When a code change **eliminates or unifies a documented divergence / special-case**, grep
  the domain doc's **Edge cases** AND **Linked artifacts** sections in the SAME pass — not
  just the diagrams and enums. A converged behavior must either delete its old edge-case
  bullet or rewrite it to the new invariant; leaving it is a contradiction, not a stale note.
- New SSOT/shared source files (here `readiness-batch.ts`) must be added to the consuming
  domain doc's Linked-artifacts list as part of the change that introduces them.
- Review checklist item: for any "single source of truth" / "kill divergence" change, the
  domain doc's Edge cases must not still describe the divergence the change removed.
