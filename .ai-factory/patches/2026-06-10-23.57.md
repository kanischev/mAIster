# Blind find-replace renumber corrupted sibling-feature ADR/migration refs

**Date:** 2026-06-10 23:57
**Files:** docs/database-schema.md, docs/api/web.openapi.yaml, docs/db/erd.md, docs/db/hitl-domain.md, docs/system-analytics/review-comments.md, web/lib/flows/graph/runner-graph.ts
**Severity:** medium

## Problem

A collision-fix commit renumbered the review-comments feature
(ADR-071→072, migration 0038→0039) because M28 run-schedules already
claimed ADR-071 / migration 0038 on `main`. The renumber was run as two
unguided global replaces and corrupted 10 references:

- **Over-reach (4):** `s/ADR-071/ADR-072/` also hit `run_schedules`
  references that legitimately point at ADR-071 — the dispatcher row in
  `database-schema.md`, its ADR link (producing `[ADR-072](#adr-071-…)` —
  visible label disagreeing with the anchor it targets), and two OpenAPI
  descriptions (`run_schedule.dispatcher` seed + two-phase fire pipeline).
- **Under-reach (5):** the `s/0038/0039/` pass was scoped narrower than the
  ADR pass and missed five `review_comments` references, leaving them on
  migration `0038` (the `review_comments` section header, the
  `review-comments.md` status block, both REVIEW_COMMENTS ERD nodes, and the
  schema-source line).
- **Corruption (1):** one line was textually mangled into
  ``migration `0039`s` stores`` — unbalanced backticks, lost
  ``.)** `review_comments``, broken markdown from line 76 onward.

## Root Cause

Blind find-replace across a **shared identifier namespace**. Two sibling
features each own an adjacent number and each mentions the *other's* number
range in prose: `run_schedules` = ADR-071 / migration 0038, `review_comments`
= ADR-072 / migration 0039. A global `s/ADR-071/ADR-072/` rewrites BOTH the
feature being renumbered AND every sibling that legitimately references the
old number. The two passes (ADR vs migration) were also scoped differently,
so they fell out of sync — ADR bumped where migration was not, and vice
versa.

No validator caught it:
- `validate-docs-mermaid.mjs` checks diagram syntax, not prose numbers.
- `validate-docs-adr-anchors.mjs` only checks that an anchor **resolves** —
  it does NOT check that the visible `[ADR-072]` link text agrees with the
  `#adr-071-…` slug it targets, so the over-reached link passed green.

## Solution

Per-occurrence **semantic** review instead of textual replace. For every hit
asked "which feature does THIS reference name?" and kept `run_schedules` at
071/0038, moved only `review_comments` to 072/0039. Restored the mangled
line. Left the `runs.…:adr071:rework-comments` artifact-id literal frozen
(it is an opaque, persisted id segment, never parsed back to a number) and
added a one-line WHY comment so a future reader does not "correct" it to
adr072 and orphan existing rows. Verified with a regression grep sweep
(zero over/under-reach, zero corruption) + the changed-files docs gate
(8/8 mermaid, 11/11 anchors).

## Prevention

- **Never renumber a shared ADR/migration namespace with an unguided global
  find-replace.** Enumerate every occurrence and classify each by the feature
  it names; only the renumbered feature's own references change.
- When two unmerged branches collide on an ADR or migration number, treat the
  renumber as a semantic edit, not a `sed`. Run ONE pass that handles both the
  ADR and the migration number together so they can't desync.
- Always end a renumber with a grep sweep: assert the OLD number now appears
  only in its legitimate sibling's context, and the NEW number only in the
  renumbered feature's.
- **Tooling gap to close (review checklist item):** extend
  `validate-docs-adr-anchors.mjs` to assert the link **text** `[ADR-NNN]`
  matches the `#adr-NNN-…` slug it points at — that single check would have
  caught the over-reached `[ADR-072](#adr-071-…)` link automatically.
