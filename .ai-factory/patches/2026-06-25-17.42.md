# Triager review fixes: auto-launch crash-loop, config validation asymmetry, give-up contract, doc drift

**Date:** 2026-06-25 17:42
**Files:** web/lib/scheduler/handlers/auto-launch-triaged.ts, web/lib/agents/{config,definition,project-links}.ts, web/lib/services/triage.ts, web/lib/queries/board.ts, web/components/board/{board,flight-card,launch-popover}.tsx, web/messages/{en,ru}.json, docs/{decisions.md, system-analytics/{triage,external-operations}.md, database-schema.md, db/{erd,runs-domain,scheduler-domain}.md, api/web.openapi.yaml}, + fixture triager.md
**Severity:** high

## Problem

`/aif-review` of the triager branch (ADR-110/111) surfaced one HIGH and several MEDIUM issues:

- **C1 (HIGH):** the `auto_launch_triaged` tick never cleared `tasks.launch_mode='auto'` on a successful launch, and `classifyTaskLaunchability` returns `launchable` again once the latest run is `Failed`/`Abandoned`. An auto-enqueued task whose flow run *failed* was re-launched every 60s with no cap or backoff — a global-slot-starving, cost-burning crash-loop. No test and no design bullet covered the launch-then-fail path.
- **C2 (MEDIUM):** per-instance config was written as `z.record(z.unknown())` → `updateAgentLink` raw write → `resolveAgentConfig` verbatim copy, with no validation against the agent's declared `config_schema`. Out-of-range enum / wrong-typed / unknown-key values reached the immutable `runs.agent_config` snapshot and the agent system prompt.
- **C3 (MEDIUM):** a declared config `default` was never type-checked against its `type` (`type: number, default: "x"` parsed clean).
- **C4 (MEDIUM):** the documented give-up ("clear `launch_mode` + post system comment + set `flagged`") was only half-built — the code just cleared `launch_mode` and wrote a `triage_requeued` activity.
- **C5 (MEDIUM):** doc drift — `runner_list` field list said `provider` (code returns `capabilityAgent`+`readinessStatus`); `triage_status += flagged` was mis-attributed to migration 0072; `job_kind` ERD enum + cron-tick filter enum missed `auto_launch_triaged`; the R7-cited canonical `external-operations.md` never documented the new tools/scopes.

## Root Cause

- **C1:** an automated, level-triggered launcher was wired to `classifyTaskLaunchability`, a classifier deliberately built to report a `Failed`-latest-run task as relaunchable (so the *manual* "Launch again" button reappears — the ralph-loop). That semantic is safe with a human in the loop; it is a crash-loop when an unattended timer drives it. The "reuse the existing launchability classifier" decision was correct for dependency-release but silently inherited unbounded manual-retry semantics.
- **C2/C3:** validation asymmetry — the config *declaration* was strictly validated (`MaisterError("CONFIG")`, no catalog row on a bad schema), but the *default value's type* and the *per-instance values* were trusted. Declaring types is pointless if the values aren't checked against them; and the trusted values flow into an immutable, prompt-injected snapshot.
- **C4:** the spec described the safer behavior (hold + explain) but the implementation shipped the weaker subset; "code wins" forces reconciliation, and here the doc was the better target to implement.
- **C5:** a Drizzle `text({enum})` widening is application-level (no Postgres CHECK), so it generates no migration — but the docs narrated it as a DB migration. And a doc that cites another as the single source of truth (R7) was updated while its cited home was not.

## Solution

- **C1 (level-triggered + cap + backoff):** kept `launch_mode='auto'` across a launch (preserves dependency-release + transient-retry), but added a failure cap (`MAX_AUTO_LAUNCH_ATTEMPTS = 3` failed flow runs → give up) and exponential backoff keyed on the last failed run's `endedAt` (`countFailedFlowRuns` + `Date.now()` gate). `Failed`/`Abandoned` count as attempts; `Crashed` does not (it is held for human recover/discard, never auto-relaunched).
- **C4 give-up (unified):** both terminal-`PRECONDITION` and cap-exhausted give-up now run ONE transaction that clears `launch_mode`, sets `triage_status='flagged'`, and posts a system `comment_create` with the reason. Added a `flagged`-guard to `triager.md` so the comment's `task.comment_added` re-trigger is a no-op (a flagged task is human-owned), preventing a comment/trigger storm.
- **C2/C3:** added pure `validateInstanceConfig(declared, instance)` (client-safe, returns `string | null`) enforced in `updateAgentLink` (the server seam covering all callers → `MaisterError("CONFIG")` → 422); added a `default`-type check to `configParamDeclSchema.superRefine`.
- **S1:** threaded `triageStatus` into the flight DTO + gated Launch on `flagged` for backlog and flight cards + a `flagged` flight chip + the popover reason map (server boundary was already safe). **S3:** `sendTaskToTriage` now clears the stale `launch_mode`. **S5:** restored the `_journal.json` trailing newline (the ext-handler reformat finding was withdrawn — prettier wants the one-liner).
- **C5:** reconciled `provider → capabilityAgent, readinessStatus`, the migration attribution, the `job_kind` ERD/filter enums, and added the tools/scopes/`flag`/`enqueue` to `external-operations.md`.

Tests: full unit 5024 green (+ new `validateInstanceConfig`, default-type, flight-card cases); integration green (tick cap/backoff/flagged-give-up/2-candidate, config-rejection, triage wire-through); tsc/eslint/redocly/mermaid/adr-anchors clean.

## Prevention

- **An automated/level-triggered launcher MUST bound retries** (attempt cap + backoff) and MUST NOT reuse a classifier whose "launchable" verdict was designed for human-initiated retry. When reusing a launch predicate for a new (unattended) caller, re-derive what each terminal run state means for THAT caller — a `Failed`-latest-run is "offer manual relaunch" for a human and "stop or back off" for a timer.
- **A consumed intent must be cleared or bounded.** A persistent `*_mode='auto'`-style intent that a sweep acts on needs an explicit lifecycle: cleared on consumption (one-shot) or bounded by a cap (level-triggered). Grep every writer/clearer of the intent column when adding a consumer.
- **If a schema declares types, validate values against it on BOTH sides** — the declaration's `default` AND every per-instance write — before anything persists. A `z.record(z.unknown())` wire type feeding a typed, snapshotted, prompt-injected column is a trust smell; validate at the service seam so all callers (not just the one route) are covered.
- **A documented MUST/edge-case bullet the code doesn't satisfy is a must-fix, not a doc-only nit** — pick the safer behavior to implement (here: hold + comment).
- **A Drizzle `text({enum})` widening is app-level (no DB CHECK → no migration)** — never attribute it to a migration number in docs; reserve migration attribution for real DDL (the sibling `task_relations` CHECK *was* a migration).
- **When a doc cites another as its R7 single-source-of-truth, update that home in the same pass** — a cross-reference to a doc that never mentions the feature is drift.
- **A control-plane comment that emits a trigger event can re-enter the emitting agent** — when a system action posts a comment on a task an agent subscribes to, ensure the agent's prompt no-ops on the resulting state (here: flagged → human-owned).
