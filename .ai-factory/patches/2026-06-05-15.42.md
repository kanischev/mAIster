# M17 branch-review remediation — permission-schema leak to browser + repark pre-CAS crash-window collision

**Date:** 2026-06-05 15:42
**Files:** web/lib/queries/hitl.ts, web/lib/queries/board.ts, web/lib/flows/runner.ts (+ 3 integration tests)
**Severity:** high

## Problem

A `/aif-review` of the `feature/m17-hitl-hybrid-surface` branch surfaced 2
blocking issues that the all-green suites had not caught:

1. (critical) **Permission HITL schema leaked to the browser.** M17 added
   `schema: row.rawSchema` to `HitlItem` (`mapRowsToHitlItems`) and
   `hitlSchema: row.rawSchema ?? null` to the board flight-card DTO — for ALL
   kinds. For `kind=permission` the raw schema is
   `{ requestId, supervisorSessionId, toolCall, options }`, written by
   runner-agent. Both read models feed `"use client"` components
   (`InboxRespond`/`FlightCardHitl` → `RunHitlResponse`), so Next serialized the
   supervisor session id + request id + tool-call into the RSC flight payload
   sent to the browser on the portfolio home and every project board. The ext
   API route already nulls this exact field with a "NEVER cross the trust
   boundary" comment; the internal UI read models missed the same guard.
   `RunHitlResponse` only ever reads `schema` when `schema.review === true`, so
   for permission the field was unused — pure over-exposure.

2. (high) **on_reject repark: crash between `markStepSucceeded` and the repark
   CAS is unrecoverable.** In the flat-runner repark block, ordering is
   `markStepSucceeded` (human step_run → Succeeded, attempt 1) → write
   rework-comments → unlink sentinels → repark CAS. A process death in that
   multi-fs-op window leaves `current_step_id` STILL on the human step (the CAS
   that moves it to the goto target had not run) with that step's `step_runs`
   row already `Succeeded`. Reconcile's new `linear-gate-orphan` path crashes
   the run with `resume_target_step_id = current_step_id` = the human step;
   operator Recover → `driveResume` → `runFlow({ crashResume: { targetStepId:
   <humanStep> } })`. On resume `lastForStep` is `Succeeded` (not `NeedsInput`),
   so the else-branch ran `createStepRun({ attempt: 1 })` — the increment fired
   only for `reparkResume`, which is null on a `crashResume` — colliding with
   the existing `(run_id, step_id, attempt=1)` row
   (`step_runs_run_step_attempt_uq`). Recover then failed on every click; the run
   was stuck `Crashed`.

## Root Cause

- **#1** is a fan-out miss: P6 added `schema` to the shared `HitlItem`/board DTO
  to enable inline form/review rendering, but the existing permission-schema
  trust-boundary guard (already present at the ext-API boundary) was never
  propagated to the internal read models. The new integration test used a benign
  permission fixture (`{ options: [...] }` only), so it asserted the schema was
  *present* and never exercised a schema carrying supervisor handles — it
  enshrined the leak instead of guarding it.
- **#2** is an atomicity/ordering gap. ADR-052 named and tested only the
  *post-CAS* crash window (`current_step_id = goto`, recovery resumes at the goto
  target and never re-creates the human step_run). The *pre-CAS* sub-window
  (markStepSucceeded committed, CAS not yet) leaves the run pointing at a step
  whose latest step_run is already terminal, and the resume attempt-number logic
  only handled the in-process repark tail-call, not a crash-recovery re-entry on
  that same step.

## Solution

- **#1** — Mirror the ext-API DTO guard in both internal read models:
  `schema: row.kind === "permission" ? null : row.rawSchema` in
  `mapRowsToHitlItems`, and
  `hitlSchema: row.kind === "permission" ? null : (row.rawSchema ?? null)` in
  `board.ts`. Permission options are already projected separately via
  `extractOptions`, so the UI is unaffected. Hardened both integration tests to
  use a permission schema carrying `requestId`/`supervisorSessionId` and assert
  `schema === null` + the sensitive values are absent from the serialized item.

- **#2** — Generalize the attempt-increment in the runner's create-step-run
  branch: re-execute = `reparkResumeStepId !== null || (isResume &&
  priorAttemptsForStep.length > 0)`, and `attempt = max(prior)+1` when
  re-executing. A first-ever execution still uses `attempt=1`, so the
  `step_runs` unique constraint keeps guarding against the promote + manual
  re-dispatch double-execution race (and concurrent re-entries still collide at
  the new attempt level — the constraint is the real guard, the number just
  picks the slot). Added a regression test (T3.1.7d) seeding the exact pre-CAS
  state and asserting Recover re-prompts (`NeedsInput`) with two distinct
  attempts {1,2}, no collision.

## Prevention

- When a field is added to a shared read-model DTO that crosses to the browser,
  enumerate every `kind`/variant and confirm each is safe — especially when a
  sibling boundary (here the ext API) already guards the same field. A guard at
  one boundary is a signal the field is sensitive at ALL boundaries.
- Security fixtures must carry the sensitive payload. A test that uses a benign
  fixture and asserts "field is present" proves nothing about leakage; assert
  the sensitive bytes are absent from the serialized output.
- For any multi-write transition with a CAS, enumerate the crash window BEFORE
  the CAS, not just after. If a partial state leaves `current_step_id`/cursor on
  a step whose ledger row is already terminal, the recovery path must tolerate
  re-entry on a completed step (fresh attempt), not assume attempt=1.
