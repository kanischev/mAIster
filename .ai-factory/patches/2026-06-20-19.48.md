# PR squash force-push decision not retry-stable (transient-push reclaim → non-fast-forward)

**Date:** 2026-06-20 19:48
**Files:** web/lib/runs/promote.ts, web/lib/runs/__tests__/promote-pr.test.ts, docs/system-analytics/execution-policy.md
**Severity:** high

## Problem

A Codex adversarial review flagged the PR-mode squash path (added for execution-policy
`commits=squash_rework` / `squash_on_promote`). The `forcePush` flag passed to the PR
branch push was derived from the CURRENT attempt's squash result (`didSquash =
squash.squashed`). On the first attempt this is correct (squash rewrites `base..HEAD`
→ `force: true`). But if `pushBranch` then fails transiently (`EXECUTOR_UNAVAILABLE`),
the promotion claim is intentionally left `claiming` (retryable) and the remote is NOT
updated. On the reclaim, the local branch is already squashed, so the squash is a ≤1-commit
no-op → `didSquash = false` → the PR push runs WITHOUT `--force-with-lease`. The remote
PR branch still holds the old unsquashed history → the push is a non-fast-forward →
`CONFLICT`. The run gets stuck or needs manual branch repair — after exactly the failure
class the code documents as retryable.

## Root Cause

The force-push signal was tied to a PER-ATTEMPT, non-idempotent side-effect result
(`squashRunBranch` returns `squashed: false` once the branch is already collapsed), but the
need to force is a property of the RUN (its branch was rewritten relative to the remote) —
a fact that outlives any single attempt. A retry/reclaim re-derived the signal from a
no-op squash and lost it. The original implementation reasoned about the happy path only
and missed that the claim is explicitly designed to be re-entered after a transient push
failure (§3.2 durable promotion claim).

## Solution

Made the force decision RETRY-STABLE by deriving it from the IMMUTABLE execution-policy
snapshot instead of the squash result. Extracted a pure predicate
`squashesRunHistory(claim)` = `runKind==='flow' && baseCommit && commitsFromSnapshot(...)
∈ {squash_rework, squash_on_promote}`, and used it for BOTH the squash guard (DRY) and the
PR `forcePush`. `squashForCommitPolicy` now performs the (idempotent) squash and returns
void; the force decision no longer depends on whether THIS attempt rewrote anything.
`--force-with-lease` is safe even when this attempt was a no-op (it only matters on a
non-fast-forward and never clobbers an unexpected remote). Updated the as-built doc.

## Prevention

- A decision that gates a RETRYABLE side-effect must derive from durable/immutable state
  (here: the policy snapshot), never from the result of a non-idempotent op that an earlier
  attempt may have already consumed. Ask "what does the SECOND attempt see?" for every flag
  computed before a side-effect that can be re-entered.
- When adding a branch to a path guarded by a durable claim (`promotion_attempt_id`,
  `claiming` reclaim), test the RECLAIM path, not just the first attempt. The regression
  test mirrors the existing stale-claim crash-window harness: stale claim + no-op squash +
  assert `force: true` + run reaches Done.
- Review checklist: any new `force`/`overwrite`/`skip`/`already-done` flag computed from a
  side-effect's return value is a retry-safety smell — pin it with a reclaim test.
