# Time-limit watchdog: split-brain kill, ledger clobber race, and stranded queue

**Date:** 2026-06-01 12:55
**Files:** web/lib/runs/keepalive-sweeper.ts, web/lib/runs/__tests__/time-limit-watchdog.integration.test.ts
**Severity:** critical

## Problem

A Codex adversarial review of the M11c branch flagged three real defects in the
`limits.maxDurationMinutes` kill-on-cap watchdog (`runTimeLimitPass`), all
verified against the code:

1. **(critical) Split-brain kill.** The watchdog looked up the live supervisor
   session only by `runs.acp_session_id`. The graph runner persists
   `acp_session_id` *after* a node's prompt returns, so a node still mid-prompt
   has a null column while its agent session is live. With a null column the
   watchdog skipped `deleteSession` but still marked the node/run `Failed` — a
   terminal run in MAIster while the agent kept running and mutating the
   worktree outside the run lifecycle.
2. **(high) Ledger clobber race.** `markNodeFailed(attempt.id, …)` updated the
   attempt with no status predicate, in a separate statement from the
   `runs.status = 'Running'`-guarded run update. If the runner completed the
   node between the watchdog's read and write, the watchdog overwrote a
   `Succeeded` attempt to `Failed` while the guarded run update matched zero
   rows — leaving a non-failed run with a watchdog-failed node attempt.
3. **(high) Stranded queue.** After a successful kill the path only incremented
   a counter and logged. Unlike the normal terminal exit (`promoteAfterExit`)
   and idle (`releaseSlotOnIdle`), it never called `promoteNextPending`, so a
   `Pending` run could stay queued indefinitely when the only freed slot came
   from a duration-cap kill.

## Root Cause

- **#1** — the candidate/teardown match used the WRONG id space. `acp_session_id`
  is a *late-persisted, derived* handle; the supervisor's authoritative
  ownership key for a live session is the *server-owned* `(runId, stepId)`,
  which `SupervisorSessionRecord` already carries. The watchdog keyed off the
  field that is null exactly during the window it most needs to act (a
  long-running, hence over-cap, prompt).
- **#2** — "compute then mutate" with the read (`fetchActiveAttempt`) and the two
  writes (`markNodeFailed`, run-status update) spread across three
  non-transactional steps. A background sweep racing a live single-writer
  runner needs one atomic claim, not three independent statements.
- **#3** — a new terminal transition was added without wiring it into the
  scheduler's slot-release contract that every other terminal/idle path honors.

## Solution

- **#1** — match the live session by `(record.runId === run.id && record.stepId
  === run.currentStepId && status === 'live')` instead of `acp_session_id`, so a
  mid-prompt session is found and actually torn down. Only mark `Failed` after
  the session is killed OR confirmed absent (no matching live record); a
  retryable `EXECUTOR_UNAVAILABLE` from `deleteSession` leaves the run `Running`
  to retry next tick (no split-brain), while a terminal failure (404 — session
  already gone) proceeds. Matching by exact `stepId` also prevents killing a
  *later* node's session when `currentStepId` advanced.
- **#2** — claim the terminal transition in a single `db.transaction`: the
  guarded `UPDATE runs … WHERE status='Running' AND current_step_id = node`
  `RETURNING`, and only on a non-empty claim `markNodeFailed(…, tx)`. A
  concurrently-finishing node moves `current_step_id` off this node, so the
  guard matches zero rows and the ledger is never clobbered.
- **#3** — after a won claim, call `promoteNextPending` via a lazy
  `import("@/lib/flows/runner")` (cycle-safe, mirroring runner-graph's
  `promoteAfterExit`), dispatching `runFlow` for the promoted run.
- Added three integration regression tests: mid-prompt kill with null
  `acp_session_id` + live record (matched by runId+stepId), retryable 5xx leaves
  the run `Running`, and queued-`Pending` promotion after a kill.

## Prevention

- When tearing down a remote resource, key the lookup on the **server-owned
  identity** (`runId`/`stepId`) the remote actually tracks, never a
  late-persisted derived handle that can legitimately be null during the active
  window.
- A terminal/abort path that mutates more than one row MUST claim the transition
  atomically (transaction + CAS guards on every row it touches), especially when
  a background sweep races a live single-writer.
- Never mark work terminally Failed until its underlying execution is confirmed
  stopped or confirmed absent — a retryable teardown failure means "retry," not
  "give up and lie about the state."
- Any new terminal transition that frees a scheduler slot MUST call
  `promoteNextPending` (or `releaseSlotOnIdle`); grep the existing terminal
  paths and match their slot-release contract.
- Watchdog/sweeper tests must cover the null-derived-handle-with-live-session
  case and the retryable-failure case, not only the happy path.
