# Adversarial-review triage: composer newline loss, supervisor URI confinement, no-op skill picker; one false positive rejected

**Date:** 2026-06-17 21:37
**Files:** web/lib/capabilities/composer-serialize.ts, web/components/capabilities/capability-composer.tsx, supervisor/src/prompt-confinement.ts (new), supervisor/src/{types,spawn,http-api}.ts, web/lib/supervisor-client.ts, web/lib/scratch-runs/service.ts, web/components/scratch/scratch-launcher.tsx, docs/api/{web,supervisor}.openapi.yaml, docs/system-analytics/scratch-runs.md, web/messages/{en,ru}.json, + 6 supervisor test fixtures
**Severity:** high

## Problem

A Codex adversarial review of `feature/unified-capability-composer` raised four
findings. They were NOT taken at face value — each was verified against the code
and git history first. Three were real; one was a false positive.

1. **(high, real)** The supervisor forwarded prompt content-block resource URIs
   verbatim with no validation (`resource_link.uri` = any non-empty string). The
   web tier confines them, but the supervisor is a separate process (may run on a
   different host) — a direct/buggy caller could send `file:///etc/passwd` or a
   remote scheme.
2. **(high, real-but-by-design-backend)** The scratch launcher rendered a
   toggleable skills picker and sent `capabilities.skillIds`, but the backend
   replaces it with ALL project skills (broad materialization, FR-C3). The UI
   control was a false affordance and the OpenAPI implied skillIds were validated.
3. **(medium, real)** The TipTap composer's `docToCanonical` flattened all text +
   chip nodes across paragraph boundaries with `""`, so a multiline prompt
   (Enter / pasted) serialized `line one` + `line two` as `line oneline two` —
   prompt data loss.
4. **(medium, FALSE POSITIVE)** Codex claimed the branch "removed"
   `node_attempts.resolved_prompt` + migration 0053. It did not.

## Root Cause

- **#4 was a diff-direction artifact.** Codex reviewed `git diff main` (2-dot:
  working-tree vs main-tip). `resolved_prompt` + migration 0053 were added to
  MAIN by commit `cd9c6282` AFTER this branch forked at `a787dadf`; the branch's
  own 3-dot diff (`main...HEAD`) never touches them. A 2-dot diff renders main's
  post-fork additions as "removed on the branch." The branch regressed nothing —
  it inherits the feature on rebase. "Restoring" it would have created a divergent
  duplicate that conflicts at merge. Lesson: always confirm "branch removed X"
  with `git diff main...HEAD` (merge-base) before acting on a 2-dot finding.
- **#1**: the verbatim-forward invariant (normalization stays web-side) was read
  as "the supervisor does no validation at all." But rejecting an out-of-bounds
  URI is validation, not normalization — compatible with verbatim-forward and
  required by the project's "internal trust assumptions rot" rule. The session
  record simply never retained the worktree/repo to confine against.
- **#2**: the broad-materialization policy (locked FR-C3) landed in the backend
  but the pre-broad skill-selection UI was never removed, so the control silently
  did nothing.
- **#3**: the composer's doc↔canonical mapping assumed a single paragraph; the
  editor schema (`Document > Paragraph > inline`) allows many.

## Solution

- **#1**: new pure `contentBlockUriViolation(blocks, roots)` (prompt-confinement.ts,
  10 tests): a `resource_link` MUST be a `file:` URI inside roots = worktree ∪
  repo ∪ run-dir; remote schemes rejected; a `resource` block's non-file logical
  uri is left alone. Roots are bound to the session AT CREATION (worktreePath +
  optional repoPath added to `SessionRecord`; repoPath threaded through
  `StartSessionRequest` → web `createSession` → scratch call site only, since
  scratch is the one path that emits file content-blocks). The prompt handler
  rejects a violation with 409 BEFORE forwarding. Owner-approved.
- **#2**: skills `CapabilityGroup` made `readOnly` ("all included" + note); the
  dead `skillIds` state/init/count/body wiring removed (launcher no longer sends
  an ignored selection); OpenAPI `skillIds` documented as ignored-for-scratch;
  scratch-runs.md broad-policy flipped Designed→Implemented + UI note. EN/RU i18n.
  Owner-approved (read-only over remove).
- **#3**: `docToCanonical` now serializes each top-level block and joins with `\n`;
  `segmentsToDoc` splits text on `\n` into paragraph nodes. Extracted pure
  `segmentsToParagraphs`/`paragraphsToCanonical` (composer-serialize.ts) so the
  multiline round-trip is unit-tested without ProseMirror (the editor itself is
  e2e-only).
- **#4**: no code change. Documented as a rebase artifact.

No `[FIX]` console logging added: the touched web modules are pure/client-safe
(no-console rule); the supervisor uses pino and already logs the 409 violation.

## Prevention

- **Verify adversarial findings before implementing.** Adversarial reviewers
  surface false positives; a "branch removed X" claim must be checked with the
  3-dot merge-base diff, not a 2-dot working-tree-vs-tip diff.
- **A path-bearing field crossing a process/host boundary must be confined on the
  RECEIVING side too**, against roots that side controls (bound at session
  creation, not taken from the per-request body) — even when an upstream already
  confines. "Internal-only" trust rots.
- **When a backend policy overrides a user selection, remove or neutralize the UI
  control AND the contract text in the same change** — a silently-ignored input is
  a false affordance and a lying contract.
- **A rich-text serializer must account for every block/break node** the editor
  schema permits, not just the common single-paragraph case; cover the multiline
  round-trip with a pure unit test.
