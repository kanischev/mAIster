# M7 adversarial review pass 3 — durable wake-up, error-code dimensionality, per-run sequence, analytics drift

**Date:** 2026-05-28 20:01
**Files:** `web/app/api/runs/[runId]/hitl/[hitlRequestId]/respond/route.ts`, `supervisor/src/http-api.ts`, `supervisor/src/spawn.ts`, `web/lib/supervisor-client.ts`, `docs/system-analytics/hitl.md`, `docs/system-analytics/runs.md`, `docs/supervisor.md`, `docs/database-schema.md`, tests
**Severity:** critical

## Problem

Pass 3 of the M7 adversarial review returned `needs-attention` with four
findings (2 critical + 2 high). The previous two rounds' fixes still held;
these were exposed by the more aggressive review surface area.

1. **Critical:** Form / human HITL responses set `respondedAt` and returned
   200 BEFORE `queueMicrotask(runFlow)` was guaranteed to have completed —
   and `runFlow` rejection was only logged. If the process crashed between
   the Phase 3 commit and the microtask (or `runFlow` itself rejected), the
   run was permanently stranded in `NeedsInput`: a same-payload retry hit
   the `already-delivered` branch and returned 200 without re-queuing the
   runner. The only recovery was an operator-level intervention.
2. **Critical:** Supervisor's `POST /sessions/:id/input` returned the same
   `404 NEEDS_INPUT` shape for both *unknown session* and *unknown
   requestId*. The web client mapped every 404 to `HITL_TIMEOUT` (terminal).
   So a supervisor restart, deploy, or in-memory registry loss during HITL
   was treated as a deliberately expired user decision — `runs.status` was
   set to `Failed`, `respondedAt` was marked, and the only recovery was
   "launch a fresh run." The "unknown session" case is a *retryable*
   recovery path; conflating it with "deferred expired" lost that
   information.
3. **High:** The pass-2 fix moved event logging to one `<runId>/run.events.jsonl`
   per run, but each new spawn still initialized `record.monotonicId = 0`.
   The web SSE bridge filters replay by `ev.monotonicId > lastSeen` — so
   after session A wrote ids 1..N, session B appended ids 1..M and every
   one of those was silently skipped by reconnecting clients. The
   observability gap the per-run log was meant to close re-opened, this
   time for slash-in-existing's multi-step flow and for new-session-per-step
   spawns.
4. **High:** `system-analytics/hitl.md`, `system-analytics/runs.md`,
   `supervisor.md`, and `database-schema.md` still described pre-M7
   behavior: no HITL response route, supervisor input returning 501,
   last-write-wins on rapid resubmit, `response` and `respondedAt`
   committed in the same TX that flips `runs.status` back to `Running`.
   None of this matched the shipped implementation; readers using docs as
   spec would be wrong on every contract that mattered.

## Root Cause

These four collapse to one design failure repeated across boundaries:
**when state, side-effect, and observability live in different processes,
the protocols between them MUST encode every distinction the downstream
needs — defaulting to "same code, same status, same path" is a contract
hole**.

- **#1:** "Marking delivered" and "wake the runner" are independent
  side-effects, but the route treated them as a fused unit. There was no
  protocol for "delivered but the wake never fired" — and `queueMicrotask`
  is in-process, not durable. The retry path then hit a stale branch that
  said "nothing left to do."
- **#2:** The supervisor's input route flattened two semantically distinct
  failures into the same HTTP shape because it had only one error code
  available for the "row missing" case. Web's status-driven classifier
  could not recover the lost dimension. Recovery options for the operator
  ("retry" vs "this was a real timeout") were collapsed into "this was a
  real timeout" because that's the only branch the protocol could carry.
- **#3:** `monotonicId` was a per-spawn counter living on the in-memory
  `SessionRecord`. When the log file became per-run, the counter's scope
  did not follow. The contract between writer (supervisor) and reader (web
  bridge) required a strictly-increasing per-run sequence — the spawn site
  was not updated to honor it.
- **#4:** Docs lived in "planning verbs" (Designed Mx) and were updated by
  separate commits. Once the code-only fix landed, the docs were
  out-of-spec until someone deliberately came back. Even the patches in
  rounds 1 and 2 had documented this as a follow-up.

## Solution

1. **`scheduleResume(runId)` helper.** The Phase 3 wake is reified as a
   named helper so the same-payload retry path can call it. The claim
   transaction now returns the locked `runStatus`; the retry path
   re-queues `runFlow` only when the run is still in `NeedsInput`. A
   crash between commit and the original microtask is now recoverable by
   the obvious user action (resubmit the same response).
2. **Distinct codes for two distinct failures.** The supervisor input
   route now returns `503 EXECUTOR_UNAVAILABLE` for "unknown session"
   (retryable; recovery is "wait for the supervisor to reconcile") and
   `410 HITL_TIMEOUT` for "known session, no pending deferred" (terminal;
   recovery is "launch a fresh run"). The web client's status-driven
   mapping was updated accordingly. The response route's existing
   retry-vs-terminal classification then does the right thing without
   further change.
3. **Tail-seed `record.monotonicId`.** New `tailMaxMonotonicId(path)` in
   `spawn.ts` reads a 64 KiB tail window of the events.jsonl, parses the
   complete records, and seeds the new spawn's counter from the highest
   id present. The per-run event sequence stays strictly increasing
   across consecutive sessions. The regression test asserts session B's
   minimum id is greater than session A's maximum AND that the merged
   sequence is strictly monotonic.
4. **Analytics docs rewritten in the same commit as the code.** Three
   system-analytics + the database-schema + supervisor.md sections that
   directly describe the M7 wire / state-machine are now Implemented M7
   prose with the actual contracts (two-phase commit, atomic claim,
   retry classification matrix, supervisor 503/410 split, per-run
   monotonicId, runner-owned NeedsInput→Running transition).

## Prevention

- **Plan a durable wake.** Any HTTP route whose success depends on a
  follow-up worker tick MUST encode "delivered but worker tick lost" in
  its retry contract. Either persist a resume-job row read by a
  separate worker, or make the same-payload retry path durably re-queue
  the wake. Returning 200 + `queueMicrotask` is a contract hole the
  moment any downstream depends on the microtask actually running.
- **Don't flatten error-code dimensions across a remote boundary.** If
  two failure modes need different recovery actions, they MUST have
  different status codes / error codes — even if the underlying root
  cause is "row missing". Code review checklist: for every 4xx/5xx in
  a remote handler, name the recovery the caller will take. If two
  branches share the same code but different recoveries, split them.
- **Counter scope follows storage scope.** When a numeric cursor backs
  a replay contract, its scope MUST match the scope of the durable
  log it indexes. A per-spawn counter writing to a per-run file is a
  guaranteed off-by-N for every consumer that reads across spawns.
  Lint rule worth considering: any `monotonicId = 0` initializer in a
  long-lived data path needs a comment justifying its scope.
- **Analytics drift IS a defect, in the same PR as the code.** The
  pattern of "code lands, docs follow-up" produced two consecutive
  rounds of "specs still describe pre-M7 behavior." Stop the pattern:
  the merge gate for any change to a wire surface, a state machine
  transition, a retry classification, or a DB column semantic MUST
  include the same-commit doc update for the artifact that names that
  surface.
