# Reconcile reattach falsely crashes live scratch dialogs via the resume driver

**Date:** 2026-06-16 22:45
**Files:** web/lib/reconcile.ts, web/lib/__tests__/reconcile-classify.test.ts
**Severity:** high

## Problem

A freshly launched scratch run landed in `Crashed` right after the agent's
first reply, so the conversational dialog could never be reached. Observed log
sequence: run `Running` → agent first turn `stopReason=end_turn` → periodic
reconcile sweep logs `reconcile: reattached` (`reason=live-session`) →
`run-resume-driver` sends a continuation prompt → agent replies "I don't have
prior context about what was being worked on before this session" → `run-state`
`NeedsInput → Crashed` with `reason="resume-prompt-no-permission"`
(`crashResumedRun: status-guard mismatch`).

## Root Cause

`classifyRunReconcile` (`web/lib/reconcile.ts`) returned
`{action:"reattach", reason:"live-session"}` for ANY `Running` run with a live
supervisor session matched by `acp_session_id`, **before** branching on
`run_kind`. The sweep's `case "reattach"` then called the resume driver
(`scheduleResumedSessionDrive` → `runResumedSession`) for that run.

The resume driver's contract is flow-only: it sends `RESUME_CONTINUATION_PROMPT`
("…please continue with the prior tool call.") and waits for the supervisor to
replay the cancelled `session.permission_request`, then auto-delivers the stored
HITL intent. That contract exists for a FLOW run recovering a checkpointed HITL
session.

A scratch run is a plain conversational dialog: after a turn ends with
`end_turn`, its supervisor session stays live waiting for the NEXT user message.
It has no prior tool call and no stored HITL intent. Driving it with the
continuation prompt produced a context-less agent reply, no permission request
ever arrived, and the resume-prompt watchdog (`MAISTER_RESUME_PROMPT_TIMEOUT_
SECONDS`) crashed the run. The grace-window guard did not help because it only
applies on the no-live-session path — the `liveSession` reattach branch wins
first.

## Solution

Two-layer, surgical:

1. **Classifier carve-out** (`reconcile.ts` `classifyInner`): when
   `liveSession` is true, return `{action:"skip", reason:"live-scratch-
   session"}` for `run_kind='scratch'`; flow runs keep the existing
   `{reattach, "live-session"}`. A live `Running` scratch dialog is healthy and
   is resumed by the next user message through the scratch message path, not by
   reconcile. Added `"live-scratch-session"` to the `ReconcileReason` union.

2. **Defense-in-depth at the apply site** (`reconcile.ts` sweep `case
   "reattach"`): refuse to call `driveResumed` for any non-`flow` run_kind
   (skip + `log.warn` with `[FIX]` prefix), so a future classifier regression
   cannot route a scratch/agent run into the resume driver.

The optional driver self-guard from the plan was intentionally skipped —
layers 1+2 fully close the bug and the minimal-changes rule discourages
over-guarding.

Tests: updated the now-incorrect "scratch + live session → reattach" case to
assert `skip / live-scratch-session`, and added an explicit regression case
"flow + live session still → reattach". Classifier suite 46/46 green; reconcile
sweep integration 11/11 green; tsc + eslint clean.

## Prevention

- When a recovery/resume mechanism is built for one run kind (flow HITL
  checkpoint resume), any shared dispatch site (reconcile classifier + sweep)
  MUST branch on `run_kind` before routing into it. Scratch/agent runs have
  different lifecycle contracts than flow runs.
- A decision branch that fires on a single coarse predicate (`liveSession`)
  before kind-specific branching is a smell — order the decision table so the
  most-specific guard is not shadowed.
- Always pair the classifier rule with a guard at the apply/side-effect site
  for irreversible actions (crashing a run): the pure classifier is unit-tested,
  but a belt-and-suspenders guard at the impure call site prevents silent
  regressions.
