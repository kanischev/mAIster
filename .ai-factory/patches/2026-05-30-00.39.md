# Supervisor readiness verification cleanup

**Date:** 2026-05-30 00:39
**Files:** web/types/platform-status.ts, web/lib/supervisor-client.ts, web/components/chrome/platform-status.tsx, web/components/chrome/top-nav.tsx, web/components/chrome/status-bar.tsx, web/components/chrome/left-rail.tsx, web/components/auth/side-panel.tsx, web/components/board/board.tsx, web/app/api/runs/__tests__/route.trust-boundary.integration.test.ts, docs/system-analytics/tasks.md
**Severity:** medium

## Problem

The supervisor readiness feature was functionally present, but verification found three completion gaps:

- Shared UI components imported `PlatformStatus` from `web/lib/supervisor-client.ts`, crossing the project boundary that keeps `components/*` browser-safe and `lib/*` server-only.
- `docs/system-analytics/tasks.md` still described launch failure as executor-only and did not include the supervisor readiness preflight.
- The route integration test asserted no run and no task transition on supervisor-unavailable launch, but did not assert that no workspace row was created.

## Root Cause

The readiness type was introduced at the same time as the server health client, so UI presentation code reused the nearest exported type instead of a browser-safe shared type location. The analytics update focused on the run lifecycle document and missed the task-launch view of the same workflow. The test covered the most visible side effects but did not mirror the full API contract wording for worktree, run, workspace, and task invariants.

## Solution

Moved the shared platform readiness types to `web/types/platform-status.ts` and re-exported them from `web/lib/supervisor-client.ts` for server-side compatibility. Updated all readiness presentation components to import from the browser-safe type module. Added the supervisor health gate to the task analytics sequence. Extended the route integration test to assert that `workspaces` remains unchanged when readiness fails before launch side effects.

## Prevention

- When adding UI-facing types from server boundary work, place shared structural types under `web/types/*` first, then let `web/lib/*` import or re-export them.
- For launch-flow changes, update both `runs` analytics and `tasks` analytics because the same user action is modeled from two product angles.
- Route tests for no-side-effect preconditions should assert every persisted table mentioned in the API contract, not only the most obvious rows.
