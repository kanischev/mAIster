# Repo onboarding: clone TOCTOU, credential persistence, stale global ERD

**Date:** 2026-05-31 13.54
**Files:** web/lib/registration-lock.ts, web/app/api/projects/route.ts, web/lib/__tests__/registration-lock.test.ts, web/components/projects/new-project-form.tsx, web/messages/en.json, web/messages/ru.json, docs/db/erd.md, docs/system-analytics/git-integration.md
**Severity:** high

## Problem

Adversarial (Codex) review of the ADR-025 repo-onboarding branch found three issues:

1. **[high] Concurrent clone deletion (TOCTOU).** In clone mode `resolveProjectSource` checks `pathExists(dir)`, runs `git clone`, then on any clone failure removes `dir`. Two concurrent registrations for the same derived target can both pass the pre-check; the second `git clone` fails because the dir now exists, and its cleanup `rm(dir, recursive)` deletes the **first** request's clone — leaving `projects.repo_path` pointing at a missing repo.
2. **[high] Credentials persisted in `repo_url`.** `validateUrl` allows `https://user:token@host/repo.git`; the raw URL is inserted into `projects.repo_url`, putting a user-supplied secret into the DB/backups even though logs are already redacted.
3. **[medium] Global ERD stale.** `docs/db/erd.md` `PROJECTS` entity lacked the new `repo_url`/`provider` columns although the migration, `database-schema.md`, and `projects-domain.md` had them.

## Root Cause

1. The pre-check → clone → cleanup sequence has a TOCTOU window, and the cleanup wasn't proven to only remove dirs created by *this* request. The design assumed single-threaded access without enforcing it.
2. URL validation was scoped to scheme/shell-safety, not to userinfo; persistence reused the body value verbatim. The "no provider secrets at rest" contract was about MAIster-managed secrets and didn't consider a user voluntarily embedding one.
3. A DB schema change updated the per-domain ERD + narrative but not the top-level consolidated ERD — a doc surface that wasn't traced from the changed column.

## Solution

1. **Single-import lock** (user's chosen approach over temp-dir/rename): new `web/lib/registration-lock.ts` exporting `withRegistrationLock(fn)` — a process-wide promise-chain mutex (`chain.then(fn, fn)` so a failed registration never wedges the queue). `route.ts` wraps the `resolveProjectSource` + `register` critical section in it, so concurrent registrations serialize; the second now hits "target exists → supply override" before cloning and never touches the first's dir. Justified because MAIster is a single-host, admin-only control plane.
2. **UI warning, not server rejection** (user's call — a credential in a user-supplied URL is the user's choice): the Add-Project form flags `://user:pass@` with a non-blocking amber `warnUrlCreds` notice (EN+RU) recommending host SSH keys / a credential helper. `git-integration.md` edge case updated to state the URL is persisted as entered (redaction still covers logs/errors).
3. **ERD sync**: added `repo_url`/`provider` to the `PROJECTS` entity in `docs/db/erd.md`.

## Prevention

- For any disk/clone side effect behind a TOCTOU pre-check on a single-host service, serialize the critical section (lock) or prove ownership before cleanup — never `rm` a path you didn't prove you created.
- When a body field can carry a secret (URL userinfo), decide explicitly whether it is stored, redacted, or rejected — and keep the spec claim ("no secrets at rest") consistent with the chosen behavior.
- DB column changes must update the **consolidated** `docs/db/erd.md` in addition to the per-domain ERD/narrative — re-derive the full doc-surface list from the changed column, not from the plan's file list (matches the aif-verify contract-surface rule).
