# Testcontainers sandbox access during integration verification

**Date:** 2026-06-02 13:56
**Files:** none
**Severity:** low

## Problem

The run timeline integration test appeared to fail with:

`Error: Could not find a working container runtime strategy`

Docker and the existing Postgres container were healthy, but the test process
could not discover a usable Testcontainers runtime in the default sandbox.

## Root Cause

The default sandbox could not connect to Docker Desktop's socket at
`unix:///Users/kaa/.docker/run/docker.sock`. Testcontainers failed before the
suite executed any application assertions. Running `docker ps` with unrestricted
Docker access succeeded, proving the container runtime itself was healthy.

## Solution

Reran the Testcontainers-backed Vitest lane with Docker socket access allowed:

`pnpm exec vitest run --project integration lib/queries/__tests__/run-timeline.integration.test.ts --reporter=verbose`

The suite passed: 1 integration file, 4 tests.

## Prevention

When a Vitest integration lane uses Testcontainers in Codex, first verify Docker
socket accessibility from the same execution context. If sandboxed `docker ps`
fails with socket permission errors but unrestricted `docker ps` succeeds, rerun
the Testcontainers-backed lane with escalated Docker access before treating the
failure as a code defect.
