# M13 Found-Issues Fix

## Problem

Verification found M13 implementation drift after merge: unit fake DBs crashed
around new assignment/artifact tables, assignment events were stored but not
visible in run-detail history, and docs still described M13 runtime/UI work as
future work.

## Root Cause

The M13 runtime widened the query/update surface, but older runner unit fakes did
not model Drizzle chaining, nullable DB defaults, or the new assignment/artifact
tables. The assignment event ledger was append-only but not included in the
run-detail timeline read model. Docs were not updated from planned persistence
language to shipped runtime behavior.

## Solution

- Extended runner test fakes for the new table surface, Drizzle `limit`,
  `returning`, and upsert chaining, and realistic nullable `step_runs` defaults.
- Added assignment event history to `getRunTimeline()` and rendered it in the
  run-detail timeline with EN/RU labels and component/integration coverage.
- Updated roadmap and DB docs to mark M13 implemented while explicitly deferring
  `human_edit` and external hook waits to later milestones.
- Cleaned ESLint/Prettier warning noise across M13-adjacent files.

## Prevention

When a milestone introduces a new append-only ledger table, add it to every
run-detail/read-model surface in the same implementation slice and run the full
unit suite plus lint before verification. Test fakes should model Drizzle chain
shape and DB-null defaults for any table they expose.
