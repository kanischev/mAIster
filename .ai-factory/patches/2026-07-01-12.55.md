## Problem

`/aif-review` of the ADR-121 priority-ordered task queue found the admission core
sound but flagged: **(C1)** two agent idle-resume paths bypassed the pool cap,
contradicting INV-1 ("a pool's live count NEVER exceeds its cap, including under a
resume burst", both pools); **(C2/C3)** INV-3 exactly-once and the agent-pool
cap-safe resume had no tests, and the INV-1 burst test omitted the mandated
`edgeDrain`-off + agent-pool cases; **(S4/S5)** INV-5 had only a static grep and no
route-level 409 assertion; **(S2)** the web `PUT /tasks/{number}` op was undocumented;
**(S3)** `QueueSettingsControl` hand-rolled native `<select>/<input>/<button>`.

## Root Cause

The D2 over-cap bypass that ADR-121 T14 closed on the *permission* idle-resume path
was NOT closed on its two siblings — `claimAndResumeAgentRun` (hook_trip approve,
ADR-108) and `scheduleBudgetBreachResume` (budget raise, ADR-106). Both flipped a
checkpointed `NeedsInputIdle` agent run straight to `Running` in one CAS with **no
cap check**. A `NeedsInputIdle` run has *freed* its slot (it is excluded from
`countLiveRuns`), so reclaiming it without a gate lets a burst of hook approvals /
budget raises push the agent pool (cap 3) over its limit. The code comments even
confessed the duplication ("Mirrors the agent-idle HITL resume CAS") — a known-drift
smell the project's own rules say to treat as must-fix. T14 scoped itself narrowly to
the permission path and left the two siblings unswept.

## Solution

Extracted ONE shared `claimAgentResumeSlot(db, runId)` that both wakes call: under the
scheduler advisory lock it reads status, flips `NeedsInput` directly (slot still held —
slot-neutral), and for `NeedsInputIdle` cap-gates the reclaim — at cap it stamps
`resume_requested_at` and returns `"queued"` (deferring to the unified C3 admission
gate, which already admits agent resumes cap-safely one-per-freed-slot), else flips to
`Running`. Same cap-safe contract T14 gave the permission path, now DRY across all
three human-answered-idle resume paths.

**Sibling sweep (per the project's fix-the-class rule).** Grepped every
`NeedsInputIdle → Running` (and slot-reclaiming) transition:
- FIXED (in scope — human-answered-idle resume): `claimAndResumeAgentRun`,
  `scheduleBudgetBreachResume` (agent branch).
- ALREADY cap-safe: `resumeRun` / the permission `NeedsInputIdle` branch (T14),
  the `promoteNextPending` gate itself.
- DELIBERATELY OUT OF SCOPE (orchestrator-engine transitions, ADR-098/099 —
  cap-gating them could deadlock a run-tree): `resumeDelegatedChild`
  (`agents/launch.ts` — delegation wake with a return contract the caller expects to
  start), `markResumedFromWait` (`WaitingOnChildren → Running`), `markReworkFromReview`
  (`Review → Running`). These reclaim a freed slot with no cap gate too, but belong to
  the orchestrator's own cap-accounting model and need an ADR-level decision, not an
  ADR-121 hotfix. **Recorded here as a known residual.**

Tests: added the mandated INV-1 burst matrix (agent pool + `edgeDrain` off), INV-3
double-fire on C1 and C3 run sources + claim-then-crash no-double-mint, an INV-5
behavioral no-effect test, a real-PG service test of the new cap-gate
(`hitl-agent-resume-cap`), and a 409-cycle assertion on the ext/agent-token relations
route. Docs: flipped one stale `Designed` tag; added the `putTask` OpenAPI op
(redocly: only the 2 pre-existing `StudioAssistantActionResult` errors remain). UI:
reworked `QueueSettingsControl` onto HeroUI `Select`/`Input`/`Button` with dirty-state
Save gating + a green-check success glyph, mirroring `delivery-policy-settings-control`.

## Prevention

- When a fix closes a bypass/guard on ONE call path, grep the WHOLE defect class in
  the SAME pass and classify every sibling as fixed / already-safe / out-of-scope —
  and RECORD the classification. A `// Mirrors X` comment is a confession of drift,
  not documentation: treat it as a must-sweep marker.
- A stated invariant (INV-N in a `system-analytics` Expectations section) is a review
  checklist item: every code path that could violate it must be enumerated, not just
  the one the feature author had in mind. "Unconditional / both pools" wording means
  *all* paths, including sibling subsystems (hook_trip, budget-raise, orchestrator).
- `NeedsInputIdle`, `WaitingOnChildren`, and `Review` are all slot-FREED states
  (excluded from `countLiveRuns`); ANY transition from them to `Running`/`NeedsInput`
  reclaims a slot and MUST be cap-gated or consciously exempted with a recorded reason.
- Doc audits that grep column names must try BOTH snake_case and the Drizzle
  camelCase field name — `database-schema.md` documents columns in camelCase, so a
  snake_case grep yields false-negative "missing column" findings.
