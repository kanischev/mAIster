# Guardrail halt swallowed on checkpoint EXECUTOR_UNAVAILABLE (ADR-108 hook-trip)

**Date:** 2026-06-24 03:08
**Files:** web/lib/runs/hook-trip.ts, web/lib/flows/runner-agent.ts, web/lib/agents/launch.ts, web/lib/runs/__tests__/hook-trip-escalate.integration.test.ts, web/lib/flows/__tests__/runner-agent-hooks.test.ts, web/lib/agents/__tests__/agent-hook-trip.integration.test.ts, .ai-factory/specs/feature-guardrail-hook-engine.md
**Severity:** high

## Problem

A halting guardrail trip (`repetition` / `no_progress`) could be silently
swallowed. When `escalateHookTrip`'s pre-tx checkpoint returned
`EXECUTOR_UNAVAILABLE` (supervisor 5xx / network), the function returned
`{ escalated: false }` with **no state mutation**. Both consumers then treated
this as benign:

- The flow consumer un-claimed the trip and fell through to
  `ok = !persistFailure && stopReason === "end_turn"`. A halt cancels the
  agent's tool call with no `session.exited` and no permission event, so `ok`
  computed to **`true` → the step reported success and the run advanced**.
- The standalone-agent consumer **discarded the result entirely** and relied on
  a `session.exited{reason:"checkpoint"}` that never fires when the checkpoint
  itself failed → the run later finalized `Done` or hung.

The supervisor halts a session **exactly once** (`acp-client.ts` `hookHalted`;
asserted by `guardrail-interceptor.integration.test.ts:218` — "trips once, then
short-circuits every later call"). So the web-side comment's "crash-reconcile /
retry next signal" safety net did not exist: there is no second
`session.hook_trip`, and crash-reconcile does not reclaim a `Running` run whose
session is still live in the supervisor. The safety mechanism the branch adds
was defeated by a transient 503.

## Root Cause

A **single sentinel return (`{ escalated: false }`) conflated two
fundamentally different outcomes**: "nothing to escalate" (run already
advanced / not Running / lost CAS — benign) and "I could NOT durably record a
LIVE halt" (checkpoint 5xx — the agent is halted but stranded). The consumers
only had the benign interpretation wired for a resolved `false`; only a *thrown*
escalate reached the recoverable-CRASH path.

The deeper trap: the pattern was copied from the budget watchdog
(`actBudgetEscalate`), which bails the same way on `EXECUTOR_UNAVAILABLE` — but
that is **safe for budget and unsafe for hooks**. The budget watchdog is
periodic and re-fires on the next token tick; the hook halt fires once and never
re-fires. Mirroring an escalation helper without re-checking whether the caller
has a retry driver carried the bug across.

## Solution

Make `escalateHookTrip` **throw** the `EXECUTOR_UNAVAILABLE` error instead of
returning `{ escalated: false }`: if it cannot durably escalate a live halt, that
is exceptional, not a benign no-op. No state is mutated before the throw, so
there is still no split-brain. Both consumers then converge on the **already
existing, already tested** recoverable-CRASH path:

- Flow consumer: its `.then(_, onRejected)` handler already sets
  `hookEscalateFailed = true` → `runAgentStep` surfaces `CRASH` → runFlow marks
  `Crashed` (recover → `session/resume`). Zero behavioral change there — only
  the comments were broadened to name the new rejection cause.
- Agent consumer: wrapped `escalateHookTrip` in try/catch → on throw,
  `finalizeAgentRun(runId, "Crashed", …)` and detach. The run surfaces as a
  recoverable `Crashed` for a human, never a silent `Done`.

`{ escalated: false }` now means **only** "nothing to escalate" (run gone / not
Running / lost CAS). The frozen SDD spec §9 gained a crash-window bullet (d) and
its stale migration number (`0063` → `0066`, the as-built file on disk) was
reconciled — the latter the second Codex finding.

Chose throw-to-CRASH over (1) retry-until-durable (adds backoff loops inside the
SSE consumers, can block event draining — rejected on KISS) and (2) a
`{escalated:false, stranded:true}` discriminated result (adds a *second* CRASH
trigger alongside the existing thrown path — rejected on DRY).

## Prevention

- **A sentinel value that encodes "success absent" must not also encode "I
  failed to do the critical thing."** When a helper for a safety/irreversibility
  action cannot complete its effect, throw (or return a distinct, consumer-handled
  variant) — never the same value used for a benign no-op. Audit every consumer's
  reaction to the sentinel, not just the producer.
- **A halt/escalation that fires exactly once has no retry driver — its single
  failure is terminal.** Before copying a `bail-on-EXECUTOR_UNAVAILABLE` pattern
  from a sibling escalation helper, check whether the new caller re-fires (a
  periodic watchdog) or fires once (a one-shot supervisor halt). Same code,
  opposite safety.
- **"The reconcile/crash sweep will handle it" must name the exact state the
  sweep filters on.** Here the sweep reclaims `Running` runs with *no live
  session*; a `Running` run whose session is still live but halted is invisible
  to it. A recovery-sweep justification is only valid if the stranded state
  actually reaches the sweep's predicate — assert it with a test.
- Tests for any "could not complete the side effect" branch must assert the run
  **cannot advance to success/Done** — assert the terminal status, not just the
  absence of a happy-path artifact.
