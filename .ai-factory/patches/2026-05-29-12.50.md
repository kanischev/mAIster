# Trust-boundary regression tests for the two M9 mutation routes

**Date:** 2026-05-29 12:50
**Files:** web/app/api/runs/__tests__/route.trust-boundary.integration.test.ts, web/app/api/projects/[slug]/tasks/__tests__/route.integration.test.ts
**Severity:** medium

## Problem

`/aif-verify` flagged a blocking gap under the project's "failure-class
regression coverage" rule: the **trust-boundary** class applied to two M9
mutation routes that had **no regression test** —
`POST /api/runs` (project membership derived from `taskId` server-state) and
`POST /api/projects/[slug]/tasks` (body `flowId` / `executorOverrideId` must
belong to the slug's project). The route code enforced both correctly, but
nothing locked that behavior against regression.

## Root Cause

Not a code defect — a test-coverage gap. The routes were authored with the
correct server-state derivation + RBAC, and unit-level authz tests cover the
ordering matrix, but no route-level test asserted the end-to-end deny path
(non-member / cross-project id rejected). The project's evolved rule treats an
applicable failure class without a matching test as blocking, precisely because
untested trust-boundary behavior tends to silently regress.

## Solution

Two testcontainers integration tests using the REAL authz layer (only `@/auth`
and `getDb` are mocked, so the actual `getProjectRole` / membership lookup runs
against a real Postgres):

- `runs/route.trust-boundary`: anonymous → 401; a member of project A launching
  a `taskId` that lives in project B → 403 `UNAUTHORIZED`, with assertions that
  no `runs` row was created and the task stayed `Backlog` (the deny precedes any
  worktree/spawn side-effect, and the project is derived from the task, not the
  body).
- `projects/[slug]/tasks/route`: non-member → 403; a `flowId` from a different
  project → 422 `CONFIG`; an `executorOverrideId` from a different project →
  422; a member with this project's flow → 201 (`Backlog`/`Backlog`).

No route/authz/schema code changed.

## Prevention

- When adding an RBAC-guarded route whose handler derives a cross-resource id
  from server state, add a route-level deny test in the same change — assert the
  "names a different resource" request is rejected AND no side-effect row is
  written. Unit authz tests are necessary but not sufficient; the route wiring
  needs its own assertion.
- `/aif-verify`'s failure-class enumeration is the backstop: every flagged
  route must show a matching test in the report before verify passes.
