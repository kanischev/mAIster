# M19 recover/GC: classifier drift between UI and backend + a swallowed GC failure

**Date:** 2026-06-02 02:50
**Files:** web/lib/runs/recover-classify.ts (new), web/lib/runs/recover.ts, web/lib/queries/run.ts, web/lib/queries/__tests__/run-recoverable.test.ts, web/lib/runs/__tests__/recover-classify.test.ts, web/lib/gc/revision-gc.ts, web/lib/gc/sweeper.ts, web/app/api/cron/gc/route.ts, web/app/api/cron/gc/__tests__/route.integration.test.ts, web/lib/gc/__tests__/revision-gc.integration.test.ts
**Severity:** medium

## Problem

A second Codex adversarial pass on the M19 branch found two real defects that
survived the first fix round:

- **#1 — the Recover UI hid a backend-supported recovery path.** Run-detail
  gated the Recover button on `detail.recoverable`, computed by
  `isRunRecoverable` (queries/run.ts), which returned true ONLY for
  `Crashed + acpSessionId + ai_coding`. But the actual backend driver
  `resumeCrashedRun`/`driveResume` classifies recovery via `classifyRecover`
  (recover.ts), which redispatches EVERY session-less node
  (cli/check/judge/guard/human/null) non-destructively. So a Crashed
  check/judge/cli run was shown as discard-only and the operator was pushed
  toward Abandon/GC even though Recover would have re-dispatched it — directly
  contradicting `reconciliation-gc.md:288` ("Recoverable via explicit human
  Recover (accepted-risk re-dispatch)").
- **#2 — cron GC reported a clean 200 when a revision cache dir failed to
  delete.** `runRevisionGcSweep` deleted the `flow_revisions` row in-tx, then
  `rm(installedPath).catch(log)` — swallowing the failure and still doing
  `deleted++`. `RevisionGcSummary` had no failure axis, so a cache-dir rm
  failure became `200 { revisionsRemoved: 1, errors: [] }`, violating the
  documented "any sub-sweep partial failure → 207" flow
  (`reconciliation-gc.md:149-152`) and hiding disk leaks from cron monitoring.

## Root Cause

Both are **duplicated-decision drift** — the same judgment was encoded twice and
the copies diverged.

- #1: recoverability was implemented as a SECOND, hand-written predicate
  (`isRunRecoverable`) instead of being derived from the single backend
  classifier (`classifyRecover`). The two were written to different mental
  models: the UI predicate to "can we `--resume`?" and the backend to "what
  recovery plan applies?". The UI's narrower rule silently shadowed the
  backend's redispatch path. Worse, the unit test `run-recoverable.test.ts`
  asserted the narrow (buggy) behavior verbatim ("non-agent node kinds are
  never recoverable"), cementing the drift so it stayed green.
- #2: the previous fix round (patch 2026-06-02-02.33) surfaced WORKSPACE
  partial failures as 207 but left the symmetric REVISION partial-failure path
  swallowed — the `rm` error was logged-and-dropped and the summary type had no
  field to carry it, so the cron route had nothing to turn into a 207.

## Solution

- **#1**: extracted the pure classifier into `web/lib/runs/recover-classify.ts`
  (`classifyRecover` + `RecoverPlan` + `NodeKind`, no `server-only`). `recover.ts`
  re-exports it (stable import path for the route + existing tests). `run.ts`
  now computes `recoverable = status === "Crashed" && classifyRecover(...) !==
  "discard-only"`, so the UI affordance is DERIVED from the same function the
  backend executes — they cannot drift again. Inverted the drifted unit
  assertions to the corrected contract (session-less + null nodes → recoverable
  via re-dispatch; only `ai_coding` without `acpSessionId` → discard-only) and
  updated the stale field/predicate doc comments.
- **#2**: added a `failed` count to `RevisionGcSummary`; the sweep now `try/catch`-
  es the `rm` and increments `failed` (keeping `deleted++` because the DB row IS
  gone — the orphaned dir is a separate axis). Added an injectable `opts.rm` seam
  so the failure path is deterministically testable. The cron route appends an
  `errors[]` entry when `revision.failed > 0` → 207, mirroring the workspace
  handling. Patched the `sweeper.ts` fallback summary literal for the new field.

## Prevention

- A user-facing capability flag (e.g. "can recover") MUST be derived from the
  same function that performs the capability, never re-implemented. If you find
  yourself writing a second predicate that "should match" existing logic,
  extract the logic into a shared pure module and call it from both sides.
- When a contract has symmetric arms (workspace sweep vs revision sweep, resume
  vs redispatch), fixing one arm MUST include an audit of the others — a partial
  fix leaves the same class of bug one file over.
- A test that asserts current behavior is only a regression guard if that
  behavior is the CONTRACT. When the contract lives in docs/another module
  (`classifyRecover`, `reconciliation-gc.md`), assert against the contract;
  re-encoding the buggy behavior makes the test complicit in the drift.
- A swept resource with a post-commit side effect (DB delete → fs rm) needs a
  failure axis on its summary; logging-and-swallowing makes partial failure
  invisible to the monitor that reads the status code.
