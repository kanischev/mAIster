# M8 review fixes — sweeper split-brain, idle resume driver, claim-before-spawn

**Date:** 2026-05-29 13:30
**Files:** web/lib/runs/keepalive-sweeper.ts, web/lib/runs/state-transitions.ts, web/lib/runs/resume.ts, web/lib/runs/resume-driver.ts (new), web/app/api/runs/[runId]/hitl/[hitlRequestId]/respond/route.ts, web/lib/runs/__tests__/keepalive-sweeper.test.ts (new), web/lib/runs/__tests__/resume.test.ts (new), web/app/api/runs/[runId]/hitl/[hitlRequestId]/respond/__tests__/route.test.ts
**Severity:** critical

## Problem

Codex adversarial review of the M8 worker-lifecycle branch surfaced
three lifecycle blockers that would have shipped with the milestone:

1. **Sweeper split-brain on supervisor unavailability** —
   `web/lib/runs/keepalive-sweeper.ts` treated any `listSessions()`
   failure as proof that no sessions were live. A transient
   supervisor/network outage would move `NeedsInput` rows into
   `NeedsInputIdle`, release scheduler slots, and allow `--resume`
   spawns even while the original worker was still alive holding the
   pending permission deferred.
2. **Idle HITL resume returned 202 with nothing actually in
   progress** — `/respond` idle branch called `resumeRun()` (spawned
   the supervisor session) and returned 202, but no code path opened
   the SSE consumer, sent a prompt to wake the adapter, or
   auto-delivered the stored intent against the re-issued
   `requestId`. The operator's response could sit indefinitely with
   `respondedAt = null`.
3. **Resume spawned before winning the atomic claim** — `resumeRun()`
   called `createSession()` BEFORE `markResumed()`. Two same-payload
   `/respond` retries could both succeed at the HITL claim CAS, both
   spawn duplicate supervisor sessions, and only one would win the
   `NeedsInputIdle → NeedsInput` flip. The loser surfaced a terminal
   410 to the operator while the duplicate worker was never deleted.

## Root Cause

For #1: the `loadSupervisorSessions()` helper returned `null` on
failure and `runPass1` interpreted `null` as "no live sessions found"
instead of "supervisor unreachable, abort tick". The original M8
design intended both branches but conflated them in code.

For #2: the M8 plan documented "Phase 3 happens asynchronously over
the next 5-60 s" inside the runner-agent, but the runner-agent's
permission-handling lives inside `runAgentStep` which is only entered
through `runFlow` at step entry. After `resumeRun`, no actor was
holding the new supervisor session — the existing runner-agent state
machine assumed the session was created by a fresh agent-step entry,
which never happened on resume.

For #3: classic order-of-operations bug. The plan D7 table described
the success-path order as "createSession → markResumed", which
matched the sequential code structure but failed to enumerate the
two-callers-race scenario. The atomic claim must precede any
external side effect to make idempotent retries safe.

## Solution

**Fix 1 (sweeper, abort on listSessions failure):** the early-return
in `runPass1` when `supervisorMap === null` skips both
`checkpointSession` and `markCheckpointed`. Candidates remain in
`NeedsInput` and the next tick re-attempts. A regression test in
`web/lib/runs/__tests__/keepalive-sweeper.test.ts` asserts
`markCheckpointed`, `releaseSlotOnIdle`, and `checkpointSession` are
NOT called when `listSessions()` rejects.

**Fix 2 (resume driver):** new
`web/lib/runs/resume-driver.ts:runResumedSession` opens
`streamSession()`, sends a continuation prompt, watches for
`session.permission_request` with an
`MAISTER_RESUME_PROMPT_TIMEOUT_SECONDS` watchdog, auto-delivers the
stored intent via `deliverPermission`, marks the original
`hitl_requests.respondedAt` with audit
`{originalRequestId, reissuedRequestId, deliveredViaResume:true}`,
transitions the run to `Review` on `end_turn`, and calls
`crashResumedRun` / `failResumedRun` on the failure paths. The
driver is scheduled via `queueMicrotask` from `/respond` after a
successful `resumeRun`, so the route still returns 202 immediately.
Without this driver the 202 response was a lie.

**Fix 3 (atomic claim before spawn):** `resumeRun()` reordered so
`markResumed` (status-guard `NeedsInputIdle → NeedsInput`) runs
BEFORE `createSession`. Lost-claim race returns a distinct
`code: "CLAIM_RACE"` outcome that `/respond` maps to 202 instead of
410 — concurrent resume is in progress, not terminal failure.
Retryable supervisor 5xx after the claim invokes the new
`rollbackResumedRun` helper (`NeedsInput → NeedsInputIdle`) so the
slot returns to a recoverable state. Terminal supervisor errors
(400/404/empty-acpSessionId) transition `NeedsInput → Failed` via
the widened `failResumedRun` guard.

## Prevention

- Every external side effect in a lifecycle path needs an
  atomic claim BEFORE the side effect. Document the order
  explicitly in plan tables — "side effect first, claim second"
  is almost always a race.
- A response code communicating progress (202) must correspond to
  an actually-scheduled background task. Returning 202 without
  scheduling work is a contract lie that hides forever.
- When a sweeper or reconciler depends on an external authoritative
  source (`listSessions()`), distinguish "source said no" from
  "source unreachable" with two separate code paths.
- Reviews should target route handlers with three lenses: (a) order
  of DB writes vs external calls, (b) what background work is
  responsible for completing the contract after a 202, (c) what
  happens if the external call fails halfway through.
- Add a regression test for "external dependency throws" on every
  reconciler/sweeper path before merging.
