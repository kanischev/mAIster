# Artifact body injection (ADR-120) — adversarial-review round 3 fixes

**Date:** 2026-07-01 12:38
**Files:** web/lib/flows/graph/gates-exec.ts, web/lib/worktree.ts, web/lib/flows/graph/artifact-inject.ts (+ tests: artifact-inject.test.ts, worktree-range.test.ts, artifact-inject.integration.test.ts)
**Severity:** high

## Problem

A third Codex adversarial-review pass on the ADR-120 artifact-body-injection
feature found three residual holes:

1. **[high] Missing-current gate content refs bypass controlled gate failure.**
   `resolveNodeArtifactContents` leaves `.content` UNSET for an included
   (non-skipped) gate ref whose artifact has no `current` row (`!row → continue`).
   Because `runNodeGates` creates the `gate_results` row `running` BEFORE the
   render and had no per-gate try/catch, an unguarded `{{ artifacts.missing.content }}`
   in a non-skipped `command_check`/`skill_check`/`ai_judgment` threw out of
   `runOneGate`, leaving the gate row stuck `running` and failing the run through
   the top-level catch — misleading recovery/readiness.

2. **[high] Bounded git-log injection ignored git exit failures.** The shared
   `streamGitTruncatedTo` resolved on stdout `end` without awaiting `close` or
   checking the exit code/stderr. A valid-shaped but nonexistent ref (or corrupt
   repo) — which writes stderr and exits non-zero WITHOUT us killing it — was
   treated as `{ text: '', truncated: false }`, so `logRangeBounded` silently
   omitted required commit evidence from an injected prompt instead of failing
   `CONFLICT` like `logRange`.

3. **[medium] Content scanner resolved unused action fields.**
   `collectContentArtifactIds` scanned BOTH `action.prompt` and `action.command`
   for every node, but execution is type-specific (`ai_coding`/`judge`/
   `orchestrator` render only `prompt`; `cli`/`check` render only `command`). The
   action schemas are `.passthrough()`, so a leftover `action.command` on an agent
   node forced the 2.2.0 floor / resolved a ref the executor never renders —
   violating the feature's own load-time/runtime parity goal.

## Root Cause

1. The prior round split content refs into skip-aware inclusion (excluding
   `checks=skip` gates), but for INCLUDED gate refs it still relied on the
   downstream `renderStrict` to surface a missing ref — and the gate execution
   path (unlike `executeNodeAction`) had no surrounding catch, so a domain throw
   escaped uncontrolled. Deferring a possible-throw to a code path that lacks a
   controlled-failure boundary is the defect.

2. A streaming reader that resolves on `stdout.end` conflates "stream finished"
   with "process succeeded." For a two-phase reader (buffered-then-stream, like
   diff) the buffered phase catches git failures first; but the DIRECT streamer
   (`logRangeBounded`) has no such phase, so a non-zero exit with empty stdout
   read as success. Exit status must gate the result.

3. A scanner mirroring an executor must mirror its FIELD SELECTION, not scan
   every passthrough field. Passthrough schemas mean "unused field" is
   representable, so scanning it introduces load-vs-runtime drift.

## Solution

1. **Per-gate render/exec guard.** Added `runGateStepGuarded(gateResultId, db,
   run)` in gates-exec: it runs the gate's `runCliStep`/`runAgentStep`, and on a
   thrown `MaisterError` calls `markGateFailed` (controlled terminal) and returns
   `null` so the caller `return "failed"` — `runNodeGates` then routes by
   blocking/advisory mode. Anything non-domain rethrows. This keeps the tolerant
   pre-resolution (so guarded `?? ''` gate refs still render a default and
   advisory gates stay non-blocking) while guaranteeing no leaked `running` row.

2. **`streamGitTruncatedTo` awaits `close` + branches on exit.** Track a
   `truncated` flag (we SIGKILL on purpose at the byte bound) and a bounded
   stderr buffer; resolve/reject only on `close`: `truncated` → success,
   `code === 0` → full success, otherwise reject `CONFLICT` with stderr. Shared by
   the diff-overflow fallback and `logRangeBounded` (both now fail loudly on real
   git errors).

3. **Type-aware action scan.** `collectContentArtifactIds` scans `action.prompt`
   only for `PROMPT_ACTION_NODE_TYPES` (ai_coding/judge/orchestrator) and
   `action.command` only for `COMMAND_ACTION_NODE_TYPES` (cli/check); an
   unknown/absent type scans both (defensive callers).

Logging via existing module-local `pino` (no `console.log`, per the `no-console`
lint rule); the gate guard's failure is recorded on the `gate_results` row.

## Prevention

- **A deferred throw needs a controlled-failure boundary at its destination.**
  Before deferring a possible domain throw to a later code path, verify that path
  wraps it into the domain's terminal FSM — else it escapes to a generic catch and
  leaks half-created rows. Prefer converting at the boundary that owns the row.
- **Stream readers must gate on process exit, not stream end.** For any spawned
  reader, resolve only after `close` and reject non-zero exits (distinguishing an
  intentional kill from a failure). Regression-test a valid-shaped nonexistent ref.
- **A scanner that mirrors an executor mirrors its field selection.** With
  `.passthrough()` schemas, scan exactly the fields the executor reads for that
  node type; test that an unused passthrough field triggers neither collection nor
  engine-floor gating.
