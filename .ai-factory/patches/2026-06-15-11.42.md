# Dirty Gate Diff Visibility

**Date:** 2026-06-15 11:42
**Files:** `web/app/(app)/runs/[runId]/layout.tsx`, `web/components/workbench/run-diff.tsx`, `web/components/runs/dirty-resolution-banner.tsx`, `web/messages/en.json`, `web/messages/ru.json`
**Severity:** medium

## Problem

A run paused on a human review gate could show a dirty-resolution banner with
uncommitted and untracked files, while the run inspector "Changes" tab showed
"No changes" and the actions panel mostly showed disabled lifecycle actions.
Deep links to the diff view did not reliably open the working-tree diff because
the client diff component always initialized to the `run` scope.

## Root Cause

The dirty banner and the inspector used different change sources. The banner
read the live worktree status through `statusPorcelain` and
`computeDirtySummary`, while the inspector always loaded
`getRunChangeSummary({ scope: "run" })`, which only describes the committed
base-to-branch range. The workbench diff API already supported
`scope=uncommitted`, including untracked files, but the client component ignored
the URL scope when initializing its local state.

## Solution

When a dirty human review gate is active, the run layout now loads the inspector
summary from `scope=uncommitted` and preserves that scope in inspector file
links. `RunDiff` initializes from the URL `scope` parameter, so
`?wb=diff&scope=uncommitted` opens the working-tree diff. The dirty banner and
inspector actions now expose explicit shortcuts to the uncommitted diff,
pending input, and agent chat, while disabled lifecycle actions are filtered
out unless they are currently meaningful.

## Prevention

Any run-detail surface that displays dirty worktree state must choose between
committed run range and working-tree scope deliberately. During review of run
UI changes, check that links preserving `scope=uncommitted` are consumed by the
client component, not only emitted by the server-rendered page.
