# Inbox-card lazy context: legacy progress, silent degradation, untested expand

**Date:** 2026-06-19 00:02
**Files:** web/lib/queries/inbox-context.ts, web/lib/queries/hitl-stage.ts, web/lib/queries/__tests__/inbox-context.integration.test.ts, web/e2e/inbox.spec.ts, docs/system-analytics/hitl.md
**Severity:** medium

## Problem

A Codex adversarial review of the inbox-card-redesign branch surfaced four
defects, all spec-compliance gaps against `.ai-factory/plans/feature-inbox-card-redesign.md`:

1. **Legacy linear runs always rendered `0/N` progress.** `loadProgress`
   computed `total` from the compiled manifest but counted "done" only from
   `node_attempts`. The plan required "Legacy run → board `buildSpine` over
   `step_runs`".
2. **Read failures were silently swallowed.** All four `inbox-context` loaders
   `catch {}` → `[]`/`null` with no log; a broken gate read was
   indistinguishable from a healthy node with no gates. The plan's Settings
   mandated "WARN (server) on `inbox-context` read failures and on an
   unresolved stage `step_id`".
3. **The expanded tier was untested.** The card unit test (static
   `renderToStaticMarkup`) explicitly deferred expand→fetch→render to e2e, but
   the e2e only toggled `aria-expanded` — a broken lazy fetch or expanded
   renderer would have passed.
4. **A doc named deleted components.** `hitl.md`'s "WI-1 (Implemented)" section
   still said `/inbox` reuses `HitlInboxBlock` + `InboxRespond` (deleted),
   contradicting the same file's redesign section.

## Root Cause

- **#1:** `runFlow` (`web/lib/flows/runner.ts:184`) dispatches graph (`nodes[]`)
  → `runGraph` → `node_attempts`, and legacy (`steps[]`) → the linear path →
  `step_runs`, with a `return` making them mutually exclusive. So a legacy run
  has **zero** `node_attempts`. The new code counted only one of the two ledgers.
  (Aggravating: the `node_attempts` schema comment at `schema.ts:1702` *claims*
  linear flows "write here too" — it's stale and contradicts the runtime
  dispatch, which likely misled the original author.)
- **#2:** Degradation was implemented as the spec required (never 500) but the
  paired observability half (WARN) was dropped, collapsing failures into the
  benign empty state.
- **#3:** Coverage was "handed off" to a layer (e2e) that never received it —
  the unit test correctly cannot exercise effects (`environment: node` +
  `renderToStaticMarkup`, no jsdom in the project), but the e2e didn't fill the
  gap. The integration test for progress only seeded the graph ledger, so #1
  was invisible to it too.
- **#4:** A superseding section was added next to the old one without retiring
  the old prose; the deleted component names were left behind.

## Solution

- **#1:** Branch `loadProgress` on `isLinear`
  (`Array.isArray(manifest.steps) && manifest.steps.length > 0`, the same
  discriminator as `resolveCurrentNodeContext`); count distinct succeeded
  `step_runs.stepId` for legacy, distinct succeeded `node_attempts.nodeId` for
  graph (extracted as `countSucceededSteps`/`countSucceededNodes`). Added an
  integration case: legacy `steps[]` manifest + one succeeded `step_run` →
  `{ done: 1, total: 2 }` (real Postgres).
- **#2:** Added a `pino` logger to `inbox-context.ts` and a `log.warn` (with
  `runId`, plus `currentStepId` for the gate path) in every loader's `catch`.
  Kept the `gates[]` (non-null) contract — the plan deliberately chose
  `[]`+WARN degradation, not a nullable contract; making `gates` nullable would
  be an out-of-plan OpenAPI + UI change.
- **#3:** Strengthened the Playwright e2e: a `page.route` mock of
  `inbox-context` makes the expanded gates/message/progress/diff render
  deterministic; assert the fetch fires (`waitForResponse`) and the four
  sections render; a second test forces a 500 then 200 to assert the
  `role="alert"` + working retry. (CI-executed like the rest of the harness;
  the Next 16 single-dev-server lock prevents running e2e in the same session.)
- **#4:** Rewrote the WI-1 paragraph to name `HitlInboxList` / `HitlCard` and
  the reused `run-hitl-response.tsx`, pointing to the redesign section (R7
  cross-reference), and fixed the sequence-diagram render line.

## Prevention

- When a read model joins a "ledger", confirm **which** ledger the executor
  actually writes for each run shape. `runFlow` keeps legacy on `step_runs` and
  graph on `node_attempts`; a stale schema comment claimed otherwise. Trust the
  dispatch (`runner.ts`), not the comment.
- "Degrade, never 500" is only half a contract — the other half is a WARN so
  the failure is observable. Ship both or neither.
- A test that "defers coverage to another layer" is a debt, not coverage.
  Verify the receiving layer actually asserts it. For interaction behind an
  effect/fetch in a `renderToStaticMarkup` (no-jsdom) project, the e2e (with a
  `page.route` mock) is the home — assert the network call AND the rendered
  result AND the error/retry path.
- When superseding a doc section, retire the old prose in the same edit; a
  deleted symbol name left in an "Implemented" section is a self-contradiction.
