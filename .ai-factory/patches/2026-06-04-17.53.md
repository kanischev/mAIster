# M18 promotion: PR-adapter repo context, drift-override decoupling, Gitea pagination

**Date:** 2026-06-04 17:53
**Files:** web/lib/runs/pr-adapter.ts, web/lib/runs/promote.ts, web/lib/runs/__tests__/pr-adapter.test.ts, web/lib/runs/__tests__/promote-service.test.ts, web/app/api/runs/[runId]/promote/__tests__/route.test.ts
**Severity:** high

## Problem

Three Codex adversarial-review findings against the M18 branch promotion path:

1. **(high) gh/glab ran against the wrong repository.** `CliPrAdapter.exec()`
   hardcoded `cwd: undefined`, so `gh`/`glab` inferred the repo from the
   MAIster web server's own process CWD instead of the target project's
   checkout — even though `repoPath` was already threaded through
   `CreateOrUpdatePrArgs`. A PR promotion could list/create PRs against the
   wrong repo (or fail). Tests recorded `opts` but never asserted `cwd`.
2. **(high) `allowTargetDrift` allowed blind promotion + skipped target
   existence.** The `reviewedTargetCommit`-required check AND the
   `resolveBaseCommit()` existence/resolve call were both nested inside
   `if (!input.allowTargetDrift)`. Setting `allowTargetDrift: true` therefore
   promoted with no reviewed SHA and no proof the target branch existed. A
   missing target then misclassified as a merge `CONFLICT` and opened a bogus
   "resolve merge conflict" assignment. This contradicted the OpenAPI contract
   (`reviewedTargetCommit` always required; `allowTargetDrift` skips only the
   SHA-equality check).
3. **(medium) Gitea PR idempotency only checked the first page.**
   `findOpenPr()` fetched one `state=open&limit=50` page; an existing PR beyond
   page 1 (busy repo / crash-window retry) was invisible, so the "idempotent"
   re-promote could POST a duplicate.

## Root Cause

1. The `repoPath` field existed in the args type and was passed by the caller,
   but the adapter never consumed it for the actual `gh`/`glab` invocation —
   `cwd: undefined` was a leftover that defeated the whole repo-targeting
   design. `gh`/`glab` are CWD-sensitive; relying on the parent process CWD is
   silently wrong when the server runs anywhere but inside the target repo.
2. Three distinct concerns were conflated into one conditional: (a) the audit
   requirement that a reviewed SHA was supplied, (b) proving the target exists
   by resolving it, and (c) the optimistic-concurrency SHA-equality check.
   Only (c) is what `allowTargetDrift` is meant to waive; bundling (a) and (b)
   under the same flag turned an "ignore a known SHA mismatch" override into a
   "skip all target validation" bypass — drifting the code away from the
   already-correct OpenAPI prose.
3. A documented-but-unfixed limitation: the comment acknowledged `limit=50` was
   the per-page max and could omit a match, but the single-fetch path was never
   replaced with pagination.

## Solution

1. `exec(args, cwd?)` now accepts a cwd; every `gh pr list/create` and
   `glab mr list/create` call passes `args.repoPath`. The repo-independent
   `--version` preflight passes none. `repoPath` is trusted server state
   (`workspace.parentRepoPath`) already validated via `absolutePathSchema` by
   the preceding `pushBranch` in the same side-effect sequence.
2. Restructured the flow-promotion gate so `reviewedTargetCommit` is required
   on EVERY promotion and `resolveBaseCommit()` ALWAYS runs (proving existence
   before the claim or any side-effect). `allowTargetDrift` now guards ONLY the
   `liveTip !== reviewedTargetCommit` equality throw. This makes the code match
   the OpenAPI contract rather than the other way round — a missing target is
   now a clean `PRECONDITION` ("base ref does not resolve to a commit"), never
   a misclassified merge-conflict assignment.
3. `findOpenPr()` pages through `&page=N` (limit=50) until the (head,base)
   match is found or a short page (`< limit`) signals the last page, bounded by
   a `PR_LOOKUP_MAX_PAGES` (200) sentinel. Exceeding the cap REFUSES
   (`EXECUTOR_UNAVAILABLE`) instead of returning null — so it bounds a
   never-terminating server without reintroducing the silent-truncation
   idempotency gap (a refusal never creates blind). *(Cap added in the
   /aif-review follow-up; the original wave shipped uncapped.)*

Regression tests added: gh/glab `cwd === repoPath` on every invocation; Gitea
match-on-page-2 with no duplicate POST; `allowTargetDrift:true` without
`reviewedTargetCommit` → PRECONDITION (no git, no claim);
`allowTargetDrift:true` with a non-resolving target → PRECONDITION existence
error, no merge-conflict assignment, no claim. Updated the route flow-dispatch
test (it had used `allowTargetDrift` to dodge the unmocked `resolveBaseCommit`)
to mock `resolveBaseCommit` and pass a matching `reviewedTargetCommit`.

## Prevention

- **A field threaded into an args type but never read at the sink is a smell.**
  When a side-effect helper shells a CWD-sensitive CLI (`gh`, `glab`, `git`),
  the test MUST assert the resolved `cwd`/`--repo`, not just the argv. "Records
  opts but never asserts it" is untested surface.
- **One override flag should waive exactly one concern.** When a boolean gates
  a block doing N things (require-input + resolve-existence + equality-check),
  splitting per concern prevents an override from silently disabling
  validation it was never meant to touch. Mirror the spec's wording: if the
  contract says "skips the equality check", the code must skip ONLY that.
- **OpenAPI prose is the acceptance checklist.** When code and an
  already-correct spec disagree, fix the code; add the spec's exact refusal
  (`reviewedTargetCommit` required; `allowTargetDrift` skips equality only) as
  a unit assertion so the drift can't recur.
- **Provider list/lookup calls must paginate or filter** — a single-page
  idempotency check is a duplicate-write waiting for a busy repo. Terminate on
  a short page, and bound the sweep with a high sentinel cap that REFUSES on
  exceed (never returns "no match") — a non-terminating server cannot then loop
  forever nor trigger a blind duplicate create.
