# Codex full-branch review fixes: agent-config version skew, intent-scoped retry cap, enqueue existing-flow

**Date:** 2026-06-25 19:58
**Files:** web/lib/agents/project-links.ts, web/lib/db/schema.ts, web/lib/db/migrations/0073_task_launch_armed_at.sql (+ meta), web/lib/services/triage.ts, web/lib/scheduler/handlers/auto-launch-triaged.ts, web/app/api/v1/ext/projects/[slug]/tasks/[taskId]/triage/route.ts, 3 test files, docs/system-analytics/{triage,agents}.md, docs/db/runs-domain.md
**Severity:** high

## Problem

Second adversarial Codex pass over the full ADR-111 branch found four holes
(F3 was a re-raise of a settled decision — see Solution):

- **F1 (version skew).** `updateAgentLink` validated per-instance agent `config`
  against `agents.config_schema` (the GLOBAL catalog row, re-keyed from the
  package's NEWEST install at every resync), while a launch resolves the agent
  from the project's PINNED package version. A catalog resync diverges the two:
  a config valid for the pinned agent gets rejected, or a newer-catalog-only key
  is persisted then silently defaulted at spawn — quietly changing triager
  behaviour.
- **F2 (retry cap counts unrelated failures).** `countFailedFlowRuns` counted
  EVERY Failed/Abandoned flow run for the task, with no scoping to the current
  enqueue intent. Old manual failures, or failures under a PREVIOUS flow, made a
  freshly re-triaged task hit the give-up cap and get flagged WITHOUT EVER
  launching the new work — which also broke the give-up's own advertised
  recovery ("re-send it to triage to retry": the old failures still counted).
- **F4 (enqueue existing-flow path).** The OpenAPI contract says `enqueue:true`
  is valid when the flow resolves from "an existing `tasks.flow_id` OR a `flowId`
  in this body", but the route rejected on `body.flowId === undefined` and never
  read the task's flow — a client following the spec got a wrongful 422 and the
  task stayed unarmed.

## Root Cause

- F1: the validation read the convenient always-present catalog projection
  instead of the authoritative pinned definition. The catalog row is a UI
  rendering aid, NOT the contract launch executes; only `resolveEffective`
  `AgentDefinition` (which reads the pinned install's `maister-agents/<stem>.md`)
  matches launch.
- F2: the cap had no notion of "current intent." A bare `count(*)` over a task's
  failure history conflates distinct intents (different flow, manual vs auto,
  before vs after a re-arm). The give-up comment even CLAIMED "auto-launched"
  while the query counted all sources.
- F4: the guard encoded "flow in THIS body" but the contract (and the comment
  right above it) meant "the task ENDS with a flow" — the existing `flow_id` was
  never consulted, and the task lookup selected only `id`.

## Solution

- **F1:** `updateAgentLink` now resolves `resolveEffectiveAgentDefinition({proj,
  agentId})` and validates against `effective.parsed.config` — the same pinned
  definition launch runs. No migration; reuses the canonical resolver (so its
  package attached+Installed+trusted gate applies, which is correct: you cannot
  meaningfully configure an agent that cannot resolve/launch). Test fixture now
  writes a real pinned `.md` via `renderAgentDefinition`; a new version-skew test
  proves a catalog-only key is rejected and a pinned-only key accepted.
- **F2:** new `tasks.launch_armed_at` column (migration 0073), stamped `= now`
  whenever `triage_set` arms `launch_mode='auto'` and cleared with it everywhere
  (verdict-without-enqueue, flag, send-to-triage). `countFailedFlowRuns` filters
  `started_at >= launch_armed_at` (null → count all, legacy-safe). The give-up
  CAS also pins `launch_armed_at` so a same-flow re-arm during a launch is not
  clobbered. (User chose the complete armed_at boundary over a flow_id-only
  filter, which would not fix the same-flow recovery loop.)
- **F4:** the route reads `tasks.flow_id` and allows enqueue when
  `body.flowId ?? task.flowId` resolves; the OpenAPI was already correct (the
  code was the drift), so only code + a regression test changed.
- **F3 (NOT implemented — settled decision):** Codex re-recommended a triage-time
  launchable-flow guard (revision installed / setup / schema / engine). Declined
  again: the tick now treats CONFIG as a terminal give-up (prior round), so a
  knowably-broken flow is HELD, not stalled. A triage-time guard duplicates the
  launch-path checks and still cannot catch a flow that breaks BETWEEN triage and
  launch — the give-up backstop is the single, drift-proof enforcement point.

## Prevention

- **Validate against the SAME source the executor reads.** When a value is
  validated at write and consumed at run, both must read the identical resolver /
  pinned artifact — never a convenient mutable projection (catalog/cache/index).
  A version pin makes "the catalog" and "what runs" two different things.
- **A retry/attempt cap MUST be scoped to the intent it bounds.** `count(*)` over
  all history conflates intents; bound it with an explicit `armed_at` (or
  intent-id) boundary so a fresh, deliberate re-arm earns a fresh budget — and so
  the documented recovery path actually recovers.
- **Code must enforce the CONTRACT, not a stricter paraphrase.** When a guard's
  comment and its OpenAPI say "resolvable from A or B" but the code checks only A,
  that is a defect; cross-check guards against the published spec, and add a test
  for the B path.
- **A new column / new give-up trigger is a contract change** — update the ERD,
  the narrative schema doc, and the domain Expectations in the same change, then
  re-run `pnpm validate:docs`.
