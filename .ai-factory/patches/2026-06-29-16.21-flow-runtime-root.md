# Flow resume must use the shared runtime root

**Date:** 2026-06-29 16:21
**Files:** web/lib/flows/runner.ts, web/lib/flows/graph/runner-graph.ts, web/lib/flows/graph/runner-core.ts, web/lib/flows/__tests__/runner-reentry.test.ts
**Severity:** high

## Problem

A fresh flow run repeatedly asked the same intake form after the user answered
it. The run stayed on the `intake` form node and accumulated multiple
`hitl_requests` rows for the same step.

The concrete run had the user response at:

```
.maister/maister-dev/runs/<runId>/input-intake.json
```

but a duplicate prompt artifact was written under:

```
web/.maister/maister-dev/runs/<runId>/needs-input.json
```

## Root Cause

HITL response handling writes input artifacts through `runtimeRoot()`, which
normalizes package cwd values like `web/` to the repository root. Background
resume entered through `runFlow()` and defaulted to raw `process.cwd()` instead.

When the Next.js server cwd was `web/`, the resumed runner looked for
`web/.maister/.../input-intake.json`, did not find the real answer, and created a
new HITL request for the same form node.

## Solution

Added `resolveFlowRuntimeRoot()` and made both `runFlow()` and the graph runner
default to the shared `runtimeRoot()` helper instead of `process.cwd()`. This
keeps HITL writes, graph reads, linear reads, and `MAISTER_RUNTIME_ROOT`
overrides on the same filesystem root.

Added a regression test that calls `runFlow()` without an explicit
`runtimeRoot`, with `MAISTER_RUNTIME_ROOT` pointing at a temp runtime directory
that already contains the human input artifact. The test asserts the runner
resumes from the artifact and does not create another HITL request.

## Prevention

- Runtime artifact readers and writers must use the same `runtimeRoot()` path
  contract.
- Do not default long-lived background runner code to bare `process.cwd()`.
- For HITL resume regressions, assert that an existing `input-<step>.json`
  prevents fresh `hitl_requests` inserts.
