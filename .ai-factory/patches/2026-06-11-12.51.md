**Date:** 2026-06-11 12:51
**Files:** docs/decisions.md, docs/api/web.openapi.yaml, docs/api/async/outbound-webhooks.asyncapi.yaml, docs/database-schema.md, docs/db/webhooks.md, docs/db/erd.md, docs/configuration.md, docs/system-analytics/outbound-webhooks.md, docs/system-analytics/scheduler.md, docs/CLAUDE.md, docs/pv/improvement-roadmap.md, CLAUDE.md, .env.example, deploy/maister.env.example, web/lib/db/schema.ts, web/lib/flows/graph/gate-store.ts, web/lib/flows/runner.ts, web/lib/flows/graph/runner-graph.ts, web/lib/scheduler/handlers/webhook-delivery.ts, web/lib/webhooks/destination.ts, + 6 test files
**Severity:** high

## Problem

`/aif-verify` over the full outbound-webhooks branch (vs merge-base 19888156)
found 3 blockers and a band of warnings AFTER the feature and a prior 6-fix
adversarial pass were "done":

1. **ADR number collision**: main merged M29 first and took ADR-075; this
   branch also claimed ADR-075 — 105 references across 20 tracked files,
   including 2 lowercase anchor slugs and a `deploy/maister.env.example`
   header that cited a *third*, unrelated ADR (ADR-071).
2. **Write+emit autocommit pairs at 3 sites** the prior atomicity fix missed:
   `createGateResult` (INSERT + `gate.decided` emit — reachable with terminal
   status via `reportExternalGate`'s supersede branch) and the
   NeedsInput-flip + `run.needs_input` emit in both `runner.ts` and
   `runner-graph.ts`. A crash between the two statements commits the state
   change but silently loses the event.
3. **DB docs asserted the opposite of the schema**: three docs claimed
   `webhook_deliveries.event_id` has NO cascade and that delivery history
   "survives independently" — migration 0040 defines ON DELETE CASCADE;
   longevity actually comes from the retention pass pruning only
   zero-delivery events.
4. Doc drift from the prior fix pass itself: kill-switch still described as a
   plain "no-op" (it became a skip pass that stamps `fanout_at`), headers
   still "MAY be env refs" (became refs-only), replay reset enumeration
   missing the outcome-field clears the fix added.
5. `finishDelivery`'s `delivered` branch left stale `last_error_kind` /
   `last_error_message` from a prior failed attempt on the row.

## Root Cause

- **Numbering**: three sibling branches all reserved "next free" ADR-075 from
  the same main snapshot; whoever merges second collides. The plan even
  documented the pre-merge re-check — but a re-check is a manual step, and
  the branch sat unmerged while main moved twice.
- **Atomicity misses**: the prior fix targeted the file the review named
  (`gate-store.transition()`) and the sites the patch log enumerated, not the
  *class*. `createGateResult` lives 30 lines above the fixed function and even
  sits under the fixed function's invariant comment; the two runner flip sites
  are textually similar to the correctly-transactional `runner-agent.ts` site
  but were wired in a different commit. Class-wide greps
  (`emitWebhookEvent` outside a tx) were prescribed by the previous patch's
  Prevention note and not actually run.
- **Cascade falsehood**: the doc was written from the *plan*'s early intent
  (no cascade was indeed DQ3's first draft) while the shipped migration used
  cascade; nobody re-derived the doc from the SQL.
- **Fix-pass drift**: behavior fixes were applied code-first under review
  pressure; the as-built docs for the *changed* semantics (skip-pass,
  refs-only, field resets) were updated only where the fixer happened to look.

## Solution

- Renumbered ADR-075 → ADR-076 across all 20 tracked files in one scripted
  pass (`sed` over an explicit file list, both `ADR-075` and the lowercase
  `adr-075` anchor slug), corrected the deploy header's wrong ADR-071 label,
  excluded `.ai-factory/plans|patches` (historical logs) and generated
  `web/.next`. Migration 0040 untouched — main gained no migration.
- Wrapped all three write+emit pairs in one `db.transaction`:
  `createGateResult` wraps INSERT + emit only when the status is terminal
  (savepoint-safe if the caller hands a tx); both runner sites wrap
  ledger-mark + status flip + emit, mirroring the `runner-agent.ts` pattern.
- Rewrote the cascade prose in `docs/db/webhooks.md` + `database-schema.md`
  (+ ERD annotation): cascade exists; the retention pass's
  zero-delivery-only predicate is what guarantees audit longevity.
- Synced kill-switch (skip-pass), headers (refs-only), and replay-reset
  (outcome fields) wording across OpenAPI/DB docs/analytics; added the
  missing `MAISTER_WEBHOOK_ALLOW_HOSTS` to `deploy/maister.env.example`;
  added the emit-sites column to the taxonomy table; added the
  disable=skip decision bullet to the ADR; flipped two stale
  "being built" → "built".
- `finishDelivery` delivered branch now NULLs `last_error_kind` +
  `last_error_message`.
- Tests added: `createGateResult` insert-at-terminal atomicity (emit failure
  rolls back the row); dual-`v1=` rotation verified on the wire against both
  secrets; 302 redirect → Location target never fetched + retryable `http`
  classification; PATCH CLEAR-half (`secondary_signing_secret_ref` → null,
  `headers` → `{}` persist); promote finalize emits pinned per path
  (local_merge / pull_request / scratch → exactly `run.promoted`,`run.done`;
  conflict/superseded paths → zero emits) via an outbox spy in the unit
  harness.

## Prevention

- **Renumber check is merge-adjacent, not implement-adjacent**: re-grep
  `git show main:docs/decisions.md` for the claimed number immediately before
  merging, every time main moves — not once at implement start. The
  reserve-the-slot renumber needs: heading, index row, lowercase anchor slugs
  (second sed pattern), and any *cross-ADR* citations in env/deploy templates
  that can silently point at the wrong decision.
- **Fix the class, not the cited line**: when a review names an autocommit
  write+emit pair, grep the whole repo for the emit helper and audit every
  call site against "is this inside the same tx as its paired write" before
  closing the fix. The previous patch SAID this and it didn't happen —
  enforcement is a grep run recorded in the fix close-out, not a sentence.
- **Derive DB docs from the migration SQL, not the plan**: any "no cascade" /
  "cascade" claim in prose must quote the actual `ON DELETE` clause from the
  shipped migration.
- **A behavior fix is not closed until its wording sweep runs**: grep docs
  for the OLD behavior's distinctive words ("no-op", "MAY be", the old reset
  field list) after changing semantics; stale phrasing in a spec is a
  contract bug, not cosmetics.
