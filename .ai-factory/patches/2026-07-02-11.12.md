# MCP facade tool schemas drifted behind their ext routes (triage_set under-forwarding, executorOverrideId ghosts) + agent token missing tasks:update

**Date:** 2026-07-02 11:12
**Files:** mcp/src/tools.ts, mcp/src/__tests__/tools.test.ts, web/types/token-scopes.ts, web/app/api/v1/ext/projects/[slug]/tasks/[taskId]/__tests__/route.integration.test.ts, web/lib/agents/__tests__/fixtures/core-package/maister-agents/triager.md, docs/system-analytics/agents.md, docs/system-analytics/external-operations.md, docs/api/external/operations.openapi.yaml
**Severity:** high

## Problem

The core:triager platform agent (ADR-112) could not perform its documented
procedure through the maister MCP facade:

1. `triage_set` forwarded only `flowId/runnerId/targetBranch/promotionMode`.
   The persona's `flag: true` hold (dedup / unroutable path) produced an EMPTY
   body → ext route 422 "at least one field is required". `enqueue`,
   `baseBranch`, and the ADR-121 `priority`/`confidence` fields were silently
   DROPPED — auto-enqueue and queue prioritization written by the agent never
   reached the platform.
2. `task_update` (clarify-mode prompt sharpening) failed 403: ephemeral agent
   tokens (`AGENT_TOKEN_SCOPES`) never carried `tasks:update` although the
   ADR-112 design (§6.2) prescribes the capability.
3. Inverse drift: `task_create` and `task_update` MCP specs advertised
   `executorOverrideId`, which both strict ext route schemas refuse with 422
   (M42 moved runner binding to `run_sessions`); the ext OpenAPI still
   documented the field on create/PATCH bodies and in `TaskDTO`.

## Root Cause

The MCP facade tool specs (`TOOL_SPECS` + the `dispatchTool` per-tool
destructuring) are a HAND-MAINTAINED mirror of the ext route zod schemas with
no shared source and no contract test tying them together. Three feature
waves changed the route side without sweeping the mirror: ADR-112 added
`flag`/`enqueue`/`baseBranch` to the triage op, ADR-121 added
`priority`/`confidence`, and M42 removed `executorOverrideId` from task
create/update — none touched mcp/src/tools.ts. The `dispatchTool` pattern of
destructuring known keys makes the drift SILENT: unknown args are dropped, so
the agent sees "success" while its fields never leave the facade. The scope
gap has the same shape: ADR-112 extended `TOKEN_SCOPES` with discovery scopes
and updated the route, but the fixed `AGENT_TOKEN_SCOPES` grant list was only
partially extended (flows:read/runners:read added, tasks:update forgotten).

## Solution

- `triage_set` inputSchema + handler now carry the full ext contract:
  `baseBranch`, `flag`, `enqueue`, `priority` (low|normal|high|urgent),
  `confidence` (0..1); description rewritten to state the verdict/flag
  mutual exclusion and the priority/confidence independence.
- `tasks:update` added to `AGENT_TOKEN_SCOPES` (maps to the pre-existing
  `editTask` action; ext PATCH body is strictly `title`/`prompt`, so the
  widened grant exposes exactly the clarify-mode capability).
- `executorOverrideId` removed from `task_create`/`task_update` MCP specs +
  handlers (run_launch keeps it — its route accepts the deprecated alias) and
  from the ext OpenAPI create/update bodies + TaskDTO.
- Tests: mcp tools.test.ts — full-surface triage_set forward, flag-only
  forward, executorOverrideId strip on both task tools (37/37 green);
  web integration — a token holding exactly `AGENT_TOKEN_SCOPES` PATCHes
  title/prompt 200 against real PG (8/8 green); triage route integration
  14/14 green; web+mcp tsc 0; scoped eslint 0; redocly external spec valid.
- Fixture `web/lib/agents/__tests__/fixtures/core-package/maister-agents/triager.md`
  re-synced byte-identical to the updated maister-plugins deliverable.

No `[FIX]` runtime logging added: the fixes are declarative contract
alignments (schemas, a scope constant, specs); repo convention (no-console,
pino-only) governs, and the behavior is locked by tests instead.

## Prevention

- When a feature changes an ext route's zod body schema, grep
  `mcp/src/tools.ts` for the corresponding tool in the SAME pass — the facade
  is a manual mirror and `dispatchTool` drops unknown args silently, so drift
  never errors on the agent side.
- Treat `AGENT_TOKEN_SCOPES` as part of any "agent gains an op" change: the
  route scope, the scope→action map, AND the fixed grant list must move
  together (the grant list is the one that gets forgotten).
- The new integration test pattern (mint a token with exactly
  `AGENT_TOKEN_SCOPES`, hit the route) is the cheap guard for grant-list
  regressions — reuse it when adding agent-reachable ops.
- Consider (roadmap) generating TOOL_SPECS from the ext OpenAPI or adding a
  contract test that diffs tool properties against route zod schemas.
