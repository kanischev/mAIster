# Consensus node role and durability fixes

**Date:** 2026-06-24 18:46
**Files:** `web/lib/flows/graph/consensus/roles.ts`, `web/lib/flows/graph/consensus/drafts.ts`, `web/lib/flows/graph/consensus/runtime.ts`, `web/lib/flows/graph/consensus/ledger.ts`, `web/lib/flows/graph/consensus/__tests__/roles.test.ts`, `web/lib/flows/graph/consensus/__tests__/runtime.test.ts`, `web/lib/flows/graph/consensus/__tests__/ledger.test.ts`, `docs/system-analytics/flow-graph.md`, `docs/system-analytics/orchestrator.md`
**Severity:** high

## Problem

The consensus node branch had review findings that would make mixed-agent or
mixed-runner consensus flows unsafe:

- Verifier turns ran on the parent run executor instead of the verifier
  participant's own agent or runner.
- Runner-backed draft and synthesizer paths used partial runner snapshots,
  losing provider and sidecar launch contract fields.
- Human resolution input was unlinked before synthesis and artifact writes
  succeeded, so retries could lose the operator's decision.
- Consensus verifier raw-output artifacts and ledger rows were written in
  separate auto-committed writes, leaving a crash window that could re-charge a
  verifier on resume.
- System analytics docs still described the M41 consensus node as designed in
  places that should describe the implemented contract.

## Root Cause

Consensus-specific code reimplemented pieces of normal agent/runner launch
resolution instead of reusing the project launch contracts. The duplicated
resolver paths missed verifier participant context and only selected a subset of
runner columns. Durability issues came from treating input consumption and ledger
recording as independent side effects rather than commit-bound state changes.

## Solution

Added a shared consensus role resolver that resolves agent roles through the
standard agent launch runtime and runner roles through a full runner plus sidecar
snapshot. Draft fan-out, verifier execution, and synthesis now use that shared
contract. Human resolution files are consumed only after successful durable side
effects, and verdict artifact plus ledger writes now happen in one transaction.
Focused tests cover role resolution, verifier runner routing, human input
retention/removal, and verdict transaction shape. Analytics docs now reference
the implemented M41 consensus node.

## Prevention

- Do not duplicate runner/agent launch resolution in feature-specific runtime
  modules; add a small adapter around the existing launch contract.
- Review mixed-agent and mixed-runner flows with at least one test that asserts
  the actual executor and supervisor runner passed to `runAgentStep`.
- Treat file deletion and database ledger writes as durability boundaries:
  delete operator input after successful side effects, and transactionally group
  artifact rows with their semantic ledger rows.
- Keep system analytics status labels in sync with implementation and acceptance
  tests during feature-branch reviews.
