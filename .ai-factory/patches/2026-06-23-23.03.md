# Edit-task modal "Properties" panel could not scroll — lower fields + Save clipped

**Date:** 2026-06-23 23:03
**Files:** web/components/board/task-card-editing.tsx, web/e2e/task-edit-fields-scroll.spec.ts, web/playwright.config.ts
**Severity:** medium

## Problem

In the "Редактировать задачу" (Edit task) modal (`TaskCardEditModal`), the
right-hand "Поля"/"Properties" `<aside>` panel did not scroll. Its lower fields
(ПРЕСЕТ / ПРОВЕРКИ / РЕВЬЮ ЧЕЛОВЕКОМ / АВТОПРОМОУШЕН / СВЯЗИ) and the
Отмена/Сохранить (Cancel/Save) buttons were clipped below the modal's bottom
edge and were unreachable on any viewport shorter than the panel content.

Measured live (viewport 1280×600): the panel grid item was 843px tall while the
modal section was capped at 568px — the bottom 275px was clipped by the
section's `overflow-hidden`, and the panel's inner `overflow-y-auto` never
activated (`scrollHeight === clientHeight`, maxScroll 0).

## Root Cause

The modal `<section>` is a height-capped grid:
`grid max-h-[calc(100vh-32px)] overflow-hidden lg:grid-cols-[minmax(0,1fr)_340px]`.

The two columns already had `min-h-0 overflow-y-auto`. But CSS Grid's implicit
row track defaults to `auto`, which sizes to the **tallest** column's content.
The right column (9 fields + relations + buttons) made the single row grow to
843px — past the section's `max-h`. `overflow-hidden` then clipped the excess.
`min-h-0` on the grid items does nothing here because the *row track itself* was
never bounded to the section height — a flex/grid child can only shrink-to-fit
its scroll area when its track has a definite size.

The misleading part: this looked like an `overflow-hidden`-vs-`overflow-y-auto`
conflict, and the project's existing two-column scroll idiom (diff-view.tsx) is
`min-h-0 + overflow-y-auto` with NO `grid-rows`. Copying that idiom alone did
NOT fix it — diff-view works because it carries a `min-h-[520px]` floor and a
different parent height context. Empirical DOM measurement (scrollHeight vs
clientHeight vs section height) was what actually pinned the cause; reasoning
from the className alone pointed at the wrong fix.

## Solution

Two layers:
1. Bound the grid row track on `lg` (where the two-column layout lives):
   add `lg:grid-rows-[minmax(0,1fr)]` to the section. This forces the single
   row to the section's bounded height instead of `auto`, so both columns'
   scroll areas get a definite height. Kept `lg:`-scoped so the mobile
   `grid-cols-1` stacked layout (two `auto` rows) is untouched.
2. Move each column's scroll into an inner `h-full min-h-0 overflow-y-auto`
   wrapper (the column itself becomes `min-h-0 overflow-hidden`), matching the
   diff-view scroll-container shape. This gives a clean, padding-preserving
   scroll surface.

Verified live (viewport 600px): panel grid item now 566px (= within section),
inner scroller scrollHeight 843 / clientHeight 566, maxScroll 277, scrolls to
bottom, Save button reachable. Mobile (500px) confirmed unchanged: single column,
two `auto` row tracks, the `lg:` override not applied.

## Prevention

- For a height-capped CSS Grid modal/panel with `overflow-hidden`, bounding the
  **row track** (`grid-rows-[minmax(0,1fr)]`) is required for child
  `overflow-y-auto` to engage. `min-h-0` on the grid item is necessary but NOT
  sufficient — the track must have a definite size. Default `auto` tracks grow
  to the tallest column and overflow a `max-h` ceiling.
- When a layout bug "looks like" a known className idiom elsewhere, measure the
  actual DOM (`getBoundingClientRect`, `scrollHeight`/`clientHeight`) before
  assuming the idiom transfers — parent height context differs between call
  sites and the same classes can behave differently.
- A scroll/clipping regression is only catchable on a viewport shorter than the
  content. The e2e test pins a short viewport (1280×600) and asserts
  `scrollHeight > clientHeight`, panel-within-section, and Save reachable after
  scrolling. Verified red (843 === 843, fails) with the fix reverted, green with
  it applied.
