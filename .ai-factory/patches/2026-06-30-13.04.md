# Local Package Runtime Metadata Path Confinement

**Date:** 2026-06-30 13:04
**Files:** `web/lib/local-packages/paths.ts`, `web/lib/local-packages/service.ts`, `web/lib/local-packages/git.ts`, `web/lib/flows/package-content.ts`
**Severity:** high

## Problem

Opening a Studio local package could fail with:

`MaisterError: artifact path escapes the working dir: .maister/capabilities/.../codex-home/.tmp/plugins-clone-*/.git/config`

The editor file listing treated assistant runtime metadata under `.maister/` as package artifacts. A nested `.git/config` below that runtime tree then reached the normal artifact path confinement guard and crashed the page.

## Root Cause

Local-package readers filtered only the top-level `.git` directory and did not treat `.maister/` as internal runtime metadata. The recursive listing walked into `.maister/capabilities/**`, and the git diff/commit path did not proactively exclude `.maister/` for existing local-package repos.

## Solution

Centralized local-package internal path detection for `.git` and `.maister` path segments. Applied it to path confinement, package file listing, installed/local package content walking, and clean-copy export/fork paths. Added a git helper that ensures `.maister/` is present in `.git/info/exclude` before local-package init, diff, commit, discard, and cut/commit gates.

## Prevention

When fixing working-dir path boundary bugs, audit all sibling readers and writers that consume the same relative paths: UI listing, direct file reads, BOM/package-content reads, export/fork copy paths, git diff, and git commit. Internal runtime directories must be pruned before traversal, not only rejected at read time.
