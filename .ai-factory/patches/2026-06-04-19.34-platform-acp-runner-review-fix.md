# Platform ACP Runner Review Fix

## Problem

The platform ACP runner implementation still carried pieces of the retired
project-scoped executor model. Runtime code, tests, OpenAPI docs, and bundled
Flow fixtures disagreed about whether launches should use `executors[]`,
`default_executor`, `recommended_executor`, or platform ACP runners with run
snapshots.

## Root Cause

The first ACP runner implementation added the new platform model but left
legacy executor compatibility paths and stale fixtures in place. Flow import
semantics were also under-specified: portable Flow runner requirements were
being conflated with local database runner ids.

## Solution

- Removed legacy executor table/column usage from production code and added a
  final drop migration.
- Made legacy `executors`, `default_executor`, `executor_override`, and
  `recommended_executor` fields fail schema validation instead of silently
  degrading.
- Preserved Flow portability through `runner_profiles` and `settings.runner`
  targets, with remap requirements carrying runner profile metadata.
- Ensured resume/recover uses immutable `runs.runner_snapshot`.
- Hardened ACP runner usage/readiness typing and removed targeted `any`/`as
  never` casts from the reviewed runner surfaces.
- Updated OpenAPI, README, root agent guidance, and high-signal system docs to
  describe platform ACP runners, CCR sidecars, launch options, remaps, and
  secret-safe DTOs.
- Added regression coverage for secret redaction, Flow runner profile remaps,
  legacy field rejection, snapshot recovery, and local reset dry-run safety.

## Prevention

- Treat runner identity as platform-scoped and historical launches as
  snapshot-scoped.
- Flow packages may export portable runner profiles, but must not export local
  runner primary keys or secrets.
- New launch/admin endpoints must have OpenAPI entries and negative tests for
  secret-bearing provider fields.
- Reset/destructive tooling must keep a dry-run smoke test.
