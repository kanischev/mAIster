# Codex adversarial review: doc/contract mismatch, popover a11y, partial redaction-test coverage

**Date:** 2026-06-18 12:57
**Files:** web/components/chrome/runners-readiness-rail.tsx, web/components/chrome/__tests__/runners-readiness-rail.test.ts, web/lib/acp-runners/__tests__/readiness-summary.test.ts, web/app/(app)/settings/__tests__/page-contract.test.ts (new), docs/screens/chrome/left-rail.md, .ai-factory/plans/runner-readiness-rail-popover-settings.md
**Severity:** medium

## Problem

Three findings from a Codex adversarial review of the runner-readiness rail popover branch:

1. **Doc/contract mismatch.** The plan + `left-rail.md` claimed `/settings` is gated by `requireGlobalRole('admin')`. The page (`settings/page.tsx`) actually SOFT-gates: `getSessionUser()` + `role === "admin"` conditional render + a localized "forbidden" panel, loading no admin data for non-admins. Secure, but not the hard 403 the docs advertised — a future reader would trust a route-level boundary that does not exist.
2. **Popover not a reliable a11y surface.** The hover popover used `opacity-0` (stays in the accessibility tree), was a descendant of the admin `<Link>` with no explicit name, and had no `aria-describedby`/stable tooltip id → each link's accessible name was polluted with every runner row even while the tooltip was visually hidden.
3. **Redaction regression test under-covered the union.** The safe-DTO test exercised only 3 of the 5 secret-bearing `PlatformRunnerProvider` variants (`anthropic_compatible`, `openai_compatible`, `google_vertex`); `google_gemini` and `google_gateway` were untested, so a provider-specific mapping change could leak their `apiKey`/`baseUrl` while the security test still passed.

## Root Cause

1. The plan asserted a convention ("admin routes enforce `requireGlobalRole`") that holds for `/mcps`, `/admin/users`, `/admin/scheduler` but NOT `/settings`, which was built with the softer forbidden-panel pattern. The claim was copied into the plan and a pre-existing docs table row without verifying the actual page.
2. CSS-only reveal was implemented with `opacity-0` (visual-only) instead of `invisible` (visibility:hidden, which also leaves the a11y tree), and the rich popover was placed inside the link's name-computation subtree without an overriding `aria-label`.
3. The redaction test enumerated provider variants by hand (sample), not by the full union, so new secret-bearing kinds were silently uncovered.

## Solution

1. **Corrected docs to match reality (owner-chosen over hardening the route).** Fixed the plan Decision #4 + Identifiers section (added a dated deviation note) and `left-rail.md:23` to state the actual soft-gate; left the `acp-runners.md` `requireGlobalRole` refs untouched (they correctly describe the admin API routes). Added `web/app/(app)/settings/__tests__/page-contract.test.ts`: non-admin → forbidden panel + zero admin-data loaders called (`getDb`/`hostToolStatus`/`checkSupervisorDiagnostics`); admin contrast → loaders called, no forbidden panel.
2. **Popover a11y:** switched hidden state to `invisible` (+ `group-hover/focus-within:visible`) so it leaves the a11y tree when closed; gave each chip a deterministic `aria-label` (`adapter: cause`) so the name no longer derives from the popover subtree; wired `aria-describedby` → a stable `runners-readiness-tip-<adapter>` id on the tooltip. Kept non-admin chips NON-focusable — that was a decision explicitly locked with the owner in the plan (Resolved-Q #3), and Codex flagged it only conditionally.
3. **Redaction test made exhaustive:** drove the sweep from a `SECRET_BEARING_RUNNERS` table covering all 5 secret-bearing variants + a guard test that fails if any `PROVIDER_KINDS` entry is neither in the secret-free allowlist nor in the sweep.

## Prevention

- **Verify a security/contract claim against the code before repeating it in a plan or doc.** When a plan says "route X is gated by Y", grep route X. Here `grep -nE "requireGlobalRole|getSessionUser|role === \"admin\"" web/app/(app)/{mcps,admin/users,admin/scheduler,settings}/page.tsx` showed `/settings` was the lone soft-gate — recorded per the skill-context "record the grep" rule.
- **Redaction/boundary tests must be union-exhaustive, not sampled.** Drive them from the type union (`PROVIDER_KINDS`) with a coverage guard, so adding a new secret-bearing variant fails CI until a case is added.
- **CSS-only tooltips: use `invisible`, not `opacity-0`, for the hidden state**, and give the trigger an explicit `aria-label` + `aria-describedby` instead of relying on subtree name computation when the trigger wraps rich content.
- **When a review flags something a plan explicitly decided (with the owner), don't fold** — confirm it's a locked decision and keep it (non-admin not focusable), fixing only the genuinely-broken part (the name pollution).
