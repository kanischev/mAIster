# Token scope contract and migration snapshot drift

**Date:** 2026-06-08 19:02
**Files:** `web/lib/db/migrations/meta/0031_snapshot.json`, `docs/api/external/operations.openapi.yaml`, `docs/database-schema.md`, `docs/system-analytics/identity-access.md`
**Severity:** high

## Problem

The token actor/scope implementation added the correct SQL migration and runtime
code, but review found two contract drifts before merge:

- Drizzle migration metadata snapshot `0031_snapshot.json` still represented
  `project_tokens` and `tasks` without the new `token_kind`, `owner_user_id`,
  and `created_by_user_id` columns.
- External OpenAPI described default-on scope enforcement in prose but omitted
  the resulting `403` response from most non-HITL routes and omitted
  `TaskDTO.createdByUserId`.

## Root Cause

The migration SQL was custom-written because normal Drizzle generation was
blocked by prior snapshot history. The SQL and schema were updated, but the
generated snapshot was not reconciled to the same post-migration state. The API
contract was updated at the security-scheme/prose level, but the per-operation
response maps and DTO schema were not swept after `handleExt` changed from
HITL-only scope enforcement to default-on scope enforcement.

## Solution

Updated the migration metadata snapshot to include exactly the columns, foreign
keys, and index added by `0031_token_actor_scope_support.sql`. Updated external
OpenAPI per-route response maps with `403`, removed stale HITL-only wording,
and added `TaskDTO.createdByUserId` to the external schema. Cleaned the touched
hard-delete docs so they no longer list the dropped `flow_graph_layouts` table
as a live reference check.

## Prevention

- When a migration is manually repaired or custom-authored, compare
  `schema.ts`, the SQL migration, and the matching snapshot before review.
- When a shared route wrapper changes response behavior, sweep every OpenAPI
  operation using that wrapper, not only the routes whose code diff is nearby.
- For DTO field additions, grep both code and OpenAPI schemas for the DTO type
  name and confirm the public projection matches exactly.
