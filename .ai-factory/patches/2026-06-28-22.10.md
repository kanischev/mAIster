# git commit without a default author fails on hosts/CI with no git identity

**Date:** 2026-06-28 22:10
**Files:** web/lib/worktree.ts
**Severity:** high

## Problem

CI (`lint + typecheck + unit`) was red on every open PR. Three unit tests failed,
all doing a real `git commit`:

- `lib/__tests__/squash-run-branch.test.ts` ×2 — `result.squashed` was `false`
- `lib/runs/__tests__/auto-resolve-dirty.test.ts` — result was `null`, expected `"commit"`

CI runner log showed the cause:

```
Author identity unknown
*** Please tell me who you are.
fatal: empty ident name (for <runner@...>) not allowed
```

The failures are environmental, not caused by any PR's changes — the GitHub
runner has no global `git config user.name/user.email`, so `git commit` aborts.
The same break would hit any freshly-provisioned host (new MAIster deploy) where
git identity was never configured: auto-commit-dirty-before-review and
squash-on-promote would both fail.

## Root Cause

ADR-093 already solved this for `commitFile` (config persist): when the host git
author is unset, it supplies a per-field default identity via
`-c user.name=... -c user.email=...`. But two later git-commit sites added for
execution-policy never reused that pattern and ran a bare `git commit`:

- `snapshotDirtyWorktree` (auto-commit a dirty worktree before a review gate)
- `squashRunBranch` (collapse run-branch commits into one before promote)

A missing default author was a silently-skipped case, not a deliberate choice —
it only surfaces where git identity is truly absent (CI / fresh host), which is
why every developer machine (global git identity set) passed it.

## Solution

Extracted the ADR-093 default-identity logic out of `commitFile` into a shared
helper `commitIdentityArgs(repo)` — reads `user.name`/`user.email`, and for each
unset field returns `-c user.<field>=<default>` (configured host values are never
overridden, only the missing one is defaulted). Applied it at all three commit
sites: `commitFile`, `snapshotDirtyWorktree`, `squashRunBranch`.

Verified by running the two failing test files with an empty `HOME` +
`GIT_CONFIG_NOSYSTEM=1` and no `GIT_AUTHOR_*`/`GIT_COMMITTER_*` env (the CI
condition): both pass (9/9) after the fix.

## Prevention

- Never run a bare `git commit` from server code. Always route through the shared
  `commitIdentityArgs` (or `commitFile`) so a host/CI with no git identity gets a
  deterministic default author instead of an aborted commit.
- When adding a new git-commit call site, grep for `commitIdentityArgs` and reuse
  it — the ADR-093 contract is "every MAIster-authored commit supplies a default
  identity when the host's is unset."
- A test that does a real `git commit` should run green under a clean git
  environment, not just on a developer machine with a global identity.
