# M39-stream adversarial fixes: flow-driving agent idempotency/branch-base + element-fork manifest id

**Date:** 2026-06-24 18.55
**Files:** web/lib/agents/launch.ts, web/lib/services/runs.ts, web/lib/local-packages/service.ts (+ 5 test files)
**Severity:** critical

## Problem

A second Codex adversarial review (of the whole merged M39 stream) found four
issues; three were fixed, one was an accepted design call.

1. **(critical)** A flow-driving agent (an agent whose definition declares a
   `flow`) bypassed trigger idempotency: `launchAgentRun` dispatched into
   `launchAgentDrivenFlowRun` BEFORE the `(agent_id, trigger_event_id)` dedup
   pre-check, and the auto-created task + `launchRun` carried no trigger
   metadata. An at-least-once domain-event/webhook redelivery created duplicate
   board tasks + duplicate flow runs.
2. **(high, accepted)** Migration 0068 opens with `DELETE FROM "agents"` — a
   pre-release reset that CASCADE-drops per-project attachments + schedules on
   upgrade. Kept intentional (single-operator pre-release; a deterministic
   re-key is impractical since the id format itself changed); caveat below.
3. **(high)** `registerFlowElementInManifest` derived the manifest `flows[].id`
   from the directory basename (`flows/dev` → `dev`) instead of the copied
   `flow.yaml` `name` (`aif-dev`). The installer requires `id === flow.yaml.name`
   → a valid source flow forked into an UNCUTTABLE local package.
4. **(high)** `branch_base` was ignored for flow-driving agents — the
   flow-driving branch returns before the standalone branch-base resolution (a
   gap in the PRIOR round's branch_base fix, which only covered the standalone
   path).

## Root Cause

1. **A dedup pre-check placed AFTER a dispatch branch only protects the branches
   below it.** The `(agent_id, trigger_event_id)` pre-check + the `runs` insert's
   `onConflictDoNothing` lived in the standalone path, below the
   `if (parsed.flow) return launchAgentDrivenFlowRun(...)` early return — so the
   flow-driving branch skipped both, and flow runs never recorded
   `trigger_event_id` for the claim to dedup against.
3. **A "convenient" derivation that diverges from a stricter downstream
   contract.** Deriving the id from the directory basename was easy but the
   installer enforces `id === flow.yaml.name`; the two silently disagree the
   moment a flow's name ≠ its directory.
4. **Same defect CLASS in a sibling code path hidden by an early return.** The
   prior round resolved `branch_base` inside the standalone worktree block —
   which the flow-driving early return never reaches. Fixing the cited line
   without sweeping every dispatch branch left the twin un-fixed.

## Solution

1. Move the trigger-event pre-check ABOVE the flow/standalone dispatch in
   `launchAgentRun` (covers both paths before any side effect). Thread
   `triggerSource/triggerEventId/triggerPayload` through `LaunchRunInput` onto the
   flow-run insert so the claim is recorded. Add `onConflictDoNothing().returning()`
   + a typed `CONFLICT` on the no-op so a CONCURRENT redelivery is a clean
   CONFLICT (worktree compensated by the existing catch), never a raw 23505→500.
   Board launches set no `trigger_event_id`, so they never conflict.
3. Parse the copied `flow.yaml` with `loadFlowManifest` (the SAME loader the
   installer uses) and set `id = flowManifest.name`; a malformed copy fails fast.
4. Resolve `link.branchBase ?? recommended.branch_base ?? null` in
   `launchAgentDrivenFlowRun` and pass it as `launchRun`'s `baseBranch` (which
   validates it + defaults `targetBranch` to it).

## Prevention

- **A claim/dedup/guard MUST sit before the FIRST side effect AND before any
  dispatch branch.** When a function early-returns into sub-paths, a check placed
  in one sub-path silently skips the others. Grep every `return <subpath>(...)`
  above the check.
- **When you fix a defect inside a function with multiple dispatch branches,
  re-apply the fix to EVERY branch in the same pass.** The branch_base twin
  (round 1 fixed standalone, round 2 had to fix flow-driving) is the canonical
  miss — the early return hid the sibling from the edit set.
- **Derive an identifier from the SAME source its downstream contract checks.**
  If install asserts `id === flow.yaml.name`, registration MUST read
  `flow.yaml.name` — never a "close enough" proxy (the dir basename).
- **Threading a new field onto a SHARED insert + adding `onConflictDoNothing`
  ripples to every fake-DB unit mock of that insert.** The three `runs-launch-*`
  mocks each needed a chainable `.values().onConflictDoNothing().returning()` —
  run the full board-launch suite, not just the new test.
- **Migration 0068 caveat (Finding 2):** the `DELETE FROM "agents"` reset wipes
  per-project agent attachments (enablement, runner override, `branch_base`,
  policy) + trigger schedules on upgrade — agents must be RE-ATTACHED to projects
  after upgrading; the startup/admin resync restores only the catalog. Acceptable
  pre-release; gate or re-key before any real upgrade path consumes it.
