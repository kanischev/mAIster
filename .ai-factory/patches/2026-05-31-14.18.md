# Repo onboarding: failed local registration left a new .git behind

**Date:** 2026-05-31 14.18
**Files:** web/lib/repo-source.ts, web/app/api/projects/route.ts, web/lib/__tests__/repo-source.test.ts, web/app/api/__tests__/projects-register-clone.integration.test.ts, docs/system-analytics/projects.md
**Severity:** high

## Problem

(Codex adversarial review, finding 2.) In existing-local registration,
`resolveProjectSource` ran `git init` on a non-git target dir **before** the
route loaded/validated `maister.yaml`. If the admin pointed at the wrong dir, or
the manifest was missing/invalid (or a later flow-install failed), registration
failed but the directory had already been mutated into a git repo — a
non-rollback-safe side effect on an arbitrary operator path. The existing
compensation only removed *cloned* dirs (`clonedByUs`), never an init'd `.git`.

## Root Cause

The mutation (`git init`) was placed in the resolve/read phase instead of being
ordered as the last, committed step. Registration's "all-or-nothing" guarantee
covered the DB rows and the clone, but not the local `git init` side effect —
the failure ordering simply wasn't considered for existing-local dirs.

## Solution

Prevent over compensate (Codex's preferred option):

1. `resolveProjectSource` no longer mutates an existing dir. For a non-git dir
   it returns `{ gitStatus: "initialized", repoUrl: null, provider: null }`
   WITHOUT running `git init`.
2. The route runs `git init` as the **last** step of `register()` — inside the
   flow-install try block, after the manifest validated and the project + flows
   committed. So `CONFIG` / `CONFLICT` / `FLOW_INSTALL` failures (which all
   precede it) leave the dir untouched.
3. If that final `git init` itself fails, the existing compensation rolls back
   the project row + `.maister/<slug>`, and a new guarded step removes the
   created `.git` — gated on `gitStatus === "initialized"` so a **pre-existing**
   repo's `.git` is never deleted.

Tests: updated the resolve unit test to assert no mutation (`isGitRepo === false`
after resolve); added an integration test (non-git dir + invalid `maister.yaml`
→ 422 `CONFIG`, dir has no `.git`). Analytics (`projects.md`) updated: resolve
diagram, atomicity expectation, and the `git init` edge case now state the
deferral + revert.

## Prevention

- Order side effects so the irreversible/external mutation is the LAST committed
  step; validate all cheap preconditions (schema, collisions) before mutating
  anything on disk.
- Every compensation that removes a path must be gated on proof THIS request
  created it (`clonedByUs`, `gitStatus === "initialized"`) — never blanket-`rm`
  a path that could be the user's pre-existing state.
- When a resolver both reads and mutates, split the mutation out so the
  transactional caller controls when (and whether) it happens.
