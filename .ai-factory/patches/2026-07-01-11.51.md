# Artifact body injection (ADR-120) — adversarial-review round 3

**Date:** 2026-07-01 11:51
**Files:** web/lib/worktree.ts, web/lib/flows/graph/artifact-content.ts, web/lib/flows/graph/artifact-inject.ts, web/lib/flows/graph/runner-graph.ts (+ tests: worktree-range.test.ts, artifact-inject.test.ts, artifact-inject.integration.test.ts; + docs: decisions.md, flow-dsl.md, configuration.md, system-analytics/artifacts.md)
**Severity:** high

## Problem

A third Codex adversarial-review pass found two remaining holes in the ADR-120
injection path:

1. **[high] Gate-only gone payloads bypass controlled gate failure handling.**
   Round-2 made gate-only refs resolve TOLERANTLY (gone → leave
   `artifacts.<id>.content` unset). For a NON-skipped gate that then rendered the
   ref, `renderStrict` threw during the gate's agent/CLI step — and the gate loop
   (`runNodeGates`) has NO try/catch around `runOneGate`, so the throw propagated
   to the top-level `runGraph` catch. The gate_results row (created `running`
   before the render) was left `running` and the run failed via the catch — an
   uncontrolled failure that hides the real cause and misleads recovery/readiness.

2. **[high] Large `git-log` injection failed instead of truncating.** The bounded
   read (Codex round-2 #2) only covered `file` locators. `git-log` still called the
   uncapped `logRange`, whose 4 MiB child-process buffer THROWS `CONFLICT` on
   overflow — so an oversized `commit_set`/git-log body failed the node before
   spawn instead of truncating, contradicting the "large bodies truncate, never
   fail" acceptance criterion.

## Root Cause

1. The round-2 tolerant-defer was a half-measure: it correctly stopped a SKIPPED
   gate from failing the node, but for a NON-skipped gate it just moved the failure
   from a clean pre-spawn `CONFIG` to an uncontrolled mid-gate render throw. The
   real distinction is not "action vs gate" (tolerant/strict) but "will this gate
   actually run?" — which depends on execution-policy check-strictness, not known
   to a policy-agnostic collector.

2. The bound was applied per-locator-kind and only `file` was covered. `git-log`'s
   overflow semantics (throw, unlike `diff`'s existing stream-truncate) were missed
   — the fix generalized the wrong subset.

## Solution

1. Replaced the tolerant-defer with **policy-aware exclusion**. `collectContentArtifactIds`
   gained an `includeGate` predicate; the runner passes
   `(gate) => !isPolicySkippedGate(checks, gate.kind)` (checks from the run's
   execution-policy snapshot). Skipped gates are excluded from the resolution set
   entirely (their refs never resolved → a skipped gate does no content work). Every
   INCLUDED ref (action + inline + non-skipped gate) is resolved STRICTLY: a gone
   payload fails the node with a controlled `markNodeFailed(CONFIG)` before spawn —
   never an uncontrolled mid-gate throw, never a half-created gate row. Removed the
   now-unneeded `collectActionContentArtifactIds` (the strict/tolerant split is gone).

2. Extracted a generic bounded streamer `streamGitTruncatedTo(repo, args, maxBytes,
   label)` (the existing `streamGitDiffTruncated` now delegates to it — DRY, diff
   behavior unchanged) and added `logRangeBounded(args, maxBytes)` that streams
   `git log` and TRUNCATES at the bound. The resolver's `git-log` case uses it on
   the injection path (maxBytes set); the payload route keeps the uncapped
   `logRange`.

## Prevention

- **Model the real decision variable.** When a value's handling depends on a
  downstream conditional (here: will the gate run under execution policy?), thread
  that condition into the decision instead of approximating it with a proxy
  (tolerant vs strict). A "tolerant" fallback that defers a failure to an
  unguarded call site converts a clean error into an uncontrolled one.
- **A resolution pass must fail CONTROLLED or not at all.** Prefer failing before a
  side-effecting step (pre-spawn `markNodeFailed`) over leaving state that makes a
  later step throw through a path with no error handling.
- **Bound EVERY overflow-prone source on a capped path, and check each source's
  overflow semantics.** `diff` stream-truncates; `log` throws. A per-kind cap must
  enumerate all kinds and their failure modes, not generalize from one.
