# Scratch transcript replay after follow-up turns

**Date:** 2026-06-29 14:59
**Files:** web/lib/scratch-runs/events.ts, web/lib/scratch-runs/__tests__/transcript.test.ts
**Severity:** medium

## Problem

Scratch conversations, including project scratch runs and the local-package
Flow assistant, could show part of the previous assistant response again after
the user sent a follow-up message.

## Root Cause

The scratch event consumer resumes supervisor SSE with `Last-Event-ID` derived
from `scratch_messages.supervisor_event_id`. Assistant, thought, tool, and usage
events are coalesced into existing rows, but row updates only changed
`content`; the stored `supervisor_event_id` stayed at the first chunk/update
event. A follow-up turn therefore resumed after a stale event id and replayed
later chunks from the previous turn. The offset query also used a
Postgres-specific cast, so SQLite dev sessions could fall back to replaying
from the start.

## Solution

Update `supervisor_event_id` every time a coalesced scratch message row is
updated, so the persisted offset always reflects the latest projected
supervisor event. Compute the maximum projected event id in TypeScript from
stored row values instead of using a dialect-specific SQL cast.

## Prevention

- Treat coalesced transcript rows as both display records and event-offset
  checkpoints; updates must advance both content and offset.
- Regression tests for replay-sensitive consumers should make fake SSE streams
  honor `Last-Event-ID`.
- Keep scratch event offset queries dialect-agnostic unless both Postgres and
  SQLite behavior are covered.
