# Non-shared promote+rework double-win (lost update) — F1 sibling-sweep miss

**Date:** 2026-06-22 14:06
**Files:** web/lib/agents/launch.ts, web/lib/runs/promote.ts, web/app/api/v1/ext/runs/__tests__/promote-rework.integration.test.ts
**Severity:** high

## Problem

`promote-rework.integration.test.ts` test `(f) concurrent promote + rework` flaked
(~40% of runs) with `expected [200, 200] to deeply equal [200, 409]` — i.e. BOTH the
`run_promote` and the `run_rework` on the same `Review` child returned 200. The memory
(and an earlier triage) had mislabeled this as testcontainer/pg contention; it is a
genuine **double-win lost-update**: the rework's `Review → Running` work is silently
clobbered to `Done` and both callers report success.

## Root Cause

A non-shared (own worktree) promote and rework had no shared serialization point:
- `reworkChildRun` fenced only the SHARED branch on the tree allocator
  `workspaces.promotion_state` (HARDENING-2 F1). The non-shared `else` did a bare
  `markReworkFromReview` CAS (`UPDATE … WHERE status='Review'`) with no workspace lock
  and no promotion_state check.
- The non-shared promote finalize flips `status='Done' WHERE id=runId` with no
  `status='Review'` re-guard, after a LOCKLESS merge.
- The promote claim's `loadRun` reads `run.status` WITHOUT a lock.

Two windows produced `[200,200]`: (during) a rework CASes `Review→Running` after the
claim mints `promotion_state='claiming'` but before the finalize, then the finalize
clobbers it to Done; (pre-claim) a rework commits `Review→Running` while the promote
waits for the workspace lock, leaving the claim's `run.status` stale-`Review` so it
proceeds and the finalize clobbers. This is the SAME defect class as the F1 fix and the
Codex finding (a gate/finalize running after a lockless side-effect) — F1 shipped the
shared half and MISSED the own-worktree sibling (backend rule 13 sibling-sweep).

## Solution

Mirror F1 for the non-shared path, plus close the stale-read window:
- `reworkChildRun`: an own/workspace-bearing child now fences on its OWN `workspaces`
  row (`loadOwnWorkspacePromotionStateForUpdate`, FOR UPDATE) — refuse `CONFLICT` while
  `promotion_state ∈ {'claiming','done'}`, else CAS in the same tx. A workspace-less
  child (`workspace='none'/'repo_read'`, no row) can't be promoted → keeps the bare CAS.
- `promoteWorkspaceRun` claim: RE-READ `run.status` under the workspace lock for a
  NON-shared promote (the row the rework fence also locks). A rework that won the lock
  first is now observed → `PRECONDITION` instead of a clobber. Scoped to non-shared: a
  SHARED tree relies on the C1 finalize re-check + the F1 fence, and re-reading there
  would turn a concurrent-promote loser's `CONFLICT` (canReclaim) into `PRECONDITION`
  (this broke T10(d) on the first pass — caught and scoped).

Deterministic test `(f3)`: a deferred-merge gate pins the during-window and asserts the
rework is refused `CONFLICT` (RED before: returned 200 then was clobbered). `(f)` is now
green 6/6 runs; `(f3)` GREEN.

## Prevention

- **A fix for a "side-effect-then-finalize" race MUST sweep every promotable path in the
  same pass.** F1 fenced shared rework-vs-promote but left the own-worktree twin; the
  miss surfaced only as a flaky `[200,200]`. When fixing one branch of an if/else that
  guards an irreversible side-effect, grep the sibling branch in the SAME pass (backend
  rule 13).
- **A flaky concurrency test asserting a single-winner invariant is a real-bug signal,
  not infra noise.** Capture the ACTUAL failure (`[200,200]` ≠ `[200,409]`) before
  blaming the harness; a double-success is a lost update. Re-run to confirm flaky, then
  read what it flaked TO.
- **Serialize concurrent mutators on ONE consistent lock, and read the decision field
  UNDER that lock.** A lock-free pre-read (the claim's first `loadRun`) defeats the lock;
  re-read after acquiring it. Avoid inverting lock order (runs-row FOR UPDATE in the
  promote vs workspace-row in the rework) → use the re-read, not a second lock.
- **Scope a re-guard to the branch that needs it.** The shared path already had C1; a
  blanket re-read regressed the shared concurrent-promote error code. Run the FULL
  affected suite (incl. the other run-kind branch) after touching shared promote logic.
