# Adversarial-review round 3: auth-first ordering on state-changing routes + OpenAPI password-gate contract

**Date:** 2026-05-29 14:38
**Files:** web/app/api/runs/[runId]/hitl/[hitlRequestId]/respond/route.ts, web/app/api/projects/[slug]/tasks/route.ts, web/app/api/runs/route.ts, docs/api/web.openapi.yaml, web/app/api/runs/[runId]/hitl/[hitlRequestId]/respond/__tests__/route.test.ts, web/app/api/runs/__tests__/route.trust-boundary.integration.test.ts, web/app/api/projects/[slug]/tasks/__tests__/route.integration.test.ts
**Severity:** high

## Problem

Third Codex adversarial pass, after the round-2 fail-closed password gate landed:

1. **(high)** Resource lookups ran BEFORE the active-session gate. The HITL
   respond route (`respond/route.ts`) read `hitl_requests` then `runs` and
   returned distinct `PRECONDITION` errors (hitl-not-found, run-mismatch,
   run-not-found) *before* calling `requireProjectAction` at line 201. Because
   `PASSWORD_CHANGE_REQUIRED` is only enforced inside
   `requireProjectRole`/`requireProjectAction`, a forced-password-change account
   (the seeded `admin@maister.local` / `maister-admin`) could still probe
   workflow identifiers and run↔hitl relationships with the well-known default
   password, and unauthenticated callers could read resource-shape errors off a
   state-changing endpoint before any auth ran. The same ordering bug existed in
   `tasks/route.ts` (resolved the URL slug → `PRECONDITION` project-not-found
   before authz) and was partially present in `runs/route.ts` (gated up front
   with the weaker `requireSession()`, so must-change callers passed the gate
   and could probe task/project existence before `requireProjectAction`).
2. **(medium)** The new `PASSWORD_CHANGE_REQUIRED` 403 was absent from the public
   OpenAPI contract. `MaisterErrorBody.code` enumerated only `UNAUTHENTICATED`
   and `UNAUTHORIZED`; every role-gated route's 403 description omitted it.
   Generated clients / contract checks would reject or mishandle a real 403
   payload the changed routes now emit.

## Root Cause

1. The round-2 fix added the third authorization gate (account-active /
   no-forced-action-pending) but wired it ONLY into the role-check functions.
   Those functions are called *after* a route resolves its URL/body resource to
   a row — so the resource read (and its shape-leaking `PRECONDITION`) happened
   before the gate. "Authenticated + authorized" were ordered correctly relative
   to each other, but the resource lookup was interleaved ahead of *both*. An
   auth gate that runs after a DB probe is not fail-closed for that probe.
2. Contract drift: the error enum and 403 descriptions were authored before the
   `PASSWORD_CHANGE_REQUIRED` code existed and were not re-derived from the
   runtime error surface when the code was added in round 2.

## Solution

1. **Auth-first ordering.** Call `requireActiveSession()` as the FIRST statement
   inside the handler body (immediately after body parsing, before any
   `getDb()`/`select`) on `respond/route.ts` and `tasks/route.ts`; upgraded
   `runs/route.ts`'s existing up-front `requireSession()` to
   `requireActiveSession()`. This makes unauthenticated → 401 and
   must-change → 403 `PASSWORD_CHANGE_REQUIRED` fire before any resource probe.
   Project-membership (`requireProjectAction`) still runs *after* the row read,
   because `projectId` is server-derived from the row (never body-supplied) —
   that ordering is correct and unchanged. **Swept all four state-changing
   routes:** `projects/route.ts` already gated up front with
   `requireGlobalRole("admin")` (which funnels through `requireActiveSession`),
   so it needed no change.
2. **OpenAPI contract.** Added `PASSWORD_CHANGE_REQUIRED` to the shared
   `MaisterErrorBody.code` enum, the global Authentication note, the project
   registration phase list, and every role-gated route's 403 description; updated
   the `postRun` description from `requireSession` to `requireActiveSession`.
   `npx @redocly/cli lint` passes (only pre-existing warnings remain).

## Prevention

- **Auth before I/O on state-changing routes.** The gate that decides "may this
  caller do anything here" must run before the first resource read, not just
  before the mutation. A `PRECONDITION not-found` answer is itself information;
  if an unauthenticated or account-locked caller can elicit it, the endpoint
  leaks. Pattern: `parse body → requireActiveSession() → resolve resource →
  requireProjectAction(serverDerivedProjectId) → mutate`.
- **When you add an authorization gate, audit call-site ORDERING, not just call
  presence.** A new gate added inside a helper only protects code that runs
  after that helper. Grep every route that resolves a URL/body resource to a row
  and confirm the session/account gate precedes the lookup. (Skill-context rule:
  sweep sibling routes that reach the same class of sink in the same pass.)
- **Re-derive API contracts from the runtime error surface whenever an error
  code is added.** A new `MaisterError` code is a contract change: enum +
  per-route response descriptions must be updated in the same change, then
  validated (`@redocly/cli lint`).
- Tests added (all real-authz or gate-ordering, not stubs):
  - respond route (unit): a mocked `requireActiveSession` rejection on an EMPTY
    db yields 403 `PASSWORD_CHANGE_REQUIRED` (not 409 hitl-not-found) and writes
    nothing — proves the gate precedes the lookup.
  - runs trust-boundary (integration): a seeded `mustChangePassword` member
    hitting a NONEXISTENT task gets 403 `PASSWORD_CHANGE_REQUIRED`, not 409
    task-not-found.
  - tasks route (integration): a seeded `mustChangePassword` member hitting a
    NONEXISTENT slug gets 403 `PASSWORD_CHANGE_REQUIRED`, not 409
    project-not-found.
