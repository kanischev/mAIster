# M23 Observatory review fixes

**Date:** 2026-06-05 11:29
**Files:** web/lib/queries/observatory.ts, web/lib/queries/observatory-core.ts, web/lib/observatory/filters.ts, web/components/observatory/*, web/e2e/m23-observatory.spec.ts, web/messages/en.json, web/messages/ru.json, web/styles/globals.css, docs/system-analytics/observatory.md
**Severity:** medium

## Problem

The first M23 Observatory implementation had several review findings:

1. Correction metrics and autonomy metrics used different run populations when
   historical runs had no `node_attempts` ledger rows, which made one dashboard
   row mix ledger-backed and non-ledger semantics.
2. The dashboard used HeroUI/Tailwind `danger` classes without a matching local
   theme token.
3. Human re-review loops needed an explicit test and worked example because
   ADR-051 counts retries for every node type under the frozen pressure ratio.
4. Dead clustering helpers remained in the core query module after the signal
   implementation moved elsewhere.
5. The bounded-query integration test seeded only one run and could not catch
   per-run loops.
6. Artifact filters existed in the query contract but were not parsed or wired
   into the dashboard controls.
7. Portfolio node grouping merged equal node ids across projects and flows.
8. RU translations and drill-down labels were incomplete.

## Root Cause

The read model was built across multiple partial contracts at once:
ledger-backed correction, HITL wait accounting, artifact evidence, and signal
harvesting. The initial implementation correctly stayed read-only but let a few
contract edges drift between query code, UI filters, docs, and tests.

## Solution

- Restricted Observatory metric rows to a single ledger-backed eligible run
  population, so correction and autonomy totals use the same runs.
- Added local `danger` and `danger-soft` theme tokens used by the M23 UI.
- Kept the frozen retry semantics: retries are `max(attempt) - 1` per
  `(run_id, node_id)` for every node type, including human review re-entry.
- Removed duplicated dead signal clustering helpers from
  `observatory-core.ts`.
- Expanded the bounded-query integration fixture to eight runs and added a
  regression for no-ledger HITL waits being excluded from the eligible
  Observatory population.
- Parsed and wired `artifactKind` and `artifactDefId` filters through pages,
  HeroUI filter controls, labels, and drill-down links.
- Grouped portfolio node heatmap rows by project, flow, node id, and node type.
- Completed the RU Observatory labels and renamed the node drill-down table to
  "Latest attempt by run".
- Updated the Observatory spec worked example to make the human re-review
  pressure ratio explicit. ADR-051 remains unchanged.

## Validation

- Focused Observatory unit/component/page tests passed.
- Focused Observatory integration tests passed in a Docker/Testcontainers
  environment.
- Web typecheck passed.
- Documentation Mermaid validation passed.
- Full repository integration still has unrelated legacy/ext-route failures;
  use the focused Observatory integration file as the M23b signal until those
  suites are cleaned up separately.

## Prevention

- Metric surfaces that combine ledgers and HITL waits must define one eligible
  population before computing independent rollups.
- Product metrics should encode frozen formula semantics in pure core helpers
  first, then keep signal harvesting and docs aligned to the same helper
  behavior.
- Query-count tests need enough rows to fail when a loop is accidentally added.
- UI filter controls should be wired in the same change that adds query contract
  fields.
