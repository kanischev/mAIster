# Cost-budget governance (ADR-101) — /aif-review finding fixes

**Date:** 2026-06-22 13:05
**Files:** web/app/api/runs/[runId]/hitl/[hitlRequestId]/respond/route.ts, web/lib/runs/budget-default.ts, web/lib/runs/budget-meters.ts, web/lib/runs/keepalive-sweeper.ts, web/lib/orchestrator/cascade.ts, web/lib/services/hitl.ts, web/components/board/run-hitl-response.tsx, docs/api/async/outbound-webhooks.asyncapi.yaml, docs/system-analytics/outbound-webhooks.md, docs/error-taxonomy.md (+ 4 test files)
**Severity:** critical

## Problem

A full review of the cost-budget governance branch surfaced two merge-blockers, two high-impact gaps, two medium logic holes, and a contract drift:

- **C1** — "Raise & resume" was dead on the wire: the respond route's Zod `bodySchema` did not declare `raiseTo`, so `z.object()` stripped it; the service then saw `Number(undefined)=NaN` and every Raise returned PRECONDITION. Service tests stayed green because they call `respondToHitl` directly, bypassing the route schema.
- **C2** — `applyDefaultBudgetForUnattended` seeded only `tree.maxTokens`, but a standalone run has `root_run_id=NULL` and the watchdog evaluates tree scope only when `rootRunId===id`. So `MAISTER_DEFAULT_UNATTENDED_BUDGET_TOKENS` silently did nothing for the common single-run case.
- **H1** — tree-scope terminate flipped DB rows terminal but never `deleteSession` the root/children, so the swarm kept spending past the hard ceiling.
- **H2** — no two-racer concurrency test for the CAS transitions (project-mandated), and the `setInterval` sweeper had no re-entrancy guard; an escalate loser unconditionally unlinked the winner's `needs-input.json`.
- **M1** — task-scope `consecutiveFailures` was dead: the live Running run heads the `started_at DESC` streak and breaks it at 0. The "passing" test faked a causally impossible ordering (live run older than its own retries).
- **M2** — a raise lifted `maxTokens` but the effective `hardMaxTokens` stayed at the stale snapshot value (per-meter `override ?? snapshot` resolution), so a raised run hard-terminated early.
- **W1** — the newly-authored `DataRunEscalated.onStuck` AsyncAPI enum omitted the reachable `ship_with_warning` value.

## Root Cause

Recurring classes:

1. **Type-vs-runtime-schema drift (C1).** A new field (`raiseTo`) was added to the service's `RespondInput.body` *type* but not to the *transport* Zod gate the route parses with. The TS type made it look wired; the runtime schema silently dropped it. Tests that bypass the transport gate cannot catch this.
2. **Config-scope vs enforcement-scope mismatch (C2).** A default seeded a *scope* (`tree`) that the enforcement path only evaluates for a subset of entities (orchestrator roots). The seed and the watchdog's `rootRunId===id` gate were never reconciled against the standalone-run shape (`root_run_id=NULL`).
3. **"Mark terminal" ≠ "stop the work" (H1).** The cascade mutates DB rows; ACP sessions live in the supervisor. Reusing the cascade for a *budget kill* (whose whole point is to stop spend) inherited the DB-only behavior without the session teardown the run-scope path already does.
4. **Trailing-streak over a set that includes the live row (M1).** `ORDER BY started_at DESC` puts the live (non-failure) run first, so a leading-failure streak is always 0. The meter's definition didn't exclude the row being evaluated.
5. **Per-meter override resolution ignores cross-meter coupling (M2).** `hardMaxTokens` resolved independently of `maxTokens`; a raise that touched only `maxTokens` left the computed hard band stale because the snapshot's explicit hard still won.

## Solution

- **C1:** added `raiseTo: z.number().int().positive().optional()` to the route `bodySchema` (transport-only; the service still re-validates fail-closed). budget_breach is human-actor-only, so the ext route needs no equivalent.
- **C2:** `applyDefaultBudgetForUnattended` now seeds BOTH `run.maxTokens` (always evaluated for a Running candidate) and `tree.maxTokens` (bounds an orchestrator swarm total at the root).
- **H1:** `cascadeAbandonRunTree` now returns `cascadedRunIds` (additive); `actBudgetTerminateTree` takes the supervisor `records`, kills the root's live session first (E5 EXECUTOR_UNAVAILABLE → leave for next tick), then best-effort tears down each cascaded child's live session.
- **H2:** the escalate lost-race path no longer unlinks `needs-input.json` (the winner depends on it); the sweeper gained a `running` re-entrancy guard so overlapping ticks can't fire; added two real two-racer tests (concurrent `runSweepTick` → exactly one escalate / one `run.failed`).
- **M1:** `consecutiveFailedRuns` gained `excludeRunId`; the watchdog passes `candidate.id` for task + tree so the streak counts failures BEFORE the live run. The misleading test was rewritten to realistic monotonic ordering + a negative case.
- **M2:** added `tokenCeilings()` (also DRYs the 3 duplicated run/task/tree token blocks) — when a raise lifted `maxTokens` without an explicit override hard, the terminate band re-derives from the raised value × multiplier.
- **W1:** added `ship_with_warning` to the AsyncAPI enum + the two analytics rows; removed a duplicated sentence in the `BUDGET_EXCEEDED` error-taxonomy row.

## Prevention

- When adding a field consumed by a service, add it to EVERY parse gate on the request path (route + ext route), and add a **route-level** test that goes through the actual `bodySchema.parse` — service-level tests that construct the input object bypass the transport contract ("half-A + half-B ≠ A∘B").
- When a default/config seeds a scoped value, verify the enforcement path actually evaluates that scope for the *target entity shape* (here: `root_run_id=NULL` standalone runs). Add a test that drives the seeded default through the enforcement path, not just the seeding function.
- A terminal-state flip that is meant to *stop external work* must also tear down the external resource (ACP session), mirroring the sibling path's discipline; reusing a DB-only cascade for a "stop spending" goal needs an explicit session-kill step.
- A "trailing streak" meter evaluated against a row set that includes the row under evaluation must exclude that row; a test asserting the meter trips must use a physically realizable ordering (monotonic `started_at`).
- Cross-coupled limits (`maxTokens` ↔ `hardMaxTokens`) must resolve together: an override of one re-derives the dependent unless the dependent is also explicitly overridden.
- A newly-authored API contract surface (Data* schema, enum) must enumerate every value the code can actually emit — grep the emit call sites, not just the variant you added.
