# Router-sidecar DELETE: orphan-on-failed-stop, check-then-delete race, and a leaky PATCH contract

**Date:** 2026-06-21 02:04
**Files:** web/app/api/admin/router-sidecars/[sidecarId]/route.ts, docs/api/web.openapi.yaml, docs/system-analytics/acp-runners.md, web/app/api/admin/router-sidecars/[sidecarId]/__tests__/delete.test.ts, web/app/api/admin/router-sidecars/[sidecarId]/__tests__/delete.integration.test.ts (new), web/app/api/admin/router-sidecars/__tests__/route.test.ts
**Severity:** high

## Problem

Adversarial review of the sidecar create/edit/delete branch surfaced three defects in `DELETE`/`PATCH /api/admin/router-sidecars/{sidecarId}`:

1. **(high) Orphan on failed stop.** `DELETE` wrapped `stopSidecar()` in a `try/catch` that swallowed the error and deleted the catalog row anyway, returning `204`. The supervisor can live on another host, so a fetch error / 503 / timeout does NOT prove the managed CCR process died — the row (the only handle to stop it later) was gone while the process kept running. A unit test explicitly enshrined this unsafe `204`.
2. **(high) Check-then-delete race.** The usage-guard read (`loadSidecarUsageReferences`) and the `db.delete` ran with no transaction/lock between them. Because `platform_acp_runners.sidecar_id` is `ON DELETE SET NULL`, a runner bound to the sidecar in that window was silently unbound by the delete — contradicting the documented "refuse rather than unbind" semantics.
3. **(medium) Leaky PATCH contract.** `patchBodySchema` (strict) still listed `readinessStatus`/`readinessReasons`; the route then overwrote them with server-computed values. They were silently accepted-then-ignored, and OpenAPI documented them as request fields while claiming readiness is server-authoritative.

## Root Cause

A deliberate-but-wrong tradeoff plus two oversights:

- (1) The author chose "a down supervisor must not block removing the config row" (a code comment said so), prioritizing config-cleanup convenience over the process-orphan leak. Owner reaffirmed safety wins: an unconfirmed stop must abort the delete.
- (2) The guard was written as a sequential read+delete; correctness under concurrency needs the read and the delete to be one serialized unit. The codebase already has the right primitive (`db.transaction` + `.for("update")`) but it wasn't applied here.
- (3) Readiness fields leaked into the input schema because the row's full shape was copied into the `.strict()` body schema instead of only the caller-mutable subset.

## Solution

- **Findings 1+2 together:** rewrote `DELETE` as ONE `db.transaction` that (a) `SELECT ... FOR UPDATE`s the sidecar row, (b) runs the usage-guard inside the lock, (c) for a `managed` sidecar stops it and lets a failure **propagate** (aborting the transaction → no delete), (d) deletes. The `FOR UPDATE` lock conflicts with the `FOR KEY SHARE` lock a runner-bind FK takes on the same parent row, so a concurrent bind serializes behind the transaction; after the delete commits the bind fails its FK. This matches the dominant codebase convention (40+ `db.transaction` sites; `.for("update")` in hitl/users/runs-recover/flows; `runner-graph.ts:179` documents it as a Postgres-only lock with SQLite relying on write serialization). No migration; FK semantics unchanged.
- Added `EXECUTOR_UNAVAILABLE → 503` to the route's `statusForCode` (the universal convention across ~20 routes; this route was the only one missing it). An unconfirmed managed stop now returns `503` and keeps the row.
- **Finding 3:** removed `readinessStatus`/`readinessReasons` from `patchBodySchema` (the `.strict()` schema now rejects them with `CONFIG`/422) and from the OpenAPI PATCH request body; updated the PATCH/DELETE descriptions.
- Tests: flipped the unit test that enshrined `204`-on-stop-failure to assert `503` + no delete; added a `422` test for a PATCH body carrying readiness fields; added `delete.integration.test.ts` against a real Postgres testcontainer covering happy/guard/**rollback** and a deterministic **race** proof (a barrier-blocked stop holds the transaction open while a concurrent bind is shown to block on the lock, then fail its FK after commit).
- Docs: updated `acp-runners.md` Expectations + Edge cases to the new serialized/confirmed-stop/strict-readiness contract.

## Prevention

- A guard that reads then mutates across an `ON DELETE SET NULL` (or any nullable FK) MUST be serialized — wrap read+mutate in `db.transaction` with `.for("update")` on the parent row. A bare check-then-act is a race by default.
- A "best-effort" side effect before an irreversible delete is only safe if the side effect failing is genuinely harmless. When the deleted row is the only handle to undo the side effect (here: stop a process), the side effect must be CONFIRMED before the delete, not swallowed.
- A `.strict()` request schema must list only caller-mutable fields. Server-authoritative/derived fields belong in the response schema only; copying a row's full shape into the input schema silently accepts-and-ignores them and drifts the OpenAPI contract.
- New error paths must reuse the project's `statusForCode` conventions (`EXECUTOR_UNAVAILABLE → 503`), never fall through to a generic 500.
- Concurrency invariants deserve a deterministic integration test that fails WITHOUT the fix — a controllable barrier (here, a blocking stub inside the locked transaction) turns a non-deterministic race into a reliable assertion.
