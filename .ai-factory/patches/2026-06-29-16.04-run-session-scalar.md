# Active run session scalar subqueries can lose outer column qualification

**Date:** 2026-06-29 16:04
**Files:** web/lib/runs/active-run-session.ts, web/lib/runs/__tests__/active-run-session.test.ts
**Severity:** high

## Problem

A fresh flow run could execute normally and create HITL assignments, but the run
detail page crashed with:

```
MaisterError: Run <runId> has no capability agent in runner snapshot
```

The concrete run had valid `run_sessions.capability_agent = 'claude'` and a
valid runner snapshot. `getRunDetail` loaded, but `getRunSettings` returned null
for the active session scalar projections.

## Root Cause

`activeRunSessionScalar` embedded the outer `runs.id` column directly inside a
raw Drizzle SQL template. In simple `FROM runs` selectors Drizzle rendered that
correlated reference as a bare `"id"`, producing:

```
WHERE rs.run_id = "id"
```

Inside the subquery Postgres resolves `"id"` against `run_sessions.id`, so the
active session lookup compared `run_sessions.run_id` to `run_sessions.id` and
returned no row.

## Solution

Build the correlated run id reference as an explicit qualified identifier with
`sql.identifier(getTableName(runIdCol.table))` and `sql.identifier(runIdCol.name)`,
so the scalar subquery renders:

```
WHERE rs.run_id = "runs"."id"
```

Added a focused unit test that compiles the query to SQL and asserts the
qualified outer reference is present.

## Prevention

- When a correlated scalar subquery is built with raw Drizzle SQL, assert the
  generated SQL for outer column qualification.
- Do not rely on `AnyColumn` interpolation inside nested raw SQL to preserve
  table qualification.
- For active run session read-model changes, smoke-check both `getRunDetail` and
  `getRunSettings`; the latter is the simple selector that exposed this bug.
