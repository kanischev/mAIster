# Gate output back-ref written after terminal transition (both declared-output paths)

**Date:** 2026-06-10 23:44
**Files:** web/lib/flows/graph/gates-exec.ts, web/lib/flows/graph/__tests__/gates-exec.integration.test.ts
**Severity:** low

## Problem

`gate_results.outputArtifactRef` was UPDATEd *after* `markGatePassed` /
`markGateFailed` in BOTH declared-output paths of the gate executor (the plain
`artifact_required` pass path and `runMutationAssertionGate`). A process death
between the terminal transition and the UPDATE leaves a terminal gate row
permanently missing its back-ref — terminal gates are never re-executed, so no
recovery path heals it. Latent today (no reader of the column yet), found by
`/aif-review` crash-window analysis on `feature/harness-loop-foundation`.

## Root Cause

The back-ref UPDATE was appended after the transition as a logical
afterthought ("set the link once the verdict is in") instead of being grouped
with the evidence writes that intentionally precede the terminal transition.
The module already had the correct local invariant — D-C4: "record the report
BEFORE the terminal gate transition; a crash between leaves the gate `running`
so a rework re-executes it" — but it was applied only to the artifact insert,
not to every write tied to the same gate result row. Any write sequenced after
a terminal status flip inherits a permanent-inconsistency window because
terminal rows are never revisited.

A second instance existed because the plain-path setter and the mutation-path
setter were written as separate hunks in the same commit; the review initially
spotted only the mutation-path one. The sibling surfaced during the fix via
the "sweep sibling sites of the same pattern" rule.

## Solution

Moved both `outputArtifactRef` UPDATEs to before their terminal transitions
(`markGatePassed` / `markGateFailed`). Now a crash in the window leaves the
gate `running`; re-execution re-records the artifact (deterministic-id upsert)
and idempotently re-sets the back-ref. No signature changes; pino `log.debug`
lines added at both sites for field verification. Regression assertion added
to the existing failing-mutation-gate integration test:
`expect(gate?.outputArtifactRef).toBe("verify-report")` on the FAIL path (the
pass path was already asserted by the pre-existing declared-output test).

Also from the same `/aif-fix` session — a review-finding RETRACTION worth
remembering: the suggested React-key fix in
`control-effectiveness-card.tsx` was a false positive. `runs.flowId` is an FK
to `flows.id` (the **per-project** PK), so `flowId:nodeId:gateId` is already
unique across projects; `flowRefId` is the **cross-project package identity**
(`ON CONFLICT (project_id, flow_ref_id)`), and "fixing" the key to use it
would have INTRODUCED the collision the review claimed to prevent. No change
applied there.

## Prevention

- Invariant to enforce in review and code: once a row reaches a terminal
  status, NO further writes to that row may be sequenced after the flip. Every
  field belonging to the outcome (back-refs, evidence links, payload columns)
  is written either in the same UPDATE as the transition or strictly before
  it, relying on the re-execution-of-non-terminal-rows recovery path.
- When a crash-window finding is confirmed on one call site, grep the column
  name across the file/module before fixing — afterthought writes ship in
  pairs when two paths were authored in one commit.
- Before flagging an identifier-collision (React keys, Map keys, dedup keys),
  verify the identifier's SCOPE in `lib/db/schema.ts` — do not infer it from
  the name. In this repo: `flows.id` (= `runs.flowId`) is per-project-install;
  `flows.flowRefId` is the package identity shared across projects;
  `flow_revisions` are global content-addressed. A "more specific-sounding"
  name is not a more specific scope.
