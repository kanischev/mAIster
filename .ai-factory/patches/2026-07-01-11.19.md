# Artifact body injection (ADR-120) — adversarial-review round 2

**Date:** 2026-07-01 11:19
**Files:** web/lib/flows/graph/artifact-inject.ts, web/lib/flows/graph/runner-graph.ts, web/lib/flows/flow-dsl-grammar.ts (+ tests: artifact-inject.test.ts, artifact-inject.integration.test.ts, flow-dsl-grammar.test.ts; + docs: decisions.md, flow-dsl.md, system-analytics/artifacts.md)
**Severity:** high

## Problem

A second Codex adversarial-review pass on the ADR-120 feature found two defects:

1. **[high] Eager artifact-content resolution can bypass skipped-gate semantics.**
   `runGraph` pre-resolves EVERY collected content ref (including gate-only refs)
   before `runNodeGates` applies execution policy. A non-review `command_check`/
   `skill_check` gate that is policy-skipped (`checks=skip`, axis A3) but whose
   command references a `current` file artifact with a gone/notfound payload
   threw `CONFIG` during pre-resolution → the node failed before the gate could be
   recorded as skipped. An explicitly-skipped/degraded gate became a run-failing
   pre-spawn error.

2. **[medium] In-product Flow authoring grammar taught the old contract.**
   `flow-dsl-grammar.ts` is the SSOT shown to the Studio assistant every turn and
   shipped as the `/flow-authoring` skill. It still listed
   `input.requires: [{ artifact, kind }]` only — omitting `inline: true`,
   `artifacts.<id>.content`, the node-type restriction, and the `2.2.0` floor — so
   generated/authored flows would keep using the pre-ADR-120 contract even though
   the runtime schema + docs had shipped it.

## Root Cause

1. Round-1 introduced ONE content-resolution pass (`resolveNodeArtifactContents`)
   with ONE failure policy (gone → hard-fail before spawn). That policy is correct
   for the ACTION surface (the agent/cli spawn genuinely needs the body) but wrong
   for GATE refs, whose evaluation is conditional on execution policy and happens
   later. The pass conflated "content needed to spawn" with "content needed if a
   gate runs" — a lifecycle/ordering mismatch.

2. The plan's contract-surface list traced `docs/` but missed the in-CODE grammar
   SSOT (`flow-dsl-grammar.ts` + its drift guard). A contract surface that lives
   in code, not `docs/`, was invisible to a docs-only surface sweep.

## Solution

1. Split the collector: `collectActionContentArtifactIds` returns only the
   action-surface ids (action.prompt + action.command + inline requires);
   `collectContentArtifactIds` (the shared floor-gate + full-resolution scan) stays
   the superset (adds gate-field refs). In `resolveNodeArtifactContents`, a
   gone/notfound payload hard-fails ONLY for action-surface ids; a gate-only ref's
   gone payload is skipped (WARN) and deferred to gate execution — which honors the
   skip decision (a skipped gate never renders it; a running gate fails at its own
   `renderStrict`). No `checks` threading needed; correct regardless of policy.

2. Updated `flow-dsl-grammar.ts` to document `inline?: true`,
   `artifacts.<id>.content`, the node-type restriction, the slug-id grammar, the
   256 KiB cap, and the `2.2.0` floor. Extended the drift-guard test
   (`flow-dsl-grammar.test.ts`) to assert those tokens are present.

Logging: reused module-local `pino` (a distinct WARN for the deferred gate-only
gone vs. the fail-before-spawn action gone) — no `console.log` (project `no-console`).

## Prevention

- **Match failure policy to lifecycle stage:** content needed to SPAWN must
  resolve-or-fail eagerly; content needed only if a downstream, policy-gated step
  runs must resolve tolerantly and fail (if at all) at that step, after its
  enable/skip decision. When one resolution pass serves multiple consumers with
  different execution conditions, partition the inputs by consumer, not by a single
  global failure mode.
- **Contract surfaces include code SSOTs, not just `docs/`:** when a schema/DSL
  changes, grep for in-code grammar/prompt/assistant SSOTs (`flow-dsl-grammar.ts`,
  authoring skills) and their drift guards, and update them in the same change —
  the "trace every contract surface" rule is not satisfied by `docs/` alone.
