# Local-package BOM path traversal + flow-rename manifest/flow.yaml desync

**Date:** 2026-06-29 01:03
**Files:** web/lib/flows/package-content.ts, web/lib/queries/package-bom.ts, web/lib/queries/packages.ts, web/lib/local-packages/bom.ts, web/lib/local-packages/rename-artifact.ts, web/lib/local-packages/__tests__/bom.test.ts, web/lib/local-packages/__tests__/rename-artifact.test.ts, docs/system-analytics/local-packages.md
**Severity:** critical

## Problem

Two defects in the ADR-115 local-package composition editor, found by an
adversarial Codex review of the branch:

1. **(critical) Path traversal in local BOM flow loading.** `localPackageSource`
   parsed `maister-package.yaml` with the *lenient* editor parser, took
   `flows[].path`, and loaded each flow via
   `loadFlowManifest(join(workingDir, flowPath, "flow.yaml"))` with no
   confinement. A crafted `flows[].path` (`../…`, absolute, symlink) made the
   Studio edit RSC read/compile a `flow.yaml` outside the package working dir and
   surface its graph/tooltip metadata to the browser.

2. **(high) Flow identity rename desynced manifest id from flow.yaml name.** The
   card-level flow rename rewrote `maister-package.yaml` (`id`/`path`) and moved
   the `flows/<old>/` dir, but left the moved `flow.yaml`'s top-level `name`
   stale. The installer enforces `flow.yaml name === manifest flow id`
   (`lib/packages/install.ts:210`), so a renamed flow saved a package that could
   never be cut/installed — discovered only at cut time, far from the action.

## Root Cause

1. The installed `PackageSource` was safe by *provenance* (its manifest paths are
   validated at install time), so when the shared builder was extended to a
   *local* source, the same `join(root, untrustedPath)` shape was copied without
   re-checking that the local manifest is untrusted (lenient parse of a
   user-editable file). The file readers (`readInstalledPackageFile`) already
   route through the canonical confinement gate; only the flow loader bypassed it.

2. The rename treated "identity" as "the manifest entry + the directory name" and
   missed that a flow's identity is bound by a *third* artifact — the `flow.yaml`
   `name` — via an installer invariant living in a different module. Editing two
   of the three coupled files is the classic partial-update bug.

## Solution

1. Added one confinement gate `resolveConfinedFlowYaml(root, flowDirPath)` in
   `package-content.ts` (reusing the existing `resolveConfinedFile` — `..`/abs/
   symlink/size-cap), and routed **all three** manifest-flow-path→`loadFlowManifest`
   sites through it: `localPackageSource.loadFlow` (the vulnerable one),
   `installedPackageSource.loadFlow`, and `getStudioPackageFlowGraphs`
   (defense-in-depth + DRY — removed the triplicated `join(...,"flow.yaml")`). An
   escaping/missing path returns null → the loader throws `PRECONDITION` → the
   per-flow `try/catch` degrades to an id-only card, and the confinement gate logs
   the reject once. The installed-source characterization snapshot stayed
   byte-identical, proving the reroute is non-regressive.

2. In the flow-rename branch, also rewrite the moved `flow.yaml`'s `name` to the
   new id using `parseDocument` (preserves comments + key order, unlike
   parse/stringify); an unparseable moved `flow.yaml` fails the rename with
   `CONFIG` instead of shipping a stale name.

Synced the analytics doc (rename flowchart, the "is a defect" edge-case bullet,
the rename expectation, and a new BOM-confinement expectation) so the spec no
longer contradicts the corrected behavior.

## Prevention

- **Sibling-sweep on any boundary-validation fix (project rule).** Grepped the
  whole `lib` tree for the sink class
  (`loadFlowManifest|join(root,…,"flow.yaml")`): the three fixed sites plus
  `service.ts:583` `registerFlowElementInManifest` — the latter is immune (it
  *derives* the id from `flow.yaml name`, the inverse-correct pattern) and not in
  this branch's diff, so left untouched. Recording the grep is the rule.
- When a "trusted-by-provenance" source abstraction (installed) is generalized to
  an untrusted one (local working dir, lenient parse), re-audit every path that
  reaches an `fs`/`loadFlowManifest`/`spawn` sink — provenance safety does not
  transfer across the new source.
- When an artifact's identity is enforced by an invariant across multiple files
  (`manifest.id === flow.yaml.name`), any "rename identity" op MUST move *all*
  coupled files in one atomic draft op; assert the invariant directly in a test.
- A characterization snapshot is the cheapest guard that a "confine/refactor the
  shared path, leave installed behavior identical" change actually stayed
  identical — keep it.
