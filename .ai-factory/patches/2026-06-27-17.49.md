# Workbench stop missed mid-prompt live agent (split-brain) — matched by run_sessions.acp_session_id instead of runId

**Date:** 2026-06-27 17:49
**Files:** web/lib/workbench-lifecycle/service.ts, web/lib/workbench-lifecycle/__tests__/{service,race,handoff,real-git.integration,workbench-stop.integration}.test.ts, docs/system-analytics/workbench-lifecycle.md
**Severity:** critical

## Problem

`stopWorkbenchRun` (and `stop-archive` / `stop-drop` / ext token cancel, for
both `flow` and `agent` runs) could mark a run terminal (`Review` / `Abandoned`)
while its agent process kept running and mutating the worktree — split-brain.
`stopLiveSupervisorSession` looked up the run's live supervisor session through
`listRunSessionAcpIds(runId)`, which reads `run_sessions.acp_session_id` and
filters out NULLs. During an in-flight graph node the column is still NULL, so
the set was empty, the helper returned `false`, no `deleteSession` fired, and the
caller parked the run anyway. Found by Codex adversarial review.

## Root Cause

The graph runner persists `run_sessions.acp_session_id` only AFTER
`executeNodeAction` returns (`runner-graph.ts` — createSession + the full prompt
turn run first, the column is written at the end). So there is a long window —
the entire duration of a node's prompt — where the supervisor has a live session
(its record carries `runId` and `acpSessionId`) but the DB row's
`acp_session_id` is NULL. Any teardown that keys off that column instead of the
always-present, server-owned `runId` is blind during exactly the window when a
user is most likely to hit Stop. The supervisor `SupervisorSessionRecord` already
carries a required `runId`; the helper just keyed on the wrong field.

The tell: the keepalive time-limit watchdog (`keepalive-sweeper.ts:580-593`)
already carried a verbatim comment describing this exact hazard and matched by
`(runId, stepId)`. gate-chat, budget-terminate, and the tree-terminate paths
also already matched by `runId`. The stop helper was the one sibling that never
got the same treatment — a missed-recorder, not a new discovery.

## Solution

Match the run's live supervisor sessions by `session.runId === ctx.run.id` on
each `listSessions()` record (status `live`), then `deleteSession`. For Stop we
want EVERY live session of the run, so we do NOT narrow by `stepId` (that is only
for the watchdog's single capped node). Removed the now-orphaned
`listRunSessionAcpIds` dep (interface + `defaultWorkbenchLifecycleDeps` impl) and
the now-unused `loadRunSessions` import. Added a `log.info` at the kill site
(matches the watchdog's logging idiom). Rewrote the two stop unit tests to drive
via `listSessions()` records keyed by `runId` and added a mid-prompt regression
test: a live record for the run/current step with no acp id, asserting
`deleteSession` runs BEFORE `markStoppedAndCloseAssignments`. Fixed the doc line
in `workbench-lifecycle.md` that still described the acp-id matching.

## Prevention

- A "find the live supervisor session for this run" lookup MUST key on the
  server-owned `runId` from the `listSessions()` record, never on
  `run_sessions.acp_session_id` — that column is NULL for the whole mid-prompt
  window. `(runId, stepId)` only when you need exact-node protection (a single
  capped/parked node); plain `runId` when you mean "all of this run's live
  sessions" (stop/terminate).
- When fixing one instance of a "match live session" defect, grep EVERY
  `listSessions()` caller and every `deleteSession`/`checkpointSession` site and
  confirm each keys on `runId` (or a persisted/in-hand supervisor `sessionId`,
  e.g. scratch's `supervisorSessionId`), not the lazily-persisted acp column.
  This bug was the last un-swept sibling of a fix already applied to the
  watchdog months earlier — the class fix never reached it.
- Split-brain check for any terminal/stop transition: the live process MUST be
  torn down BEFORE the run is flipped terminal. Assert call ordering in the test.
