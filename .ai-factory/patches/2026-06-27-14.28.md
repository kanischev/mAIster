# M42 run_sessions cutover left non-flow paths + the entire integration seed surface writing dropped columns

**Date:** 2026-06-27 14:28
**Files:** web/lib/scratch-runs/service.ts, web/lib/agents/launch.ts, web/lib/flows/graph/consensus/drafts.ts, web/lib/services/gate-chat.ts, web/app/api/scratch-runs/[runId]/recover/route.ts, web/app/api/scratch-runs/[runId]/route.ts, web/app/api/runs/[runId]/hitl/[hitlRequestId]/chat/route.ts (+ 37 test files, +8 docs)
**Severity:** critical

## Problem

M42 (ADR-114) moved per-run runner identity + the ACP resume handle off the
`runs` table (migration 0082 dropped `runner_id, runner_resolution_tier,
capability_agent, runner_snapshot, acp_session_id`) onto a new `run_sessions`
table. The cutover was applied to the FLOW launch path, all read-models, the
graph runner, and resume/recover — but NOT to the scratch-run, standalone-agent,
and consensus-draft paths, nor to gate-chat / the scratch recover+detail routes.
Those still wrote and read the dropped columns. The branch shipped with
"gates green" — but the green was **unit-only**; the full integration suite was
never run and was broadly red (baseline: 76 failed files / 439 failed tests).

## Root Cause

Two interacting blind spots:

1. **Drizzle hides dropped-column INSERTs but not UPDATEs.** `db.insert(runs)`
   builds its column list from the *table definition*, so a stale
   `runnerSnapshot:` key is silently dropped — no error, no data. But
   `db.update(runs).set({ acpSessionId })` (acpSessionId the only key) reduces
   to an empty `SET` clause (the table has no `onUpdate` column) → Postgres
   "syntax error at or near WHERE" → a crash AFTER the supervisor session spawns
   (leaking it). Reads off a `runs` row just return `undefined`. So tsc (handles
   typed `any`) AND mocked unit tests both pass while real Postgres breaks.

2. **Raw-SQL test seeds hard-fail; Drizzle test seeds silently mis-seed.** ~73
   integration files seeded the dropped columns: 9 via raw `INSERT INTO "runs"
   (... "runner_snapshot" ...)` (hard "column does not exist") and ~64 via
   Drizzle `insert(runs).values({...})` (silent drop → reads via `run_sessions`
   return null → assertions fail). Nobody ran `pnpm test:integration`, so this
   was invisible.

## Solution

- **Production reads:** adopted the established `resume.ts`/`recover.ts`
  merge-at-load idiom — at each `const run = db.select().from(runs)` site that
  feeds a dropped-column read, merge the active session's fields once
  (`loadActiveRunSession`) so every downstream `run.*` read (and cross-function
  `args.run.*`, e.g. the consensus-draft child) sees live values. gate-chat
  dropped its duplicate local `activeRunSessionAcpId` helper for the shared one.
- **Production writes:** replaced every `.update(runs).set({ acpSessionId })`
  with `persistRunSessionAcpSessionId(db, runId, "default", acp)`, and stripped
  the now-dead runner keys from the `insert(runs)` calls (the `run_sessions`
  row was already inserted alongside via `defaultRunSessionValues`).
- **Test seeds:** moved the runner fields from each `runs` insert into a
  parallel `run_sessions` insert (`session_name: "default"`, FK-ordered after
  the run). For raw-SQL seeds, dropped the columns + renumbered `$N` only when
  the runner param wasn't trailing. Fixed two M42-behavior assertions
  (`flow_runner_remaps` re-keyed `stepId`→`slotKey: "session:implement"`; a
  Mapped remap resolves on the `binding` tier, not `stepTarget`) and the
  `loaded.sessions` Map the engine now requires in hand-built `runGraph`
  fixtures. Unit route tests mock `@/lib/runs/active-run-session`
  (importOriginal + override) to derive the session from the run fixture.

Verified: web tsc 0, eslint 0, unit 5162/5162 (484 files), integration
1875/1875 (241 files), docs validate green, grep-to-zero on dropped-column
writes to `runs`.

## Prevention

- A column DROP is a **whole-tree** sweep, not a diff sweep: grep every reader
  AND writer of the dropped name (camelCase property + snake_case SQL +
  raw-SQL strings) across `web/` and test seeds — `tsc` is blind because DB
  handles are typed `any`, and Drizzle silently drops unknown INSERT keys.
- "Gates green" must mean `test:integration` ran, not just `test:unit`. The
  integration suite is the only thing that exercises real Postgres against a
  post-migration schema; mocked unit tests cannot catch a dropped column.
- When migrating per-row state to a side table, follow the established
  merge-at-load idiom (`resume.ts`) at the load site rather than touching each
  read; for writes, route through the one persist helper.
- A re-keyed table (here `flow_runner_remaps` → `slot_key`) is a contract
  sweep: grep tests for the old key columns and the old resolution tier.
