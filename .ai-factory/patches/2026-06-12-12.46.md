**Severity:** high

## Problem

Verification found three blockers in the multi-run cost and delivery-policy branch:

- `ai_rebase_merge` was exposed in policy schemas and UI but still rejected at promotion time with `PRECONDITION`.
- The full web unit suite failed because old launch service fixtures lacked `flow.projectId`, and stale assertions still expected manual run-again states to be non-launchable after the feature changed that contract.
- The branch's feature migration used `0045`, colliding with newer `main` migrations.

## Root Cause

The branch advanced across several contract layers without one final consistency pass:

- Task 14 intentionally left `ai_rebase_merge` as a placeholder, but the verification gate treated Task 15 as part of the branch completeness contract.
- `launchRun` had moved to the manual relaunch classifier, while two service-level tests still described the older board/latest-run classifier and hid the drift behind an early flow/project fixture failure.
- The migration was numbered against an older `main`; current `main` had since claimed `0045` and `0046`.

## Solution

`ai_rebase_merge` now maps to the existing durable rebase-merge side-effect path while preserving the `ai_rebase_merge` policy/response mode for API and webhook audit. Rebase conflicts surface through the existing `merge_conflict` assignment path, with structured INFO/WARN logs carrying `runId`, `projectId`, `promotionAttemptId`, source branch, target branch, parent repo path, and failing command.

Launch service tests now seed `flow.projectId`, clear fake latest-run state between seeded cases, and assert the manual run-again allow-list for `Done`, `Abandoned`, and `Crashed` states. The feature migration was renumbered to `0047`, the current `main` migration tail was restored into the worktree, and docs/journal references were aligned.

## Prevention

- When a UI exposes a policy enum value, route/service tests must prove it is executable or explicitly disabled with a documented state.
- When a route switches from board launchability to manual run-again launchability, migrate the service tests in the same change; do not let a different precondition mask stale assertions.
- Before final verification on long-lived branches, compare migration filenames and `_journal.json` against current `main` and renumber before running broad gates.
