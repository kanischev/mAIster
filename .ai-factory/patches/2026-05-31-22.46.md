# Multi-write state transitions stranded runs; CAS-after-insert raced to 500; new status/route not propagated to all read models/specs

**Date:** 2026-05-31 22:46
**Files:** web/app/api/runs/[runId]/takeover/return/route.ts, web/app/api/runs/[runId]/takeover/claim/route.ts, web/lib/queries/portfolio.ts, docs/api/web.openapi.yaml, docs/system-analytics/manual-takeover.md
**Severity:** critical

## Problem

An adversarial review of the M11b takeover feature (after it had passed
per-phase implementor→reviewer review and a full green gate) found 4 real
defects:

1. **(critical)** The two-phase takeover **return** could permanently strand a
   run. A crash between the committed side-effect transaction and the later
   status flip left the run `HumanWorking` with an *ended* takeover ledger row
   (`getActiveTakeover` filters `ended_at IS NULL` → retry threw `PRECONDITION`;
   the recovery sweep filters `status='Running'` → could not rescue it). A crash
   between the status flip and the cursor repark left the run `Running` with
   `current_step_id` still at the review node → the recovery sweep re-dispatched
   the graph runner at the **wrong node**, skipping the re-validation gates.
2. **(high)** Two concurrent takeover **claims** could return HTTP 500 instead of
   the documented 409. The route appended a `node_attempts` row (attempt =
   `max+1`) *before* the status CAS; with `UNIQUE(run_id,node_id,attempt)`, the
   losing INSERT raised a raw Postgres 23505 that was not a `MaisterError`.
3. **(medium)** A new real run status (`HumanWorking`) was added to the board
   read model but **not** to the portfolio (cross-project home) read model, so a
   claimed run vanished from the home grid while still holding a capacity slot.
4. **(medium)** A new state-changing route (`POST /api/runs/{runId}/abandon`)
   shipped with **no OpenAPI contract**.

## Root Cause

- **#1 — partial atomicity + asymmetric recovery.** An earlier fix had made *two
  of the four* writes atomic (the ledger side-effects) but left the status flip
  and the cursor repark as separate auto-commit writes. The recovery sweep was
  designed only for the *last* partial state (`Running` after the flip), so the
  *earlier* partial state (`HumanWorking` after the side-effect commit) had no
  rescuer. "Two-phase commit" was reasoned about as exception-handling, not as
  *process-crash* windows between independent commits.
- **#2 — side-effect before the guard.** The uniqueness guard that should make a
  concurrent loser deterministic was placed *after* a write that could itself
  collide on a different unique constraint, so the failure surfaced as an
  unmapped DB error instead of the intended domain conflict.
- **#3/#4 — incomplete fan-out of a new invariant.** A new enum value / new route
  was propagated to the surfaces named in the plan's file list, but not to *every*
  consumer (a second read model; the API spec). The plan/contract-surface table
  was the checklist, and it predated the later-added surfaces.

## Solution

- **#1:** fold all four writes — `recordTakeoverReturn`, `markDownstreamStale`,
  the `markReturnedToRunning` CAS, and the `current_step_id` repark — into the
  **single** side-effect transaction. Now there is no committed intermediate
  state: either the run is fully returned (`Running` + cursor at the re-entry +
  ended takeover row + staled gates) or fully not (still `HumanWorking`,
  retryable). Git ops stay *before* the tx (so a git failure is a clean 409 with
  no ledger write); the async runner resume stays *after* the commit (a death
  there is recovered by the sweep, which now reliably sees `Running` ⟺ correct
  cursor).
- **#2:** run the status CAS *first* inside the transaction; only the CAS winner
  performs the unique-constrained INSERT. The loser rolls back at the CAS →
  deterministic 409, never a raw 23505 → 500.
- **#3:** add the new status to the portfolio active-status set + map it to an
  existing workspace/agent presentation consistent with the board.
- **#4:** document the as-built route in the OpenAPI spec (read the handler
  line-by-line; do not guess statuses/body/auth) and reconcile the analytics
  edge-case tables.

## Prevention

- **Atomicity rule for state transitions:** when a single logical transition
  performs N persistent writes (ledger + status + cursor + …), commit them in
  **one** transaction/CAS. If they cannot be atomic, every reachable partial
  state must have an explicit, *tested* recovery path — enumerate the crash
  windows (process death *between* each pair of commits), not just the
  exception paths. A recovery sweep that filters on one status only covers
  partial states that reach that status.
- **Guard-before-side-effect:** when a uniqueness/contention guard exists
  (status CAS), run it *before* any other write that has its own unique
  constraint, so the loser fails at the guard (mapped domain error) rather than
  at a raw DB violation. Map error contracts (409 vs 500) and prove them with a
  real concurrency test, not a single-threaded one.
- **Fan-out a new invariant to ALL consumers:** a new enum value / status must be
  grepped across *every* read model (board AND portfolio AND any rail/home
  query), the scheduler/cap, the recovery sweeps, and the API spec — not only the
  files named in the plan. A new state-changing route must get an OpenAPI path in
  the same change.
- **Adversarial review pays for itself even after a green gate:** per-phase
  reviews caught the exception paths but missed the process-crash windows, the
  race-dependent 500, and the cross-surface drift. Budget one adversarial,
  whole-feature pass before merge; reproduce each finding against the code before
  accepting or rejecting it.
