# Authored Flow package validation boundary fixes

**Date:** 2026-06-08 13:05
**Files:** `web/lib/flows/package-authoring.ts`, `web/app/(app)/flows/actions.ts`, `web/app/api/projects/[slug]/catalog/caps/[capId]/publish-local/route.ts`, `web/lib/flows.ts`, `web/lib/flows/__tests__/package-authoring.test.ts`, `web/app/api/projects/[slug]/catalog/caps/[capId]/publish-local/__tests__/route.test.ts`, `docs/flow-dsl.md`, `docs/system-analytics/capability-catalog.md`
**Severity:** high

## Problem

Authored Flow package validation was applied on the `/flows` UI publish path
but not on the existing catalog API publish route. The authored install bridge
also entered the generic executable Flow installer without first enforcing the
authored package file contract. Package path validation normalized paths before
checking for `..`, so paths such as `schemas/../setup.sh` were accepted after
collapsing to `setup.sh`. Documentation also claimed project-context reference
validation for authored publish/export even though those paths do not receive
project role or capability registries.

## Root Cause

Validation was introduced at the newest UI boundary rather than centralized as
an authored Flow package contract. The bridge reused the generic Flow installer,
which validates `flow.yaml` but intentionally does not validate the authored
package file inventory. The path validator checked the normalized result rather
than the original path segments, and docs described the broader project-context
installer/load behavior as if it were part of authored publish/export.

## Solution

Added a shared `assertPublishableAuthoredFlowRevision` helper and wired it into
both UI publish and catalog API publish. The authored install bridge now reads
and validates the authored package directory before invoking the generic
installer. Package path validation rejects `..` in original slash/backslash path
segments before normalization. Regression tests cover API publish refusal,
bridge refusal for invalid package bytes, and original `..` path rejection.
Docs now state that project reference checks run only on install/load/launch
paths that provide project registries.

## Prevention

When a review finds a validation gap, audit every entrypoint that reaches the
same state transition: UI actions, HTTP routes, CLIs, and bridge functions. For
path validators, test both normalized and original input forms. For docs, keep
authored package validation separate from project-context Flow loading and
launch validation unless the code passes those registries explicitly.
