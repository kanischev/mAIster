# M17 adversarial-review remediation — rework evidence binding + stale ext HITL list + OpenAPI drift

**Date:** 2026-06-05 15:30
**Files:** web/lib/flows/runner.ts, web/lib/flows/graph/default-artifacts.ts, web/lib/queries/hitl.ts, docs/api/external/operations.openapi.yaml, docs/api/web.openapi.yaml (+ 2 integration tests)
**Severity:** high

## Problem

A Codex adversarial branch review of M17 surfaced 4 valid issues that all-green
unit/integration suites had NOT caught:

1. (high) Linear rework evidence recorded under attempt 1. After P3 made the flat
   runner re-execute steps on `on_reject` (incrementing `step_runs.attempt`), the
   success path still called `recordDefaultArtifacts({ attempt: 1 })` — so a
   reworked step's artifacts attached to the wrong attempt. Worse, the human_note
   lookup used `limit(1)` with NO `orderBy`, so after reject→approve it could bind
   the artifact to the earlier *reject* instead of the final approve.
2. (high) `getHitlRequestsForRun` (the new ext `GET …/hitl` read) returned every
   unresponded HITL row regardless of `runs.status`, so an external/MCP client
   could see stale prompts from Running/HumanWorking/Failed/Abandoned/Done runs.
3. (medium) The external OpenAPI `ExtHitlRequestDTO.options` was `array<string>
   nullable`, but the route returns `{optionId,label}[]` (never null).
4. (medium) The session OpenAPI `respondHitl` body still carried M7/M8 prose
   saying the runner does NOT route on `on_reject.goto_step` — contradicting the
   P3 repark implementation.

## Root Cause

- The aggregate suites were green because no test exercised the *combination* of
  a new behavior (P3 rework re-execution) with an older code path
  (`recordDefaultArtifacts(attempt: 1)` + an order-less HITL lookup from M12).
  Each was individually correct; their interaction was the bug.
- `getHitlRequestsForRun` was extracted in P1 as a thin "rows for a run" reader
  and only gained project-scope; the pending-status invariant that the board/
  inbox queries enforce was never propagated to it (a fan-out miss).
- The two OpenAPI drifts were FROZEN in Phase 0 with a guessed shape (options as
  strings) and an un-flipped legacy paragraph (M7 text not tagged `Designed — M17`,
  so the P7 as-built flip's M17-only grep didn't catch it).

## Solution

- runner.ts: track the executing step_run's real `attempt` per loop iteration
  (`currentAttempt`) and pass it to `recordDefaultArtifacts` instead of `1`.
- default-artifacts.ts: the human_note lookup now `orderBy(desc(createdAt))
  .limit(1)` — binds to the LATEST responded HITL (the final decision), fixing
  both the linear and graph rework paths.
- queries/hitl.ts: `getHitlRequestsForRun` now filters the run to
  `status ∈ {NeedsInput, NeedsInputIdle}` (returns `[]` otherwise), matching the
  inbox contract.
- Both OpenAPI files reconciled to the implemented shapes (options as objects;
  the M17 on_reject repark semantics incl. reject payload, comments_var,
  maxLoops, CONFIG on cap).
- Regressions: ext route "non-pending run → empty list" + options-shape contract
  assertion; default-artifacts "human_note binds to latest responded HITL +
  current attempt".

## Prevention

- When a phase makes a previously-single-execution code path RE-EXECUTE (rework,
  retry, repark), audit every consumer that hardcodes `attempt`/`1` or does a
  `limit(1)` without `orderBy` over per-attempt rows — they silently bind to the
  wrong pass. Add a reject→rework→approve regression whenever a loop is added.
- A "rows for X" reader that feeds an external surface MUST carry the SAME
  pending/visibility invariant as the in-app reader it mirrors (here: filter by
  `runs.status`). Treat the contract, not just the FK scope, as the spec.
- Freezing an OpenAPI shape in a docs-first phase is a guess until code exists —
  add a contract test asserting the EXACT response DTO (object vs string array)
  so spec↔route drift fails CI. The as-built doc flip must also sweep legacy
  (non-milestone-tagged) prose in the same contract, not just `Designed — <milestone>`
  markers.
