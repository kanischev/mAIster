# External-gate report: terminal-run guard was checked outside the lock (TOCTOU)

**Date:** 2026-06-03 00:52
**Files:** web/app/api/v1/ext/runs/[runId]/gates/[gateId]/report/route.ts, web/app/api/v1/ext/runs/[runId]/gates/[gateId]/report/__tests__/route.integration.test.ts (+ DB_URL capture/restore swept across 9 sibling integration tests: discard, takeover-lifecycle-fixes, takeover-resume, cron/gc, scheduler.integration, reconcile-sweep, scheduler-crash-promote, recover, crash-resume)
**Severity:** critical

## Problem

An adversarial Codex review of the M16 external-operations branch found that the
terminal-run guard on `POST /api/v1/ext/runs/{runId}/gates/{gateId}/report` was
checked **before**, not **under**, the run-row lock.

The route read `runs.status` in a pre-check outside the transaction (returning
409 for `Done|Abandoned|Crashed|Failed`), then opened a transaction whose first
statement was `SELECT id FROM runs WHERE id = $1 FOR UPDATE`. That lock query
selected only `id` and the code never re-read `status` after acquiring the lock.
If another request finalized the run in the window between the pre-check and the
lock, the report still wrote a gate result + `test_report` artifact on a
now-terminal run — directly violating the documented invariant that a report on
a terminal run returns 409 and mutates no gate/artifact state.

A second, lower finding: the route's integration test overwrote
`process.env.DB_URL` in `beforeAll` and **unconditionally `delete`d** it in
`afterAll`, corrupting the env for any process/worker that started with a
`DB_URL` set (e.g. a dev running `DB_URL=file:./dev.db`).

## Root Cause

- **#1 (the substantive one):** classic time-of-check-to-time-of-use. The
  immediately-prior patch (2026-06-02-18.44) added the terminal guard, but only
  at the pre-check layer — outside the lock. A transaction + a `FOR UPDATE` row
  lock gives mutual exclusion for the *act*, but the *check* it guards was still
  the stale pre-transaction read. Holding a lock is worthless if you don't
  re-read the guarded state under it: the `FOR UPDATE` serialized this report
  against a concurrent finalizer, but because `status` was never re-selected
  after the lock, a finalizer that committed `status = Done` first was invisible
  to the report, which proceeded to write anyway.
- **#2:** the test treated `delete process.env.DB_URL` as symmetric with the
  `beforeAll` assignment. It is only symmetric when `DB_URL` was originally
  unset. The established sibling pattern (`lib/__tests__/scheduler.test.ts`,
  `lib/db/__tests__/client.test.ts`) captures and restores; this file diverged.

## Solution

- **#1:** after the `FOR UPDATE`, re-read `{ id, status }` for the run UNDER the
  lock (`eq(runs.id, runId)` + `eq(runs.projectId, ctx.projectId)`), inside the
  transaction, before `reportExternalGate`. Missing row → throw
  `RunNotReportableError(404, NOT_FOUND)`; terminal status → throw
  `RunNotReportableError(409, CONFLICT)`. The throw rolls the transaction back
  (no gate/artifact write) and the route's `catch` maps the carried
  `httpStatus`/`code` to the response. The outer pre-check is kept as a cheap
  fast path; the locked re-read is the authoritative guard. `handleExt` still
  writes the failure audit on the >=400 response. The portable re-read runs on
  both dialects (SQLite serializes writes; the `FOR UPDATE` line stays
  Postgres-only).
- **#2:** capture `originalDbUrl = process.env.DB_URL` before the overwrite;
  in `afterAll`, `delete` only when it was originally `undefined`, otherwise
  restore the captured value.

Two regression tests were added to the report suite (now 14 tests):
1. **Deterministic TOCTOU:** spies `db.transaction` with `mockImplementationOnce`
   to flip the run to `Done` exactly in the window between the route's pre-check
   SELECT and the locked re-read, then delegates to the real transaction. Asserts
   409 `CONFLICT`, gate still `pending`, no `test_report` artifact, 409
   failure-audit row. Red/green verified: with the in-transaction terminal throw
   disabled the route returns 200 (writes on the terminal run) and the test
   fails; restored, it passes.
2. **Genuine two-racer:** fires a run-finalizing `UPDATE … status=Done` and the
   report concurrently via `Promise.all`. Asserts the invariant rather than a
   fixed winner — the response is 200 **or** 409, and the gate/artifact state is
   consistent with whichever the `FOR UPDATE` serialization produced (200 →
   gate `passed` + 1 artifact; 409 → gate `pending` + 0 artifacts), with the run
   terminal at rest either way. Proves no interleaving writes a gate on an
   already-terminal run.

The DB_URL capture/restore fix (#2) was swept across all 9 sibling integration
tests that overwrote `process.env.DB_URL` (2 unconditionally `delete`d it; 7
leaked the container URI by never restoring). All now capture the prior value
and restore it (deleting only when originally unset). Full suite green: report
14/14, the 9 siblings 52 tests total.

## Prevention

- "Wrap in a transaction" ≠ "the check is safe", and even "take a `FOR UPDATE`
  row lock" ≠ "the check is safe". For any check-then-act on mutable row state,
  the authoritative check must be **re-read under the same lock** that guards the
  act. A pre-check outside the lock is a fast-path optimization only — never the
  decision.
- When a guard (terminal/ownership/version) is added, confirm it sits on the
  locked, in-transaction read path, not just the optimistic pre-read. Grep for
  the guard near the `FOR UPDATE`, not only near the route entry.
- Test a TOCTOU window deterministically by injecting the conflicting mutation in
  a seam that executes precisely between check and act (here: a one-shot
  `vi.spyOn(db, "transaction")` that mutates then delegates to the real tx).
  A plain `Promise.all` two-racer cannot reliably hit the sub-statement window.
- Env-var test fixtures (`process.env.*`) must capture-and-restore, never
  unconditionally `delete` (which corrupts the env when the value pre-existed)
  and never leak (overwrite-without-restore strands later code on a stopped
  container). Both failure modes existed across the sibling integration tests and
  were swept in this pass; `lib/__tests__/scheduler.test.ts` and
  `lib/db/__tests__/client.test.ts` already used the correct capture+restore
  idiom and are the reference.
