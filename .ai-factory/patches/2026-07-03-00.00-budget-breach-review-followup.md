# Budget breach review follow-up

## Problem

Review follow-up found that the budget-breach four-way fork still had
contract drift in crash-window recovery, UI availability rendering, and the
progress DTO budget math.

## Root causes

- Restart recovery had a staged marker but no durable way to find a new run
  that was already launched before the final HITL response stamp.
- Park recovery did not persist the preserved ref before the terminalization
  boundary, making a retry indistinguishable from an unrecoverable half-step.
- The budget HITL UI could synthesize local fallback actions instead of
  trusting server-computed availability.
- The progress DTO had a second budget-spend path instead of using the same
  metering helpers as the sweeper.

## Fix

- Added a deterministic `budget_restart` trigger payload and retry lookup so
  restart retries recover the existing new run instead of relaunching.
- Recorded the preserved park ref at the terminalized stage and made terminal
  park retries resolve without replaying preservation.
- Removed local budget-option synthesis from the HITL controls.
- Reused sweeper limit helpers and budget-meter helpers for read-side budget
  progress, while preserving field-level degradation.
- Added focused unit and real-Postgres integration regressions for the fixed
  paths.

## Prevention

- Crash-window composites need a durable idempotency key before any external
  side effect that cannot be rolled back.
- UI tests for HITL decisions must assert that availability comes from the
  server DTO, not from client-side fallback logic.
- Budget display code should import watchdog/sweeper budget helpers rather than
  reimplementing budget math.
