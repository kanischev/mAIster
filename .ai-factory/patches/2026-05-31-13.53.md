# M11b plan staled the wrong node set + shared a mutable e2e fixture under fullyParallel

**Date:** 2026-05-31 13:53
**Files:** .ai-factory/plans/feature-m11b-manual-takeover-timeline.md, .ai-factory/plans/feature-m11c-node-settings-enforcement.md
**Severity:** high

## Problem

An `/aif-improve` pass rewrote M11b's takeover-return staleness to call
`downstreamOf(graph, 'checks')` then `markDownstreamStale(runId, downstreamNodeIds, db)`.
A Codex adversarial review (F1) caught that this **excludes the re-entry node
`checks` from the stale set**, directly contradicting the same plan's AC-4 /
Verify #4 / Context text, which require `checks`/`judge`/`review` gate rows to
flip stale. Result: on a real takeover return the re-entry validation node could
keep an old `passed` gate that validated *pre-takeover* code, weakening the core
guarantee that the human's commits are re-validated before a fresh review. The
literal `'checks'` also hard-coded the aif graph, so non-aif flows would stale
the wrong region. A second finding (F2): broadening the `authed` Playwright
project to `m11a/b/c-*` specs while reusing one shared global seed fixture under
`fullyParallel: true` would let specs race on the same mutable run/worktree.

## Root Cause

**F1 — referenced a helper's semantics without reading them.** The plan author
assumed `downstreamOf(start)` *included* `start`. The as-built helper
(`web/lib/flows/graph/runner-graph.ts`) ends with `visited.delete(startNodeId)`
and a comment: "Exclude the start node itself — it is the rework target, not
downstream." That exclusion is *correct for rework* (the rework target is a
gate-free `ai_coding` node that simply re-runs) but *wrong for takeover return*,
where the re-entry is a **gate-bearing** node whose prior PASS must be
invalidated. The asymmetry between the two callers was never reasoned about —
the rework call site was pattern-matched without checking the new caller's
different requirement.

**F2 — extended a fixture's consumer set without re-checking its concurrency
contract.** The M11a seed was written for exactly one spec; the plan broadened
the consumer set (more authed specs) but kept `fullyParallel: true` and a single
shared parked run/workspace, introducing order-dependent races.

## Solution

F1: derive the re-entry node from the current `human_review` node's
`transitions.takeover` (not a literal id), and stale
`[reentryNode, ...downstreamOf(graph, reentryNode)]` — explicitly including the
gate-bearing re-entry node. Updated every dependent statement (Context, locked
decision #4, hard-dependency, AC-4, Phase 3.2/3.3/3.6, Verify #4) so the
mechanism and the stated outcome agree, and added a regression to the plan: the
re-entry node's prior `gate_results` must flip `stale` BEFORE the rerun creates
fresh attempts.

F2: prescribe per-spec isolated fixtures — parameterize `seed-e2e.ts` by a
unique key (worker index / spec slug) so each authed spec seeds its own
project/run/worktree, keeping `fullyParallel` safe. Applied the same requirement
to the M11c seed extension.

## Prevention

- When a plan/code references a shared helper, **read the helper's actual
  boundary semantics** (does it include/exclude endpoints, off-by-one, empty
  cases) before reusing it for a *new* caller — especially when copying a call
  site whose requirements differ from the new one.
- Treat "node ids" in a plan as **derived from the pinned manifest**, never
  hard-coded to one bundled flow's graph.
- Cross-check that a plan's prescribed **mechanism** matches its own
  **acceptance/verify** prose — a contradiction between the two is a logical hole
  a reviewer will (and did) find.
- When broadening the consumers of a shared test fixture, re-validate the
  **concurrency contract** (`fullyParallel`, shared mutable state) and require
  per-spec/per-worker isolation.
