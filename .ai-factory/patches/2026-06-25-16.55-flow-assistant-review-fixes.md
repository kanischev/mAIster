# Flow assistant write-time lock and package layout review fixes

**Date:** 2026-06-25 16:55
**Files:** web/lib/studio/flow-assistant/apply.ts, web/lib/studio/flow-assistant/turn.ts, web/lib/scratch-runs/service.ts, web/app/api/studio/local-packages/[id]/assistant/route.ts, web/app/api/studio/local-packages/[id]/assistant/[runId]/messages/route.ts, web/lib/flows/authoring-skill.ts, docs/system-analytics/studio-ai-assistant.md, docs/api/web.openapi.yaml, web/lib/studio/flow-assistant/__tests__/actions.test.ts, web/lib/scratch-runs/__tests__/local-package-assistant.integration.test.ts
**Severity:** high

## Problem

The Flow Studio assistant route checked the local-package editor lock before
sending a prompt to the ACP runner, but structured action application happened
after the runner turn without rechecking that the same editor session still held
the lock. A concurrent lock takeover could let a stale assistant response write
into a working directory the user no longer owned.

The flow-authoring skill and assistant analytics docs also taught the assistant
to create flat paths such as `flows/<id>.yaml` / `flows/default.flow.yaml`,
while the runtime package contract recognizes `flows/<id>/flow.yaml`.

The OpenAPI message route described structured action validation refusals as
HTTP 409s even though the implemented UX returns a normal assistant turn with a
sanitized `actionResult.status`.

## Root Cause

The lock token was treated as a route-level admission check instead of a
write-time precondition. The service and post-processing layers did not carry
the editor `sessionId` to the apply boundary, so the most important side effect
had no current lock assertion.

The assistant prompt docs were not tested against the same package path
recognition rules used by `validatePackageArtifacts` and package installation.
That let old layout examples drift after the canonical multi-flow package shape
settled on directory paths.

The API contract mixed transport errors with structured assistant action
results. Stale hashes, unsafe paths, invalid artifacts, and lost write locks are
expected assistant-turn outcomes, not necessarily route failures.

## Solution

`sessionId` is now part of the server-side local-package assistant launch and
message inputs. The scratch service asserts it before launch/turn and passes an
`assertCanApply` callback through `postProcessFlowAssistantTurn` into
`validateAndApplyFlowAssistantAction`.

The apply loop calls that assertion immediately before each write operation. If
the first operation loses the lock, it returns a sanitized `rejected`
`flow_action_result`; if the lock is lost after earlier operations, it returns
`interrupted`. Both paths append action-log records and structured pino WARN
logs without exposing raw action JSON to the user.

The authoring skill and analytics docs now use `flows/<id>/flow.yaml` and
manifest paths like `flows/bugfix`. OpenAPI now documents route-level 409s as
lock/run-package conflicts and states that structured action refusals return
`200` with `actionResult.status`.

## Prevention

- Treat body/session fields as concurrency tokens that must reach the final
  side-effect boundary, not just the first route handler.
- For AI-generated structured edits, test both the pure apply helper and the
  service path where the external agent response arrives after a concurrency
  window.
- Keep assistant prompt examples aligned with runtime validators; add a small
  regression test for canonical generated paths whenever prompt docs teach file
  layout.
- In OpenAPI, separate HTTP transport failures from successful turns that carry
  structured domain-result payloads.
