# Raw authored Flow YAML rejected during invalid draft save

**Date:** 2026-06-08 13:46
**Files:** web/app/(app)/flows/actions.ts, web/app/(app)/flows/__tests__/actions.test.ts, web/CLAUDE.md, .ai-factory/ARCHITECTURE.md
**Severity:** high

## Problem

The authored Flow editor contract says draft save may persist invalid content
while the author is working, but the update server action rejected an empty
`flow.yaml` editor with `CONFIG missing form field: flowYaml`. It also trimmed
raw YAML before storing it, changing the authored bytes that later validation,
hashing, and export depend on.

## Root Cause

`updateAuthoredFlowAction` reused the generic non-empty form-string helper for
`flowYaml`. That helper is correct for identifiers and titles, but wrong for a
source-text field whose raw value is the authored artifact. The action also read
several user-controlled fields before project authorization, which weakened the
auth-first boundary expected for server actions.

## Solution

The action now reads only `projectSlug` first, authorizes the server-resolved
project, and then parses the remaining fields. `flowYaml` uses a raw string
helper that accepts empty text and preserves whitespace. Regression tests cover
empty YAML saving as an invalid draft and raw YAML preservation. Stale context
docs were synchronized with the system-font fallback currently used by
`web/config/fonts.ts`.

## Prevention

- Treat source/artifact text fields differently from identifiers: validate that
  they are strings, but do not trim or require non-empty content unless the
  product contract says empty content is unsavable.
- Server actions should authorize from the smallest required project locator
  before parsing expensive or content-bearing request fields.
- Review authored-package fixes against both behavior tests and the docs that
  describe draft-validity semantics.
