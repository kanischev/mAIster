# Flow run selected-node layout overflow on long agent events

**Date:** 2026-06-30 14:12
**Files:** web/components/runs/flow-run-center.tsx, web/components/runs/node-transcript-panel.tsx, web/components/runs/__tests__/flow-run-center.test.ts
**Severity:** medium

## Problem

The flow run detail screen layout could break when some agent activity events
contained long, unbroken command or event text. The selected-node area expanded
horizontally and visually ran underneath the inspector column instead of keeping
the inspector and main content in separate columns.

## Root Cause

The selected-node layout used a CSS grid track ending in `1fr`. In CSS Grid,
`1fr` keeps an automatic minimum size unless explicitly constrained, so long
descendant content in transcript rows or result summaries could force the right
column wider than the available run-shell main column. Several intermediate
containers also lacked `min-w-0`, so existing truncation and overflow handling
inside `TranscriptView` could not fully contain the content.

## Solution

Changed the selected-node grid to use `minmax(0,1fr)` and added shrink-safe
container classes (`min-w-0`, `max-w-full`, and local `overflow-hidden`) around
the flow center, graph wrapper, node list, selected-node panel, result rows,
attempt prompt disclosure, and transcript panel. Compact metadata rows now use a
bounded second track and truncation so long metadata cannot reintroduce layout
expansion.

## Prevention

- Any dashboard/workbench grid column that contains logs, command lines,
  transcript messages, diff snippets, or artifact names should use
  `minmax(0,1fr)` instead of bare `1fr`.
- Intermediate flex/grid children in these surfaces need `min-w-0`; otherwise
  inner `truncate`, `overflow-auto`, or `break-words` classes may not take
  effect.
- Regression tests should assert the shrink-safe grid track and containment
  classes for components that render long agent output.
