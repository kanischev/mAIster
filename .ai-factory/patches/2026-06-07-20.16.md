# ADR number squatting + wrong migration number: M22 layout-reversal and M26 referenced ADRs that were never written

**Date:** 2026-06-07 20:16
**Files:** docs/decisions.md, docs/database-schema.md, docs/db/erd.md, docs/db/projects-domain.md, docs/system-analytics/workbench.md, docs/error-taxonomy.md, docs/api/web.openapi.yaml, docs/CLAUDE.md, docs/pv/{flow-authoring-assistant,improvement-roadmap}.md, web/CLAUDE.md, web/lib/db/schema.ts, web/lib/db/migrations/0030_m22_drop_flow_graph_layouts.sql, web/lib/config.schema.ts, web/lib/flows/graph/presentation-layout.ts, web/app/(app)/runs/[runId]/page.tsx, web/app/api/runs/[runId]/graph/__tests__/route.test.ts, web/e2e/{_seed/seed-e2e.ts,m22-workbench.spec.ts}, .ai-factory/ROADMAP.md, .ai-factory/specs/feature-m26-structured-output-run-context.md, .ai-factory/plans/feature-m26-structured-output-run-context.md
**Severity:** high

## Problem

On `feature/m26-structured-output-run-context`, two milestones referenced ADRs
that did not exist in `docs/decisions.md`:

- **M26** cited `ADR-063` 15× (spec, plan Task-2 marked `[x]` done, configuration.md,
  flow-dsl.md, flow-graph.md) with a dead anchor — but `decisions.md` was unchanged
  vs `main` and its max ADR was `062`. Phase-0 "spec-freeze gates everything; ADR-063
  complete" was claimed done while the ADR was absent.
- **M22 layout reversal** labelled itself `ADR-062` across ~25 sites (migration
  comment, `schema.ts`, 8 docs), but `decisions.md` `ADR-062` is *"Platform user
  administration"*. The layout-reversal decision had no ADR at all, and its anchor
  `#adr-062-authored-flow-graph-layout...` resolved to nothing.

Compounding it, the layout-table drop was documented as *"migration `0029`"* in 5
places, but the actual drop ships as `0030_m22_drop_flow_graph_layouts.sql`
(`0029` = `identity_admin_audit`, the admin-surface migration).

## Root Cause

1. **ADR numbers were narrated in specs/plans/docs/code before being reserved in
   `decisions.md`.** The plan even wrote "current max is 062" and reserved 063 for
   M26 — but the M22 layout work, developed in parallel, independently grabbed 062
   (already taken by the admin-surface ADR) and copied that ADR's `migration 0029`
   coordinates wholesale. Nothing forced the number to be allocated atomically with
   the prose that cites it.
2. **The Phase-0 docs gate could not catch it.** `pnpm validate:docs` only parses
   Mermaid blocks; it does not resolve cross-file `[ADR-NNN](decisions.md#...)`
   anchors. A missing ADR and a dead anchor both pass green, so the plan's
   "validate:docs:all green" checkpoint gave false assurance.
3. **A `[x]` checkbox is not evidence.** Plan Task-2 (write ADR-063) was marked done;
   the actual commit (`docs(m26): Phase 0 spec-freeze`) silently omitted the ADR.
   No verification re-derived "does the cited ADR exist at HEAD".

## Solution

- Wrote both ADRs in `decisions.md` with house-style headers whose generated anchors
  match the already-existing references: `### ADR-063: Structured node output channel
  (P1) + run-context file (P7)` and `### ADR-064: Authored flow-graph layout in the
  flow.yaml presentation section` (marked **Supersedes ADR-051**), plus index rows.
- **Picked numbers to minimize churn:** M26 kept `063` (already consistent everywhere,
  zero edits); the layout reversal — whose refs were all wrong anyway — became `064`.
  ADR numbers reflect *recording* order in `decisions.md`, not feature chronology.
- Repointed every **layout** `ADR-062` → `ADR-064` (token + the `#adr-062-authored...`
  anchor) across docs and code; left the **admin-surface** `ADR-062` refs (users /
  members error rows, audit columns, project-membership.md) untouched — verified by
  classifying each occurrence, never a blind `replace_all` in mixed files.
- Corrected the layout-drop migration `0029` → `0030` in all 5 sites; left the
  admin-surface `migration 0029` references intact.
- Reconciled two stale plan self-contradictions: `run.json` path
  (`<runDir>` → `<worktreePath>/.maister/run.json`, the B3 re-freeze location) and
  the env-var wiring ("compose" removed → `.env.example` + `configuration.md`, NOT
  `compose.yml`).
- Verified: `tsc --noEmit` 0, 26 affected tests green, `validate:docs` 8/8 mermaid,
  no remaining layout `ADR-062` / dead anchor / layout-`0029`.

## Prevention

- **Reserve the ADR number in `decisions.md` first** (even as a one-line stub), then
  cite it. Treat "cite an ADR that has no header at HEAD" as a build break, not a
  doc nit. The next free number is `max(### ADR-NNN)` in `decisions.md`, not the
  plan author's mental count.
- **Cross-file anchor validation added to the docs gate (DONE this session).**
  `scripts/validate-docs-adr-anchors.mjs` asserts every `](…decisions.md#adr-NNN-…)`
  link resolves to an existing `### ADR-NNN` header (bare `#adr-NNN` citation
  shorthand is accepted when the number exists; a full-title anchor must match the
  header slug exactly — which is what catches a missing ADR or a squatted number).
  Wired into `pnpm validate:docs` / `validate:docs:all` (so the Claude Stop hook runs
  it). Running it surfaced **2 pre-existing** stale anchors of the same class, now
  fixed: `executors.md` linked `#adr-045-platform-acp-runners…` (that title is
  ADR-050) and `workspaces.md` had a malformed `#adr-012-…-no-ff-…` slug (missing the
  `--no-ff` double hyphen). This is the gate that would have caught all three M26/M22
  defects at Phase 0.
- **Review-exit check (skill-context "re-derive every spec surface"):** for any diff
  touching a `### ADR-NNN` header, a migration file, or an `ADR-NNN`/`migration NNNN`
  citation, grep the cited number against `decisions.md` HEAD and the
  `migrations/meta/_journal.json` tag — never trust a `[x]` checkbox or a commit
  subject.
- **When two milestones develop in parallel,** allocate their ADR numbers up front
  from a single source so neither squats the other's number.
