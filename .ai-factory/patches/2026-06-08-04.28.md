# Reopened platform Flows plan after review-gated acceptance gaps

**Date:** 2026-06-08 04:28
**Files:** .ai-factory/plans/feature-platform-flows-management.md
**Severity:** medium

## Problem

The platform Flows management plan marked the first UI/action slice as
complete even though review findings showed unresolved acceptance gaps:
project-scoped create authorization, TDD coverage, publish validation,
EN/RU enum translation, specs-first ordering, AIF package artifact scope,
query style, parse-before-authz ordering, and listing filters.

## Root Cause

The plan treated a working prototype as an accepted phase. That allowed
deferred package-authoring invariants to contradict exposed Phase 1 behavior,
and it let broad test and i18n requirements sit in later tasks instead of in
the acceptance gate for the slice that introduced the UI.

## Solution

Reopened Phase 1 and rewrote the plan around explicit acceptance gates:
specs first, red -> green tests, project `manageCatalog` authorization,
authorize-before-parse ordering, publish validation, translated enum/status
copy, URL-synced filters, and typed installed-package read paths. Added a
concrete task to materialize AIF skills, rules, agents, schemas, setup, README,
and CLI/helper scripts into portable package artifacts.

## Prevention

- Do not mark plan tasks complete when implementation exists but acceptance
  criteria, tests, or required docs are still open.
- For review-fix runs, address all findings by default, including advisory
  findings that change task ordering or acceptance gates.
- Keep TDD, i18n, and authorization-axis requirements in the same phase as the
  UI/action surface they constrain.
- When a user asks for a portable package, schedule the real package artifacts
  explicitly instead of relying on future importer placeholders.
