# M11c settings.executors validation never ran in production (tested-but-unwired)

**Date:** 2026-06-01 12:16
**Files:** web/lib/config.ts, web/app/api/runs/route.ts, web/lib/runs/keepalive-sweeper.ts, web/app/api/runs/__tests__/route.enforcement.integration.test.ts, web/lib/runs/__tests__/time-limit-watchdog.integration.test.ts, docs/configuration.md, docs/api/web.openapi.yaml, docs/decisions.md
**Severity:** high

## Problem

AC-4 of M11c and the `flow-settings.md` Expectations state that manifest
validation MUST reject `settings.executors[]` ids absent from the project's
`maister.yaml executors[]`. The validation logic existed in
`config.ts:validateNodeSettings` and had passing unit tests — but the only
production caller of `loadFlowManifest` (`flows.ts:399`, install-time) invoked it
WITHOUT the optional `executorIds` set, and the launch path
(`POST /api/runs`) validated the already-pinned `revision.manifest` via
`compileManifest` without ever re-invoking the loader with an executor set. So
the rejection branch was dead in production: a manifest naming a non-existent
executor installed, enabled, and launched silently. The unit tests passed
because they supplied `executorIds` themselves — a false green.

Four secondary gaps were fixed in the same pass: judge-shape doc drift
(configuration.md listed `ai_coding`-only fields under `judge`), no test pinning
`maxCostUsd` as record-only, no test pinning trust-gate-before-enforcement
ordering, and a watchdog candidate query that required `acp_session_id IS NOT
NULL` (a capped node hanging before reporting a session id escaped the cap).

## Root Cause

A validation was implemented behind an OPTIONAL parameter (`opts.executorIds`)
and unit-tested by passing that parameter directly. No production call site
supplied it, and nobody traced whether one did. The deeper cause: the executor
*ref* id space (`executors.executor_ref_id`, the `maister.yaml` ids) is
project-scoped, while a flow package is generic across projects — so the
parse-time loader (package-scoped, project-agnostic) is the WRONG place to
resolve those refs. The correct place is launch (project-scoped), which the plan
noted ("cross-checked at launch") but which was never wired.

## Solution

Extracted a shared pure helper `firstUnknownExecutorRef(refs, set)` in
`config.ts` (used by both the loader path and the new launch path). Added the
authoritative check to the `POST /api/runs` enforcement gate: resolve the
project's `executor_ref_id` set, and for every `ai_coding` node reject an unknown
`settings.executors` ref with `MaisterError("CONFIG")` (400) before any
worktree/run/workspace side-effect. Added integration tests for the unknown-ref
refusal (no side-effect), the valid-ref non-false-positive, and the
trust-before-enforcement ordering. Relaxed the watchdog candidate query to drop
the `acp_session_id` filter (deleteSession stays best-effort). Added a
cost-cap-never-kills test. Corrected the judge settings table and broadened the
OpenAPI 400 description.

## Prevention

- A `verify` finding "logic exists + unit test passes" is NOT proof the behavior
  runs in production. For any validation gated behind an optional parameter,
  grep ALL production call sites and confirm at least one supplies it — trace
  from the diff, not from the test.
- When an id lives in two spaces (a DB PK and a config "ref" id), state which
  space a reference field belongs to and validate against THAT set. Here
  `settings.executors` ⊆ `executors.executor_ref_id`, never `executors.id`.
- Validation whose inputs are project-scoped must run at a project-scoped seam
  (launch), not at a package-scoped seam (install) — a generic package has no
  single project's executor set.
