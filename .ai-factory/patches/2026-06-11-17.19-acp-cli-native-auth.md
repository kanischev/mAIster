# AIF Fix Learning: ACP CLI-Native Auth

## Context

During verification of the ACP OpenCode/Gemini adapter widening, the review
initially treated missing `GEMINI_API_KEY` / `GOOGLE_API_KEY` as a Gemini smoke
blocker. The user clarified that Gemini CLI, OpenCode, Codex, and Claude should
be configured in their own tools, not through mandatory MAIster-owned API keys.

## Improvement

Future `/aif-verify` and `/aif-fix` passes for ACP adapter work should check
whether the adapter provider kind is CLI-native before flagging missing env
refs:

- `anthropic`, `openai`, `google_gemini`, and `agent_native` default to the
  tool's own configured auth.
- Optional env refs are valid only as explicit operator overrides, compatible
  provider routing, gateway config, or sidecar config.
- Readiness for new CLI-native adapters should be proven by binary diagnostics
  plus cached SDK smoke evidence, not by requiring host API key variables.

## Verification Pattern

When local tools are available, run the real SDK smoke command and record the
result in the spike/spec before deciding readiness language:

```bash
pnpm -C supervisor smoke:acp --cache /tmp/maister-adapter-smoke-cache.json gemini opencode
```

If sandboxing blocks `tsx` IPC or local state directories, rerun the same
command with approved unsandboxed execution and preserve the exact reason in
the verification notes.
