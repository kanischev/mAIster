# Authored Flow Package Boundary Fixes

**Date:** 2026-06-08 12:15
**Files:** `web/app/(app)/flows/actions.ts`, `web/scripts/import-flow-package-draft.ts`, `web/lib/flows/package-authoring.ts`, `web/lib/catalog/authored-types.ts`, authored Flow tests, package docs
**Severity:** high

## Problem

The authored Flow management branch still accepted unsafe package slugs at server boundaries, decoded package files as UTF-8 before validating text-ness, allowed package paths that collide as file and directory, and documented package artifact kinds inconsistently.

## Root Cause

The create action and import CLI trusted browser/argv validation for package slugs even though those values become DB identity and `flow.yaml` content. Package import used `readFile(..., "utf8")`, which replaces invalid bytes before validation can reject binary payloads. File validation checked absolute paths, traversal, and exact duplicates, but not normalized parent/child collisions that fail during export.

## Solution

Added a shared authored Flow package slug parser, used it in server create and import override paths, and quoted generated YAML scalars. Package directory reads now decode buffers with fatal UTF-8 validation. Package file normalization now reports `path_conflict` for file-vs-directory collisions. Docs now list `asset` consistently and describe UTF-8/path-collision validation.

## Prevention

For authored packages, validate every external identifier server-side before it reaches DB identity, YAML, SQL, or filesystem sinks. Read package bytes as bytes first, then decode with strict text validation. Review package path checks for both duplicate normalized paths and ancestor/descendant collisions before export.
