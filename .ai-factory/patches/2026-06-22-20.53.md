# M38 routing/rework fail-open: producer-failure default-route + on_mismatch ignored workspace policy

**Date:** 2026-06-22 20:53
**Files:** web/lib/flows/graph/gates-exec.ts, web/lib/flows/graph/runner-graph.ts, web/lib/flows/graph/__tests__/runner-graph-decide-routing.integration.test.ts, web/lib/flows/graph/__tests__/runner-graph-on-mismatch.integration.test.ts
**Severity:** high

## Problem

Two fail-open holes in the M38 (ADR-103) flow routing/rework paths, found by a
Codex adversarial review of the `feature/flow-routing-runcontext` branch:

1. **A failed verdict gate silently default-routed the node.** On a
   `decide:{from:verdict}` node, when the verdict-producing gate emitted
   unparseable output (or had no prompt), `runOneGate` returned `"failed"`
   without surfacing a verdict. `runNodeGates` suppressed the blocking failure
   purely because it was a routing-input gate (`!routingInput`), returning
   `{ ok: true, verdict: undefined }`. `computeDecideOutcome` then saw an empty
   verdict context, matched no `when` case, and returned the **`default`**
   target — so a broken/non-producing gate drove whatever default transition
   the author configured, including an approve/promote path. A producer failure
   became silent business routing.

2. **`on_mismatch` engine-rework ignored the flow's workspace policy.** The
   malformed-structured-output rework path recorded `workspacePolicy: "keep"`
   hardcoded and never applied a checkpoint. A flow declaring
   `workspacePolicies: ["rewind-to-node-checkpoint"]` (no "keep") kept the dirty
   workspace AND persisted a policy the manifest never allowed — while the
   human-review and auto-retry rework paths correctly derived and applied the
   author's policy.

## Root Cause

Both are the same failure mode: **a new code path was added without mirroring
the fail-closed discipline its sibling paths already had.**

1. The `routingInput` suppression conflated two distinct states: "the gate
   produced a verdict to route on" vs "the gate failed to produce one". The
   `default` case of a decide table means *"a value was produced but matched no
   explicit `when`"* — it is NOT a catch-all for *"no value was produced at
   all"*. Routing on `default` when the producer failed is fail-open: the safe
   reading of "no verdict" is "I cannot decide", which must fail closed, not
   pick the author's fallback branch.

2. `on_mismatch` was a THIRD rework entry-point added alongside the existing
   human-review rework (3313) and auto-retry (1859) paths, both of which
   resolve the workspace policy from the author's declaration
   (`workspacePolicies[0]` / `retryPolicy.workspace`) and apply it via
   `applyWorkspacePolicy`. The new path skipped that wiring and hardcoded
   "keep". `workspacePolicies` is `z.array(...).min(1)` (required, no "keep"
   default), so `[0]` is always the author's intended engine default — the
   hardcode silently overrode it.

## Solution

1. `gates-exec.ts`: suppress the blocking failure of a routing-input verdict
   gate ONLY when it actually produced a verdict
   (`verdictRouting && isVerdictGate && verdictSink.verdict !== undefined`). A
   failed routing-input gate (no verdict) now sets `blockingFailedGateId` like
   any other unproducible blocking gate → the node Fails (PRECONDITION) instead
   of falling through to `default`.

2. `runner-graph.ts` on_mismatch path: derive `onMismatchPolicy =
   node.rework.workspacePolicies[0] ?? "keep"` and apply it against the rework
   target's pre-attempt checkpoint via the existing `applyWorkspacePolicy`
   primitive (same degrade-to-keep-on-null-checkpoint + fail-node-on-git-error
   handling as the human/auto-retry paths), recording the derived policy on the
   reworked attempt. Inlined (not extracted to a helper) to match the codebase's
   existing convention — the apply pattern is already inlined at the two sibling
   sites; reusing the shared `applyWorkspacePolicy` IS the DRY unit.

Two regression integration tests added: unparseable-verdict → run Failed (no
default-route, `escalate` never runs); non-keep `on_mismatch` → reworked attempt
records `"rewind-to-node-checkpoint"`, not "keep".

## Prevention

- **When a producer's output drives routing, a producer FAILURE must fail
  closed.** Never let "no value produced" share a code path with "value
  produced but unmatched". The `default`/fallback branch is for the latter only.
  Test the empty/unparseable-producer case explicitly — it is the one most
  likely to be skipped, and it is exactly where fail-open hides.
- **A new instance of an existing pattern (rework site, gate-suppression,
  routing arm) must be diffed against the existing instances for parity before
  merge.** Grep every sibling (`markNodeReworked` had 3 call paths;
  `applyWorkspacePolicy` had 2) and confirm the new one resolves/applies the
  same author-declared inputs. The bug here was not a wrong line — it was a
  MISSING line that every sibling already had.
- A hardcoded constant where siblings read a config value (`workspacePolicy:
  "keep"` next to `workspacePolicies[0]`) is a parity smell — treat it as a
  must-check, not a default.
- Sibling-sweep grep recorded (skill-context rule): both defects were confirmed
  single-site — `markNodeReworked` (2 call sites: on_mismatch + human-review),
  hardcoded `workspacePolicy: "keep"` (1 site), `routingInput` suppression (1
  site, 1 consumer at runner-graph.ts:2688). No further siblings to fix.
