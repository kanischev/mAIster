# M7 adversarial review pass 4 — SSE event framing, atomic resume, cursor namespacing, spec/runner contract drift

**Date:** 2026-05-28 20:32
**Files:** `web/app/api/runs/[runId]/stream/route.ts`, `web/lib/flows/runner.ts`, `web/lib/flows/__tests__/runner-reentry.test.ts`, `docs/system-analytics/hitl.md`, `docs/database-schema.md`, `docs/api/web.openapi.yaml`
**Severity:** critical

## Problem

Pass 4 of the M7 adversarial review returned `needs-attention` with four
findings. The first three rounds' fixes all held; these surface once the
obvious bugs are out of the way.

1. **Critical:** The web SSE bridge framed every event as
   `id: N\nevent: <type>\ndata: <json>\n\n`. Browser `EventSource`
   dispatches named events ONLY to `addEventListener(<eventName>)`; it
   never delivers them to `onmessage`. The shipped `useRunStream` hook
   used `onmessage` exclusively, so every live event — `session.line`,
   `session.update`, `session.permission_request`, terminal events —
   silently vanished. The UI would show "connected" while missing all
   run output and HITL prompts.
2. **Critical:** `runFlow`'s NeedsInput→Running transition was an
   unconditional `UPDATE runs SET status='Running' WHERE id=runId`.
   With the same-payload-retry resume-requeue fix from pass 3, two
   near-simultaneous resume calls could both load NeedsInput, both
   flip to Running, and both execute the resumed step plus every
   step after it — duplicating CLI/agent side-effects and potentially
   tripping the `step_runs (run_id, step_id, attempt)` unique
   constraint.
3. **High:** The synthetic `session.stream_timeout` event emitted by
   the web bridge used `lastSeen + 1` as its SSE `id:`. The event is
   NOT appended to `run.events.jsonl`. A reconnecting client sends
   `Last-Event-ID: lastSeen + 1`; the next real durable event has
   `monotonicId == lastSeen + 1` and is filtered out by the bridge's
   `> lastSeen` guard. The first real event after a quiet timeout
   could be permanently hidden, including a `permission_request` or
   `session.crashed`.
4. **High:** `system-analytics/hitl.md` and `database-schema.md`
   described `kind="human"` as a full `on_reject.goto_step` loopback
   flow. The runner does not branch on `on_reject` in M7 — it stores
   the response and advances to the next step. A reviewer reading
   the spec would think they could send work back; the runner just
   moves on with `{rejected: ...}` as ordinary vars.

## Root Cause

All four share a pattern: **a producer-consumer contract had two
participants and only one was updated when the underlying surface
moved**.

- **#1 (event framing):** The supervisor SSE stream uses named events
  and its consumer (`supervisor-client.ts`'s `streamSession` async
  generator) parses lines manually — both ends agreed. When the web
  bridge introduced a browser consumer (`EventSource`), the framing
  was copy-pasted from the supervisor side without considering that
  the new consumer dispatches differently. The producer's contract
  was "named events"; the consumer's contract was "default `message`
  events". No place named the agreement, so neither side knew it
  was wrong.
- **#2 (resume claim):** The original runFlow shape predated the
  same-payload-retry re-queue. Pass 3 added a second caller without
  the original `WHERE` clause growing the safety conditions it now
  needed. The transition wrote `status='Running'` based on local
  knowledge (`isResume`); the race was invisible because the original
  caller was single-instance. Adding a second caller revealed the
  hole the unconditional UPDATE had always had.
- **#3 (cursor namespacing):** The `monotonicId` cursor lives in
  exactly one place — the supervisor's per-spawn counter — and
  drives the bridge's replay filter. The bridge's synthetic event
  reused the cursor namespace without ever touching the durable log.
  The next durable event collided with the synthetic id at the
  consumer's filter, not at the producer's writer.
- **#4 (spec drift):** Each round of fixes touched code; the docs
  were updated to match the parts that changed. The `on_reject`
  loop description was carried forward from the original plan
  unchanged because no code in the M7 scope DELETED it — but no
  code IMPLEMENTED it either. The doc described a future state.

## Solution

1. **Drop `event:` from the web bridge.** `formatSseEvent` now emits
   only `id:` and `data:`. Every event arrives on the default
   `message` dispatch, which is where `onmessage` (and the
   `useRunStream` hook) listens. The `type` field carried INSIDE
   `data` is the discriminator. The supervisor's `/sessions/:id/stream`
   is unchanged — its consumer parses lines manually and works with
   either framing.
2. **Atomic resume claim.** The NeedsInput→Running transition now
   runs inside a transaction that SELECTs the row, verifies
   `status === "NeedsInput" && currentStepId === targetStepId`, then
   issues a conditional UPDATE. The boolean return tells the caller
   whether it won. Losing callers return immediately without touching
   step state. New regression test races two concurrent `runFlow`
   calls and asserts exactly one transition + zero loser-side step
   work.
3. **Cursor-namespacing helpers.** New
   `formatSyntheticSseEvent(data)` emits `data:` only — no `id:` —
   so `Last-Event-ID` is not advanced by web-only events. The
   distinction at the call site makes the namespacing contract
   visible to the next person who adds a synthetic event.
4. **Doc downgrade.** `hitl.md`, `database-schema.md`, and
   `web.openapi.yaml` all now state plainly that `on_reject.goto_step`
   rerouting is Designed M8. M7 stores the response and advances the
   runner; the row kind (`hitl_requests.kind = "human"`) is preserved
   so M8 can light up rerouting without a schema change. The OpenAPI
   spec includes a direct note for UI implementers: do NOT present
   rejection as a loop-back action in M7.

## Prevention

- **Name the producer-consumer contract for every cross-process
  surface — including browser ↔ server.** SSE has two valid framings;
  HTTP has two valid auth schemes; JSON has two valid number ranges.
  The contract is not "use the obvious one." For every wire surface,
  one document (the API spec, the AsyncAPI doc, or a clearly named
  README) must say which variant the surface uses and which consumer
  shapes it supports. Adding a new consumer without checking against
  this document is a defect.
- **Every new caller of a state transition must justify the transition's
  guard.** When you add a second caller to a state mutator that
  previously had one, audit the `WHERE` clause: does it still uniquely
  identify the intended state? If the answer is "the old caller's
  preconditions implied it," widen the guard to make the conditions
  explicit.
- **Cursors and identifiers are namespaces.** Synthetic events that
  reuse a remote-system identifier namespace are bugs waiting to fire
  at consumer-side cursor logic. If a producer emits something the
  consumer might index, either (a) persist it where the consumer's
  index reads from, or (b) emit it without an index field. There is
  no safe middle.
- **"Spec describes future state" is a defect, not a TODO.** If a
  spec describes behaviour the code does not implement, the spec is
  wrong NOW — readers will write client code against it. Either
  implement, or mark the section explicitly as Designed Mx with a
  link to the milestone. Plan reviews on every spec PR should
  include "does this section describe code that exists at HEAD?"
