# Takeover lifecycle edges: a non-terminal status bypassed a HITL guard, return trusted commit-range over the working tree, and a status-only release left the ledger open

**Date:** 2026-05-31 23:49
**Files:** web/app/api/runs/[runId]/hitl/[hitlRequestId]/respond/route.ts, web/app/api/runs/[runId]/takeover/return/route.ts, web/lib/worktree.ts, web/lib/flows/graph/ledger.ts, web/lib/runs/state-transitions.ts
**Severity:** critical

## Problem

A second adversarial review of the manual-takeover feature (after the feature
plus a first remediation round were all green) found 3 lifecycle defects that
the per-phase reviews and the first adversarial pass missed:

1. **(critical)** The HITL form/human response route only blocked *terminal* run
   statuses. The new `HumanWorking` status is non-terminal, so a reviewer could
   submit the *original* review approve/rework **while a takeover was active** —
   storing a stale decision (and a step-keyed `input-review.json` artifact) that
   the post-return rerun could consume, replaying a pre-edit decision and
   bypassing fresh human review of the returned work.
2. **(high)** The return route recorded `git log/diff base..branch` (commit refs)
   and never checked the working tree, so a reviewer who edited but forgot to
   commit got a "successful" Return with empty/incomplete returned commits/diff —
   silent loss of their actual work and misleading evidence.
3. **(medium)** Abandon/release of a `HumanWorking` run flipped `runs.status` but
   never closed the takeover `node_attempts` row, so `getActiveTakeover` still
   reported an active handoff on a released/abandoned run.

## Root Cause

- **#1 — a guard enumerated the wrong axis.** It asked "is the run terminal?"
  when the real question was "is the run *awaiting this specific HITL*?" Adding a
  new non-terminal status (`HumanWorking`) silently widened the set of states in
  which a stale response was accepted. Guards that allow-list by a coarse
  property (non-terminal) instead of the exact valid states drift open every time
  a new status is added.
- **#2 — trusting a proxy for the real artifact.** "Returned work" was modeled as
  the committed branch range, but the actual artifact the human produces is the
  *working tree*. `base..branch` silently excludes uncommitted/untracked changes,
  so the contract ("the reviewer commits, then returns") was *assumed* rather than
  *enforced* — and an unmet assumption became silent data loss, not an error.
- **#3 — a state machine split across two stores updated only one.** The takeover
  lifecycle lives in BOTH `runs.status` and the `node_attempts` ledger; the
  release/abandon transition updated the status store and left the ledger store
  describing an active handoff. A lifecycle event must close every store that
  represents that lifecycle.

## Solution

- **#1:** accept a form/human HITL response ONLY when the run is in the exact
  pending states for it (`NeedsInput` | `NeedsInputIdle`); reject everything else
  (notably `HumanWorking`) with 409. Place the guard on the fresh-write branch,
  *after* the idempotent-recovery branches (already-delivered / same-payload
  re-claim) so the recovery 200 contract is preserved. Because no artifact is
  written during the takeover (and the runner unlink-consumes the artifact each
  review cycle), the post-return rerun mints a fresh HITL with nothing stale to
  replay — so no claim-time HITL closure was needed.
- **#2:** add a read-only `git status --porcelain` worktree check; in the return
  route, *before* any ledger write, reject a dirty/untracked worktree and a
  zero-commit range with retryable 409s, and document both failure modes.
- **#3:** add a ledger helper that closes the active takeover row (`ended_at`),
  and call it inside the *same transaction* as the release status flip, so the
  status and ledger stores transition atomically.

## Prevention

- **Guard by exact valid states, not by a coarse complement.** A precondition
  guard should allow-list the specific states an operation is valid in
  (`status ∈ {NeedsInput, NeedsInputIdle}`), never deny-list a coarse property
  (`!terminal`). The deny-list silently admits every future status. When adding a
  run status, grep every state guard, scheduler/cap predicate, recovery sweep,
  AND HITL/response guard for whether the new status belongs.
- **Enforce, don't assume, the human-side contract.** If a flow says "the user
  commits before returning," the server must verify it (clean-worktree check) and
  reject the violation — an unenforced precondition becomes silent corruption.
  Model the artifact as what the human actually changed (working tree), not a
  convenient proxy (commit range).
- **A lifecycle transition must close every store that represents it.** When state
  lives in more than one place (status column + ledger row + artifact), a
  terminal/release transition must update all of them atomically; otherwise reads
  that key off the not-updated store report a contradiction.
- **Re-review after remediation.** The first adversarial pass found atomicity/race
  bugs; fixing them did not surface these interaction-layer bugs. Budget a fresh
  adversarial pass after a remediation round — fixes change the surface, and the
  next layer of edges (cross-feature interaction, input validation, multi-store
  consistency) only becomes visible once the first layer is closed.
