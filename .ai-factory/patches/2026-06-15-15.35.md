# Scratch-run rename: OpenAPI contract diverged from the route; rail RBAC is a query+403 boundary, not a UI gate

**Date:** 2026-06-15 15:35
**Files:** docs/api/web.openapi.yaml, web/app/api/scratch-runs/[runId]/__tests__/route.test.ts
**Severity:** low

## Problem

A Codex adversarial review of the active-workspaces rail branch raised two findings:

1. The `PATCH /api/scratch-runs/{runId}` OpenAPI body schema declared
   `additionalProperties: false` + `name { minLength: 1 }`, but the route
   trims-then-validates and reads only `body.name`. So the contract and the
   handler disagreed in two opposite directions: a whitespace-only name
   (`"   "`) satisfies `minLength: 1` yet the route returns 400, and extra
   body properties violate `additionalProperties: false` yet the route ignores
   them. Generated clients / acceptance tests built from the spec would be
   wrong both ways.

2. A project `viewer` sees the rail's scratch-rename pencil and lifecycle
   icon-actions, which all 403 at the route (`renameScratchRun` /
   `operateScratchRun` = `member`). The review recommended projecting a
   per-row capability and hiding the controls.

## Root Cause

1. **Code-first drift.** The route was written with sensible hand-validation
   (trim, non-blank, ≤200, tolerate unknown keys) but the OpenAPI was authored
   from the *intended* shape, not the *implemented* behavior. JSON Schema
   `minLength`/`maxLength` measure the raw string; the route measures the
   trimmed string — so the schema can never express "trim then check" exactly,
   and `additionalProperties: false` contradicted a handler that ignores extras.

2. **Mis-located boundary in the finding.** The finding framed the viewer
   seeing a 403-only control as a defect requiring UI capability-gating. But
   the rail query is already membership-scoped (non-admin → inner-join on
   `project_members`; admin → all) and every action re-checks
   `requireProjectAction` server-side. The owner's call: that query scope +
   the route 403 *is* the authorization boundary; the rail should just show
   the accessible projects' workspaces. UI capability-gating (and the
   "count-only for viewers" / "aggregate other projects" variants explored)
   was over-engineering for a rare role with no security gain.

## Solution

1. Made the spec follow the code (docs/CLAUDE.md R3 — code is executable
   truth): dropped `additionalProperties: false` (extras are ignored), added
   `pattern: "\\S"` so the machine-readable schema rejects blank/whitespace-only
   names like the route does, and documented in `description` that surrounding
   whitespace is trimmed before the 1..200 check and unknown properties are
   ignored. Added a route test asserting unknown body properties are ignored
   and the trimmed name is stored (the whitespace-only→400, >200→400, trim,
   and viewer→403 cases were already covered).

2. Finding 1 → no code change. The membership-scoped query + per-route 403
   is the accepted boundary. Reverted a speculative `canOperate` capability
   projection (authz helper, query field, rail row/`+` gating) after the owner
   chose the simpler model.

## Prevention

- When a route hand-validates a body, audit the OpenAPI schema in the same
  pass: a `transform-then-validate` server (trim, coerce, normalize) usually
  cannot be expressed by raw `minLength`/`maxLength` alone — reach for
  `pattern` + a `description` that states the transform, and never declare
  `additionalProperties: false` unless the handler actually rejects extras.
- Before building UI capability projection for an RBAC finding, confirm the
  intended boundary with the owner: visibility-scoped query + server-side
  action check is often the whole boundary, and a control that 403s for a
  rare role may be an accepted trade-off rather than a defect.
- When reverting speculative changes, reconcile the *whole* working tree, not
  just the files you edited: out-of-band modifications can leave the tree
  inconsistent with the source and break the build. Here a parallel agent
  session — most likely the long-running Codex desktop sandbox (pid 29172,
  `--working-dir` = repo root, `:project_roots = write`), which has write
  access to this worktree since it sits under that root — had propagated the
  reverted `canOperate` design into a doc + an integration test with no
  Edit/Write call from this session. Diff `HEAD` for the full file set, restore
  stragglers, and check `ps`/mtimes for a concurrent writer when files change
  unexpectedly.
