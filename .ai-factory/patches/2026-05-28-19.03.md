# M7 adversarial review pass 2 — concurrency, validation, crash propagation, spec drift

**Date:** 2026-05-28 19:03
**Files:** `web/app/api/runs/[runId]/hitl/[hitlRequestId]/respond/route.ts`, `web/lib/flows/runner-agent.ts`, `web/lib/flows/hitl-validate.ts` (NEW), `docs/api/web.openapi.yaml` (NEW), `docs/api/async/web-runs.asyncapi.yaml` (NEW), tests
**Severity:** critical

## Problem

After fixing the first round of M7 adversarial findings, the verification pass
returned `needs-attention` again with five new findings:

1. **Critical:** Two concurrent permission responses raced. Both passed the
   `respondedAt IS NULL` check, both overwrote `response`, both called
   `deliverPermission`. The first succeeded on the supervisor side, the second
   got 404 and the route's 410 terminal branch flipped `runs.status = Failed`
   — destroying the first request's successful approval.
2. **Critical:** The form/human path wrote `input-<stepId>.json` BEFORE the
   transaction that re-checked `respondedAt`. A concurrent loser returned 409
   from the TX but had already corrupted the artifact the runner would later
   read. DB audit state said response A delivered; disk said response B.
3. **High:** `bodySchema` accepted `response: z.unknown()`. The form/human
   handler only checked presence before writing and marking `respondedAt`.
   `runner-human` requires a JSON object; the route would happily commit a
   scalar/array or a payload that violated the stored form schema. Once
   `respondedAt` was set, the user could not retry through the normal route.
4. **High:** When `runner-agent.handlePermissionRequest` failed to INSERT the
   hitl_requests row, it cancelled the supervisor deferred and set
   `runs.status = 'Crashed'`, but it did NOT propagate the failure to the
   `EventConsumer` result. `runNewSession`/`runSlashInExisting` computed `ok`
   solely from `promptResult.stopReason === 'end_turn'`. An agent that
   gracefully ended after the cancelled tool call made the step look
   successful; `runFlow`'s final transition to `Review` overwrote the
   `Crashed` row — losing the crash signal entirely.
5. **Medium:** The web tier's new routes (`POST /api/runs/[runId]/hitl/[hitlRequestId]/respond`
   and `GET /api/runs/[runId]/stream`) had no OpenAPI / AsyncAPI specs.
   Generated clients, contract tests, and external operators had nothing to
   read.

Every finding except #5 was a real correctness bug exploitable by a
double-click or a network retry.

## Root Cause

All four code findings collapse to one design failure: **the response route
treated DB writes and external side-effects as independent steps without
naming an atomic claim**. Each step did its own check on a stale snapshot of
the row, so retries and races could enter pathways the author didn't model.

- **#1 & #2 (concurrency):** Phase 1 was a non-CAS UPDATE. Two transactions
  could both pass the `respondedAt IS NULL` check, both UPDATE, and only after
  the cross-process side-effect (supervisor or filesystem) commit would the
  conflict be visible — by which point the loser had already mutated shared
  state. The fix is a row-level lock + a CAS guard with explicit
  classification of the four post-claim states (unclaimed, claimed-same-payload,
  claimed-different-payload, already-delivered).
- **#3 (validation):** The body schema (`z.unknown()`) was a placeholder
  intended to be tightened later. "Later" never came. The runner's contract
  with the artifact was `{ ... fields per schema ... }`; the route's contract
  with the body was "anything". The gap was load-bearing — anyone could mark
  a HITL row delivered with garbage and brick the resume. Validation needs to
  happen at the entry boundary, not somewhere downstream after `respondedAt`
  was already set.
- **#4 (crash propagation):** The failure mode was async (handled inside the
  for-await loop of the event consumer), but the outcome was synchronous
  (whether the step succeeded). The two paths had no shared state. The
  consumer logged loudly, mutated the DB, and returned silently. Logging
  isn't propagation; the step result IS the contract with the rest of the
  runner.
- **#5 (spec drift):** Treating doc work as a follow-up turned wire contracts
  into folklore. By the time the third reviewer arrived, the only source of
  truth was the implementation.

## Solution

1. **Atomic claim model.** The HITL response route now uses a row-level
   `SELECT ... FOR UPDATE` (postgres; sqlite is implicitly serialized by its
   single-writer lock) and tags the claim outcome explicitly:
   `claimed | noop-idempotent | already-delivered`. The caller branches on the
   tag instead of inferring from row state across transactions.
2. **Concurrent-winner detection on supervisor 404.** Before marking the run
   `Failed` on `HITL_TIMEOUT`, the route re-locks the row and checks
   `respondedAt`. If a concurrent winner already finished, return `200` and
   leave the run alone.
3. **DB-first for form/human.** Validation → claim → artifact write →
   respondedAt. A losing concurrent submission gets 409 before touching the
   filesystem. The artifact is written from the STORED response (not the
   body) so even an idempotent retry writes byte-identical data.
4. **`hitl-validate.ts`.** Small validator over MAIster's form schema (no
   dependency drag): `string | number | boolean | enum | array`, `required`,
   enum `options`. Throws `MaisterError("NEEDS_INPUT")` → 422 with field
   path. Runs BEFORE any state mutation, so retries with a fixed payload
   replay cleanly.
5. **Persist-failure propagation.** `handlePermissionRequest` now returns
   `{ok:true} | {ok:false, reason}`. The `EventConsumer` captures the first
   failure into a `permissionPersistFailure` field exposed via a getter.
   Both step entry points check it after `sendPrompt` returns and force
   `ok:false, errorCode:"CRASH"` so `runFlow` does not promote to Review.
6. **Web specs in the same commit as the runtime fixes.** No more "deferred
   docs": `docs/api/web.openapi.yaml` and `docs/api/async/web-runs.asyncapi.yaml`
   describe every M7 web-tier route including the retry-vs-terminal
   classification matrix. Both validate clean.

Tests:
- HITL response route: +10 cases covering concurrent retry with same vs
  different optionId, concurrent winner detection on supervisor 404, schema
  validation per field type, concurrent form double-submit (no artifact
  corruption), artifact-write failure leaving the row claimed but
  respondedAt-null.
- runner-agent: existing INSERT-failure regression tests updated to assert
  `result.ok === false` and `result.errorCode === 'CRASH'`. The previous
  assertions were silently encoding the bug as expected behavior.

## Prevention

- **Name the claim.** Any HTTP route whose body triggers a cross-process
  side-effect MUST name an atomic claim step explicitly: a single SQL CAS
  whose return value determines whether the route owns the work or must
  abort. Treat "we don't have racing requests today" as a planning fiction;
  double-clicks and connection retries arrive in production day one.
- **Validation at the entry boundary, not downstream.** If the runner expects
  a shape, the route MUST validate that shape before any irreversible state
  mutation. `z.unknown()` placeholders are debt; a TODO comment is not a
  validation strategy. Add a lint rule rejecting `z.unknown()` /
  `z.any()` in route bodies that flow into atomic claims.
- **Outcome propagation is part of the contract.** When an async handler
  mutates external state, its return value MUST encode the outcome the
  caller's contract depends on. Logging the failure is operator observability;
  it is not the same as telling the caller. Functions that "succeed by
  default" while logging errors are an antipattern.
- **Specs ship with the code that changes them.** Mark a milestone done only
  after the OpenAPI / AsyncAPI files have been touched. Add a pre-merge
  gate that diffs route filenames / Zod schemas vs spec paths. "Deferred
  docs" on a wire contract is a defect, not a follow-up.
- **Adversarial review IS part of the green-light gate.** Two passes were
  needed to find these. A green test suite is necessary, not sufficient.
