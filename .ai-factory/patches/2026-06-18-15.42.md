# Adversarial-review fixes: ownership-safety, error-classification, fail-fast, auth-ordering (ADR-093 onboarding/git)

**Date:** 2026-06-18 15:42
**Files:** web/lib/repo-source.ts, web/lib/git-remotes.ts, web/app/api/projects/route.ts, web/app/api/projects/[slug]/persist-config/route.ts (+ 5 test files)
**Severity:** high

## Problem

A Codex adversarial review of the ADR-093 onboarding/git-access branch surfaced
four defects that all passed the green test suite:

1. **Data loss** — new-empty registration did `mkdir(dir, { recursive: true })`
   then set `createdByUs: true`. Recursive mkdir succeeds even if the dir
   appeared concurrently between the absence check and the mkdir, so the route's
   failure cleanup (`rm -rf resolved.dir` for any `createdByUs` dir) could delete
   a host directory MAIster never created.
2. **Contract hole** — `pushProjectRemote`/`fetchProjectRemote` caught *every*
   `MaisterError` and returned `{ ok: true, warning }`. That swallowed validation
   failures (bad remote name/branch → `PRECONDITION`) and unknown-remote failures
   as HTTP 200 "success", contradicting the OpenAPI contract (those are 409). The
   sibling `add` path correctly 409'd — an invisible asymmetry.
3. **Silent fallback** — `pathExists` caught *all* `stat` errors → `false`, so a
   present-but-unreadable `maister.yaml` (`EACCES`/`ELOOP`/IO error) was treated
   as "absent" and silently registered from DB defaults, skipping the declared
   flows/packages/setup.
4. **Auth ordering** — `POST /api/projects` and the persist-config route parsed
   the request body (Zod) before the auth gate, so an unauthenticated caller got
   a schema-specific 422 instead of 401/403, leaking the route contract.

## Root Cause

The unifying root cause: **a green test suite proves the happy path, not the
adversarial edge.** Each defect lived in a branch the existing tests never drove:

1. The race window (concurrent create) is invisible to a single-threaded test;
   recursive mkdir's idempotence (its convenience) is exactly what erased the
   "did *I* create this?" signal that the destructive cleanup depends on.
2. A broad `catch (err instanceof MaisterError)` conflates *input/precondition*
   errors with *genuine network/push advisories* because both are `MaisterError`.
   The advisory intent ("a push failure is non-fatal") was over-applied to the
   whole call instead of only the network/non-fast-forward outcomes.
3. A catch-all `try/stat → false` is the universal "treat any failure as absent"
   anti-pattern — convenient, but it converts *every* IO error into the one
   branch (fallback) the contract reserved for a *single* condition (ENOENT).
4. Parsing-before-auth reads naturally (validate input, then act) but inverts the
   security order: authentication must gate before any contract detail is exposed.

## Solution

1. **Atomic ownership claim** — `mkdir(dirname(dir), { recursive: true })` for
   parents, then `mkdir(dir)` (non-recursive, exclusive). `EEXIST` ⇒ refuse with
   `PRECONDITION`; `createdByUs` is set only when *we* exclusively created the dir.
2. **Classify, don't blanket-catch** — `pushProjectRemote`/`fetchProjectRemote`
   pre-check existence (`getRemoteUrl` → `PRECONDITION` if null; it also validates
   the name), then catch *only* `GitPushRejectedError` + `EXECUTOR_UNAVAILABLE`
   as advisory and rethrow everything else (validation `PRECONDITION` → 409).
3. **Allow-list the fallback condition** — `pathExists` returns `false` only for
   `ENOENT`/`ENOTDIR`; any other `stat` error throws `CONFIG` (fail fast).
4. **Auth before parse** — move `requireGlobalRole`/`requireActiveSession` +
   `requireProjectAction` ahead of `req.json()`/Zod on every changed route.

Each fix carries a deterministic regression test, including real-fs symlink
tricks for the untestable-by-timing cases: a **dangling symlink** (stat ENOENT
but the entry exists → exclusive-mkdir EEXIST) for #1, and a **self-referential
symlink** (stat ELOOP) for #3.

## Prevention

- **Destructive compensation needs proof of ownership.** Any `rm -rf` /
  branch-delete gated on a "we-created-it" flag must set that flag only from an
  *exclusive* create (non-recursive mkdir / `mkdtemp`+rename / `clone -b`), never
  from an idempotent op that can't distinguish create from already-exists.
- **A catch that maps to a success/advisory must filter by error class, not by
  base type.** Catch the *specific* outcomes (`GitPushRejectedError`,
  `EXECUTOR_UNAVAILABLE`); let validation/precondition errors propagate. Test
  invalid-input and unknown-target paths return the contract's non-2xx.
- **`catch → false` is a smell.** A presence/existence probe must allow-list the
  exact "absent" errno (`ENOENT`/`ENOTDIR`) and rethrow the rest, or it silently
  routes IO failures into the fallback branch.
- **Authenticate before parsing.** The session/role gate runs before
  `req.json()`/Zod on every state-changing route; add an unauth + malformed-body
  test asserting 401/403, not 422.
- **Adversarial review catches what green tests miss** — run it on new
  trust-boundary / filesystem-mutating / error-classifying surfaces before merge.
