# M13 review fixes for role launch, merge conflicts, and HITL assignment atomicity

**Date:** 2026-06-02 15:32
**Files:** web/app/api/runs/route.ts, web/app/api/runs/[runId]/promote/route.ts, web/lib/flows/runner-human.ts, web/lib/flows/graph/runner-graph.ts, related tests
**Severity:** high

## Problem

Review found three fair M13 issues:

- Launch-time Flow role validation rejected manifests with role annotations even
  when the project had no configured active `flow_roles[]`, contradicting the
  documented back-compat contract.
- Repeated scratch-run promotion attempts with merge conflicts created multiple
  open `merge_conflict` assignments for the same run.
- Linear form/human HITL and graph `human_review` inserted HITL rows and
  assignments as separate DB operations, so assignment creation failure could
  leave an actionable HITL row without an assignment.

## Root Cause

The install-time role validator treated an omitted or empty role registry as
lenient, but launch-time validation did not share that empty-registry escape
hatch. Merge-conflict assignments are not keyed by `hitl_request_id`, so the
unique constraint cannot deduplicate them. Human HITL assignment creation had
drifted from the permission path, which already writes HITL row plus assignment
inside one transaction.

## Solution

- Launch validation now skips Flow role ref checks when the active project role
  registry is empty, and keeps rejecting unknown role refs when a registry
  exists.
- Merge-conflict assignment creation now first checks for an existing active
  `merge_conflict` assignment for the run and reuses it instead of creating
  another one.
- Linear form/human HITL and graph `human_review` now insert the HITL row and
  create the assignment through the same DB transaction where available, and
  remove the just-written `needs-input.json` artifact if persistence fails.
- Added regression tests for empty-registry launch back-compat, non-empty
  registry rejection, merge-conflict deduplication, and transaction rollback.

## Prevention

When adding a validation contract at both install and launch boundaries, add a
paired test for the lenient and strict modes so the two paths cannot drift. For
assignment creation without a natural unique key, query for an existing active
assignment before creating a new one. For any user-actionable HITL row, write
the row and assignment in a single DB transaction, and fail closed by removing
the wait artifact if durable assignment creation fails.
