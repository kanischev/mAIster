# M8 review pass 2 fixes — stale supervisor-id retry, retry-safe driver, flow-runner handoff

**Date:** 2026-05-29 15:10
**Files:** web/app/api/runs/[runId]/hitl/[hitlRequestId]/respond/route.ts, web/lib/runs/resume.ts, web/lib/runs/resume-driver.ts, web/lib/runs/state-transitions.ts, web/lib/runs/__tests__/resume-driver.test.ts (new), web/app/api/runs/[runId]/hitl/[hitlRequestId]/respond/__tests__/route.test.ts
**Severity:** critical

## Problem

Codex adversarial review pass 2 surfaced three lingering bugs after
pass 1 fixes had landed:

1. **Stale-id retry can falsely Fail the run.** When `/respond` falls
   into the `noop-idempotent` branch with `runStatus = "NeedsInput"`
   (a same-payload retry while an idle resume is in flight), the M7
   Phase 2 path calls `deliverPermission(schema.supervisorSessionId,
   schema.requestId, optionId)` against the ORIGINAL (pre-checkpoint)
   ids. That deferred was cancelled with `reason:"checkpoint"`, so
   the supervisor 404s as `HITL_TIMEOUT` and the catch block flips
   the run to `Failed` and writes `respondedAt` — even though the
   background driver was happily delivering against a fresh
   requestId.
2. **Retryable driver prompt failure strands the stored intent.**
   `runResumedSession`'s error branch called `markIntentAbandoned`
   BEFORE classifying retryability. `markIntentAbandoned` sets
   `respondedAt`, so on a retryable `EXECUTOR_UNAVAILABLE` from
   `sendPrompt` the row looks "already delivered" to the next
   `/respond` retry — no further `resumeRun` ever fires, the
   operator's intent is silently lost, and the run sits in
   `NeedsInput` with no live worker.
3. **Driver bypasses runFlow on the happy path.** On
   `stopReason === "end_turn"` the driver directly wrote
   `runs.status = "Review"`. That skips `step_runs` persistence
   (no stdout/vars/durationMs for the resumed step), skips post-step
   handling, and skips every step that follows the resumed step.
   For any multi-step flow the resumed step would silently look
   complete while later steps never ran.

## Root Cause

For #1: the M7 catch block was written assuming a `HITL_TIMEOUT`
from the supervisor means the deferred genuinely expired in the
supervisor's `pendingPermissions` registry. The M8 idle-resume path
introduced a second source of 404: the original deferred was
cancelled-as-checkpoint and the new requestId is owned by the
fresh resumed session. The route had no way to tell those apart.

For #2: the driver's error handler did `markIntentAbandoned →
classify retryable` in the wrong order. The semantics
"abandoned = respondedAt set" and "retryable = keep state for retry"
are mutually exclusive, but the code wrote the abandon marker first
and then tried to roll back logically, which was too late.

For #3: the driver was written as a single-purpose terminator
because the only test scenarios the author had loaded were
single-agent-step flows (the M8 POC's aif/review case). The runner's
existing resume invariant — that `step_runs` rows are the source of
truth for "what work has happened" and that `runFlow` is the only
function that may transition a run terminally — was not being
honored.

## Solution

**Fix 1 (`/respond` route).** In the existing `HITL_TIMEOUT` catch
branch, distinguish "claim was a same-payload retry"
(`claim.kind === "noop-idempotent"`) from a fresh claim. On a
same-payload retry the 404 is most likely the stale checkpointed
deferred — return `202 resume-in-progress` and leave the run /
`respondedAt` untouched. Subsequent retries hit the
`already-delivered` branch (if the driver finished) or hit the
`in-flight-resume-202` branch again (if it hasn't).

**Fix 2 (driver).** Classify retryability BEFORE any write to
`hitl_requests`. For retryable (`EXECUTOR_UNAVAILABLE`): call the
new `rollbackResumedRun` helper to flip the run back to
`NeedsInputIdle` and return immediately. The stored intent stays
pending with `respondedAt = null`, so the next `/respond` invocation
sees it as in-progress and can fire a fresh resume.

**Fix 3 (driver).** Replace the direct `transitionToReview()` call
with `completeResumedStepAndHandoff()`: record the resumed step's
`step_run` via `markStepSucceeded` (with the captured
`session.update`/`session.line` stdout), look up the flow's
manifest, and either:
- if no next step → terminal `Review` (preserves runFlow's terminal
  semantics for the last-step case);
- if a next step exists → advance `currentStepId` to it and
  schedule `runFlow(runId)` via `queueMicrotask`. runFlow's existing
  resume-claim path (`NeedsInput → Running` atomic CAS) picks up
  cleanly from there and continues the flow.

## Prevention

- A response code expressing progress (202) must correspond to an
  ACTUAL background task that completes the contract. A 202 that's
  semantically "we're done if everything goes right" rots into the
  bug class Codex flagged here.
- For any retryable side-effect, the rollback ordering should be:
  **classify first → write state second**. Writes that close the
  retry door must come AFTER the decision to give up on the retry.
- Any code path that terminates a run (sets `Review` / `Failed` /
  `Crashed`) must go through `runFlow` if it's not the runner
  itself. The runner is the single source of truth for terminal
  transitions. Side-channel transitions silently skip step_runs and
  flow continuation.
- For the next adversarial review: explicitly diff every error
  branch's DB writes against the retry / classification ordering;
  explicitly grep for any `runs.status = "Review" / "Failed"`
  outside `runFlow` and justify it.
