# Scratch run launch and HITL inbox drift

**Date:** 2026-06-24 03:55
**Files:** web/lib/runs/launch-progress.ts, web/components/scratch/scratch-launcher.tsx, web/lib/scratch-runs/service.ts, web/app/api/runs/[runId]/stream/route.ts, web/lib/queries/hitl.ts, web/lib/queries/portfolio.ts
**Severity:** high

## Problem

Scratch runs did not feel live after launch: the page opened only after the launch stream completed, and some scratch/HITL states did not appear in the HITL inbox. Local-package assistant runs also reused the run SSE hook but had no project row.

## Root Cause

The scratch launch stream emitted the run id only in the final result frame, after the first prompt turn could finish. The client therefore could not navigate at `session_ready`. Separately, run stream and inbox read models still assumed every scratch-related run had a project/workspace/flow row. Project-less local-package assistant runs have no project, and project scratch runs have `flow_id = NULL`, so old `innerJoin` paths filtered out valid live rows.

## Solution

The `session_ready` progress frame now carries `runId` and `dialogUrl`, and the launcher navigates once that frame arrives while the launch request continues. The run SSE route now authorizes project-less assistant runs by launching user and resolves their event-log slug from `local_packages`. HITL inbox queries left-join flows and label flow-less scratch HITL as `scratch`.

## Prevention

When adding a new run ownership shape or nullable run relation, audit every run read model that joins `projects`, `workspaces`, or `flows`. Scratch runs with `flow_id = NULL` need explicit inbox coverage, and launch-progress tests should assert the earliest frame that unblocks navigation.
