# ADR-121 admission-gate: 4 adversarial-review defects (resume state, claim capacity, settings UX + auth order)

**Date:** 2026-07-01 11:45
**Files:** web/lib/scheduler.ts, web/lib/scheduler/c2-eligibility.ts, web/lib/scheduler/handlers/auto-launch-triaged.ts, web/components/board/panels/queue-settings-control.tsx, web/components/board/panels/settings-panel.tsx, web/app/api/projects/[slug]/settings/route.ts
**Severity:** high

## Problem

A Codex adversarial review of the ADR-121 unified admission gate found four defects:

1. **[high] Queued flow HITL resumes stranded.** The gate's C3 path flipped every
   answered-idle run (agent AND flow) `NeedsInputIdle → Running` before dispatching
   the resume. The flow resume driver's completion transitions
   (`completeResumedStepAndHandoff`, `failResumedRun`, `rollbackResumedRun`) are
   status-guarded on `NeedsInput`, so a flow resume admitted as `Running` would
   auto-deliver the stored permission then fail its completion CAS → stuck `Running`
   holding a slot forever.
2. **[high] C2 bursts over-admit.** The reserve guard (`reserveAllowsC2`) used only
   `countLiveRuns`, which does not include outstanding C2 claims (a claim reserves a
   flow slot before its run row exists). Lock-serialized concurrent freed-slot gate
   calls each saw the same unchanged live count and could mint more auto runs than
   `flowCap − reserve`.
3. **[medium] Queue-settings UI could not inherit/clear.** `edgeDrain` initialized
   to `true` when no override existed and always saved an object — diverging from the
   `null ⇒ inherit env` contract; on an env-`off` deployment, opening+saving wrote
   `{edgeDrain:true}` and there was no way back to inheritance.
4. **[medium] Settings PATCH parsed body before auth.** `req.json()` + zod ran before
   `requireActiveSession`, so an unauthenticated caller could probe the schema /
   force body parsing instead of getting 401 — violating auth-first convention.

## Root Cause

1. The C3 path was modeled on the C1 Pending-resume flip (`→Running` + `driveResume`),
   but a flow HITL resume is a *different* lifecycle: the resume driver keeps the run
   `NeedsInput` and its terminal writes are `NeedsInput`-guarded. The generalization
   ignored that the flow resume driver, unlike crash-recover, owns `NeedsInput→Review`.
2. The two-phase claim decouples "slot reserved" (`tasks.queue_claimed_at`) from
   "slot occupied" (`runs` row). The reserve accounting only counted the latter, so
   the in-flight-claim window was invisible to a concurrent admitter.
3. A boolean checkbox cannot represent three states (inherit / on / off); defaulting
   the absent state to `true` silently invented an override.
4. Pre-existing route ordering (body-first) was extended with a new field without
   re-checking the auth-first convention for a changed handler.

## Solution

1. C3 is now kind-specific: **agent** → `→Running` + `startAgentSession` (unchanged,
   matches the existing agent idle-resume); **flow** → `markResumed` (`→NeedsInput`,
   fresh keepalive, checkpoint cleared) + clear `resume_requested_at`, then
   `driveResume` re-issues the session while the run stays `NeedsInput` — identical to
   the immediate (uncapped) HITL resume path, so the driver's completion CAS matches.
2. Added `countOutstandingC2Claims` (global `tasks.queue_claimed_at NOT NULL`); the
   gate's reserve check now uses `liveCount + outstandingClaims`, and the 60s poll's
   baseline `liveFlow` includes it too. Serialized under the scheduler advisory lock,
   each admitter sees the prior committed claim as consumed capacity.
3. `edgeDrain` is a tri-state `<select>` (Inherit / On / Off); the server passes the
   env-resolved default so "Inherit" shows the effective value; `save` builds the
   override from only explicitly-set keys and PATCHes `taskQueueSettings: null` when
   nothing is set (clears the override).
4. Reordered the settings PATCH: `requireActiveSession` → resolve project from the
   URL slug → `requireProjectAction(editSettings)` → THEN parse the body (a parse
   failure throws `CONFIG`, caught by the existing handler → 422).

## Prevention

- When reusing a "resume/recover" dispatcher across lifecycles, verify the target
  status matches EVERY status-guarded transition the downstream driver performs —
  grep the driver for `eq(runs.status, ...)` before choosing the claim state.
- A capacity guard that gates a two-phase claim MUST count the in-flight claim
  markers, not just the committed occupancy rows — otherwise lock-serialized
  concurrent claimers over-admit.
- Any settings whose server contract has an "inherit/null" state needs a tri-state
  control (never a boolean defaulting to a hard-coded value) and a way to clear.
- Auth-first is mandatory for changed route handlers: authenticate + authorize
  before reading/parsing the request body. Add a regression test asserting an
  unauthenticated malformed request returns 401/403, not 422.
