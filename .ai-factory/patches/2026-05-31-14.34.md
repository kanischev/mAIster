# Docker-gated integration suite silently red (seed omitted NOT-NULL flow_id) + masked exit code

**Date:** 2026-05-31 14:34
**Files:** web/lib/runs/__tests__/state-transitions.integration.test.ts, web/lib/__tests__/scheduler.integration.test.ts, web/lib/__tests__/authz.integration.test.ts, web/lib/flows/graph/__tests__/runner-core.test.ts, .ai-factory/plans/feature-m11a-flow-graph-lifecycle.md
**Severity:** medium

## Problem

`pnpm test:integration` was red: 20 tests failed across 3 files
(`state-transitions`, `scheduler`, `authz`) — all OUTSIDE the M11a diff. Two
distinct root causes:
1. `state-transitions` + `scheduler`: `seedRun` inserted `tasks`/`runs` with
   `flowId: null`, but `tasks.flow_id` / `runs.flow_id` have been `NOT NULL`
   since migration `0000` → `null value in column "flow_id" … violates
   not-null constraint`.
2. `authz`: the "role ordering" test reused `member@test.com` already created
   by an earlier test under a shared `beforeAll` (no per-test user cleanup) →
   `duplicate key value violates unique constraint "users_email_unique"`.

A separate, dangerous meta-issue: the failures were almost reported as "exit 0"
because the verify run piped vitest through `| tail` — the pipeline's exit code
is `tail`'s (0), NOT vitest's (1). The red suite was nearly missed.

## Root Cause

- **flow_id:** these integration tests are Docker-gated (testcontainers
  Postgres). They predate or ignored the `tasks.flow_id`/`runs.flow_id` NOT NULL
  + FK introduced in `0000`. The reference M11a graph integration seed
  (`runner-graph.integration.test.ts`) seeds a real `flows` row and threads
  `flowId`; the older M8/M9 seeds never did. Because the suite needs Docker, the
  failure stayed invisible whenever Docker/testcontainers wasn't exercised.
- **authz:** test isolation — `beforeAll` (not `beforeEach`) accumulates users
  across tests, and two tests hard-coded the same email. Shared mutable fixture
  state + a global unique constraint = order-dependent collision.
- **Masked exit code:** wrapping a test/lint/typecheck command in `… | tail`
  (or any pipe) makes the shell report the LAST stage's exit status. A failing
  command followed by a successful `tail` looks green.

## Solution

1. Seed a real `flows` row in each failing suite's `beforeAll` (mirroring the
   M11a graph seed: id/projectId/flowRefId/source/version/installedPath/
   `manifest: {}`/schemaVersion) and replace `flowId: null` with the seeded
   `flowId` in both the `tasks` and `runs` inserts. The NOT NULL invariant is
   correct (every task/run has a flow); the fix is to the test seed, not the
   schema.
2. `authz`: give the ordering test unique emails (`*-ordering@test.com`).
3. Added `runner-core.test.ts` — a deferred-release regression (M11a plan 3.6):
   asserts `cleanupSlashSession` releases an active slash session exactly once,
   clears the handle (no double-delete on retry), no-ops when idle, and swallows
   a `deleteSession` rejection so the terminal/failure path is never derailed.
4. Reconciled the M11a plan: tasks 3.4/3.8 now record the as-built dispatch —
   `steps[]` flows stay on the linear runner (only `nodes[]` → `runGraph`), so
   the planned `step_runs`→graph resume seed (3.8) is moot by design.

## Prevention

- A test seed that inserts a row with a foreign-key/`NOT NULL` column set to
  `null` is a bug even if TS accepts it (loosely-typed `drizzle(pool)` without
  the schema generic does not enforce insert non-nullability). Seed the
  referenced row.
- Prefer `beforeEach` cleanup OR unique fixtures per test; never share mutable
  rows that a unique constraint can collide on across tests in one file.
- NEVER trust the exit code of a command piped through `tail`/`grep`/`head`.
  Run the command bare and check `$?` (or `set -o pipefail`) before asserting
  pass/fail. A passing-looking `| tail` is the worst false-green.
- When a plan task's as-built behavior diverges (here: linear flows kept on the
  linear runner), reconcile the plan text in the Phase-7 as-built pass so
  downstream readers don't implement the superseded design.
