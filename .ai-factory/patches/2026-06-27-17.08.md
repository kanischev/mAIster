# M42 migration behavior changed but its ADR/schema/OpenAPI/analytics contracts didn't

**Date:** 2026-06-27 17:08
**Files:** docs/decisions.md, docs/database-schema.md, docs/api/web.openapi.yaml,
docs/api/supervisor.openapi.yaml, docs/system-analytics/{hitl,workbench-lifecycle,orchestrator,observatory}.md,
docs/architecture.md, docs/flow-dsl.md, docs/screens/chrome/active-workspaces.md,
web/lib/db/migrations/0081_m42_remap_slot_key.sql
**Severity:** high (operationally dangerous contract — could cause unnecessary abandon of recoverable live runs)

## Problem

Codex adversarial review of the M42 (ADR-114) `run_sessions` cutover surfaced three
spec/impl inconsistencies, all where the **implementation moved but the contract
didn't**:

1. Migration `0082` was changed to **backfill** `run_sessions` from the
   about-to-be-dropped `runs` mirror columns (preserving live runs' resume
   handles), but ADR-114 + schema docs still said "clean cutover, **no run-data
   migration**" and "existing in-flight runs **must be terminal / abandoned before
   deploy**". An operator following the stale ADR would needlessly abandon
   recoverable runs.
2. The gate-chat / stop / resume docs + both OpenAPI specs still gated availability
   on `runs.acp_session_id ≠ null` — a column `0082` **drops**. The code already
   reads the run's active `run_sessions` row (`loadActiveRunSession`,
   live-handle-first else newest).
3. Migration `0081`'s abort message told operators to "clear the table and re-map
   via the UI **after upgrade**, then re-run migrations" — but the abort **blocks**
   the upgrade, so the order is impossible.

## Root Cause

A behavior-level change to a migration (adding the `0082` backfill; the `0081`
abort guard) was landed without re-sweeping the **contract surface** that
describes that behavior. The stale claim "no run-data migration" was **duplicated
across 4 prose sites** (3 in ADR-114, 1 in database-schema.md) — single-line grep
even missed two because the phrase wrapped across lines. The dropped-column
reference was spread across **~14 current-contract sites**. Duplicated contract
prose drifts the moment one copy is the only one updated. The `0081` message was
written describing the happy-path UI remap without reasoning about the failure
ordering it actually executes in.

## Solution

- Rewrote ADR-114 (`#4`, the engine-bump rationale, and the Consequences bullet)
  + the `database-schema.md run_sessions` section to describe the **real**
  contract: a schema clean-cutover **with a one-shot run-data preservation
  backfill**; in-flight runs **survive** the deploy; the **single** remaining
  deploy precondition is non-empty `flow_runner_remaps`.
- Swept every **current-contract** `runs.acp_session_id` / `runs.runner_snapshot`
  reference to the run's **active `run_sessions`** row (OpenAPI ×2, hitl ×3,
  orchestrator ×3, workbench-lifecycle, observatory, architecture diagram,
  flow-dsl, active-workspaces). Left `runs.delegation_snapshot`, `runs.run_kind`,
  and `step_runs.acp_session_id` (all **stay**) untouched.
- Corrected the `0081` abort message + comment to the only feasible order:
  **export/record → clear → re-run migrations → re-map via UI after upgrade**.
  Added the operator runbook to the ADR Consequences + the schema `flow_runner_remaps`
  subsection.
- **Left historical records frozen** (R4/R9): pre-ADR-114 ADR bodies (ADR-005/078/
  etc.) and the dated `kaa-maister-m8-spike-findings` doc still name the old
  columns *as the then-current state* — rewriting them would falsify history. The
  ADR-114 design-motivation "finding" (`acp_session_id lives only in
  runs.acp_session_id`) was also left: it describes the **pre-decision** state it
  migrates away from.

## Prevention

- **A migration's error/abort message is a contract.** Its operator instructions
  must be **order-correct** against the failure path it fires in. When a guard
  blocks the upgrade, "do X after upgrade" is a bug — it must be "do X before
  re-running migrations".
- **When a migration's behavior changes, grep the contract surface, not the diff.**
  Sweep ADR(s), `database-schema.md`, both OpenAPI specs, and every
  `system-analytics/*` reader for the affected column/claim. A behavior change with
  a one-line code diff can still have a >10-site contract footprint.
- **Distinguish frozen-historical from current-contract before editing.** Pre-this-ADR
  ADR bodies and dated findings docs describe the past and stay; only "what is true
  now" docs get rewritten. (Same discipline as the renumber grep-to-zero rule.)
- **Grep-to-zero, accounting for line-wrapping.** Verify with both single-line and
  whitespace-tolerant (`grep -z` / multi-line) patterns — the false claim "no
  run-data migration" wrapped across two lines and evaded a naive grep.
- **Grep recorded (close-out):** `no run-data migration` → 0; `abandoned before
  deploy` → 0; stale `runs.acp_session_id ≠ null` in api/system-analytics → 0; all
  residual `runs.{acp_session_id,runner_snapshot}` hits classified as
  historical-ADR / `step_runs` / already-correct "moved to run_sessions" narrative.
  Validators: `validate:docs:all` 306/306 mermaid + 594 ADR anchors; redocly web +
  supervisor specs unchanged error/warning counts vs `main` (prose-only edits).
