# Per-node cost: camelCase fix re-introduced a double-count of sub-agent usage

**Date:** 2026-06-29 22:16
**Files:** supervisor/src/cost.ts, supervisor/src/__tests__/cost.test.ts, web/lib/runs/run-transcript-projector.ts, web/lib/runs/__tests__/run-transcript-projector.integration.test.ts, web/components/runs/node-transcript-panel.tsx, docs/system-analytics/runs.md
**Severity:** high

## Problem

The Phase-D cost fix widened `extractCost` to accept the ACP adapter's
camelCase `result.usage` (it was previously dropped → every node after the
first recorded zero tokens). But `attachCost` runs `extractCost` on EVERY raw
`session.line` and `reconcileRunCostRollups` SUMS every record per
`nodeAttemptId`. The fix's only anti-double-count guard was per-FIELD within one
`usage` object — not per-turn across lines.

Empirical check against a real flow run
(`.maister/maister-dev/runs/5e486911…/plan.log`, raw JSON-RPC adapter stdout):
the `plan` node's stream contained, in one node session, **1 snake `usage`**
(`input_tokens:1, output_tokens:1648`, nested in a Task tool response —
`...:69044,"totalToolUseCount":39,"usage":{...}` inside a `session/update`) plus
**2 camelCase `result.usage`** end-turn totals
(`{"jsonrpc","id":2,"result":{"stopReason":"end_turn","usage":{inputTokens…}}}`).
Old code recorded only the snake one (the 69,044 incident total). New code would
record all three and sum them — counting the sub-agent's tokens on top of the
end-turn total that already includes them.

## Root Cause

The plan/AC (T-D2) said "result.usage is canonical PER TURN, never count both",
but the implementation only deduped snake-vs-camel keys inside a single object.
The two shapes are SEPARATE lines: the camelCase `result.usage` is the adapter's
end-turn JSON-RPC `result` total; the snake `usage` is a per-message / sub-agent
object nested in a `session/update` notification — a component already subsumed
by the turn total. Counting both double-counts. The fix matched its AC only
under the unstated assumption that no turn emits both, which real data falsifies.

## Solution

Classify each usage by POSITION, not snake/camel (robust for any adapter):
`locateUsage` returns `source: "result"` when the usage sits under the top-level
JSON-RPC `result`, else `source: "stream"`. `extractCostWithSource` exposes that;
`extractCost` stays a thin wrapper (callers/tests unchanged).
`createTurnUsageRecorder` (pure, unit-tested) collapses the per-session stream so
each turn writes exactly one record: a `result` usage is written immediately and
discards any buffered same-turn `stream` usage; a `stream` usage is buffered and
flushed at session close only if no `result` superseded it (a streaming-only turn
is still recorded). `attachCost` wires the recorder and flushes on `detach`.

Also fixed two smaller review findings in the same pass: the transcript projector
now computes its resume cursor only over attributable lines (a run-level
`run.needs_input` line no longer defeats the `unchanged` short-circuit during
NeedsInput), and the node-transcript panel got a latest-wins fetch guard.

## Prevention

- When a fix "accepts a new shape", check every WRITER and AGGREGATOR of that
  value, not just the parser. `extractCost` is one of N callers; the real risk
  was `attachCost` (per-line emit) × `reconcileRunCostRollups` (sum). Fix the
  class (the emission/aggregation path), not the cited line.
- When an AC says "never count both / per-turn", a per-field guard is NOT that —
  it must dedup across the actual emission boundary (here: stateful per session).
- Replay the REAL artifact the bug came from. The original T-D2/T-D3 tests used
  synthetic one-record-per-node fixtures and never exercised a turn that emits a
  nested usage + an end-turn result.usage. The on-disk `*.log` files are the
  `session.line` source — grep them before trusting a usage-shape assumption.
