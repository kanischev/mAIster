# M42 cutover left a raw-SQL reader on dropped columns + destructive migrations with no preservation

**Date:** 2026-06-27 16:03
**Files:** web/lib/queries/runs-list.ts, web/lib/db/migrations/0081_m42_remap_slot_key.sql, web/lib/db/migrations/0082_m42_drop_runs_runner_mirror.sql, web/e2e/_seed/seed-e2e.ts, web/e2e/_seed/db.ts, web/e2e/{run-task-context,m13-assignments,active-workspaces,multi-run-cost-policy}.spec.ts, web/lib/flows/graph/runner-graph.ts, web/lib/queries/__tests__/runs-list.integration.test.ts (new), web/lib/db/__tests__/m42-backfill.integration.test.ts (new)
**Severity:** critical

## Problem

A Codex adversarial review of the M42 branch (run_sessions = sole source of truth for per-run runner/resume state) surfaced four defects, all confirmed real against the code:

1. **(critical)** `listRunsPage()` — the `/runs` page reader — still selected and filtered `r.capability_agent` / `r.runner_snapshot`, both DROPPED from `runs` by migration 0082. The page would 500 with `column r.capability_agent does not exist` on any load after migration.
2. **(critical)** Migration 0080 created an EMPTY `run_sessions`, then 0082 dropped `runs.{runner_id, runner_resolution_tier, capability_agent, runner_snapshot, acp_session_id}` with no backfill — every existing run silently lost its resume handle + runner snapshot (recovery / stop / gate-chat / diagnostics can no longer target the ACP session).
3. **(high)** Migration 0081 `DROP TABLE ... CREATE TABLE`'d `flow_runner_remaps`, wiping all per-project runner bindings.
4. **(medium)** The e2e seed (`seed-e2e.ts`, ~22 sites) + 5 spec files still INSERT/SELECT the dropped `runs` columns — would fail before exercising the UI.

## Root Cause

- **#1:** The M42 reader fan-out swept every Drizzle *query-builder* reader to the `activeSession*()` scalar helpers, but missed the ONE reader written in raw `sql\`...\`` template SQL. A grep over schema-symbol accessors (`runs.capabilityAgent`) does not match a raw string (`r.capability_agent`), so the straggler was invisible to a symbol-based sweep. Its only test was a **mocked-DB unit test** (`listRunsPage` accepts an injected `db`), which never sends the SQL to Postgres — so the green suite was blind to a column that does not exist in the real schema.
- **#2 / #3:** The contract migration was authored as "create new shape + drop old shape" without the preservation step. "Drop a column / re-key a table" was treated as a pure schema edit, ignoring that the columns/rows held LIVE state (the product's core resume/recovery promise) — destructive-by-omission, not by decision.
- **#4:** The test-seed sweep fixed the *integration* seeds (gated by `test:integration`) but not the *e2e* seed or spec files, because the e2e suite is not in the green gate set (Next 16 single-dev-server lock), so the breakage produced no failing gate.

## Solution

- **#1:** Added a local `activeRunSessionField(field)` helper in `runs-list.ts` that emits the same correlated `run_sessions` scalar subquery as `activeRunSessionScalar` (live-handle-first, else newest), correlating on the query's `r` alias with inner alias `sess` (the outer `rs` is the run_schedules lateral). Rewrote the two selects and the agent filter to source from it. Output shape unchanged → consumers + the unit test untouched.
- **#2:** Prepended an `INSERT INTO run_sessions (...) SELECT ... 'default' ... FROM runs WHERE <any mirror non-null> ON CONFLICT DO NOTHING` backfill to 0082, BEFORE the drops — preserves `acp_session_id` + snapshot for every existing run.
- **#3:** `slot_key` (`session:<name>` / `consensus:...`) is derived from the flow's COMPILED session structure, NOT reconstructable in SQL from the old `(step_id, source_runner_id)` (multiple old rows collapse onto one session slot with conflicting runners). A deterministic migration would be a wrong guess, so prepended an abort-if-nonempty guard (`RAISE EXCEPTION`) to 0081 — converts silent loss into a loud, documented precondition (operator clears + re-maps via the existing UI).
- **#4:** Moved runner/resume fields out of every e2e `runs` INSERT into a parallel `run_sessions` row via shared `seedDefaultRunSession` / `e2eClaudeRunnerSnapshot` helpers (in `e2e/_seed/db.ts` for specs; a self-contained copy in standalone `seed-e2e.ts`). Fixed the one `SELECT r.runner_id` read to a `run_sessions` subquery. Removed a `runnerSnapshotJson` helper my edits orphaned.
- Also fixed a stale `runner-graph.ts` comment claiming the dropped `runs.acp_session_id` mirror was "still maintained."

Verification (recorded per the sweep rule): grep-to-zero for `\b(r|runs)\.(runner_id|runner_resolution_tier|capability_agent|runner_snapshot|acp_session_id)\b` over `lib`+`app` (production) and over `e2e` → both clean. New integration tests prove #1 (reverting the fix reproduces `column r.capability_agent does not exist`; restored → pass) and #2 (acp_session_id preserved; empty runs skipped). The #3 guard was driven against a seeded row → raises. `tsc` clean, eslint clean, runs-list unit 4/4, portfolio integration 36/36 (no regression), `db:migrate` clean on fresh Postgres 16.

## Prevention

- When dropping or moving a column, grep for BOTH the camelCase schema accessor (`runs.capabilityAgent`) AND the raw snake_case SQL forms (`r.capability_agent`, `runs.capability_agent`, `INSERT INTO runs`, `SELECT ... FROM runs`). Raw-SQL readers are invisible to symbol/IDE refactors and to schema-accessor greps — they are the ones the sweep misses.
- A raw-SQL reader (`db.execute(sql\`...\`)`) MUST have an **integration** test against the migrated schema; a mocked-`db` unit test never executes the SQL and cannot catch a dropped column. Treat "mocked-DB unit test" as zero coverage for the SQL itself.
- A contract migration that DROPs a column or re-keys a table holding live state must either **backfill** (`INSERT ... SELECT` before the drop) or **abort-guard** (`RAISE` if non-empty) — never silently drop. If the new key is not derivable in SQL from the old, the guard is the honest fix, not a guessed mapping.
- Sweep test seeds across ALL harnesses (integration + e2e + per-spec), not just the ones in the CI gate. e2e-seed breakage is invisible precisely because e2e is not gated — confirm with a grep-to-zero over `e2e`, not by relying on a green suite.
