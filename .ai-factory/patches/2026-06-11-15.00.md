# Gate-chat/dirty-resolution claim ordering races + chat authz + OpenAPI drift (Codex adversarial findings)

**Date:** 2026-06-11 15:00
**Files:** web/lib/runs/dirty-resolution.ts, web/lib/services/gate-chat.ts, web/app/api/runs/[runId]/hitl/[hitlRequestId]/chat/route.ts, docs/api/web.openapi.yaml, docs/system-analytics/hitl.md, docs/decisions.md
**Severity:** critical

## Problem

Four adversarial-review findings on the M30 branch:

1. **[critical]** `resolveDirtyWorktree` ran the git side-effect (including an
   irreversible `discard`) BEFORE the write-once CAS on
   `hitl_requests.dirty_resolution`. Two concurrent requests both passed the
   unserialized precheck; the loser got CONFLICT only after its git op had
   already mutated/destroyed worktree state — DB recording one reviewer
   choice while the worktree reflected another.
2. **[high]** Idle gate-chat `chatResume` spawned a supervisor session before
   any durable claim and ignored the `markResumed` result — a racing
   `/respond` resume left a duplicate ACP session prompting a stale pause.
3. **[high]** The chat GET route authorized with `readBoard` (viewer) while
   OpenAPI + ADR-075 promised `answerHitl` (member+) — HITL transcripts
   leaked to project viewers.
4. **[high]** OpenAPI described a different gate-chat API than implemented:
   `202 {ok, seq}` + SSE-only reply vs the actual synchronous
   `200 {runId, hitlRequestId, userMessage, agentMessage, resumed}`; GET
   envelope and dirty-resolution response also drifted; 400/502 statuses
   missing.

## Root Cause

1. The "idempotency marker lands AFTER the side-effect" idiom (correct for
   *idempotent, non-destructive* side-effects like the respond flow's
   supervisor call) was transplanted onto a path whose side-effect is
   **destructive and mutually exclusive across racers**. For
   commit-vs-discard, marker-after means the loser's git op runs before it
   learns it lost. The codebase already had the correct pattern
   (promote.ts 3-phase claim; resumeRun claim-before-spawn) — the new path
   simply didn't reuse it. Notably ADR-079/OpenAPI *documented* the
   claim-first order; the implementation violated its own spec.
2. Same class: `chatResume` was written as spawn-then-claim while its
   sibling `resumeRun` is claim-then-spawn with rollback; the
   `markResumed` return value was discarded, so the lost-race signal the
   helper deliberately returns was thrown away.
3. GET was added with the board-read default instead of the
   contract-specified action; no route-level test pinned the authz action,
   so nothing failed.
4. The OpenAPI section was authored at plan time (202+SSE design) and never
   reconciled after the implementation went synchronous; no route-contract
   test asserted the documented envelope.

## Solution

- `resolveDirtyWorktree`: write-once CAS **claim first** (atomically
  re-guarded on `dirty_resolution IS NULL` + `responded_at IS NULL` + run
  status via `EXISTS`), git side-effect after; on side-effect failure the
  claim rolls back to NULL and the error rethrows (gate open, unrecorded).
  For `discard`, the gate-chat L3 baseline ref is deleted immediately after
  the tree is cleaned, BEFORE re-materialization (DD12: no un-discard even
  when remat fails).
- `chatResume`: `markResumed` claim BEFORE `createSession` when idle; a
  lost claim throws `CONFLICT` (409) with no spawn; a failed spawn calls
  `rollbackResumedRun` (best-effort) and rethrows — mirroring `resumeRun`.
- Chat GET: `requireProjectAction(run.projectId, "answerHitl")`; viewer UI
  degrades to the panel's empty initial state (composer already gated by
  `canAct`).
- OpenAPI: rewrote chat GET/POST + dirty-resolution sections to the
  implemented envelopes, added shared `GateChatMessage` schema, added
  400/502 + CONFLICT statuses; synced hitl.md + ADR-075 idle-order phrasing;
  removed the false "audit row" claim from hitl.md.
- Tests: 4 new race/rollback integration tests (mocked git, real PG,
  barrier-gated two-racer), 3 new claim-order integration tests (delegating
  spies over real transitions, one-shot rival), 12 route-contract unit
  tests pinning envelopes, the answerHitl action, and the error-status map.

## Prevention

- A write-once marker is a valid race guard ONLY if it is claimed BEFORE
  the side-effect when the side-effect is destructive or racers' effects
  diverge. Marker-after is acceptable only when every racer's side-effect
  is idempotent AND identical.
- When adding a sibling of an existing flow (chat-resume vs respond-resume),
  diff the orderings; any deviation from the established
  claim→side-effect→finalize shape needs an explicit justification.
- Never discard a `StateTransitionResult` — every `{ok:false}` needs a
  mapped behavior (CONFLICT, rollback, or documented no-op).
- Every new route lands with a route-contract test pinning (a) the authz
  action string and (b) the response envelope keys + error-status map; this
  is what catches spec drift and authz defaults at review time.
- When an API design changes during implementation (async→sync), reconcile
  the OpenAPI section in the same commit — the spec is a contract, not a
  plan artifact.
