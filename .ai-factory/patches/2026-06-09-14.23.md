# Misread a fork-before-feature branch as a "feature revert" (git diff direction artifact)

**Date:** 2026-06-09 14:23
**Files:** (process patch — no code root-caused; touched docs/* + web/lib/db/schema.ts during the mistaken fix-pass, reverted/superseded at rebase)
**Severity:** medium

## Problem

A `/aif-review` of branch `claude/zen-allen-944361` concluded it "cleanly
reverts the token-actor-scope feature" (deleted migration `0031`,
`web/lib/tokens/scopes.ts`, `web/types/token-scopes.ts`, schema token columns,
`token-actions.tsx`). A follow-up `/aif-fix` then "corrected doc drift" in eight
token-feature files (`database-schema.md`, `identity-access.md`,
`external-operations.md`, `assignments-domain.md`, `erd.md`,
`integrations-domain.md`, `tasks.md`, `schema.ts`).

Both conclusions were WRONG. The branch never reverted anything. It forked from
`6c89f31c` BEFORE `20b42ece feat(tokens): add scoped user-owned project tokens`
landed on `main`. `git diff main..HEAD` therefore rendered every main-only file
as a "deletion" on the branch side, which read like a deliberate removal.

## Root Cause

`git diff main..HEAD` (and `--stat`) shows files present in `main` but absent in
`HEAD` as deletions — indistinguishable, at a glance, from the branch having
deleted them. The reviewer never checked the fork relationship:

- `git merge-base main HEAD` = `6c89f31c`, but `main` = `20b42ece` (one commit
  AHEAD of the merge-base). That single commit was the entire "feature" the
  review thought was reverted.
- `git diff 6c89f31c..HEAD` (branch-only changes) for every token file was
  EMPTY — proof the branch never touched them. This was the check that would
  have caught the misread, and it was not run until commit time.

Worse, the `/aif-fix` "drift fixes" edited the branch's STALE pre-feature copies
of those docs to re-derive corrections that `main` already shipped correctly
with the feature. On rebase onto `main` those edits are redundant-and-conflicting
(resolved by taking `main`), and one (`external-operations.md` "scope
enforcement is inert except 2 HITL routes") is flatly wrong once the real
scoped-token feature is in the base.

## Solution

- Recognized the fork-before-feature relationship at commit/rebase time and
  rebased the branch onto `main`, resolving the stale token-file edits to
  `main`'s authoritative versions and merging the genuinely branch-side work
  (the diff-truncation refactor) into the doc files both touched.
- Kept only the truly branch-side fix-pass edits (CodeEditor initial-value
  comment, editor-stack dep exact-pinning, lowlight-stub version guard).

## Prevention

- Before calling any branch a "revert" or "removal," verify the fork
  relationship: run `git merge-base <base> HEAD`; if `<base>` is AHEAD of the
  merge-base, the base has commits the branch never had, and `git diff
  base..HEAD` deletions are diff-direction artifacts, not branch removals.
- Re-derive "what did THIS branch change" from `git diff $(git merge-base
  base HEAD)..HEAD`, never from `git diff base..HEAD`, whenever the base may
  have advanced.
- Never "fix doc drift" against a branch baseline that is behind the integration
  target — rebase/merge first, then reconcile docs against the integrated tree.
- A reviewer claim of "feature removed" MUST be backed by a diff that shows the
  removing hunk in a branch commit (`git log -p --follow <file>`), not by its
  mere absence in `base..HEAD`.
