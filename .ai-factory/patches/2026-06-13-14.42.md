# Fix Patch: Agent Comment Context and Observatory Read Path

## Problem

- Agent launches triggered by `task.comment_added` did not include the
  triggering comment body or bounded task-thread tail in the prompt, so the
  agent could see only generic trigger JSON.
- Observatory portfolio/project reads reconciled cost rollups inline, causing
  query count growth and mutating derived state from a read path.

## Root Cause

- Domain-event prompt assembly treated all events uniformly and did not
  special-case comment events that require human-authored context.
- The Observatory cost query mixed reconciliation and presentation concerns,
  even though rollups are a derived read model maintained by lifecycle code.

## Solution

- For `task.comment_added` runs, load the exact triggering comment by
  `taskId + commentId`, fail fast if the payload or row is missing, and append
  a bounded recent thread tail to the prompt.
- Keep Observatory cost aggregation read-only by querying stored run/node cost
  rollups only.
- Add focused prompt tests for the comment context and missing-comment
  precondition.

## Prevention

- When reviewing domain-event agents, verify that prompt context contains the
  domain object that caused the trigger, not just event metadata.
- Treat analytics/Observatory query functions as read-only unless the spec
  explicitly names a write-through reconciliation boundary.
- Watch for query-count assertions that detect hidden per-row work in dashboard
  read paths.
