# Force-relaunch verdict ignored flow-setup issues; duplicate EN launch copy

**Date:** 2026-06-30 13:02
**Files:** web/app/api/runs/launch-options/route.ts, web/messages/en.json, docs/decisions.md, docs/system-analytics/tasks.md, docs/api/web.openapi.yaml, .ai-factory/specs/relaunch-from-task-history.spec.md, web/app/api/runs/__tests__/relaunch-gate.integration.test.ts
**Severity:** low

## Problem

Two minor findings from the `/aif-review` of the ADR-119 force-relaunch branch:

1. **[S1 â€” UX gap]** `GET /api/runs/launch-options` computed the additive
   `relaunch` (force) verdict from `classifyForceRelaunchLaunchability` alone,
   which only refuses on the TASK gates `flagged`/`blocked`. Unlike the sibling
   manual `launchability`, it did **not** layer flow-setup issues
   (`not_enabled`/`flow_missing`/`incompatible`/â€¦). So if a task's flow was
   disabled or dropped *after* the task's first run, the runs-history "Run again"
   button stayed enabled, the user clicked Create, and the launch failed
   server-side with `PRECONDITION` surfaced as a dialog error â€” instead of the
   button disabling up-front with the flow reason.
2. **[S2 â€” answered, not coded]** A question on whether manual relaunches sharing
   the ralph-loop `attempt_number` budget is intended (it is â€” total attempts per
   task are bounded regardless of who fired them).
3. **[nit]** EN `runAgainConcurrent` = "Run again", byte-identical to the existing
   `runAgain` key (RU already differentiated).

## Root Cause

1. The force classifier was deliberately scoped to the TASK gates with a
   documented rationale ("the button only renders once a task has â‰Ą1 run â†’ already
   configured"). That rationale conflated *has a flowId* (configured) with *flow
   currently launchable* â€” a flow can become non-launchable (disabled / package
   dropped) **after** the first run, a state the "already configured" framing did
   not cover. The manual `launchability` already had the `flowIssue ?? "launchable"`
   layering; the force verdict was a parallel computation that was never given the
   same layer.
2. Copy oversight â€” the new key was seeded as a copy of the adjacent `runAgain`.

## Solution

1. In `launch-options/route.ts`, layer `flowIssue` into the `relaunch` verdict
   exactly like the manual `launchability`
   (`relaunchReason = relaunchLaunchability === "launchable" ? (flowIssue ?? "launchable") : relaunchLaunchability`),
   and gate `relaunch.launchable` on `flowIssue === null`. Run status is still
   never consulted; the server's flow-enablement checks in `launchRunStaged`
   remain the authoritative backstop (this is purely a UI-affordance improvement,
   so no `[FIX]` console logging was added â€” the project uses pino logger
   boundaries, `no-console` is a lint rule, and `launchRunStaged` already
   `log.debug`s the verdict). Expanded the openapi `relaunch.reason` enum to the
   full `FlowLaunchIssue` set and synced the ADR-119 decision text, `tasks.md`
   wire-up, and the spec FR-4.
2. Documented the ralph-budget interaction as an intended Consequence in ADR-119.
3. Changed EN `runAgainConcurrent` to "Run another".
4. Added a real-PG regression: a task with a `Disabled` flow returns
   `relaunch.launchable === false` / `reason === "not_enabled"` (mirroring the
   manual verdict).

**Sibling sweep (recorded):** the force verdict is computed in 3 sites â€”
`launch-options` (fixed), `launch-popover` (consumes the fixed field, auto-correct),
and `page.tsx` `forceLaunchability` (page-button `disabledReason`). The page site
was left unchanged on purpose: it mirrors the manual page button
(`launchDisabledReason`), which also does not layer flow issues at page level â€”
both open the dialog and rely on the dialog verdict to disable Create. Changing
only the force page button would break that parity.

## Prevention

- When a feature adds a SECOND classifier/verdict parallel to an existing one
  (here: force vs manual launchability), diff the two end-to-end: every layer the
  original applies (flow issues, supervisor readiness, archived) is a candidate
  the parallel one must either apply or consciously document as excluded â€” a
  "verdict B mirrors verdict A" claim must be checked against A's *composition*,
  not just its base classifier.
- "Already configured" (has an id/fk) is not the same invariant as "currently
  valid/launchable" (the referenced resource still resolves + is enabled). A gate
  that relies on the former silently admits the latter's failure modes.
- When seeding a new i18n key next to an existing one, give it distinct copy or
  alias it explicitly â€” a duplicate value is a latent "why are there two keys"
  question.
