# Adopt+launch atomicity, multi-package rollback, and a publish mutex (Codex adversarial review)

**Date:** 2026-06-25 20.14
**Files:** web/lib/services/runs.ts, web/lib/local-packages/versions.ts, web/lib/local-packages/publish.ts, web/lib/local-packages/lock.ts, web/lib/db/schema.ts, web/lib/db/migrations/0072_publish_mutex.sql, docs/decisions.md (+ 3 test files)
**Severity:** critical

## Problem

A Codex adversarial review of M39 Stream B (`feature/studio-package-authoring-runtime`)
returned **needs-attention** with 2 critical + 1 high finding — and a *prior* fix round
(`patches/2026-06-25-17.19.md`) had already claimed finding #1 was addressed:

1. **[critical]** A version-adopt launch advanced the shared project pin, but the only
   compensation (`revertPackageVersionChoices`) lived inside the catch that begins *after*
   `addWorktree`. So the dropped-flow refusal, the second supervisor health check, the
   deterministic preconditions, AND `addWorktree` itself all threw with the pin left
   advanced. The prior patch even *claimed* "worktree-add failures roll the pin back" — false.
2. **[critical]** `applyPackageVersionChoices` applied each package's `upgradeAttachment` and
   only `return`ed the accumulated reverts on full success. A second package's failure threw,
   discarding the first package's undo handle → partial commit with no way to compensate.
3. **[high]** Publish rewrote a single shared git remote (`maister-publish`) then pushed by
   name. Two concurrent publishes interleaved → committed content pushed to the wrong repo.

## Root Cause

All three are the same class: **a durable/shared-state mutation sequenced ahead of fallible
work, with compensation that is narrower than the failure surface.**

- #1: the compensation boundary was drawn at the wrong place (post-worktree), and the
  "accepted residual" reasoning was wrong — the pin is *shared project state*, so advancing it
  on a failed launch corrupts every future launch of that package by any task, not just a
  re-launch of the same one. "The user re-launches into the same gate" does not hold for
  shared state.
- #2: a function that hands the caller an undo handle (`AdoptRevert[]`) only returned it on the
  success path. On the throw path the handle vanished, so the all-or-nothing contract the
  caller depended on did not exist.
- #3: a module-level mutable resource (the named remote) shared across concurrent invocations,
  with no per-operation isolation and no serialization.

## Solution

1. **#1 — widen the compensation boundary (nested trys).** An outer `try` now spans the whole
   post-advance remainder of `launchRunStaged`; its catch calls `revertPackageVersionChoices`
   on ANY failure. The existing worktree-cleanup `try` was moved up to start right after
   `addWorktree` (so a capability-snapshot failure also removes the orphan worktree). Nesting
   keeps both catches scope-safe: the outer catch reads only pre-advance vars
   (`adoptReverts`/`project`/`_db`), the inner reads `worktreePath`/`runId` (declared before
   it). `runId` was hoisted above the outer try (the post-catch `tryStartRun`/return needs it).
2. **#2 — make `applyPackageVersionChoices` all-or-nothing.** Phase 1 validates every choice
   without mutating; phase 2 applies with a `try/catch` that best-effort reverts the already-
   advanced attachments (reusing `revertPackageVersionChoices`) before rethrowing.
3. **#3 — serialize publish per package (owner-chosen over a unique remote).** New
   `local_packages.publishing_started_at` (migration 0072) + `acquirePublishLock` /
   `releasePublishLock` in `lock.ts`: an atomic CAS claim (strict mutex — NO same-user escape,
   unlike the editor lock), a claim-timestamp release token so a stale-takeover's late release
   can't drop a reclaimer, and staleness auto-recovery so a crashed publish never wedges. The
   route surfaces the `CONFLICT` as "publish in progress" (non-blocking, no queue).

## Prevention

- **Sequence durable/shared-state mutations as late as possible; when one must precede fallible
  work, the compensation `try` must span the ENTIRE fallible remainder — not just the
  convenient tail.** Drawing the boundary at "after the worktree" missed half the failures.
- **If a function returns an undo handle, it must return/apply it on the throw path too.** A
  rollback handle that only exists on success is not a rollback.
- **"Deterministic, the user chose it, they'll re-run" is NOT a valid reason to leave SHARED
  state mutated on failure** — shared state affects unrelated future operations. Per-request
  determinism ≠ safe.
- **Concurrent ops on a shared named resource (a git remote, a temp branch, a singleton row)
  need per-operation isolation OR a mutex.** A reused mutable name is a cross-request race.
- **A CAS mutex needs a release *token* (here the claim timestamp), not an unconditional clear**
  — otherwise a stale-takeover + late release drops the live holder's claim.
- **Re-review a "fixed" finding against the actual code, not the patch note.** The prior round's
  note asserted a guarantee (`addWorktree` rollback) the code did not provide.
- **Blast-radius grep recorded (skill-context rule):** `applyPackageVersionChoices` →
  1 caller (`runs.ts:428`); `revertPackageVersionChoices` → 1 caller (`runs.ts`);
  `publishLocalPackage` → 1 caller (the publish route); `PUBLISH_REMOTE`/`gitSetRemote` →
  `publish.ts` + `git.ts` only. No other launch/publish entry point adopts or publishes, so the
  fix surface is exactly these files.
- **Tests:** version-adopt integration (a later package's failure rolls back an earlier
  advance), publish integration (concurrent publish → one `CONFLICT`; stale lock reclaimable),
  and a launch unit test (dropped-flow → the outer catch re-pins). Verified: tsc 0, eslint 0,
  full web unit 4981 passed, the 2 integration suites + the launch suite green, migration 0072
  applies clean (the integration tests migrate over it), ADR anchors 219 resolved.
