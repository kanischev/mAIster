# M15 readiness SSOT: kill the read-model/merge-guard divergence + the duplicated batch classifier

**Date:** 2026-06-04 12:33
**Files:** web/lib/flows/graph/readiness-core.ts, web/lib/flows/graph/evidence-readiness.ts, web/lib/queries/readiness.ts, web/lib/queries/readiness-batch.ts (new), web/lib/queries/board.ts, web/lib/queries/portfolio.ts, web/lib/queries/project.ts, web/components/readiness-badge.ts (new), web/components/run/readiness-summary.tsx, web/components/board/flight-card.tsx, web/components/portfolio/project-card.tsx, docs/api/external/operations.openapi.yaml, docs/api/web.openapi.yaml, docs/system-analytics/readiness.md, web/lib/queries/__tests__/readiness-ssot-consistency.integration.test.ts
**Severity:** medium

## Problem

A milestone whose entire thesis was "single source of truth for readiness" shipped three SSOT
violations that an `/aif-review` caught:

1. **Merge guard vs every readiness badge could disagree for the same run.** `assertEvidenceReady`
   (the authoritative promote gate) re-evaluated a `failed` `artifact_required` gate against current
   artifacts and could treat it as clear, but `getRunReadiness` / board / portfolio mapped
   `gateStatusContribution("failed") → "failed"` flatly. A run the engine would merge showed a red
   "failed" badge. Conservative direction (UI over-states blocking), but a real lie. Acknowledged
   in-code as an unclosed "Task 21".
2. **`computeReadinessByRun` was byte-identical** (modulo one param name) between `board.ts` (private)
   and `portfolio.ts` (exported) — ~136 lines × 2. `project.ts` already imported portfolio's copy.
3. **The 6-state badge class map was copy-pasted three times** across the run/board/portfolio
   components.

Plus drift: the external readiness OpenAPI enum omitted `overridden` (which M15 added to the DTO),
and the analytics flowchart had a phantom "OR required artifact stale → stale" branch that contradicts
the code (a stale-only artifact yields `blocked`, never `stale`).

## Root Cause

The SSOT was applied to the *primitive* (`readiness-core` held `gateStatusContribution`,
`liveBlockingGates`, `rollupReadiness`) but **not to the composition**. The per-surface call sites each
re-assembled those primitives, so the one piece of logic that lived in only one surface —
`assertEvidenceReady`'s `failed artifact_required → re-check current refs` re-eval — silently diverged
from the others. "Shared helpers" is not "shared verdict": as long as each consumer hand-wires the
helpers, a nuance added to one consumer is invisible to the rest. The duplicated `computeReadinessByRun`
is the same failure one level up — two copies of the *composition*, with a `Task 21 reconciles both`
TODO sitting in both as the drift fuse.

## Solution

- Lifted the re-eval into one **pure** helper `blockingGateContribution(gate, currentDefIds)` in
  `readiness-core` (no DB — each caller supplies its own already-fetched `currentDefIds` set, so purity
  is preserved and there's no N+1). `assertEvidenceReady`, `getRunReadiness`, and the batch classifier
  all route through it. The failed-artifact_required nuance now lives in exactly one place.
- Extracted the duplicated `computeReadinessByRun` into a new shared server module
  `web/lib/queries/readiness-batch.ts`; board/portfolio/project all import it. Net −317 lines in the
  two query modules.
- Extracted the badge map into `web/components/readiness-badge.ts`, imported by all three components.
- While retyping the gate-row fetches to feed the helper, the inline row `status` was typed
  `GateResultStatus` (instead of `string`), which dissolved all 6 unflagged `as any` casts the review
  flagged — the casts existed only because the rows were loosely typed.
- Doc fixes: added `overridden` to the external OpenAPI readiness enum + bullet; dropped the phantom
  branch from the flowchart; noted the merge-readiness refusal in the web promote 409.
- Tests: added a `stale` gate case and two failed-artifact_required re-eval cases (ready + still-blocked)
  to the SSOT-consistency suite, each asserting `assertEvidenceReady` AND all three read models agree —
  so the merge-guard-vs-badge invariant is now executable, not just intended.

## Prevention

- When a milestone claims "single source of truth," verify the **composition** is shared, not just the
  leaf helpers. A read model that re-assembles shared primitives is a divergence waiting to happen — grep
  for the same primitive call-chain (`liveBlockingGates` + `gateStatusContribution` + `rollupReadiness`)
  appearing in more than one place and collapse it into one function.
- A `// Task N reconciles both` / `byte-equivalent to X` comment is a **review finding, not a note** —
  it is the author flagging known duplication that will drift. Treat it as a must-fix.
- Cross-surface invariants ("the merge guard and the badge agree") need a test that drives BOTH the
  authoritative path and the display path against one seeded row. Asserting only that the three display
  surfaces agree with each other misses the case where all three agree with each other but disagree with
  the gate.
- A pure classifier's purity (`no DB/IO`) is an asset to protect: when a re-eval needs DB state, factor
  the **decision** into the pure module and keep the **fetch** in the caller — don't drag IO into the
  SSOT.
- An enum value added to a DTO union (`| "overridden"`) is a wire-contract change: grep the value into
  every `*.openapi.yaml` `enum:` that types that field.
