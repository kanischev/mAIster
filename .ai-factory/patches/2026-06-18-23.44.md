# CCR Start reported success but left dependent runners NotReady (readiness never persisted)

**Date:** 2026-06-18 23:44
**Files:** web/lib/acp-runners/native-defaults.ts, web/app/api/admin/router-sidecars/[sidecarId]/start/route.ts, web/app/api/admin/router-sidecars/[sidecarId]/stop/route.ts, docs/system-analytics/acp-runners.md, docs/decisions.md (ADR-094)
**Severity:** high

## Problem

After an admin clicked **Start** on the CCR router sidecar (ADR-094), the route
spawned the process and returned `{ok, state:"ready"}` — but a `claude-code-ccr`
runner that routes through that sidecar stayed `NotReady`, so launching it kept
failing with `EXECUTOR_UNAVAILABLE` ("ACP runner … is not ready; refusing
fallback", `web/lib/acp-runners/resolve.ts:130`). The Start button looked
successful while the feature it exists to enable remained blocked, with no
on-screen reason and no way forward except an undocumented sidecar PATCH.

Separately (low severity), the ADR-094 analytics named the wrong MiMo native
default runner id: docs said `mimo→mimo-native` while code/preset use
`mimo-code-native` — anyone asserting against the documented id would look for a
row that does not exist.

## Root Cause

Two writers, one of them missing.

`evaluateRunnerReadiness` gates a sidecar-backed runner on the **stored**
`platform_router_sidecars.readinessStatus` (readiness.ts:252). That stored value
was only ever written by the sidecar **create** (`POST`) and **edit** (`PATCH`)
routes. `reconcilePlatformRunners` — documented as "the single writer of
`readiness_status` outside create/edit" — recomputed **runner** readiness from
live diagnostics but read sidecar rows **as-is**, never recomputing sidecar
readiness. The Start/Stop routes were pure supervisor proxies and persisted
nothing.

So the chain had no path from "process is now running" → "stored sidecar
readiness = Ready" → "dependent runner = Ready" except a manual create/edit. The
seed ships `ccr-default` as `NotReady`, so a fresh install could never make a
CCR runner launchable through the intended Start flow. The "single writer" claim
was true for runners and silently false for the sidecars those runners depend
on.

The MiMo drift was a plain copy error in the ADR/analytics prose; the code was
always correct.

## Solution

Make `reconcilePlatformRunners` the single diagnostics-driven writer of **both**
runner and sidecar readiness: it now recomputes each `platform_router_sidecars`
row via `evaluateSidecarReadiness` from the same diagnostics **before** the
runner loop (and updates the in-memory row, so a sidecar-backed runner in the
same pass sees the fresh status), persisting only on change — mirroring the
existing runner update-if-changed pattern.

Added `reconcilePlatformRunnersFromSupervisor({db})` — fetch diagnostics, then
reconcile — and the Start/Stop routes call it after the supervisor acks, so the
just-changed sidecar and its dependents converge immediately instead of only at
the next `/settings` load. It is best-effort: a diagnostics outage is a no-op, a
reconcile failure is logged (pino `warn`), never thrown — a successful spawn is
never reported back as a failure. Honesty is preserved: readiness only flips to
`Ready` when diagnostics actually report the sidecar `state:"ready"` (a
started-but-misconfigured CCR healthchecks red and stays `NotReady`).

Docs (ADR-094 + acp-runners.md) updated to match: reconcile writes sidecar
readiness too, Start/Stop trigger it, and `mimo→mimo-code-native` corrected in
both sites.

## Prevention

- **A readiness/derived-state value has exactly one authoritative recompute
  path; every input that can change it must route through that path.** When a
  cached status (`readinessStatus`) gates behavior, grep every writer AND every
  event that changes the underlying truth (here: process start/stop). A
  "single writer" that only covers one of two coupled tables (runners but not
  the sidecars they depend on) is a half-truth that strands the dependent rows.
- **A state-changing proxy route whose success implies a derived-state change
  must persist that derived state in the same request** (the repo's two-phase
  rule, `rules/backend.md`). Returning the downstream `state` verbatim while the
  DB the UI reads stays stale is a defect, even when a later reconciler would
  eventually heal it.
- **Test the convergence, not just the proxy.** The route unit test asserted the
  state echo; it never asserted the dependent runner became launchable. The new
  integration test seeds `NotReady` sidecar + runner, reconciles with
  diagnostics `ready`, and asserts BOTH flip — it fails without the recompute
  step.
- **Lock identifier invariants that docs/asserts depend on.** Added a unit
  assertion that every `nativeDefaultRunnerByAdapter` id resolves to a real
  preset row (a drift makes the reconcile upsert a silent no-op). Docs are not
  machine-checked, so the code-internal invariant is the real guard.
- **Adversarial-review findings carry premises — verify the premise, not just
  the line.** A sibling finding claimed an "auth-first repo rule" violation; a
  grep showed parse-before-auth is the universal convention across all 15 admin
  routes, so the cited routes followed convention rather than breaking it. Check
  the asserted rule actually exists before "fixing" code to satisfy it.
