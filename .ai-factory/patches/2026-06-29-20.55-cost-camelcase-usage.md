# Root cause: per-node token spend is 0 for every node after the first

## Symptom
A multi-session flow run records tokens only for the first node. `cost.jsonl`
held exactly one record (`plan#1`); `improve`, `plan#2`, `improve#2`,
`implement` ran and emitted usage but wrote NO cost record, so the run total =
first session only and every later node shows 0 tokens.

## Root cause (confirmed by reading the supervisor cost path)
`supervisor/src/cost.ts#extractCost` is the only writer of `cost.jsonl`. It:
1. fires only on `session.line` events (raw adapter stdout), and
2. reads ONLY snake_case usage keys: `input_tokens`, `output_tokens`,
   `cache_creation_input_tokens`, `cache_read_input_tokens` (cost.ts:153-164).

The ACP adapter's end-turn `result.usage` is **camelCase** — `inputTokens`,
`outputTokens`, `cachedReadTokens`, `cachedWriteTokens`. `findKeyObject` locates
the `usage` object (the line contains `"usage"`), but none of the snake_case
keys match, so every field stays `undefined` and `extractCost` returns `null`
(the all-undefined guard, cost.ts:166-173). The camelCase end-turn usage is
silently dropped → no record for that node session.

The first node happened to surface a snake_case usage line, which is why exactly
one record survived.

## NOT the cause
`attachCost` is attached **per spawn** with that session's `nodeAttemptId`
(`spawn.ts:307`), so attribution is correct *given a captured record*. The M42
multi-session attach path is sound; no web reconcile change is needed
(`reconcileRunCostRollups` already attributes per `nodeAttemptId` from
`cost.jsonl`).

## Fix (T-D2)
Widen `extractCost` to accept the camelCase shape and normalize to the canonical
snake_case record, per field with a snake-case-wins fallback so a usage object
carrying both shapes is never double-counted:
- `input_tokens`              ← `input_tokens` ?? `inputTokens`
- `output_tokens`             ← `output_tokens` ?? `outputTokens`
- `cache_read_input_tokens`   ← `cache_read_input_tokens` ?? `cachedReadTokens`
- `cache_creation_input_tokens` ← `cache_creation_input_tokens` ?? `cachedWriteTokens`

Single-session/snake-case runs are unchanged (regression-guarded).
