# Secret-redaction boundary (RailRunnerDTO) shipped without a mapping-layer regression test

**Date:** 2026-06-18 12.07
**Files:** web/lib/acp-runners/__tests__/readiness-summary.test.ts
**Severity:** medium

## Problem

`/aif-verify` flagged the runner-readiness-rail change as a blocker: the new
`toRailRunnerDTO` redaction function (maps a `platform_acp_runners` row, which
carries `provider` secret refs like `authToken: "env:X"`, down to a safe
client-shippable `RailRunnerDTO` that exposes only `providerKind`) shipped with
**no regression test at the mapping layer**. The `summarizeAdapterReadiness`
unit suite never asserted on the new `summary.runners` field at all ‚Äî it only
extended the `runner()` factory so existing cases still compiled. The component
test "never renders provider secret references" fed an *already-safe*
`RailRunnerDTO` literal, so it passed regardless of whether `toRailRunnerDTO`
actually redacts ‚Äî vacuous as a redaction guard. The redaction code was correct,
but a future edit to `toRailRunnerDTO` (e.g. `{ ...runner }`) would have leaked
`env:NAME` refs into client HTML with zero test failing.

The plan's Task 1 explicitly scoped four ADD cases (grouping, redaction,
per-runner firstReason, empty list); the implementer marked Task 1 ‚úÖ but landed
only the factory migration. A task marked complete that is not is the recurring
failure mode `/aif-verify` exists to catch.

## Root Cause

Test coverage for a security boundary was asserted at the wrong layer. The
redaction logic lives in the server-side mapping function `toRailRunnerDTO`, but
the only "secret" assertion lived in a component test that receives the *output*
DTO type ‚Äî by construction that input can never contain secrets, so the test can
never fail on a redaction regression. The mapping function (the actual boundary)
had no test exercising a secret-bearing input.

## Solution

Added a `describe("summarizeAdapterReadiness ‚Äî safe runner DTO projection")`
block to the unit suite with five cases driving the REAL mapping
(`summarizeAdapterReadiness` ‚Üí `toRailRunnerDTO`), not a literal:

- **Redaction (the boundary):** feed runners whose `provider` carries every
  secret-bearing variant ‚Äî `anthropic_compatible` (`authToken`/`baseUrl`),
  `openai_compatible` (`apiKey`/`baseUrl`/`wireApi`), `google_vertex`
  (`projectId`/`location`/`apiKey`) ‚Äî then assert each resulting DTO's key set
  equals exactly `{id, capabilityAgent, model, providerKind, enabled,
  readinessStatus, firstReason}`, has none of the secret keys, and its JSON has
  no `env:` substring.
- Per-adapter grouping (no cross-adapter bleed), per-runner `firstReason` picks
  the first non-empty reason, empty adapter ‚Üí `runners: []`, and full
  many-runner mapping.

Implementation untouched (it was already correct). Verified bare: scoped suite
15/15, full unit 4020/4020 (+5), `tsc --noEmit` clean, scoped eslint clean.

## Prevention

- A redaction / safe-projection boundary MUST have a test at the **mapping
  function**, fed an input that actually contains the secrets, asserting the
  output's **exact** key set (not just "doesn't contain X"). A test fed the
  already-safe output type is vacuous ‚Äî it cannot regress.
- When a safe DTO is a discriminated union over provider/source kinds, the
  redaction test must cover EVERY secret-bearing variant in one pass (sweep all
  siblings of the defect class), not just the first kind.
- During verify, treat a `‚úÖ`-marked task whose plan listed explicit ADD test
  cases as PARTIAL until each listed case is found in the diff ‚Äî re-derive the
  test list from the plan, not the checkbox.
