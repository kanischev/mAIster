# Project-membership routes: auth-ordering existence oracle + silent stale-write races

**Date:** 2026-06-07 18:35
**Files:** web/app/api/projects/[slug]/members/route.ts, web/app/api/projects/[slug]/members/[memberId]/route.ts, web/app/api/projects/[slug]/members/candidates/route.ts, web/lib/project-members.ts, web/components/project/project-members-panel.tsx, web/messages/{en,ru}.json, docs/api/web.openapi.yaml, docs/system-analytics/project-membership.md, docs/error-taxonomy.md
**Severity:** high

## Problem

A Codex adversarial review of the new platform-user / project-member admin
surface found two high-severity defects in the freshly added membership API:

1. **Existence oracle.** Every new membership route
   (`GET`/`POST /members`, `GET /members/candidates`,
   `PATCH`/`DELETE /members/{memberId}`) resolved `getProjectBySlug(slug)`
   *before* any auth gate. An unauthenticated caller got `PRECONDITION`/409
   for a missing/archived slug but an auth error (`UNAUTHORIZED`/403) for an
   existing one — turning the route into a project-slug existence oracle.

2. **Silent stale writes.** `changeProjectMemberRole` / `removeProjectMember`
   did `SELECT … FOR UPDATE` (existence only) then `UPDATE`/`DELETE`
   predicated solely on `id` + `project_id`. Two admins on the same stale
   roster could both "succeed": the second role write won silently, and a
   stale delete could remove a row another admin had just re-roled. No
   `CONFLICT` was ever raised, contradicting the domain's own concurrency
   contract.

## Root Cause

1. The membership routes were written resolve-first instead of mirroring the
   established project-route convention (`tasks/route.ts`:
   `requireActiveSession()` → resolve slug → `requireProjectAction`). The
   author re-derived the handler shape from scratch rather than copying a
   sibling route, so the auth-first ordering was lost.

2. The locked plan (§2.3) specified an **observed-role CAS**
   (`UPDATE … WHERE … AND role=<observed> RETURNING`, 0 rows → `CONFLICT`),
   but the implementation simplified it to a `FOR UPDATE` existence check plus
   an unconditional `UPDATE`. The existence check caught a fully-removed row
   but not a stale *role*, because no `expectedRole` was ever carried from the
   client. Optimistic concurrency degraded to last-writer-wins, and the
   integration tests asserted only the existence-race (raced remove), so the
   gap stayed green.

## Solution

1. **Auth-first ordering.** Added `await requireActiveSession()` as the first
   statement in every membership handler, before `getProjectBySlug`. An
   unauthenticated/inactive/must-change caller now gets the auth error
   regardless of slug existence; the slug is never resolved. Route unit tests
   assert `getProjectBySlug` is not called on the unauthenticated path.

2. **Observed-role CAS.** Added a required `expectedRole` carried from the
   roster: PATCH body field, DELETE `?expectedRole=` query param. The service
   collapses to a single atomic
   `UPDATE/DELETE … WHERE id AND project_id AND role=expectedRole RETURNING`;
   0 rows → `MaisterError("CONFLICT", …)`. The transaction + separate
   `FOR UPDATE` select were removed (a single CAS statement is atomic and is
   the correct optimistic-concurrency primitive). The panel sends the role it
   rendered; CONFLICT now surfaces a dedicated `staleConflict` message
   ("changed by someone else — refresh"). OpenAPI, error-taxonomy, and the
   project-membership analytics doc updated to match; added role-vs-role and
   role-vs-delete race integration tests.

## Prevention

- **Copy a sibling route for cross-cutting order.** New `app/api/**/route.ts`
  handlers must mirror an existing same-shape route for the auth-vs-resolve
  ordering (auth-first), not re-derive it. Auth gate runs before any
  server-state lookup that can leak existence.
- **A CAS needs the observed value, not just a lock.** `SELECT … FOR UPDATE`
  + unconditional write is NOT optimistic concurrency — it is last-writer-wins
  with a lock. If the contract says "concurrent stale mutation → CONFLICT",
  the write predicate MUST include the client-observed value
  (`… AND role=expectedRole RETURNING`, 0 rows → CONFLICT).
- **Test the race the contract promises, not a proxy.** A "raced remove"
  (existence) test does not cover "raced role-change" (stale value). For every
  CONFLICT class in the spec, write a test that drives the *specific* stale
  precondition.
- **Review check:** any route handler that returns different status codes for
  "resource missing" vs "not authorized" to an *unauthenticated* caller is an
  existence oracle — gate auth first.
