# Observatory artifact filters did not narrow metric totals

**Date:** 2026-06-05 13:57
**Files:** web/lib/queries/observatory.ts, web/lib/queries/__tests__/observatory.integration.test.ts, .ai-factory/patches/2026-06-05-11.29-m23-observatory-review-fixes.md
**Severity:** high

## Problem

The Observatory dashboard exposed `artifactKind` and `artifactDefId` as GET
filters, but the query layer only applied them to the artifact bucket query.
Correction totals, Autonomy Score, HITL waits, gates, and node heatmap rows
still used every eligible ledger-backed run.

## Root Cause

The read model computed the eligible run set before reading artifact rows and
never fed artifact matches back into the shared run population. This split the
dashboard into two scopes: filtered artifact buckets and unfiltered metrics.

## Solution

Artifact rows are now read immediately after the ledger-backed run population
is known. When an artifact filter is present, matching artifact rows derive the
effective run set used by runs, attempts, gates, HITL waits, and artifact
summaries. The query shape remains batched and read-only.

The stale M23 patch note was corrected so it no longer claims non-human-only
retry semantics or ADR-051 changes.

## Prevention

- Treat user-visible filters as scope filters until explicitly labeled as a
  per-section filter.
- Add integration tests that assert totals and autonomy `runIds`, not only the
  filtered child bucket list.
- Review patch learning artifacts for semantic drift before committing them.
