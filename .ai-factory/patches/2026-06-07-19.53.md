# Flow graph count labels crash server translation

**Date:** 2026-06-07 19:53
**Files:** web/app/(app)/runs/[runId]/page.tsx, web/components/board/flow-graph-view.tsx, web/messages/en.json, web/messages/ru.json, docs/api/web.openapi.yaml, .ai-factory/plans/codex-flow-graph-visual-drawings-plan.md
**Severity:** high

## Problem

Flow graph count labels used `{count}` in message catalogs, while run detail
loaded them via `tWorkbench("...")` without values to pass as client-side
templates. `intl-messageformat` treats `{count}` as an ICU variable and throws
when no `count` is provided, so flow run detail could fail before the graph
rendered. The review also found OpenAPI required-field drift and unresolved plan
decisions.

## Root Cause

Client-side template strings were stored in the same syntax as server-formatted
ICU messages. The server translation layer eagerly formats ICU variables; it
cannot safely pass `{count}` through as a raw template. The OpenAPI/schema drift
came from documenting additive fields without updating required arrays, and the
plan retained earlier questions after code/tests had resolved them.

## Solution

Changed graph count templates to `$count`, updated `formatCount()` and component
tests, marked emitted graph DTO fields as required in OpenAPI, and replaced
unresolved plan questions with explicit decisions.

## Prevention

Review translated strings passed as client-side templates for ICU placeholder
syntax. If the server loads a string with `t(key)` and no values, avoid `{name}`
placeholders unless the intent is immediate ICU formatting. API response fields
emitted unconditionally by TypeScript DTOs should be reflected in OpenAPI
`required` arrays. SDD plans should turn resolved open questions into decisions
before merge.
