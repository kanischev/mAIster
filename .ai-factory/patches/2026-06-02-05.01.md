# Non-atomic artifact record+supersede leaves dual-current rows on crash

**Date:** 2026-06-02 05:01
**Files:** web/lib/flows/graph/artifact-store.ts, web/lib/flows/graph/runner-graph.ts
**Severity:** critical

## Problem

The graph runner recorded each declared output as two independently
auto-committed statements — `recordArtifact({...validity:"current"})` then a
separate `supersedePrior(...)` — on the bare `db` handle (the file's only
`db.transaction` blocks wrapped the resume-claim CAS, not this loop). A process
death between the two writes on a def RE-RUN (rework / takeover re-entry, where
attempt N+1's row id differs from attempt N's) left BOTH rows at
`validity='current'`. `getCurrentArtifact` then read `.limit(1)` with no
`ORDER BY`, returning an arbitrary one — so `assertEvidenceReady` and the
`artifact_required` gate could approve review/merge on a superseded attempt's
stale payload. No unique constraint and no recovery sweep existed.

## Root Cause

A logical "replace the current row" operation was implemented as two physical
writes with no transaction boundary. The one-current-per-def invariant the
entire evidence model rests on was enforced only by sequencing, which a crash
breaks. The single-row read compounded it by being non-deterministic when the
invariant was (transiently) violated. A partial unique index was NOT a viable
backstop here: the insert-then-supersede ordering + the `superseded_by_id`
self-FK make a non-deferrable partial unique index fire at insert time, and the
default-artifact path legitimately keeps multiple current rows per def.

## Solution

Added `recordCurrentArtifact(args, db)` that wraps the insert + `supersedePrior`
in ONE `db.transaction`, and replaced all four runner call-site pairs with it.
Made `getCurrentArtifact` deterministic latest-wins
(`ORDER BY created_at DESC, id DESC`) so even a residual multi-current case (the
default-artifact path) returns the newest. Regression test asserts that
re-recording a def across two attempts leaves exactly one current row with the
prior superseded and `superseded_by_id` pointing at the new row.

## Prevention

- When a state transition writes a "new current" row AND retires the prior
  row(s), the two writes MUST share one transaction — review any
  `record*()` + `supersede*()`/`markStale()` pair for a missing transaction
  boundary and a reachable process-death partial state.
- A single-row "get the current X" read MUST have an explicit `ORDER BY`; a
  `.limit(1)` with no ordering is a latent nondeterminism bug the moment its
  uniqueness invariant is violated.
- Before adding a partial unique index as a safety net, check it against the
  insert/supersede ORDERING and any self-referential FK — it can fire at insert
  time and reject the legitimate flow.
