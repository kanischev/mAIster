# Auth bootstrap + RBAC trust-boundary hardening (adversarial findings 1 & 2)

**Date:** 2026-05-29 10:34
**Files:** web/lib/db/schema.ts, web/lib/db/migrations/0005_smart_bishop.sql, web/lib/db/seed.ts, web/app/(auth)/actions.ts, web/auth.config.ts, web/auth.ts, web/lib/authz.ts, web/types/next-auth.d.ts, web/app/(app)/layout.tsx, web/app/change-password/{page,actions}.tsx, web/components/auth/change-password-form.tsx, web/messages/{en,ru}.json, docs/*
**Severity:** critical

## Problem

Two findings from the M9 adversarial review:

1. **(critical) Concurrent first-user registration could mint multiple global
   admins.** The public `register()` action decided admin bootstrap with a
   non-transactional `select … limit 1` then inserted the user. Two emails
   registering against an empty DB could both observe zero users and both
   insert `role = "admin"`, granting full global authority before any operator
   could lock the instance down.
2. **(high) Global role revocation was ignored until JWT expiry.** The `jwt`
   callback copied `user.role` into the token only at sign-in; authz trusted
   that cached token role. With a 30-day JWT, a demoted admin kept `admin`
   authority (incl. the global-admin project bypass) for up to 30 days.

## Root Cause

1. Read-then-write race: "is this the first user?" was a TOCTOU check with no
   serialization, and `register()` was the bootstrap path for admin — a
   privileged decision made on a public, unauthenticated endpoint.
2. Authorization treated a **cached, long-lived credential** (the JWT role
   claim) as authoritative. A JWT is a point-in-time snapshot; using it for
   authZ means privilege changes only take effect at token expiry — a
   trust-boundary inversion.

## Solution

Finding 1 — remove the privileged decision from the public path:
- Added `users.must_change_password` (boolean, default false). Migration
  `0005` seeds **one** bootstrap admin (`admin@maister.local`, bcrypt of the
  documented default, `must_change_password = true`) via idempotent
  `INSERT … ON CONFLICT (email) DO NOTHING`, so every deploy has exactly one
  admin after `db:migrate` — no race window.
- `register()` now always creates `role = "member"` (no first-user promotion).
- The `(app)` layout redirects users with `must_change_password = true` to a
  new `/change-password` flow (page + `changePassword` server action that
  sets the new hash and clears the flag), so the well-known default password
  cannot be used past first login.

Finding 2 — make the server-side authz path DB-authoritative:
- `lib/authz.ts:getSessionUser()` now re-reads the live `users` row (role,
  `mustChangePassword`, existence) by the JWT-supplied `id` on every call;
  a deleted user resolves to `null` (denied). The cached `session.user.role`
  is never used for an authorization decision.
- The Node `auth.ts` `jwt` callback also re-reads role/mustChange on refresh
  and returns `null` (forces sign-out) when the user row is gone — token-level
  invalidation. The edge `auth.config.ts` jwt stays DB-free (middleware safe).

## Prevention

- Never make a privilege-granting decision on an unauthenticated endpoint, and
  never gate "first/only" semantics on a non-serialized read-then-write. Seed
  privileged principals deterministically (migration/seed), not as a side
  effect of public sign-up.
- Authorization MUST read the authoritative store at decision time. Treat JWTs
  (and any long-lived cached claim) as identity hints, not authZ truth. Review
  checklist: "does any `require*`/guard branch on a token/session field that
  the DB owns?" — if yes, re-read the DB.
- Tests added: bootstrap admin is the single admin with mustChange; public
  registration is always member; authz uses the live DB role (stale session
  role ignored); deleted user → null. (Finding 3 — partial-project-registration
  compensation — was explicitly deferred by the user; not addressed here.)
