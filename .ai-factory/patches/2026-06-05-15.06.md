# Workbench flow-run diff used 3-dot merge-base range, under-reporting a history-rewritten branch

**Date:** 2026-06-05 15:06
**Files:** web/lib/worktree.ts, web/lib/__tests__/worktree-diff-namestatus.test.ts
**Severity:** medium

## Problem

The M22 workbench Diff tab (`GET /api/runs/{runId}/diff`) computed the run's
delta with a **3-dot** git range — `diffRunWorkspace` ran
`git diff ${baseCommit}...${branch}` and `diffNameStatus` ran
`git diff --name-status ${base}...${branch}`. A 3-dot range diffs from
`merge-base(base, branch)`, not from `base` itself. If a run branch is rewritten
(rebase/reset) so its stored `baseCommit` is no longer an ancestor of the branch
tip, the 3-dot range silently OMITS committed branch changes whose content
happens to match that older merge-base — so the Diff tab and changed-files
summary show LESS than what is actually on the branch. Surfaced by a Codex
adversarial review.

Two existing contracts already said it should be 2-dot:
- `docs/system-analytics/workbench.md` documents the diff as
  `diffRunWorkspace base..branch` (2-dot).
- `docs/api/web.openapi.yaml` describes it as "the diff between the run
  workspace branch and its **base commit**" — which is the 2-dot semantic; the
  3-dot code diffed from the merge-base, not the base commit.
- The binding review/promotion/artifact surfaces (`diffRange`, used by the M18
  review panel, takeover-return, and artifact payload) were already **2-dot**.

So the code had drifted from its own documented contract and from every sibling
diff surface.

## Root Cause

`diffRunWorkspace` predates M22 (it served the scratch-run diff) and used a
3-dot range. When M22 reused it for flow runs, `diffNameStatus` was written to
**deliberately match** that 3-dot range ("3-dot to match diffRunWorkspace's
symmetric-difference range so the file list lines up with the rendered diff").
The author optimized for internal consistency between the two helpers and never
re-questioned whether 3-dot itself was the right semantic for a fixed-SHA base.
3-dot is the right tool when comparing two *moving* branches (PR base vs head,
where you want to exclude what the base added); it is the wrong tool when `base`
is a fixed fork commit and you want the literal "how does the branch tree differ
from the stored base" delta. In the common case (branch forked from base, base
is an ancestor) 3-dot == 2-dot, so the divergence never showed up in normal
testing or normal runs — it only manifests on history-rewritten branches.

## Solution

Changed both `diffRunWorkspace` and `diffNameStatus` to the **2-dot**
`base..branch` range — the literal stored-base→branch tree delta — matching the
documented contract and the M18 review-panel `diffRange`. Added a WHY comment on
the `diffRunWorkspace` range explaining the 2-dot vs 3-dot choice so it is not
"corrected" back. Updated the `diffNameStatus` comment + error message
accordingly. (Both helpers are used only by the diff route, so blast radius is
contained; scratch and flow both now use 2-dot consistently.)

Added a regression test that builds a real temp git repo where the stored base
is NOT an ancestor of the branch (they diverge at the root and the branch resets
a file back to the root content). Verified by flip-testing: under the old 3-dot
range the new tests FAIL (the raw diff omits `shared.txt` entirely); under 2-dot
they pass and report the full `feature.txt` (A) + `shared.txt` (M) delta.

No `[FIX]` console logging was added: the helpers already log via pino, the
project bans `console.*` (eslint `no-console`), and the change is a pure range
correction with a covering test — logging would be noise in a hot git helper.

## Prevention

- **A fixed-SHA base means 2-dot.** When diffing against a stored fork commit
  (an immutable SHA), use `base..branch` (2-dot — literal tree delta). Reserve
  3-dot (`base...branch`) for comparing two moving branch *tips* where excluding
  the base's own advancement is intended. Whenever you see `...` in a `git diff`,
  ask "is the left side a fixed point or a moving branch?".
- **Cross-check sibling surfaces for the same concept.** This project has
  multiple "base→branch diff" call sites (`diffRange`, `diffRunWorkspace`,
  `diffNameStatus`). They must share one semantic; a new one that diverges is a
  review finding. Grep the family before shipping a new diff surface.
- **"Match the other helper" is not a semantic justification.** The 3-dot choice
  was justified only as "match diffRunWorkspace" — a consistency argument, not a
  correctness one. When a comment justifies a choice by reference to a sibling
  rather than by intent, re-derive the intent.
- **Doc-vs-code drift can run code→behind→doc.** Here the docs were RIGHT
  (`base..branch`) and the code was wrong. The "code wins" rule assumes code is
  the executed truth, but when code violates an explicit documented range/format,
  treat it as a code bug, not a doc-update.
- **Edge-case-only divergence needs a constructed test.** A bug that is invisible
  in the common case (3-dot == 2-dot when base is an ancestor) only regresses
  safely with a test that forces the divergent topology. Flip-test the fix
  (revert, watch it fail) to prove the regression guard is real, not tautological.
