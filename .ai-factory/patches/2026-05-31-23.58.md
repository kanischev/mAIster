**Severity:** high

## Problem

External review found that scratch-run code and documentation still had
merge-blocking contract drift and lifecycle holes. Project registration tests
mocked only `loadProjectConfig` after registration started loading platform MCP
capabilities, the scratch detail API leaked internal supervisor/materialization
handles, scratch capacity was checked outside the inserting transaction, and
stop/discard had weak idempotency around missing supervisor sessions. The docs
also overstated native MCP/skill activation and contradicted themselves on
implemented versus designed surfaces.

## Root Cause

The feature added capability seeding and scratch lifecycle APIs across several
hot paths without keeping the integration mocks, public API projections, and
state-machine docs aligned. The original launch flow had a preflight capacity
check, but the durable `Running` insert happened later in a separate
transaction. Discard also mixed supervisor deletion, database state changes, and
filesystem removal in an order that made retries and partial failures harder to
reason about.

## Solution

Added the missing config mock and ran the full web integration suite. Resolved
the platform MCP registry path through accessible candidates instead of a single
CWD-relative parent path. Moved scratch capacity enforcement into the same
transaction that inserts the running scratch run, while preserving the early
preflight cleanup path. Projected scratch detail responses to omit internal
session IDs, adapter launch argv/env, materialized paths, and worktree paths.
Made stop/discard treat supervisor 404-style responses as idempotent, kept
terminal runs terminal, and changed discard to durably mark the run abandoned
before best-effort worktree removal. Updated docs and OpenAPI to describe the
implemented recovery route, scratch timeout edge, sanitized detail payload, and
instruction-only capability handoff truthfully.

## Prevention

- Run `pnpm --filter maister-web test:integration` before merge whenever
  project registration, config loading, or capability seeding changes.
- Keep public route responses as explicit projections instead of serializing DB
  rows with server-only operational fields.
- Document capability activation as snapshot and instruction handoff until the
  supervisor adapter natively loads selected MCPs and skills.
- Put capacity checks that protect durable state in the same transaction as the
  protected insert/update.
