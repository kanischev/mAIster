# Comprehensive MCP facade ↔ ext-route contract sweep + an OpenAPI-anchored drift guard so the mirror can't silently rot again

**Date:** 2026-07-02 14.14
**Files:** mcp/src/tools.ts, mcp/src/__tests__/tools.test.ts, mcp/src/__tests__/tool-contract.test.ts (new), mcp/package.json, docs/api/external/operations.openapi.yaml
**Severity:** high

## Problem

The earlier same-day patch (`2026-07-02-11.12`) fixed THREE specific MCP facade
drifts (triage_set under-forwarding, executorOverrideId ghosts, tasks:update
scope) and its Prevention note explicitly deferred the real fix: "adding a
contract test that diffs tool properties against route zod schemas." Nothing
guaranteed the OTHER 25 tools were in sync. A full sweep across all three
surfaces (MCP `TOOL_SPECS`/`dispatchTool` ↔ ext route zod ↔ the external
OpenAPI) found **seven** more live drifts, four of them agent-visible
capability gaps:

1. **run_launch** advertised only `taskId` + `executorOverrideId`. The route
   (`ExtLaunchRunBody`) also accepts `runnerId` (the canonical field;
   `executorOverrideId` is its deprecated alias), `baseBranch`, and
   `targetBranch` (M18 branch targeting). An agent launching through the facade
   could NOT pick a runner by the canonical name or set base/target branch —
   the fields were silently unreachable.
2. **gate_report.status** was a free-form `string`; the route + OpenAPI restrict
   it to `enum [passed, failed]`. The closed set was undiscoverable, so an
   agent guessing "success"/"ok" got a 422 with no schema hint.
3. **triage_set.priority / .confidence** were non-nullable in the tool; the
   route (`.nullable()`) and OpenAPI (`nullable: true`, null clears) let `null`
   reset them. The "clear priority/confidence" capability was unreachable via
   the facade.
4. **OpenAPI ExtRelationOpBody.kind** omitted `duplicate_of` (the ADR-112 dedup
   annotation the route zod + DB CHECK + tool all support); **ExtRelationView**
   omitted both `duplicate_of` and the system-created `requires` that
   `relation_list` returns unfiltered.
5. **relation toNumber** was typed `number` in the tool vs `integer` (min 1) in
   the route + OpenAPI — a float would 422.
6. **hitl_inbox** — the OpenAPI declared `limit`/`offset` query params (+ an
   identifier-table row for each) that the route never reads
   (`getCrossProjectHitlInbox` takes no paging) and the tool never exposed; the
   route's `count` response field was undocumented.
7. **OpenAPI ExtRunPlanTask** omitted the optional `title` the route zod + the
   `run_plan` tool both carry (found only by the new contract test — it was
   invisible to eyeballing).

## Root Cause

`TOOL_SPECS` + the `dispatchTool` per-tool destructuring are a
HAND-MAINTAINED mirror of the ext route zod schemas with no shared source and,
until now, no contract test. Every feature wave (M18 branch targeting, M37
delegation, ADR-112 dedup, ADR-121 priority) that changed a route body left the
mirror — and often the OpenAPI — untouched, and `dispatchTool` drops unknown
args silently, so the drift never errors on the agent side. The OpenAPI drifted
the same way in the opposite direction (fields the code has but the spec lost,
and phantom params the spec has but the code never implemented).

## Solution

- **run_launch**: added `runnerId` (canonical), `baseBranch`, `targetBranch` to
  the inputSchema + `dispatchTool` forwarding; kept `executorOverrideId` as the
  documented deprecated alias.
- **gate_report.status**: added `enum: ["passed", "failed"]`.
- **triage_set.priority/.confidence**: made both nullable in the tool
  (`type: ["string"|"number", "null"]`, priority enum incl `null`).
  `dispatchTool` already forwards `null` (it drops only `undefined`), so the
  clear path now round-trips end to end.
- **relation toNumber**: `type: "integer", minimum: 1`.
- **OpenAPI**: added `duplicate_of` to `ExtRelationOpBody.kind` and
  `requires`+`duplicate_of` to `ExtRelationView.kind`; added `title` to
  `ExtRunPlanTask`; removed the phantom `limit`/`offset` params (+ table rows)
  from `hitl_inbox` and documented its `count` response field (code-wins, docs
  R3).
- **New contract test** (`mcp/src/__tests__/tool-contract.test.ts`, 83 cases):
  parses the canonical external OpenAPI with `yaml` (added as an mcp devDep,
  pinned to web's `^2.9.0`) and, for every one of the 26 tools, asserts the tool
  inputSchema's property-name set, required set, per-field base type, and enum
  values (incl. `null`) equal the operation's path params ∪ query params ∪
  request-body schema — with explicit nested checks for `run_delegate.target`
  and `run_plan.tasks[]`, and a dedicated null-clear assertion for
  triage priority/confidence. Written RED-first: it failed on all seven drifts
  (incl. the ExtRunPlanTask `title` gap no human spotted), then went green.
- **Dispatch tests**: run_launch canonical-surface + deprecated-alias
  forwarding; triage null-clear forwarding.
- Verified: mcp unit 142/142 green, `tsc --noEmit` 0, scoped eslint 0, redocly
  lint of the external spec **valid with 0 warnings** (the hitl_inbox example
  gained `count` to satisfy the now-required field). Follow-up hygiene pass
  cleared the 6 pre-existing redocly warnings while here: added the MIT
  `info.license`; templated the server as `{origin}` (localhost moved into a
  variable default, off the flagged `url`); gave the hitl_inbox example
  valid-uuid ids; removed the dead unused `TokenScope` schema (unreferenced;
  its `*` semantics already live in `info` and per-op prose, enforcement in
  `token-scopes.ts`).

No `[FIX]` runtime logging added: the changes are declarative contract
alignments (a JSON schema, a spec, a test) locked by tests, per the repo's
no-console/pino-only convention.

## Prevention

- The `tool-contract.test.ts` guard now fails CI on ANY future drift in either
  direction — a tool that omits/ghosts a field, a wrong enum/type, or a spec
  that loses a field the code has. It supersedes the earlier patch's "consider
  a contract test" note; that roadmap item is now DONE.
- Anchor is the external OpenAPI (docs R3: the spec is the source of truth for
  the surface, and the routes are themselves linted/contract-tested against it),
  so one green test transitively ties tool ↔ spec ↔ route.
- When adding a new ext tool, add its `{method, path}` to the test's `TOOL_OP`
  map (the "maps every registered tool" case fails until you do), then let the
  per-field assertions drive the inputSchema.
- Out of scope (noted, not silently skipped): a full response-DTO ↔ OpenAPI
  audit. This sweep locks the INPUT contract (what the facade sends); response
  shapes beyond the two touched here (`hitl_inbox.count`, `ExtRelationView`)
  were not swept.
