# /aif-review follow-up: onExhaustion loop-back compile guard + ADR-118 comment residue

**Date:** 2026-06-30 00:47
**Files:** web/lib/flows/graph/compile.ts, web/lib/flows/graph/__tests__/compile-rework-reset.test.ts, web/lib/db/schema.ts, web/lib/flows/graph/__tests__/runner-graph-rework-reset.integration.test.ts
**Severity:** low

## Problem

A `/aif-review` of `feature/flow-rework-reset-loop` (ADR-118) surfaced three
non-blocking issues:

1. `verifyReworkReset` validated `rework.onExhaustion ∈ transitions` but did NOT
   reject an `onExhaustion` whose target is one of the node's own
   `rework.allowedTargets` — i.e. routing exhaustion back INTO the loop it just
   exhausted (without a reset). It fails closed at runtime (the loop-top backstop
   throws CONFIG on re-entry), but as a confusing runtime failure rather than an
   authoring-time refusal.
2. `node_attempts.rework_baseline` comment said "byte-identical to pre-0085" —
   stale prose left over from the migration `0085 → 0086` renumber (the adjacent
   `migration 0086` reference on the line above WAS swept; the `pre-0085` phrase
   was not).
3. An AC-8 integration-test comment said "(attempt 3 …)" where the loop-top
   backstop actually fires on the PRE-append count (2 existing → effective 2 >
   maxLoops 1) before the would-be 3rd visit is appended.

## Root Cause

1. The validation surface was incomplete: the feature's design intent is that
   `onExhaustion` routes OUT of the loop (typically to a human node that may
   re-baseline via `resetTargets`). Routing back to an `allowedTarget` is never
   legitimate, but the compile pass only checked transition-key membership, not
   the loop-back shape — so a statically-detectable misconfiguration was deferred
   to a runtime CONFIG.
2. The 0085→0086 renumber sweep matched `migration 0085` / `0085_name` forms but
   missed a `pre-0085` prose phrasing on the same comment block — exactly the
   "grep-to-zero must cover ALL forms" hazard the project's renumber rule warns
   about.
3. Test-comment drift: the arithmetic narration didn't match where the guard
   actually evaluates (pre-append count vs would-be attempt number).

## Solution

1. Added a compile-time CONFIG refusal in `verifyReworkReset`: after the
   transition-key check, reject when `transitions[onExhaustion] ∈
   rework.allowedTargets`, with a message pointing the author to route OUT of the
   loop. Added a `compile-rework-reset.test.ts` case (`onExhaustion: "redo"` →
   `work` ∈ allowedTargets → CONFIG); the existing accept case (onExhaustion →
   `done`, not an allowedTarget) still passes, proving the intended pattern is
   unaffected. The runtime backstop stays as defense in depth.
2. `pre-0085` → `pre-ADR-118` (matching the `rework-baseline.ts` helper wording,
   which is the migration-number-agnostic phrasing that won't drift on the next
   renumber).
3. Reworded the AC-8 comment to describe the pre-append count (2 existing →
   effective 2 > maxLoops 1, refused before the 3rd visit is appended).

Findings 4 (DRY: onExhaustion forward-advance block vs the normal forward-advance
block) and 5 (a unit test for the runtime onExhaustion-∉-transitions guard) were
evaluated and deliberately NOT actioned: the `markNodeSucceeded + if(materialized)
cleanupNodeMaterialization` pair is an established 3-site idiom in `runner-graph.ts`
and a clean extraction would thread ~8 loop-local closure vars into a helper for
marginal benefit (against surgical-changes); and with the new compile guard the
runtime guard is unreachable defense-in-depth, testable only by fabricating a
compile-invalid FlowGraph (the hacky internals bypass to avoid).

## Prevention

- A validation feature that fails closed at RUNTIME for a misconfiguration should
  ALSO reject it at COMPILE/load time when the condition is statically decidable
  from the manifest — keep the validation surface consistent (compile rejects;
  runtime re-asserts as defense in depth), don't split it across phases.
- When renumbering a shared migration/ADR namespace, grep-to-zero must include
  PROSE forms — `pre-NNNN`, `since NNNN`, `as of NNNN` — not only `migration
  NNNN` / `NNNN_name` / bare `` `NNNN` ``. Prefer migration-number-agnostic
  phrasing (`pre-ADR-NNN`) in long-lived comments so they survive the next
  renumber.
- A test comment that narrates arithmetic must name the exact value the guard
  evaluates (pre-append count vs would-be attempt number); a drifted narration
  misleads the next implementer about where a bound fires.
