# Hook-trip resume/escalation strand fixes + path_guard bash limitation doc

**Date:** 2026-06-23 23:16
**Files:** web/lib/services/hitl.ts, web/lib/flows/runner-agent.ts,
docs/system-analytics/guardrail-hooks.md,
.ai-factory/specs/feature-guardrail-hook-engine.md, +2 test files
**Severity:** high

## Problem

A Codex adversarial review of the G4 guardrail-hook engine (Phase 3/4) flagged
three durability/safety gaps. Two were real strand-on-crash bugs; the third was
a real gap but a design limitation consistent with the frozen spec.

1. **hook_trip resume not rescheduled on retry.** The `respondToHookTripHitl`
   `already-delivered` branch returned 200 without re-queuing the resume. A
   crash between the `respondedAt` commit and the post-commit
   `scheduleResume`/`startAgentSession` strands a flow run in `NeedsInput` with
   **no reconcile backstop** (the crash-reconcile sweep only catches `Running`
   rows with no live session). The same defect existed verbatim in two
   pre-existing siblings: `handleInfraRecoveryResponse` and
   `handleBudgetBreachResponse`.
2. **Swallowed escalation → false checkpoint.** In `runner-agent.ts`, the
   `session.hook_trip` consumer set `hookEscalated = true` then `.then()`-reset
   it only on a *resolved* `{escalated:false}`. If `escalateHookTrip` **rejects**
   (its tx throws after the pre-tx checkpoint already SIGTERM'd the agent),
   `Promise.allSettled(pendingWork)` swallows the rejection, `hookEscalated`
   stays `true`, and the runner returns `STEP_CHECKPOINTED` — reporting a clean
   pause while the run is stranded `Running` with a dead session and no HITL.
3. **bash bypasses `path_guard` under auto-approve.** `path_guard` is kind-based
   (`WRITE_KINDS` excludes `execute`), so a single out-of-lane bash write under
   `unattended` (`permissions=auto_approve`) is auto-approved. Confirmed at both
   layers (supervisor + native claude `Edit|Write|MultiEdit|NotebookEdit`).

## Root Cause

1/2 — **the established self-heal pattern was not replicated in new code.** The
`permission` and `form/human` response paths already self-heal on
`already-delivered` (re-queue `scheduleResume` when the run is still
`NeedsInput`; see hitl.ts `scheduleResume` doc comment). The newer `hook_trip`
escalation path (Phase 3) and the inherited `infra_recovery` / `budget_breach`
paths each re-implemented the response handler without carrying over the
self-heal — and the runner's escalation-claim un-set only the *resolved*
non-success case, leaving the *rejection* case (checkpoint already fired, tx
failed) reporting a false clean checkpoint.

3 — **aspirational framing vs kind-based mechanism.** The doc bills the engine
as the "structural floor … writes outside its lane," but `path_guard`'s frozen
acceptance is scoped to *write-class* tool calls and `execute` is deliberately
excluded (bash can be read-only; neither layer statically proves a command
stays in-lane). The gap is real but matches the frozen spec — it is a
documented limitation, not a code defect. (`no_progress` still bounds bash
*loops*; a single destructive command is not blocked.)

## Solution

- **Finding #1 (whole class).** In all three response handlers
  (`respondToHookTripHitl`, `handleInfraRecoveryResponse`,
  `handleBudgetBreachResponse`), the `already-delivered` tx branch now reads the
  **fresh** run status under the hitl-row lock, and the post-tx branch calls
  `scheduleResume(runId)` when that status is `NeedsInput`. Idempotent —
  `runFlow`'s resume gate (`runner-graph.ts:1534/1618`) is `NeedsInput`-only and
  no-ops if the run advanced; agent resume CAS'd to `Running` in-tx so it is
  never `NeedsInput` here (a stranded `Running` agent run is recovered by
  crash-reconcile). Predicate `=== "NeedsInput"` matches the form/human
  precedent and `runFlow`'s own gate exactly.
- **Finding #2.** The runner's escalation `.then` gained an `onRejected` handler:
  it un-claims `hookEscalated` and sets a new `hookEscalateFailed` signal
  (exposed on the consumer). Both `runNewSession` and `runSlashInExisting` check
  it FIRST and return `errorCode: "CRASH"` (with the retained `acpSessionId`) so
  `runFlow` marks the run `Crashed` and recover can `session/resume` — instead of
  the false `STEP_CHECKPOINTED`. New flag checked before the `hookEscalated` and
  `checkpointed` branches so it takes precedence without disturbing their
  existing order.
- **Finding #3 (document, no behavior change).** Added a "Known limitation"
  note to the SDD spec (§1.1) and an Edge-case bullet to `guardrail-hooks.md`
  stating `execute`/bash is out of `path_guard`'s kind-based scope, the single
  out-of-lane bash write is auto-approved under `unattended`, `no_progress`
  bounds loops, and true confinement is an OS-sandbox concern (Phase 2). The
  Expectations bullet was already correctly scoped to "write-class".

Tests: `runner-agent-hooks.test.ts` gains an escalate-rejection case (expects
`CRASH`, no `markCheckpointedFromExit`); `hitl-hook-trip.integration.test.ts`
gains a flow crash-window self-heal case (retry re-queues `runFlow`) and a
negative case (advanced run does NOT re-queue). The pre-existing `budget_breach`
/ `infra_recovery` / `hitl` suites pass unchanged — no overlapping duplicate
tests added (the `hook_trip` case is the representative of the shared pattern).

## Prevention

- **When a response/handler family shares a recovery invariant, grep the whole
  family before shipping a new member.** The self-heal-on-`already-delivered`
  rule lived in `permission` + `form/human` but was missed by `hook_trip`,
  `infra_recovery`, and `budget_breach`. Fix-the-class grep recorded:
  `rg 'transition === "already-delivered"' web/lib/services/hitl.ts` → 5 response
  handlers; the 2 self-healing (`permission`/`form`) are the reference, the 3
  escalation-style (`hook_trip`/`infra_recovery`/`budget_breach`) were the
  laggards now fixed.
- **A `.then(onFulfilled)` on a promise pushed into `Promise.allSettled` is a
  swallowed-rejection trap.** Any flag mutated only in `onFulfilled` keeps its
  optimistic value when the promise rejects. Always pair the success reset with
  an `onRejected` (or `.catch`) when the flag drives downstream control flow.
- **Distinguish a spec-conformant limitation from a bug before "fixing" it.**
  Finding #3's recommended hardening (deny/HITL `execute`) would break
  legitimate read-only bash (tests/git/grep) in unattended path-guarded runs —
  a product regression. The honest fix was to document the kind-based scope, not
  to mechanically apply the reviewer's suggestion.
