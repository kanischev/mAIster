# Codex adversarial-review fixes: auto-launch give-up taxonomy, authoritative enqueue intent, give-up CAS

**Date:** 2026-06-25 19:13
**Files:** web/lib/scheduler/handlers/auto-launch-triaged.ts, web/lib/services/triage.ts, web/lib/scheduler/handlers/__tests__/auto-launch-triaged.integration.test.ts, web/app/api/v1/ext/projects/[slug]/tasks/[taskId]/triage/__tests__/route.integration.test.ts, docs/system-analytics/triage.md
**Severity:** high

## Problem

Three holes in the ADR-111 `auto_launch_triaged` path, surfaced by an adversarial
Codex review:

1. **Infinite-retry / no-silent-stall breach.** A triaged+`auto` task whose flow
   has an enabled-but-structurally-unlaunchable revision (unsupported manifest
   `schemaVersion`, incompatible engine, unknown role/capability/mcp ref) made
   `launchRun` throw `CONFIG`. The tick's `isTerminalLaunchRefusal` matched only
   `PRECONDITION`, so `CONFIG` fell into the "transient" branch → the task stayed
   `launch_mode='auto'` and the tick re-attempted + WARN-spammed every 60 s
   forever, never holding the task.
2. **Stale enqueue intent.** `applyTriageVerdict` set `launch_mode='auto'` only
   when `enqueue` was true and never cleared a pre-existing `'auto'`;
   `applyTriageFlag` never touched it. A task armed once could later be
   re-triaged without `enqueue` (or flagged) yet keep `launch_mode='auto'`,
   auto-launching work the latest decision did not authorize. `sendTaskToTriage`
   already cleared it — the three write-paths were inconsistent.
3. **Background-sweep stale write.** `giveUp` updated the task by `id+project`
   only. Between candidate selection and the give-up write (a window spanning
   `launchRun`'s git/worktree/supervisor I/O), an external action (human clears
   the flag, re-triage, flow change, `sendToTriage`) could land — and the
   unconditional write clobbered it and posted an obsolete system comment.

## Root Cause

1. The retry-loop's terminal/transient split enumerated only the *obvious*
   non-retryable code (`PRECONDITION`) and treated "everything else" as
   transient. But `CONFIG` is, by the taxonomy's own semantics, a
   misconfiguration that retrying can NEVER fix — it is non-retryable, exactly
   like `PRECONDITION`. The author's model (PRECONDITION=terminal,
   EXECUTOR_UNAVAILABLE/CONFLICT=transient) simply did not account for the
   `CONFIG` refusals `launchRun` raises *after* a flow passes the triage-time
   enablement+trust gate. Triage-time validation can't prevent this either: a
   flow healthy at triage can break (package upgrade / schema drift) before the
   tick fires, so the *launch-time* classification is the only place that closes
   it completely.
2. `launch_mode` is a shared "intent" column written by four paths
   (`applyTriageVerdict`, `applyTriageFlag`, `sendTaskToTriage`, the tick's
   `giveUp`). Only one of the triage write-paths was made authoritative; the
   others were write-on-arm-only, leaving the prior value to drift.
3. A read-then-write across a long async window without a compare-and-set
   predicate is a classic TOCTOU. The singleton lease only prevents two *ticks*
   racing — it does nothing against concurrent *route* mutations.

## Solution

1. `isTerminalLaunchRefusal` now returns true for `PRECONDITION` **or** `CONFIG`
   → a structurally-unlaunchable flow is HELD (give-up: clear `launch_mode`, set
   `flagged`, post a comment) instead of looping. `EXECUTOR_UNAVAILABLE` /
   `CONFLICT` stay transient. (Chose this tick-side classification over a
   triage-time revision guard — it fully closes the hole including post-triage
   drift, with ~3 lines and no duplication of launch-path checks.)
2. `applyTriageVerdict` sets `launch_mode = enqueue ? 'auto' : null` (clears stale
   arm); `applyTriageFlag` also clears it. All triage write-paths are now
   authoritative + consistent with `sendTaskToTriage`.
3. `giveUp` writes under a CAS (`triage_status='triaged' AND launch_mode='auto'
   AND flow_id=<candidate>`), uses `.returning()` to detect 0 rows, and on a miss
   writes nothing / posts no comment / returns `false` so the caller counts it as
   `skipped`, not `gaveUp`.

Regression tests added for all three (real-Postgres integration); the doc
contract (`triage.md` flowcharts + Expectations) updated in the same pass so the
"give-up only on PRECONDITION" statements match the code.

## Prevention

- **Retry/give-up loops must classify the FULL non-retryable set, not the
  obvious member.** When a loop branches terminal-vs-transient on a
  `MaisterError.code`, enumerate every code the callee can throw and classify
  each explicitly — `CONFIG` (misconfiguration) is non-retryable and belongs with
  `PRECONDITION`, not in a `default: transient` bucket. A `default → transient`
  fallback in a no-silent-stall loop is a latent infinite-retry.
- **A shared "intent"/state column needs ONE authoritative writer semantics
  across ALL write-paths.** When fixing one, grep every sibling writer (here: the
  four `launch_mode` writers) and make them consistent in the same pass — a
  single path that clears while siblings leave-stale is the tell.
- **Read-then-write across an async I/O window needs a CAS** over the selected
  identity tuple + a row-count check before any side-effect (comment/event). A
  singleton lease guards tick-vs-tick, never tick-vs-external-route.
- **A behavior change to a documented give-up/terminal classification is a
  contract change** — update the `system-analytics` Expectations + flowcharts in
  the same change (docs/CLAUDE.md: code wins, fix docs in the same PR) and re-run
  `pnpm validate:docs`.
