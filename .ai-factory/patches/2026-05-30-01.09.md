# Supervisor readiness docs contract drift

**Date:** 2026-05-30 01:09
**Files:** docs/error-taxonomy.md, docs/system-analytics/tasks.md, docs/system-analytics/runs.md
**Severity:** medium

## Problem

Verification found that the supervisor readiness implementation was functionally correct, but two documentation surfaces still contradicted the implemented `POST /api/runs` contract:

- `docs/error-taxonomy.md` listed executor-not-registered under `PRECONDITION`, while the route and OpenAPI use `EXECUTOR_UNAVAILABLE`.
- `docs/system-analytics/tasks.md` said missing executor returns `503 + registered list`, but the route returns the standard `MaisterErrorBody`.
- `docs/system-analytics/runs.md` described the readiness failure as leaving no worktree/run/task change, omitting the workspace row covered by the OpenAPI and regression test.

## Root Cause

The original documentation had older launch semantics, and the readiness feature updated the primary API/OpenAPI path before every secondary analytics wording was normalized. Because the implementation reuses the same `EXECUTOR_UNAVAILABLE` code for missing executor registration and supervisor unavailability, stale examples around `PRECONDITION` and richer UI hints became contract drift.

## Solution

Aligned the taxonomy and analytics wording with the actual route behavior:

- Removed executor-not-registered from the `PRECONDITION` meaning.
- Replaced the task analytics `503 + registered list` wording with `503 EXECUTOR_UNAVAILABLE`.
- Updated the run analytics no-side-effects note to include workspace rows.

## Prevention

- During verification, compare every error-code table against the actual `throw new MaisterError(...)` sites, not only OpenAPI responses.
- Treat analytics sequence diagrams as contract surfaces when they mention response payloads or side-effect invariants.
- When documenting no-side-effect launch gates, enumerate every persistent artifact named by the route contract: worktree, run, workspace, and task.
