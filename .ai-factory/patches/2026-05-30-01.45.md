# M10 Flow package trust boundary, launch state, backfill collision, and missing API contract

**Date:** 2026-05-30 01:45
**Files:** web/lib/flows/lifecycle.ts, web/lib/queries/flow-packages.ts, web/app/api/runs/route.ts, web/scripts/backfill-flow-revisions.ts, web/app/api/projects/[slug]/flow-packages/[flowRefId]/revisions/[revisionId]/remove/route.ts, web/app/api/projects/[slug]/flow-packages/[flowRefId]/upgrade-preview/route.ts, docs/api/web.openapi.yaml
**Severity:** critical

## Problem

A Codex adversarial review of the M10 Flow-package branch surfaced four defects:

1. (critical) A project could enable/rollback to a `flow_revisions` row installed
   by a *different* project from a *different, untrusted* source, because the
   revision lookup matched on `flowRefId` only and the trust check read the
   project flow row, not the candidate revision's source.
2. (critical) The legacy backfill keyed revisions on `(flowRefId, flow.revision
   ?? "unknown")`. Two projects sharing a flow id but different local manifests
   collided on `(flowRefId, "unknown")`; the second reused the first revision id
   and pointed its enablement + historical runs at the wrong manifest/cache path.
3. (high) After installing an untrusted package (state `Installed`,
   `enabledRevisionId` set), pressing Trust made it launchable — the launch guard
   rejected only missing pointer / Disabled / Failed / untrusted, never requiring
   an explicitly launchable state, collapsing the trust+enable lifecycle.
4. (medium) The 8 new `/api/projects/{slug}/flow-packages/*` routes had no entry
   in `docs/api/web.openapi.yaml`, which still declared milestone M9.

## Root Cause

The shared mental model "revisions are globally content-addressed" was correct
for *bytes* (a git SHA is the same bytes everywhere) but was applied to *trust
and adoption*, which are project- and source-scoped. Two coupled mistakes
followed: trust was evaluated on the denormalized `flows` row instead of the
candidate revision's source, and there was no source binding between a project
and the global revision set. The backfill inherited the old loader's
non-content-addressed "unknown" sentinel, so identity collapsed for distinct
local manifests. The launch state machine documented `Installed -> Enabled` as
an explicit step but the guard never enforced it, so trust alone satisfied every
remaining condition. The OpenAPI gap was a docs-after-code sync that was deferred
and not finished.

## Solution

- **Source-scope the revision boundary.** `loadRevisionForFlow(db, flowRefId,
  revisionId, expectedSource)` now requires `rev.source === expectedSource` (the
  project's declared flow source); enable/rollback/remove/upgrade-preview all
  thread the project's `flows.source`. `upgradeFlow` rejects a source that
  differs from the flow's source. `getFlowPackages` filters listed revisions to
  the project's source. This binds the global revision set to the project's own
  source without a new table.
- **Content-address the backfill.** Legacy non-40-hex revisions derive
  `resolvedRevision = manifestDigest(manifest).slice(0,40)`; real SHAs are kept.
  Distinct manifests now map to distinct revisions; runs with the "unknown"
  text simply stay unlinked and fall back to `flows.manifest` rather than
  mis-linking.
- **Explicit launchable allow-list.** `LAUNCHABLE_ENABLEMENT_STATES =
  {Enabled, UpdateAvailable}`; launch refuses anything else. Trust no longer
  implies launchability — an untrusted install stays `Installed` and must be
  `/enable`d.
- **OpenAPI.** Added all 8 routes (paths, bodies, shared `responses`,
  `UpgradePreview`/`ContractDiff` schemas) and bumped the spec to M10.

## Prevention

- When a resource is **globally shared/content-addressed**, decide explicitly
  whether *trust/ownership* is also global. If trust is per-tenant, every
  read/enable/list/remove path must filter by the tenant's binding (here:
  source), not just the shared key. Add this as a review checklist item for any
  "global cache + per-project pointer" design.
- A documented state machine transition (`Installed -> Enabled`) must be backed
  by an **allow-list guard**, never by "reject the known-bad states." Enumerate
  launchable/actionable states positively.
- Migrations/backfills that assign identity MUST be content-addressed when the
  source data has a non-unique sentinel; add a smoke test with two tenants
  sharing a key but differing payloads.
- Treat OpenAPI/AsyncAPI sync as part of the same task as the route, not a
  trailing docs chore — `/aif-verify` re-derives the route list from the diff.
- Regression coverage to add in the Docker-gated integration suite:
  install-untrusted -> trust -> launch still refused until enable; cross-source
  enable/rollback rejected with PRECONDITION; backfill produces distinct
  revisions for two projects sharing a flow id with different manifests.
