# CCR admin start/stop dead in production — composition root never wired the injected manager

**Date:** 2026-06-18 20:43
**Files:** supervisor/src/main.ts, supervisor/src/http-api.ts, supervisor/src/__tests__/main-wiring.test.ts, supervisor/src/__tests__/sidecars-route.test.ts, web/app/api/admin/router-sidecars/route.ts, docs/api/supervisor.openapi.yaml
**Severity:** high

## Problem

The new admin CCR start/stop feature (ADR-094) returned `409 PRECONDITION
"CCR manager is not configured"` on every call in the real supervisor. The
admin "Start sidecar" / "Stop sidecar" buttons could never start CCR. The full
test suite was green, so the defect was invisible until a code-reading review.

Two smaller boundary findings rode along (from the same review):
- `configPath` on the supervisor start body was an open `z.string().min(1)`
  flowing into `fs.access`/`readFile`, while its siblings `baseUrl`/
  `healthcheckUrl` carried `.url()` and `id` carried a regex.
- `POST /sidecars/:id/stop` never validated `req.params.id` (the documented
  OpenAPI `SidecarId` pattern was unenforced; harmless Map-key lookup, but a
  spec-vs-impl gap and asymmetric with the start route).

## Root Cause

`registerRoutes` reads the CCR manager from `opts.spawnOverrides?.ccrManager`.
The two new sidecar routes treated `undefined` as a hard `409`. But the
production composition root (`supervisor/src/main.ts` `start()`) built the
`registerRoutes` options **without** a `spawnOverrides` field at all — so
`opts.spawnOverrides` was always `undefined` in prod.

Why it stayed hidden: the *existing* session-spawn path reads the same option
but falls back to the module singleton in `spawn.ts` (`opts.ccrManager ??
defaultCcrManager`), so it works without explicit wiring. The new routes copied
the read but not the fallback. And every test exercised the routes with a
**mocked** manager injected via `spawnOverrides` (`sidecars-route.test.ts`) or
against a **stub** supervisor (e2e/web integration) — nothing exercised the real
`main.ts` wiring, so the green gate certified a feature that 409s in prod. This
is the classic "unit-tests-the-handler, never-tests-the-composition-root" gap
for an injected dependency that has no fallback.

## Solution

- **C1 (the blocker):** extracted `buildRegisterRoutesOptions(deps)` from
  `main.ts` `start()`, which now includes `spawnOverrides: { ccrManager }` (the
  same singleton already handed to the model-catalog source, so process state
  stays shared). `start()` calls it; production wiring now flows through one
  tested function. Honored the author's intended "explicit-wire-or-409" design
  (it is in the plan, OpenAPI, and tests) rather than redesigning to a silent
  fallback.
- **Regression guard:** new `main-wiring.test.ts` asserts
  `buildRegisterRoutesOptions(...).spawnOverrides?.ccrManager` is the passed
  manager. Importing `main.ts` is made safe by guarding its auto-start on
  `!process.env.VITEST` (vitest sets `VITEST` in every worker; prod/dev via tsx
  never set it) — no port bind on import.
- **S1 (configPath):** constrained `configPath` to reject `..` traversal
  (`/^(?!.*\.\.).+$/`) at both the supervisor `SidecarStartBodySchema` (the fs
  sink) and the upstream web admin-create route (where an admin first sets the
  value that is later forwarded). The regex still allows the seeded
  `~/.claude-code-router/config.json` (no `..`). Mirrored the pattern in
  `docs/api/supervisor.openapi.yaml` `CcrInstanceConfig.configPath`.
- **S2 (stop param):** extracted a shared `SidecarIdSchema` and validate
  `req.params.id` on the bodiless stop route (→ ZodError → 409 via the existing
  `setErrorHandler`); the start route already constrains it via the body schema
  + `body.id === params.id`.

Added behavior tests: start with `configPath: "../../etc/passwd"` → 409 +
`ensureRunning` not called; stop with `id="a@b"` → 409 + `stop` not called.

## Sibling sweep (recorded per project rule)

`configPath` → fs-sink class, every site audited in this pass:
- supervisor `SidecarStartBodySchema.configPath` — **fixed** (the fs boundary).
- web `POST /api/admin/router-sidecars` create schema (`route.ts:41`) — **fixed**
  (entry boundary; the stored value reaches the fs sink via the start route).
- web start route — forwards the DB value, no body schema of its own; covered by
  the create-time + supervisor-time guards. No change.
- web `SidecarInstanceConfig` (supervisor-client type) — compile-time type only,
  no runtime validation; relies on the supervisor schema. No change.
- siblings `id` (regex), `baseUrl`/`healthcheckUrl` (`.url()`) already
  constrained at both boundaries. No gap.

## Prevention

- When a route handler reads an **optional injected dependency** and treats
  absence as an error (no fallback), a unit test that injects a mock is NOT
  sufficient coverage — it certifies the handler, not the wiring. Add a test on
  the composition root (the option/DI builder) OR mirror the codebase's
  established fallback (`x ?? defaultX`). Grep every `opts.<dep>?` read for a
  matching fallback or a wiring test.
- A "409 when not configured" test passing with `boot(undefined)` is a smell: it
  encodes the broken-prod state as expected. Pair it with a test proving the
  real entrypoint configures the dependency.
- Boundary-validation fix = sweep every sibling reaching the same sink in the
  same pass, across BOTH the producing (create) and consuming (use) boundaries,
  and mirror the constraint into the OpenAPI contract.
- Make the composition root importable (guard auto-start) so DI wiring is
  unit-testable without binding ports.
