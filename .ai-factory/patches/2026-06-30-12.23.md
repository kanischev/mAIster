# Atomic counter bumped before validation burns retry budget; body parsed before auth

**Date:** 2026-06-30 12:23
**Files:** web/lib/services/runs.ts, web/app/api/runs/route.ts, web/app/api/runs/__tests__/relaunch-concurrency.integration.test.ts, web/app/api/runs/__tests__/post-branch.test.ts, .ai-factory/specs/relaunch-from-task-history.spec.md, docs/decisions.md, docs/system-analytics/runs.md
**Severity:** high

## Problem

ADR-119 added an atomic `UPDATE tasks SET attempt_number = attempt_number + 1
RETURNING` to make concurrent force-launches reserve distinct branch names. Two
defects surfaced in adversarial review:

1. **[high]** The bump was placed at the point where the run branch was derived —
   which sat **before** the branch allow-list validation (`knownBranches.has(base/target)`)
   and `resolveBaseCommit`. A member could POST an invalid `baseBranch`, get a
   `PRECONDITION` after the bump, leave no run row, and still have burned a
   number. Because `ralph-loop` reads `tasks.attempt_number` as the max-attempt
   high-water mark, repeated invalid launches erode (and eventually exhaust) a
   task's auto-relaunch budget — a cheap, user-controllable DoS of the retry loop.
2. **[medium]** `POST /api/runs` parsed `postBodySchema` (now carrying the new
   `allowConcurrent` field) **before** `requireActiveSession()`, so an
   unauthenticated/password-gated caller with a malformed body got a body-shape
   `CONFIG` 400 instead of the auth response — contradicting the route's own
   "auth-first" comment.

## Root Cause

1. The original code computed `newAttempt`/`branch` early (a stale read), with
   branch validation downstream. When I replaced the stale read with an atomic
   bump **in place**, I inherited that ordering — moving a *write* (the side
   effect) ahead of cheap, throwing preconditions. The plan's crash-window
   analysis treated a burned number as "meaningless," which is only true if
   nothing reads `attempt_number` as a budget. It does (ralph-loop). The
   "irreversible/external mutation is the LAST step after all cheap preconditions"
   rule (git-mechanics skill rule) applies to a DB counter bump too, not just to
   `git`/`fs` mutations — any write whose reversal leaves an observable gap.
2. Pre-existing ordering: body parse was first because the route was written to
   fail fast on shape. Adding an auth-gated behaviour flag to that schema made the
   ordering a real contract issue (auth must precede any body interpretation).

## Solution

1. Moved the atomic allocation + branch derivation to **after** the launchability
   gate, branch allow-list validation, and `resolveBaseCommit`, immediately before
   `addWorktree`. Only the irreducible `addWorktree`/tx window can now burn a
   number (a genuine git/db failure, not user input). Added a real-PG regression:
   a launch refused by branch validation leaves `attempt_number` untouched and
   creates no run row.
2. Reordered `POST /api/runs` to run `requireActiveSession()` before
   `req.json()`/`postBodySchema.parse`. Added a unit regression: an unauthenticated
   request with an invalid body returns `UNAUTHENTICATED` (401), never `CONFIG`,
   and never reaches `launchRun`.
3. Corrected the crash-window narrative in the spec, ADR-119, and `runs.md` to
   state that cheap-precondition refusals never burn (only the addWorktree/tx
   window does).

## Prevention

- A counter/sequence bump that another feature reads as a budget or quota is a
  **side effect**, not a free derivation — sequence it after every cheap,
  throwing precondition (validation, allow-list, ref resolution), right before the
  point of no return. Apply the "irreversible mutation is the last committed step"
  rule to DB writes that leave observable gaps, not only to git/fs operations.
- When a "burned value is meaningless" claim is made, grep for every reader of
  that column first (`rg "\.attemptNumber|attempt_number"`) — a single consumer
  that treats it as a cap invalidates the claim.
- A route that gates a new body field on auth MUST authenticate before parsing the
  body; add a test asserting the unauthenticated-invalid-body path returns the
  auth error, not a schema error.
- Prove burn/no-burn semantics against real Postgres — the failure-after-write
  ordering is invisible to a mocked unit db.
