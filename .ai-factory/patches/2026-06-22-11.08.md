# Shared-tree promote/allocation hardening (3 Codex adversarial-review findings)

**Date:** 2026-06-22 11:08
**Files:** web/lib/agents/launch.ts, web/lib/domain-events/auto-launch.ts, web/lib/runs/run-status-sets.ts, web/lib/runs/promote.ts, web/lib/runs/shared-tree.ts (new), web/lib/reconcile.ts
**Severity:** critical

## Problem

A `/codex:adversarial-review` of the ADR-101 shared-worktree branch found three real holes in the tree-level promote/allocation path:

1. **Target mutated before the safety fence (critical).** `promoteWorkspaceRun` runs the git merge (+ optional push) LOCKLESS, then the finalize tx re-checks the settled-gate and aborts `CONFLICT` if a sibling left `Review`. The abort protects the DB (runs stay `Review`) but NOT the already-merged target branch — a `run_rework` during the merge window leaves the target mutated while the run reports "not promoted."
2. **Failed siblings auto-merge their partial work (critical).** `SETTLED_RUN_STATUSES` (used by the settled-gate) includes `Failed|Crashed|Abandoned`, so a failure-terminal shared sibling is "settled" and does not block. The as-plan auto-promoter then merges a tree containing that sibling's partial, unreviewed commits UNATTENDED when a sibling reaches `Review`.
3. **Allocation trusts the filesystem, not the DB (high).** The allocator-vs-reuser decision keyed on `listWorktrees` (filesystem); the `workspaces` row is inserted later. A crash between `git worktree add` and the insert orphans the path (on disk, no row) → every later sibling "reuses" it and skips the insert → the tree NEVER gets a row → unresolvable for promote/diff/GC.

## Root Cause

- **#1 — a safety re-check placed AFTER an irreversible side-effect cannot protect that side-effect.** The lock held during the claim was released before the lockless merge; the finalize re-check ran after the merge already committed to git. The fence was on the wrong side of the irreversible operation.
- **#2 — one status set was reused for two different concerns.** `SETTLED_RUN_STATUSES` answers "is this sibling done WRITING?" (writer-safety). It was silently reused to answer "is this tree safe to AUTO-merge?" (promotability). A failure-terminal sibling is settled-for-writing but its half-finished work is NOT safe to auto-ship — the two predicates diverge.
- **#3 — ownership/allocation state was derived from a non-durable observation.** The git worktree on disk is a side-effect, not the source of truth; the DB row is. Deriving "already allocated?" from the filesystem opened a crash-window where the durable record (the row) and the side-effect (the dir) disagree, with no recovery.

## Solution

- **F1 — fence the rework BEFORE any git side-effect.** `reworkChildRun`, for a shared writable child, opens a tx that locks the tree allocator `workspaces` row FOR UPDATE (the SAME row the promote claim/finalize lock) and refuses `CONFLICT` while `promotion_state ∈ {claiming, done}`, else CASes `Review→Running` in that tx. Rework and promote now serialize on one row → a rework can't open during the merge window. The finalize re-check stays as a backstop.
- **F2 — distinct promotability predicate on the auto path (Option B).** New `FAILURE_TERMINAL_RUN_STATUSES` (= TERMINAL minus `Done`). `autoPromoteAsPlanChild` SKIPS (benign log) a shared tree with any failure-terminal sibling, leaving it for human attention. Manual `run_promote` stays allowed (a human reviews the whole tree-diff). The writer-safety settled-gate is untouched.
- **F3 — DB is the allocation source of truth + orphan recovery.** `launchAgentRun` decides reuser-vs-allocator from the `workspaces` row (not `listWorktrees`); no row + path on disk ⇒ ORPHAN-CLAIM (reuse dir, insert row, `base_commit=null`); the insert is `onConflictDoNothing(worktree_path)`; an `allocatedWorktree` flag makes teardown remove only a dir THIS launch created. `reconcile.ts` gains `recoverOrphanSharedTrees` (wired into the sweep) to re-create a missing row for an on-disk orphan tree. The shared-tree resolver/predicates were extracted into `web/lib/runs/shared-tree.ts` (DRY — promote, rework, auto-promote, launch all reuse them).

No new `MaisterError` code (reused `CONFLICT`/`PRECONDITION`, ADR-008). No migration.

## Prevention

- **A gate that guards an irreversible side-effect MUST run before (or in the same tx as) that side-effect — never as an after-the-fact re-check.** Ask: "if this gate fails, can I undo what already happened?" If not, move the gate earlier.
- **Never reuse a status/state set across concerns without checking the semantics align.** Writer-safety ≠ promotability ≠ recoverability. Name the predicate after the concern (`countFailureTerminalSharedSiblings` vs `countUnsettledSharedSiblings`).
- **Ownership/allocation/identity state belongs in the DB, not in a filesystem/side-effect observation.** When a durable record and a side-effect can diverge (TOCTOU/crash window), add explicit claim + reconcile recovery; make the insert idempotent (`onConflictDoNothing`).
- **Review check:** for any "fence then mutate" sequence on a shared resource, confirm the lock spans the mutation, or the mutation is reversible/idempotent on a later re-attempt.
- **Test the race directly** (deferred-side-effect stub + concurrent actor), not just the sequential happy path.
