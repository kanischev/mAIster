# Shared-tree promote mutated the target before the failure-terminal gate (Codex adversarial review)

**Date:** 2026-06-22 13:40
**Files:** web/lib/runs/promote.ts, web/lib/runs/__tests__/promote-shared-tree.integration.test.ts
**Severity:** critical

## Problem

ADR-101 shared-tree promote (the M37 follow-up on `claude/priceless-goldstine-33d1f1`).
A NON-human promote of a shared writable tree (`workspace_mode='shared'`,
`agent_workspace='worktree'`) could **merge / push the target branch and only
then abort**, leaving the target mutated with unreviewed work while returning
`CONFLICT` — which §8 documents as "merged nothing".

Two cases:
- **Case A (deterministic):** the orchestrator `run_promote` ext route had **no**
  failure-terminal pre-check. With a sibling ALREADY `Failed`/`Crashed`/`Abandoned`,
  the merge ran, then the finalize-tx FIX B re-check aborted → target already merged.
- **Case B (race):** a `Review` sibling that failure-terminalizes DURING the
  lockless merge window. (Out of this fix's scope — see Solution; still caught,
  non-silently, by FIX B post-merge.)

## Root Cause

The promote claim-tx settled-gate uses `countUnsettledSharedSiblings`, whose
predicate is the complement of `SETTLED_RUN_STATUSES` (terminal + Review). The
FAILURE-terminal statuses (`Failed | Crashed | Abandoned`) **are settled**, so the
claim-tx gate by design does NOT block a tree that already contains a
failure-terminal sibling. The ONLY failure-terminal guard was FIX B, located in the
**finalize tx — which runs AFTER `promoteMergeSideEffect` + `maybePushTargetBranch`**
(side-effects intentionally run lockless, between the claim and finalize txns). So
detection happened only post-merge, after the target was already mutated. The
auto-launcher had its own pre-merge F2 pre-check; the orchestrator `run_promote`
route had none, so Case A was reachable with no concurrency at all.

## Solution

Add the failure-terminal guard to the **claim tx, before any git side-effect**,
scoped to non-human promotes (`!isHumanPromotion(ctx)`), reusing the existing
`countFailureTerminalSharedSiblings` helper. It lives inside the same shared-tree
`if` block as the settled-gate, so an already-failed sibling throws `PRECONDITION`
before the claim is even minted — the merge fn is never reached and the target is
never touched. A human manual promote stays exempt (the whole tree-diff, incl. the
failed sibling's work, was reviewed — F2 Option B). The finalize FIX B is kept as
the under-lock backstop for Case B.

Scope decision (owner-approved): claim-tx guard only. Case B (a sibling failing
mid-merge) stays caught — non-silently — by FIX B (`CONFLICT`, no settle). Closing
it fully needs either fencing ~10 `Review→failure-terminal` transition sites
(grep-to-zero hazard) or a temp-ref two-phase merge (shifts the inconsistency to
DB=Done/target-not-advanced). Both trade a narrow, detected race for a new failure
class — rejected under the zero-margin mandate.

Test: `promote-shared-tree.integration.test.ts` "FIX C" — parametrized over
`Failed`/`Crashed`/`Abandoned`, asserts `PRECONDITION` + the merge spy is **never
called** (the "target unchanged" proof in the stubbed-git harness) + claim never
minted (`promotion_state` stays `'none'`).

## Prevention

- **When a gate must prevent an external/irreversible side-effect, it MUST run
  BEFORE the side-effect, not just detect-and-abort after.** A post-side-effect
  re-check that returns an error does NOT undo a git merge/push. "Detect during the
  window" ≠ "prevent the mutation." Audit any finalize-tx guard that sits after a
  lockless side-effect for what it can actually still prevent.
- **Complement-based status predicates silently admit new/adjacent classes.**
  `notInArray(status, SETTLED)` treats failure-terminal as settled — a separate
  allow-list (`inArray(status, FAILURE_TERMINAL)`) was needed. Mirrors backend
  rule 44 (allow-list exact states, never a coarse deny-list complement).
- **Sibling sweep (backend rule 13):** the defect class = "failure-terminal gate
  runs only post-merge for non-human promotes." Grep recorded:
  `rg "promoteWorkspaceRun|promoteChildRunForToken|promoteRun"` → all non-scratch
  promotes funnel through `promoteWorkspaceRun`; `promoteChildRunForToken`
  (orchestrator run_promote + auto-launcher) → `promoteRun` → `promoteWorkspaceRun`;
  scratch goes to `promoteScratchRun` (never a shared tree). Fixed the class once at
  the central claim-tx choke point — covering every current AND future non-human
  caller — instead of patching the ext route in isolation (DRY; no per-route drift).
- Verify a finding before fixing: Codex marked this an inference from execution
  order — the order (side-effect 666-672 vs finalize gate 815) and the missing ext
  pre-check were both confirmed in source before any edit.
