# Token creation auth-order leak

**Date:** 2026-06-08 19:18
**Files:** `web/app/api/projects/[slug]/tokens/route.ts`, `web/app/api/projects/[slug]/tokens/__tests__/route-auth.test.ts`
**Severity:** high

## Problem

`POST /api/projects/{slug}/tokens` parsed and validated the request body before
checking `requireActiveSession()`. An unauthenticated caller with an invalid
body could receive `422 CONFIG` instead of the expected `401 UNAUTHENTICATED`,
revealing the token-create body schema before the session gate.

## Root Cause

The route had grown from a simple token-create handler into a scoped
user/project token surface, but body parsing remained outside the main
authenticated block. The existing integration tests covered successful token
creation and authorization, but did not pin the auth-first ordering for invalid
request bodies.

## Solution

Moved `requireActiveSession()` to the first operation inside `POST` before
`req.json()` or Zod parsing. Added a focused unit regression test that mocks the
auth and DB boundaries, sends an invalid body with no active session, and
asserts `401 UNAUTHENTICATED` while proving no DB or project authorization work
ran.

## Prevention

- For session-auth Route Handlers, authenticate before parsing request bodies or
  resolving resource rows.
- Add route-level auth-order tests for security-sensitive management endpoints,
  especially when the body schema changes.
- Prefer fast unit tests for pre-auth ordering bugs so the regression is not
  hidden behind Testcontainers availability.
