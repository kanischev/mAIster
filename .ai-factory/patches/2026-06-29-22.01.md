# Gemini scratch prompt auth should use native CLI home plus workspace skills

**Date:** 2026-06-29 22:01
**Files:** web/lib/acp-runners/adapter-support.ts, web/lib/capabilities/adapter-home.ts, web/lib/scratch-runs/service.ts, web/lib/capabilities/__tests__/adapter-home.test.ts, web/app/api/scratch-runs/__tests__/route.test.ts, supervisor/src/acp-client.ts, supervisor/src/__tests__/adapter-compatibility.integration.test.ts, supervisor/test/fixtures/mock-acp-compatibility.mjs, docs/system-analytics/acp-runners.md, docs/configuration.md
**Severity:** medium

## Problem

A Gemini scratch run could create its ACP session successfully, then fail the
first prompt with `EXECUTOR_UNAVAILABLE` / `API key not valid` even though the
Gemini CLI was already authorized on the host. The launch catch also marked the
scratch run `Crashed`, forcing the user into the recover flow even though the
supervisor session and worktree were still valid.

## Root Cause

Scratch capability materialization redirected Gemini to a fresh per-run
`GEMINI_CLI_HOME`. That detached `gemini --acp` from the same user/workspace
settings merge used by a normal console launch. Gemini CLI discovers workspace
skills from `<worktree>/.gemini/skills` and keeps user auth/config in native
`~/.gemini`; MAIster was using the wrong materialization surface. Follow-up
scratch messages already treated `EXECUTOR_UNAVAILABLE` prompt failures as
retryable, but the initial launch path used the generic post-commit crash
handler for every prompt error after `session_ready`.

After the home/materialization fix, the next failure layer was Gemini ACP's auth
selection. `session/new` reads `~/.gemini/settings.json`; if
`security.auth.selectedType` is `gemini-api-key`, Gemini uses API-key auth even
when MAIster did not configure a runner key. A stale inherited key can therefore
produce `API_KEY_INVALID` while the same CLI is otherwise intended to run with
its native login. MAIster must not force an ACP `authenticate` call here because
Gemini's `oauth-personal` method may start an interactive browser login.

## Solution

Gemini now uses a cwd-dir materialization target: project bundle skills are
copied to `<worktree>/.gemini/skills`, and MAIster no longer injects
`GEMINI_CLI_HOME` for Gemini scratch sessions. The CLI-managed login remains
authoritative. The initial launch catch also routes `EXECUTOR_UNAVAILABLE`
through `markScratchPromptRetryable`, leaving the dialog in `WaitingForUser`
with the error recorded. Non-retryable failures still use
`markScratchCrashed`.

For a `google_gemini` runner without `apiKeyEnv`, the supervisor now leaves auth
selection entirely to Gemini CLI's native user configuration. It does not pass a
Gemini API key and does not call ACP `authenticate`; `session/new` reads the
operator's `~/.gemini` defaults.

## Prevention

- Regression-test that Gemini materialization writes project skills to
  worktree `.gemini/skills` and returns no redirect env.
- Regression-test that a Gemini CLI-native runner does not call ACP
  `authenticate` before `session/new`.
- Keep initial scratch launch prompt handling aligned with follow-up message
  prompt handling.
- Regression-test post-`session_ready` prompt failures separately from
  pre-session launch failures.
