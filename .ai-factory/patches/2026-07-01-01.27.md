# Artifact body injection (ADR-120) ‚Äî adversarial-review fixes

**Date:** 2026-07-01 01:27
**Files:** web/lib/flows/graph/artifact-inject.ts, web/lib/flows/graph/artifact-content.ts, web/lib/config.ts, web/lib/flows/graph/runner-graph.ts (+ tests: artifact-inject.test.ts, artifact-content.test.ts, config.artifact-inline.test.ts, artifact-inject.integration.test.ts; + docs: decisions.md, flow-dsl.md, configuration.md, system-analytics/artifacts.md)
**Severity:** high

## Problem

A Codex adversarial review of the ADR-120 artifact-body-injection feature found
three defects that all unit/integration tests had passed over:

1. **[high] Gate content scan wired to the wrong gate field.**
   `collectContentArtifactIds` scanned only `gate.prompt` for `ai_judgment` AND
   `skill_check`. But the gate executor renders `ai_judgment` from `prompt`,
   `skill_check` from `command` (or `/${skill}`), and `command_check` from
   `command` (via `runCliStep`). So a real `skill_check`/`command_check` `command`
   using `{{ artifacts.x.content }}` was never collected ‚Üí its content was never
   resolved into the node context ‚Üí `renderStrict` threw `CONFIG` at gate
   execution (false negative); and an unused `skill_check.prompt` content ref
   spuriously forced the engine floor + body resolution (false positive). This
   broke the documented "load-time/runtime detection cannot drift" contract.

2. **[high] Inline cap did not bound the file read.** `resolveFile` read the
   WHOLE artifact file into memory (`readFile`) before `capForInline` truncated
   it. A multi-GB `file` artifact under the run dir could OOM/stall the web
   process before the 256 KiB cap was reached ‚Äî violating the contract that
   `MAISTER_ARTIFACT_INLINE_MAX_BYTES` bounds injected bodies and "never fails the
   run on a large body".

3. **[medium] Valid manifests could be unrenderable.** The schema accepts any
   non-empty string for `output.produces[].id` / `input.requires[].artifact`, but
   a body-injectable id is interpolated into an XML attribute AND a dotted Mustache
   path. An id with `.`/`:`/quotes passed validation yet produced malformed XML /
   an unresolvable template / a scan miss at run time.

## Root Cause

1. The scan was written against a single assumed field (`prompt`) for "verdict
   gates", without tracing which field each gate KIND actually feeds to
   `renderStrict` in `gates-exec.ts`. The plan's D10 prose said "ai_judgment/
   skill_check gate prompts", and the implementation copied the prose instead of
   the executor's real field selection ‚Äî and missed `command_check` entirely
   (which renders `command` via `runCliStep`, exactly like a `cli` node, whose
   `action.command` WAS scanned ‚Äî an inconsistency).

2. The SRP split (D7: "resolver returns RAW, cap at injection seam") was applied
   to the TRUNCATION but not to the READ. The cap was a post-read string trim, so
   the expensive step (loading the file) happened unbounded before the cheap trim.

3. The new injection surface reused the existing M12 id fields, which were only
   ever slug-shaped by convention (the M12 metadata accessors `{{ artifacts.<id>
   .uri }}` already assume slugs). The convention was never encoded as a
   constraint, so the new XML/Mustache interpolation inherited the latent gap.

## Solution

1. **Field-specific gate scan.** Added `renderedGateField(gate)` returning the
   field the executor renders (`ai_judgment`‚Üí`prompt`, `skill_check`/
   `command_check`‚Üí`command`); `collectContentArtifactIds` scans that. This is the
   shared scan used by BOTH the load-time floor gate (config.ts) and the runtime
   resolver, so they cannot drift. Swept the WHOLE defect class (all gate kinds),
   not just the two cited.

2. **Bounded injection read.** Added an optional `maxBytes` to
   `ResolveArtifactContentCtx`; `resolveFile` reads at most `maxBytes` bytes via a
   positioned `FileHandle.read` (no full-file load) when set. The runner passes
   `cap + 1` (enough for `capForInline` to detect over-cap and truncate); the
   payload route passes nothing ‚Üí full read, HTTP contract unchanged. `capForInline`
   gained an optional `maxBytes` so the runner reads the cap ONCE and uses it for
   both the read bound and the truncation.

3. **Injectable id grammar.** Added `isInjectableArtifactId` (`^[A-Za-z0-9_-]+$`,
   the scan's character class) and a load-time refusal (`firstInvalidInlineArtifactId`
   ‚Üí `CONFIG`) for any `inline: true` id that isn't a slug. Scoped to the inline
   opt-in surface (not the shared `produces[].id` contract) to avoid breaking
   external packages that may use a non-slug id purely as an `artifact_required`
   reference.

Logging: the feature already logs via module-local `pino` (WARN on gone/notfound,
INFO on inject). Per project convention (`no-console` lint rule) the `[FIX]`
`console.log` pattern is replaced by pino; no new console logging was added ‚Äî the
grammar refusal surfaces as a `CONFIG` `MaisterError`, the bounded read is
transparent.

## Prevention

- **Scan/emit parity with the executor:** when a load-time scanner mirrors a
  runtime renderer, derive the scanned fields from the executor's actual
  per-branch field selection (read `gates-exec.ts`), never from plan prose. A
  shared helper used by both the gate (load) and the runner (runtime) is the
  enforcement; add a unit test per gate KIND √ó field (rendered vs unused).
- **Cap at the source, not after the read:** a "size cap" contract must bound the
  READ/allocation, not just trim an already-loaded buffer. For any unbounded
  external read (file/stdout/network) on a capped path, pass the bound down to
  the read primitive; regression-test that an over-cap input returns ‚â§ cap bytes.
- **Encode interpolation grammars:** any value interpolated into a structured sink
  (XML attribute, dotted template path, regex) must be validated against that
  sink's grammar at the boundary that introduces the interpolation ‚Äî a "non-empty
  string" schema is shape, not invariant (mirrors the skill-context "validate at
  the SINK's invariant" rule).
