# Flow assistant review follow-up fixes

**Date:** 2026-06-25 17:16
**Files:** `web/lib/scratch-runs/service.ts`, `web/app/api/scratch-runs/__tests__/route.test.ts`, `web/lib/scratch-runs/transcript.ts`, `web/lib/scratch-runs/__tests__/transcript.test.ts`, `web/components/studio/studio-ai-tab.tsx`, `web/messages/en.json`, `web/messages/ru.json`, `docs/api/web.openapi.yaml`, `docs/decisions.md`, `docs/system-analytics/studio-ai-assistant.md`
**Severity:** critical

## Problem

A parallel review found that the Flow Studio assistant safety flag
`readOnlySession: true` was also present in the generic project scratch launcher.
That would make ordinary `/api/scratch-runs` coding sessions unable to perform
write/edit/execute tool calls. The same review also found that structured Flow
assistant action fences could appear transiently during streaming, a duplicate
non-validating action-result encoder existed, runner DTO fields advertised
disabled runners that were filtered out, and the `intent` semantics were not
explicit enough.

## Root Cause

The assistant launch path was added beside the existing generic scratch launch
path in the same service module. A read-only ACP session invariant that belongs
only to local-package assistant runs leaked into the older generic create-session
call. The raw action JSON issue had the opposite shape: persisted post-turn
sanitization existed, but the live transcript renderer reused the incremental
assistant chunks before that post-processing pass completed.

## Solution

Removed `readOnlySession: true` from the generic project scratch launcher and
kept it on the local-package assistant launch. Added a route-level regression
assertion that generic scratch launch does not create a read-only session.
Added live display stripping for complete and incomplete
`maister-flow-assistant-action` fences before assistant markdown reaches the
primary transcript. Deleted the duplicate scratch transcript encoder so
action-result encoding stays centralized in the validating Flow assistant
protocol module. Simplified assistant runner responses to enabled Ready options
only, removed the orphaned `tabProperties` translation key, and documented that
`intent` is prompt-only in V1.

## Prevention

- When adding a safety flag to one launch path, grep every sibling create-session
  call and pin both the intended positive and intended negative contracts in
  tests.
- Hide machine-readable protocol blocks at every render boundary, not only in
  final persistence.
- Keep protocol encoders in the schema-owning module so serialization cannot
  drift from validation.
- If an API filters out disabled records, do not keep disabled-state fields in
  the DTO unless the endpoint actually returns those records.
