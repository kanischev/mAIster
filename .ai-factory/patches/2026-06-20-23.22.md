# Orchestrator wake-on-Review + promote/rework + auto-launcher exactly-once backstop (M37 review fixes)

**Date:** 2026-06-20 23:22
**Files:** web/lib/agents/launch.ts, web/lib/domain-events/{taxonomy,outbox,orchestrator-resume,auto-launch}.ts, web/lib/runs/{state-transitions,promote,run-status-sets}.ts, web/lib/flows/graph/runner-graph.ts, web/lib/reconcile.ts, web/lib/db/schema.ts (+ migrations 0059/0060), web/app/api/v1/ext/runs/{promote,rework,delegate}/route.ts, mcp/src/tools.ts, web/types/token-scopes.ts, docs/*
**Severity:** high

## Problem

`/aif-review` of the M37 orchestrator branch found a deadlock (C1): a delegated child
whose agent declares `workspace: worktree` exits a clean turn to `Review`, not `Done`.
`Review` (a) emitted no domain event and (b) counted as perpetually "pending" in the
three child-pending counters. So a parked orchestrator (`WaitingOnChildren`) whose
children produce diffs was never woken and never completed — the headline coding-
delegation use case did not close. Plus: the `workspace` axis was parsed then dropped
(C2); concurrency contracts had only single-threaded tests (C3); a shared-worktree
allocation TOCTOU surfaced as a raw 500 (C4); and analytics drift (C5).

A C3 concurrent two-racer test then exposed a SECOND latent race: the auto-launcher's
`hasAnyRun` exactly-once guard is a check-then-act whose claimed `(agent_id,
trigger_event_id)` index backstop is INERT for auto-launches (`trigger_event_id` is
null), so two concurrent dispatches double-launched a released dependent.

## Root Cause

1. **State-vs-event mismatch.** `Review` is a real run status but was never modeled as
   a child-settled SIGNAL. The wake/complete machinery enumerated child-terminal as
   exactly `{Done,Failed,Crashed,Abandoned}` — the spec's Expectations did too — so a
   diff-producing child (the normal coding case) fell through every path. The e2e test
   masked it by using `workspace: none` children (which exit `Done`).
2. **Check-then-act with no DB constraint.** The auto-launcher relied on a `SELECT`
   (`hasAnyRun`) plus the singleton dispatcher for exactly-once; the documented unique-
   index backstop did not actually apply (null `trigger_event_id`). The project's own
   concurrency rule forbids exactly this (an insert gated by a read-then-write SELECT
   instead of a DB constraint + `onConflictDoNothing`).
3. **Accepted-but-ignored input.** `workspace` was added to the route schema and the
   `delegationSpec` but never threaded into `launchAgentRun`, and `startAgentSession`
   re-derived the axis from the agent def — so a delegation override would have silently
   diverged from what the worktree allocation used.

## Solution

- **C-2 settle model:** one shared `SETTLED_RUN_STATUSES = terminal + Review`
  (`run-status-sets.ts`) DRY'd across the three child-pending counters. A new
  `run.review` domain event (settled-not-terminal, carries `parent_run_id`, migration
  0059) wakes the parked coordinator; `finalizeAgentRun` emits it for a delegated Review
  child and PRESERVES `acp_session_id` on that flip so rework can `session/resume`.
- **promote/rework:** `run_promote` (new `runs:promote` scope) merges a reviewed child
  via the existing promote core behind an ADDITIVE non-user `actor` (owner-less,
  conflict→typed CONFLICT, no human assignment); `run_rework` re-opens a Review child
  (Review→Running CAS + resume). as-plan auto-promotes on `run.review` so the auto-DAG
  flows unattended.
- **Exactly-once DB backstop:** partial unique index `runs_auto_task_uq (task_id) WHERE
  launch_mode='auto'` (migration 0060). `launchAgentRun`'s targetless
  `onConflictDoNothing()` already dedups the second concurrent insert.
- **C2/C4:** `workspace` is now an honored `LaunchAgentRunInput` override (and
  `startAgentSession` reads the persisted `agent_workspace`); shared-tree allocation
  catches the concurrent-allocator git error, re-checks the registry, and reuses-or-
  CONFLICTs instead of raising a raw 500.

## Prevention

- **Model a status that gates a loop as BOTH a state and a signal.** When a run status
  controls whether a coordinator advances, ask "what wakes the coordinator when a child
  reaches it?" — if nothing emits, it deadlocks. Enumerated-terminal sets are a smell:
  prefer a derived `SETTLED`/`PENDING` predicate from ONE source so every counter agrees.
- **A check-then-act insert needs a DB constraint, not just a SELECT + a singleton
  assumption.** If the code comment claims a unique-index backstop, write the CONCURRENT
  two-racer test that proves it — a single-threaded "call it twice" test would have
  passed while the index was inert. The real two-racer is what found the bug.
- **An accepted request field that never reaches a side-effect is a silent contract
  lie.** Grep every consumer of a new input (here every `launchAgentRun(` caller) and
  confirm it is applied or drop it from the schema.
- **Don't trust a spec's status tags or its Expectations enumeration** — re-derive the
  child-terminal set from the code; a documented edge-case/enumeration that omits a
  reachable status (`Review`) steers implementers wrong.
