# Read-model / validation drift — board failed-gate re-eval, requires.kind unvalidated, Postgres-only @> in readiness

**Date:** 2026-06-02 05:04
**Files:** web/lib/config.ts, web/lib/queries/board.ts, web/lib/flows/graph/evidence-readiness.ts
**Severity:** medium

## Problem

Three smaller drifts surfaced in the same review:
1. The board `mergeBlocked` badge counted ANY live `failed` blocking
   `artifact_required` gate, but the authoritative `assertEvidenceReady`
   RE-EVALUATES a failed gate (skips it if its inputs are current again). The two
   readiness surfaces disagreed → a card could stay merge-blocked after a clean
   re-run.
2. `docs/flow-dsl.md` promised an `input.requires` `{artifact, kind}`
   kind-mismatch CONFIG rejection, but `validateArtifacts` only checked the
   artifact id existed — a wrong `kind` was silently accepted (and the bundled
   aif flow uses the object form throughout).
3. `assertEvidenceReady` filtered `required_for` with the Postgres-only `@>`
   jsonb operator, which would throw on the documented SQLite ("ultra-light
   dev") dialect.

## Root Cause

1. The readiness predicate was duplicated across two consumers (the authoritative
   `evidence-readiness.ts` and the board read model) and only one carried the
   failed-gate re-evaluation — classic read-model drift from copying a predicate
   instead of sharing it.
2. A validation rule existed in the spec but not the validator — partial
   implementation of a documented contract, with the live flow already relying on
   the unchecked form.
3. A dialect-specific operator was used with no guard, against a project that
   explicitly supports two dialects.

## Solution

1. Board `mergeBlocked` now mirrors `assertEvidenceReady` Check 2: a `failed`
   gate whose `inputArtifactRefs` are all current again no longer blocks (reuses
   the board's existing `currentPairs` set); `stale` always blocks.
2. Added Rule 2b to `validateArtifacts`: an id→producedKind map; an object-form
   `requires` whose declared kind != the produced kind throws CONFIG. Tests for
   mismatch (reject) and match (accept).
3. Dialect guard in `assertEvidenceReady`: Postgres keeps the server-side `@>`
   filter; otherwise fetch the run's rows and filter `requiredFor` in JS. Same JS
   filter used by the new `getCurrentRequiredForGitArtifacts` so neither depends
   on `@>`.

## Prevention

- When two code paths compute the SAME verdict (readiness, authz, eligibility),
  a new diff that refines one must fan the refinement out to every consumer — or
  the predicate should be shared. Grep the new value/rule into every consumer.
- A validation rule asserted in a contract doc (flow-dsl, OpenAPI, error
  taxonomy) must have a matching code check AND a test, especially when a shipped
  artifact already exercises the unchecked form.
- A dialect-specific SQL operator (`@>`, `ILIKE`, `jsonb_*`) in code that targets
  more than one dialect needs a guard or a dialect-agnostic equivalent.
