# Takeover return left requiredFor:[review] evidence stale-but-current

**Date:** 2026-06-02 05:02
**Files:** web/app/api/runs/[runId]/takeover/return/route.ts, web/lib/flows/graph/artifact-store.ts
**Severity:** critical

## Problem

After a manual takeover at the review node, the human committed work locally and
returned. The return route staled only `[reentryNode, ...downstreamOf(reentry)]`.
In the bundled aif flow the re-entry is `checks`, which is DOWNSTREAM of
`implement`, so `implement`'s `impl-diff` (`requiredFor:[review,merge]`, pinned
to an immutable PRE-takeover SHA) was never staled — it stayed `current`. The
reviewer's actual commits were recorded as separate `takeover:*` defs with NO
`requiredFor`. On loop-back, `assertEvidenceReady("review")` found `impl-diff`
current and approved — validating the pre-takeover diff, decoupled from the work
the reviewer actually produced.

## Root Cause

Staleness was propagated only DOWNSTREAM of the re-entry node, but a takeover
mutates the whole branch — including the range captured by UPSTREAM
`requiredFor` evidence. "Downstream of the rework point" is the right rule for
re-executable nodes, but `requiredFor` git artifacts upstream of the re-entry
are NOT re-executed, so they silently froze the pre-takeover state. The evidence
contract (review validates the cumulative diff) was not connected to the
takeover mutation.

## Solution

On takeover return, AFTER `markDownstreamStale`, re-pin every STILL-current
`requiredFor:[review|merge]` git artifact (kind diff|commit_set) to the
post-takeover branch tip — `baseRef..tip` = the FULL cumulative diff including
the reviewer's commits — superseding the pre-takeover row in the same
transaction. Running after the stale step means downstream requiredFor artifacts
(already staled, to be re-produced by the re-run) are excluded; only upstream
ones that won't be re-produced are refreshed. Helper
`getCurrentRequiredForGitArtifacts` selects the rows (requiredFor filtered in JS
— dialect-agnostic). Integration test asserts `impl-diff` is re-pinned to the
new tip, the prior row superseded, and non-`requiredFor` diffs untouched.

## Prevention

- When a flow mutates the workspace OUTSIDE the normal node-execution path
  (manual takeover, rewind, external edit), audit EVERY `requiredFor` evidence
  artifact — not just the ones downstream of the resume point. Evidence pinned
  to an immutable ref upstream of the mutation goes stale-but-`current` and will
  pass readiness on a frozen snapshot.
- A "downstream-only" staleness sweep is suspect whenever upstream artifacts
  capture a RANGE (diff/log) rather than a point — the range moved.
