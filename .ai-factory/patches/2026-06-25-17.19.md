# Version-adopt advanced a shared project pin before launch preconditions could fail

**Date:** 2026-06-25 17:19
**Files:** web/lib/services/runs.ts, web/lib/local-packages/versions.ts, web/lib/local-packages/git.ts, web/lib/local-packages/publish.ts, docs/decisions.md
**Severity:** high

## Problem

M39 Stream B added "version-adopt launch": when a Backlog task launches against a
centralized package that has a newer cut, `launchRunStaged` advances the project's
pinned `package_installs` row (`upgradeAttachment`) so the run executes the adopted
version. That pin mutation happened **early** in `launchRunStaged` — right after the
flow loaded — but ~15 launch preconditions still run *after* it (supervisor health,
enablement/trust, schema/engine-compat, host-requirement checks, settings
materialization, branch-target resolution, the worktree `git worktree add`, and the
run-row insert). If any of those threw, the function aborted but **left the project's
pin advanced**. The pin is shared project state, so an unrelated launch failure
silently moved every future launch of that package onto the newer version — a state
corruption that outlives the failed run.

A secondary defect: the PR-to-source publish hardcoded `main` as the PR base, so a PR
or compare-URL against an upstream whose default branch is `master`/`develop` targeted
a wrong/absent base.

## Root Cause

The adopt step mutated durable, shared state (the project pin) **before** the operation
it enabled (the launch) was known to succeed. This is the classic "commit the side
effect, then discover you can't finish" ordering bug: an irreversible-by-default write
sequenced ahead of fallible work, with no compensating action on the failure path.
Wrapping the whole ~500-line remainder of `launchRunStaged` in one try/catch to revert
was rejected as non-surgical (a massive re-indent touching unrelated code), so the early
version simply had no rollback at all.

The PR-base bug was a plain assumption: "default branch == main" is false for many real
repos, and `package_sources` stores no per-source default-branch hint.

## Solution

Made adopt+launch **atomic by compensation**, proportionate to where failures actually
arise rather than a blanket wrap:

1. **Hoist the dominant failure ahead of the mutation.** `checkSupervisorHealth()` takes
   no arguments and is the single most common launch-blocker, so a pre-adopt supervisor
   gate now runs *before* the pin advances — `EXECUTOR_UNAVAILABLE` can no longer leave a
   moved pin.
2. **Record + revert what was advanced.** `applyPackageVersionChoices` now returns
   `AdoptRevert[]` (`{ attachmentId, priorInstallId }` per advance, `[]` on no-op) instead
   of a boolean. A new `revertPackageVersionChoices(reverts, …)` best-effort re-pins each
   attachment back to its prior install (catches + logs per-revert failures so one bad
   revert can't mask the original error). It is invoked in the **existing** worktree-cleanup
   catch in `launchRunStaged`, so materialize / worktree-add / run-insert failures (the
   post-mutation, non-deterministic ones) roll the pin back alongside the worktree removal.
3. **Refuse, don't silently mis-run, on a dropped flow.** If the adopted cut no longer
   ships the launched flow, throw `PRECONDITION` ("adopted package version no longer ships
   flow X") rather than proceeding against a missing flow.
4. **Document the residual.** The remaining deterministic preconditions
   (enablement/trust, schema/compat, host-requirements, settings, branch-target) that run
   between the pin advance and the worktree step still leave the pin advanced on failure —
   but those are deterministic and the user explicitly chose to adopt that version, so a
   re-launch hits the same gate with the same outcome. ADR-107's crash-window note records
   this as the accepted, defensible boundary.

PR base: added `gitRemoteDefaultBranch(dir, remote)` (`git ls-remote --symref <remote>
HEAD`, batch-mode env so it never prompts for credentials, returns `null` on failure) and
`publishLocalPackage` resolves `prBase = (await gitRemoteDefaultBranch(...)) ?? DEFAULT_PR_BASE`
for both the PR target and the compare-URL.

## Prevention

- **Sequence durable/shared-state mutations as late as possible, after all cheap,
  deterministic preconditions.** When a side effect must precede fallible work, pair it with
  an explicit compensating action on every failure path — return the undo handle (here
  `AdoptRevert[]`) rather than a boolean so the caller *can* compensate.
- **Code review check:** any write to shared project/tenant state (pins, defaults,
  enablement) inside a multi-step operation must answer "what reverts this if step N+k
  throws?" If the answer is "nothing," that's the bug.
- **Don't hardcode `main` as a git default branch** — resolve it from the remote's HEAD
  symref and fall back, the same way clones do.
- **Tests:** the version-adopt integration suite now asserts the revert re-pins the
  attachment after a simulated failed launch (adopt advances → `revertPackageVersionChoices`
  → pin back at the prior install), and the boolean→`AdoptRevert[]` contract change is
  covered by `toHaveLength(...)` assertions instead of truthiness.
