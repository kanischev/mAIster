# M37 orchestrator: post-merge verification gap + a guard on only ONE of two entry paths

**Date:** 2026-06-21 03.05
**Files:** web/lib/scheduler.ts, web/lib/runs/promote.ts, web/lib/{config,services/runs}.ts, web/lib/flows/graph/runner-graph.ts, web/lib/agents/tokens.ts, web/lib/domain-events/__tests__/emit-terminal.integration.test.ts (+ docs/db ERDs, board affordance)
**Severity:** high

## Problem

Fixing the /aif-review findings on `feature/orchestrator-engine` AFTER merging
main exposed three recurring failure classes:

1. **A contention guard installed on only ONE of two entry paths.** The
   one-active-writer-per-shared-worktree guard (`sharedWriterSiblingActive`)
   lived only in `promoteNextPending`, but a delegated shared child first reaches
   Running through `launchAgentRun → tryStartRun` — which had no guard. With the
   agent cap not full, 2-3 shared children raced straight into Running on one
   worktree. Both tests stubbed `tryStartRun`, so they were green AND wrong.

2. **A merge made a shared-path field required → main's tests of that path went
   stale, and a TARGETED post-merge test run missed it.** The branch made
   `parent_run_id` required on run-terminal `emitDomainEvent` (folded into the
   payload). main's `emit-terminal.integration.test.ts` asserted the exact
   payload via `toEqual({...})` with no `parentRunId`. The merge committed
   "gates green" after running web unit + a TARGETED orchestrator integration
   subset + docs — not the full integration suite — so two stale assertions
   shipped in the merge commit and only surfaced in the next phase.

3. **An autonomous code path that omits the flag its sibling sets.**
   `promoteChildRunForToken` called `promoteRun({mode:"local_merge"})` without
   `autoOnReady`, so the human-review target-drift gate threw PRECONDITION
   "reviewedTargetCommit is required" — dead-ending BOTH `run_promote` AND the
   as-plan auto-promoter. The sibling autonomous path (`auto-delivery.ts`) sets
   `autoOnReady: true`; this one silently didn't. The only existing test stubbed
   `promote`, so no test drove the real service.

## Root Cause

1. The guard was written where the author was looking (the queue promoter), not
   derived from "every transition into the guarded state." `tryStartRun` and
   `promoteNextPending` are TWO independent Pending→Running edges; a guard that
   protects a shared-tree invariant must sit on both (or in a shared helper both
   call), exactly like the run-status fan-out rule.

2. Merge verification scope was chosen by "what this branch touched" instead of
   "what now depends on a changed shared contract." When a branch widens a shared
   emit/DTO/required-field, EVERY consumer's exact-shape assertion across the
   WHOLE repo is implicated — including tests the branch never opened (they came
   from main). A targeted run cannot see them.

3. Two callers of one core (`promoteRun`) with different required-flag
   combinations and no shared wrapper — the classic "// the other path sets X"
   divergence. The autonomous-vs-human distinction lived as an ad-hoc flag at
   each call site instead of one `promoteAutonomously()` seam.

## Solution

1. Added the `sharedWriterSiblingActive` check to `tryStartRun` (fetch
   `workspace_mode`/`root_run_id`, block the flip when a sibling is active),
   mirroring `promoteNextPending`. Added an S-1 test that drives the REAL
   `tryStartRun` path with the cap NOT full (so the block is provably the guard,
   not the cap).
2. After ANY merge that widens a shared contract, run the FULL `vitest run
   --project integration` (and full unit) — not a hand-picked subset — BEFORE
   claiming the merge green. Fixed the two stale `emit-terminal` assertions to
   expect `parentRunId: null`.
3. Added `autoOnReady: true` to `promoteChildRunForToken` (readiness re-gate
   still runs). Regression-locked by a promote-rework integration test that
   drives the real service to Done + asserts the `run.done` payload's
   `parent_run_id`.

## Prevention

- **Guard-on-every-edge:** when adding a contention/invariant guard, grep every
  transition INTO the guarded state (here: every `status: "Running"` write) and
  put the guard on each, or factor a shared admit() helper. A guard on one of N
  edges is a guard on none. A test that stubs the unguarded edge is false green.
- **Merge verification = full suites, scoped by CONTRACT not by diff.** If a
  merge changes a required field on a shared emit/DTO/payload, the blast radius
  is every exact-shape (`toEqual`) assertion in the repo, including main-authored
  tests the branch never opened. Run the full unit AND full integration suite
  before stamping a merge "green"; a targeted run is for iteration, not sign-off.
- **Two callers of one core with different required flags = a missing seam.**
  When `foo.ts` passes `{autoOnReady:true}` and `bar.ts` doesn't, that's a code
  review smell: extract `promoteAutonomously()` so the flag can't be forgotten,
  and pin each public entry route with a test that drives the REAL service (never
  a stub) at least once.
