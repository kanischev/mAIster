# Required token audit and scope-contract drift

**Date:** 2026-06-08 21:13
**Files:** web/lib/tokens/ext-handler.ts, web/lib/tokens/__tests__/ext-handler.test.ts, docs/decisions.md, docs/configuration.md, docs/error-taxonomy.md, .ai-factory/ROADMAP.md, .ai-factory/plans/feature-token-actor-scope-support.md
**Severity:** high

## Problem

The shared `/api/v1/ext/*` token handler caught and discarded
`recordTokenAudit()` failures. A token-authenticated route could therefore
return a success or expected authorization response even when the mandatory
`token_audit_log` write failed.

Current-state docs also still described token scopes as full-project or
HITL-only opt-in enforcement after the code had moved to default route-level
scope enforcement.

## Root Cause

The M16 handler originally treated audit writes as best effort so endpoint
responses would not be blocked by audit persistence. That assumption became
invalid once token audit became a required accountability boundary for named
human-owned and project automation tokens.

The scope model changed incrementally across M16/M17/follow-up work, but not
every current contract document was updated from "scope label/audit-only" to
"enforced allow-list with `*` compatibility".

## Solution

Added `recordRequiredTokenAudit()` in the shared external handler. It logs
structured `[FIX:token-audit-required]` context and rethrows audit write
failures, so routes fail closed instead of returning success without a
required audit row. Non-critical `last_used_at` updates remain asynchronous but
now log warnings instead of failing silently.

Added focused unit regression tests for success-path and insufficient-scope
audit failures. Updated current ADR/config/error-taxonomy/roadmap/active-plan
text to describe route-level scope enforcement and `*` as the broad
compatibility scope.

## Prevention

- Treat token audit writes as required security/accountability writes in code
  review; do not use `.catch(() => {})` around `recordTokenAudit()`.
- Add a focused failure-injection test whenever a route relies on audit writes
  for accountability.
- When a security contract moves from planned/partial to implemented, search
  ADRs, configuration docs, error taxonomy, roadmap, and active plans for old
  scope/auth language.
