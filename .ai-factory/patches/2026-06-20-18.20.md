# Execution-control policy: review-finding fixes (orphan escalate, silent enum, squash-PR)

**Date:** 2026-06-20 18:20
**Files:** web/lib/flows/graph/runner-graph.ts, web/lib/runs/execution-policy.ts, web/lib/services/runs.ts, web/lib/runs/promote.ts, web/lib/runs/__tests__/execution-policy.test.ts, web/lib/flows/graph/__tests__/runner-graph.integration.test.ts, web/lib/runs/__tests__/promote-pr.test.ts, docs/system-analytics/execution-policy.md
**Severity:** medium

## Problem

`/aif-review` of the `feature/execution-control-policy` branch surfaced six findings; the load-bearing ones:

- **F1 (orphan state):** the rework-exhaustion `escalate` path (`runner-graph.ts`) called `runReviewHuman(...)`, **discarded its return**, then unconditionally flipped the run to `NeedsInput`. Its comment claimed `runReviewHuman` "creates a fresh HITL request + assignment" — but under an `unattended` policy (`humanGate=auto_pass`) the same function can auto-pass / ship-with-warning and create **no** HITL row. That would leave a `NeedsInput` run with no HITL request and no assignment — invisible in the inbox, unresolvable.
- **F2 (silent no-op enum):** `crashRetry=auto_retry` was accepted by the launch zod schema but had no implementation, so the ralph consumer silently skipped it → it behaved as `fail`. Same shape for the documented aliases `commits=defer` (==`keep_all`) and `commits=squash_on_promote` (==`squash_rework`).
- **F4 (doc/code drift):** the analytics-doc mermaid showed `squashRunBranch → merge OR PR`, but the code `return`ed for PR mode **before** the squash block, so `commits=squash_rework` was silently a no-op for PR-mode promotions.

## Root Cause

- **F1:** two policy axes that fire at the **same node** gave contradictory instructions — `reworkExhaustion=escalate` ("pause for a human") vs `humanGate=auto_pass` ("don't pause"). The escalate caller assumed the pause-creator always pauses, but the pause-creator honors a *different* axis that can short-circuit. The bug was masked by an emergent invariant (an `auto_pass` node only reaches its escalate route when evidence is not-ready, and re-entry finds the same not-ready state → it pauses correctly), so it was unreachable today — but undocumented, unverified, untested, and one refactor from a stuck run. **A caller that ignores a callee's return value is asserting an invariant it does not enforce.**
- **F2:** an enum value reached a typed schema before it had an implementation. Validation passed, so the value flowed to a dispatcher that only matched the implemented arms and silently fell through to the default — a no-op trap rather than a loud failure.
- **F4:** an early `return` for one branch (PR) placed below the diagram-described step (squash) meant the diagram described intended behavior the code never executed.

## Solution

- **F1:** added an optional `forcePause` flag to `runReviewHuman`'s ctx that skips the B2/B3 auto-pass/on-stuck short-circuit and always creates the HITL request + assignment. The escalate caller passes `forcePause: true` (the machine is stuck — it MUST reach a human even under `auto_pass`). The misleading comment was corrected.
- **F2:** added `assertExecutionPolicySupported(policy)` (throws `MaisterError("CONFIG")` for `crashRetry=auto_retry`), called in the launch path right after `assertNoBlindShip`. The value now fails loudly (UI-visible) instead of degrading to `fail`. `defer`/`squash_on_promote` are correct (they do what their names imply) so they were kept, documented as aliases, and pinned with tests. Owner decision: implement `auto_retry` later (filed as a follow-up), reject until then — "no silent troubles."
- **F4:** owner chose to extend squash to PR. Hoisted the squash into a `squashForCommitPolicy(claim, runId)` helper that runs **before** either side-effect (merge AND PR), returning whether the history was rewritten; the PR push force-updates (`--force-with-lease`) **only** when a squash actually rewrote `base..HEAD`. The mermaid is now accurate.
- **F3 / F5 / F6** (owner: leave member-level / intentional seam / rebase at merge time) were resolved by documenting the intentional choices in code comments + the analytics doc; no behavior change.

## Prevention

- **Never discard the return of a function that may pause/escalate.** If a caller flips state to `NeedsInput` after calling a pause-creator, it must either (a) honor the callee's "did I actually pause?" return, or (b) pass an explicit force flag so the callee's policy short-circuits cannot fire. A code comment asserting "X always creates a HITL" is a smell when X reads a policy axis that can change that.
- **When two policy axes can act on the same site, test their interaction explicitly.** The single most valuable missing test (per the review) was exactly the A1×B2 cross-axis case; it had full single-axis coverage and zero interaction coverage.
- **A schema must not accept a value with no implementation.** Either implement it, or reject it at the entry boundary with a typed error. A dispatcher that `=== "known"` and falls through to a default turns an unimplemented enum into a silent no-op. Grep every consumer of an enum's resolver when adding a value (here: `crashRetryFromSnapshot` → ralph consumer; `commitsFromSnapshot` → promote).
- **A diagram step below an early `return` is drift.** When a doc describes step S feeding both branch A and branch B, verify neither branch `return`s before S.
- **Sibling sweep paid off:** grepping all `runReviewHuman` callers confirmed only the escalate path ignored the return (the normal node path handled it), scoping the fix precisely.
