# Path traversal via runId/stepId in supervisor StartSessionRequestSchema

**Date:** 2026-05-26 12:45
**Files:** supervisor/src/types.ts, supervisor/src/__tests__/types.test.ts, web/lib/supervisor-client.ts
**Severity:** medium

## Problem

In `supervisor/src/types.ts`, the `StartSessionRequestSchema` Zod validator
accepted any non-empty string for `runId` and `stepId`. Both values flow
straight into `path.resolve()` in two places:

- `supervisor/src/spawn.ts` at the log path:
  `resolve(runtimeRoot, ".maister", projectSlug, "runs", runId, stepId + ".log")`
- `supervisor/src/cost.ts` at the cost.jsonl path:
  `resolve(runtimeRoot, ".maister", projectSlug, "runs", runId, "cost.jsonl")`

A caller with HTTP access to the supervisor could set
`runId: "../../../etc"` (or similar) to escape `.maister/` and write log
content under whatever directory the supervisor process had write access
to. The supervisor is internal-network in POC topology and has no auth,
which mitigates blast radius — but defense-in-depth matters.

Two minor cleanups landed in the same patch:

- `web/lib/supervisor-client.ts:162` had `if (!res.ok && res.status !== 204)`
  — the `&& res.status !== 204` is dead because 204 is always `res.ok`.
- `web/lib/supervisor-client.ts` SSE parser kept a `currentEvent` variable
  that was written but never read (the typed `data:` JSON already carries
  the event type). Removed.

## Root Cause

The Zod validator was authored with the **shape** of the payload in mind
(runId, stepId are "non-empty strings") but did not enforce a
**filename-safe** invariant. The code consuming the values
(`spawn.ts`, `cost.ts`) trusted that callers would only pass safe segments,
which is a classic "trust at the wrong boundary" mistake: validation at
the type level (`z.string().min(1)`) is not the same as validation at the
**use-site contract** (path component without `..` or `/`).

The dead `currentEvent` variable is residual scaffolding from an earlier
SSE-parser design where the event-type header was tracked separately;
once the supervisor's `data:` payload was made fully self-describing
(`{type, sessionId, monotonicId, ...}`), the variable became orphaned but
wasn't pruned.

The redundant `204` check came from defensive coding: `204` was listed as
a "success" status alongside the implicit `res.ok` check, not realizing
`res.ok` (`status >= 200 && status < 300`) already covers it.

## Solution

Added a `SAFE_PATH_SEGMENT` regex (`^[A-Za-z0-9._-]+$`) and a `.max(128)`
length cap to both `runId` and `stepId` in `StartSessionRequestSchema`.
Now `..`, `/`, `\\`, null bytes, spaces, and any other shell- or
path-metacharacter inputs are rejected at the boundary with a 409
`PRECONDITION` response from the supervisor — the path-resolution code
in `spawn.ts` / `cost.ts` never sees them.

Added six new unit tests to `types.test.ts` covering:

- `runId` with path-traversal segment (`../../../etc`)
- `runId` with forward slash
- `stepId` with path-traversal segment
- `stepId` with whitespace
- `runId` longer than 128 characters
- Happy case: both fields with dots, dashes, underscores

Removed the dead `currentEvent` variable and the redundant `204` branch.

All 111 tests across both packages still pass; both packages lint and
typecheck clean.

## Prevention

- **Validate use-site contracts, not just type shape.** When a string
  flows into `path.resolve` / `path.join` / `fs.*` / `child_process.spawn`
  argv / SQL / HTML, validate against the **target's invariant**, not
  just the string's syntactic shape. A regex constraint at the Zod
  schema is the cheapest enforcement point.
- **Treat every HTTP boundary as untrusted**, even when the service is
  internal-network. Internal services often gain auth later (or get
  exposed by mistake), and defense-in-depth costs ~5 minutes per field.
- **Code-review checklist for path operations**: trace every `path.resolve`
  / `path.join` back to the validator. If the input is `z.string()` or
  any other "open" type, that's the flag.
- **ESLint rule candidate**: `no-unused-vars` should catch dead local
  variables like the removed `currentEvent`. It's already at `"warn"` in
  the supervisor config — the warning was visible during lint but slipped
  past review. Tighten to `"error"` in a future cleanup if false-positive
  rate is acceptable.
- **Boolean simplification reviews**: when a condition has the shape
  `A && X !== K` where `X` is constrained by `A`, check whether the
  right side is implied by the left. This is a frequent source of
  dead code that lint won't flag.
