# Authored Flow package trust-boundary and kind normalization

**Date:** 2026-06-08 12:02
**Files:** web/app/(app)/flows/actions.ts, web/app/(app)/flows/[projectSlug]/[capId]/page.tsx, web/lib/flows/package-authoring.ts, web/app/(app)/flows/__tests__/actions.test.ts, web/lib/flows/__tests__/package-authoring.test.ts, web/messages/en.json, web/messages/ru.json
**Severity:** critical

## Problem

The authored Flow update action accepted `capabilitySlug` from a hidden form
field and stored it into package metadata. A tampered form could therefore make
the portable package identity diverge from the catalog capability identity.

Unsupported package file kinds were also preserved after validation. Invalid
drafts could be stored with raw unknown kinds and later rendered through a
dynamic translation key.

## Root Cause

The update action treated hidden form fields as authority even though hidden
fields are still client-controlled request body data. The catalog capability
slug was already available server-side, but the action did not derive package
metadata from it.

The file-kind validator recorded `unsupported_kind` issues but returned the
original kind unchanged. That preserved invalid enum values across the storage
and rendering boundary.

## Solution

The update form no longer submits `capabilitySlug`. The server action loads the
authored capability after project authorization and uses the server-owned slug
when building package metadata.

Package validation now exposes a shared kind guard, records unsupported kinds
as invalid, and normalizes the stored kind to `asset`. The detail page also uses
a localized unsupported-kind fallback for stale stored data that predates
normalization.

## Prevention

- Treat hidden inputs as locators or concurrency tokens only, never as
  authority for resource identity.
- When validation allows invalid drafts to be saved for editing, normalize any
  UI-visible enum to a safe stored value or use an explicit localized fallback.
- Add regression tests that submit tampered hidden fields and unsupported enum
  values through the same server action boundary users hit.
