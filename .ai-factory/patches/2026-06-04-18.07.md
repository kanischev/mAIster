# Calibration outcome table in configuration.md missing `invalid_confidence` row

**Date:** 2026-06-04 18:07
**Files:** docs/configuration.md
**Severity:** low

## Problem

The M15 verdict-calibration hardening added a fourth fail outcome,
`invalid_confidence` (a `confidence` present but outside the `0..1` domain —
non-finite, `<0`, or `>1` — fails closed). It was wired through the code
(`web/lib/flows/graph/gates-exec.ts` `calibrateVerdict`), the type
(`web/lib/db/schema.ts` `GateVerdict` calibration union), and the analytics
SSOT doc (`docs/system-analytics/readiness.md` — flowchart node, outcome
table row, and the MUST bullet). But the parallel outcome table in the
user-facing config reference `docs/configuration.md` (the `maister.yaml` /
`flow.yaml` / calibration reference) still listed only the three original
fail/pass outcomes (`above_threshold` / `below_threshold` / `no_confidence` /
`missing_confidence_allowed`). Two docs describing the same calibration table
disagreed; the config reference understated the shipped behavior.

## Root Cause

Two surfaces document the same feature: the analytics doc (`readiness.md`)
and the config reference (`configuration.md`). The implementing change updated
`readiness.md` (the obvious analytics surface for a state/classifier change)
but missed `configuration.md`, because the calibration outcome table is
duplicated across both files rather than living in one canonical place with a
cross-reference. Per `docs/CLAUDE.md` R7 ("cross-reference, do not duplicate"),
this duplication is the structural smell that let the drift happen — when the
same table exists twice, a single-surface edit silently diverges.

## Solution

Added the missing row to the `configuration.md` calibration outcome table,
placed with the other "confidence present" case (right after `< min` /
`below_threshold`, before the "absent" rows) so the present-vs-absent grouping
reads cleanly:

`| yes | yes | out of \`0..1\` (non-finite, \`<0\`, or \`>1\`) | — | failed | invalid_confidence |`

Also appended one clause to the adjacent "fail-closed on missing confidence"
paragraph clarifying that `allow_missing_confidence` rescues only an *absent*
confidence, never one *present* but out of domain — directly relevant because
that paragraph is exactly where a reader would (wrongly) assume the flag also
covers invalid values. Wording mirrors `readiness.md` verbatim so the two
surfaces now match. Verified: `pnpm validate:docs` green (3/3 mermaid blocks),
table is well-formed (6 cells/row), and the row matches both the code
(`calibrateVerdict`) and the `GateVerdict` schema union.

## Prevention

- When a change adds/changes a gate verdict outcome, run status, enum value,
  or any classifier state, grep the new identifier across BOTH `docs/system-analytics/`
  AND `docs/configuration.md` (and `error-taxonomy.md` for error codes) — the
  same value is frequently tabulated in more than one doc surface.
- The deeper fix is the `docs/CLAUDE.md` R7 collapse: a calibration-outcome
  table that lives in two files will drift again. **Done in a same-session
  follow-up** — the duplicated table was removed from `configuration.md`,
  which now links to the canonical **calibration truth table** in
  `system-analytics/readiness.md` (anchor
  `#verdict-calibration-at-gate-execution-ai_judgment--skill_check`) and
  deliberately does NOT re-enumerate the outcome strings, so there is no
  remaining drift surface. `configuration.md` keeps only the config-knob
  reference (`confidence_min`, `verdict_calibration`, `allow_missing_confidence`).
- Review-side: the `/aif-review` "re-derive every analytics/spec surface"
  rule already caught this. Keep treating cross-surface doc tables as a
  reconciliation set, not just the primary analytics doc.
