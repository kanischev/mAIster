# Dispatcher batch cap overshoot + tick/trigger double-launch window + policy copy drift

**Date:** 2026-06-10 11:42
**Files:** web/lib/run-schedules/dispatch.ts, web/lib/run-schedules/__tests__/dispatch.integration.test.ts, web/messages/en.json, web/messages/ru.json, docs/system-analytics/run-schedules.md
**Severity:** high

## Problem

Three adversarial-review findings against the just-shipped M28 run-schedule
dispatcher:

1. With cap 3 and 10 due `skip` schedules, ONE dispatcher batch launched all
   10 — 3 started, 7 landed `Pending` with worktrees created — even though
   `skip` means "no run when at capacity".
2. A cron tick arriving while a trigger-now was between its tx1
   (`dispatching` committed) and tx2 (final outcome) could claim the same
   still-due row and launch a second attempt; the loser's fast
   `launch_failed` overwrote the marker, so the winner's tx2 CAS dropped and
   a REAL run ended up invisible (`launch_failed`, no `last_run_id`).
3. The `start_anyway` modal hint said "always launch a new run, even in
   parallel" — but the overlap matrix only lets it bypass the GLOBAL cap; a
   busy task is always skipped.

## Root Cause

1. **Per-row cap reads ignore intra-batch state.** `decideAndStage` computed
   `capFull` from `countLiveRuns(tx)` — but staged launches don't create run
   rows until AFTER the claim transaction commits, so every row in the batch
   saw the same pre-batch count. The "benign race" rationale (external
   concurrent launch wins a slot → `queued_pending` recorded honestly) was
   wrongly stretched to cover pressure the dispatcher itself created — a
   knowable, self-inflicted overshoot, not a race.
2. **Asymmetric staleness guards.** `dispatchScheduleNow` refused fresh
   `dispatching` rows (300s window) but the tick's due-claim predicate did
   not — the guard was written once at the surface where the hazard was
   first imagined (the button) instead of at the shared claim layer where
   both paths intersect. Worse, the first fix attempt
   (`AND NOT (outcome = 'dispatching' AND ...)`) silently excluded EVERY
   fresh row: `NULL = 'dispatching'` is NULL, `NOT NULL` is NULL, and a NULL
   WHERE term drops the row — SQL three-valued logic.
3. **UI copy written from the policy NAME, not the policy MATRIX.** The
   hint paraphrased "start anyway" instead of the DQ7 row, and the catalog
   pass transcribed it without cross-checking the spec.

## Solution

1. `dispatchDueSchedules` passes `reservedSlots: intents.length` into
   `decideAndStage`; `capFull = countLiveRuns(tx) + reservedSlots >= cap`.
   Launches staged earlier in the batch now count as occupied slots for
   later rows.
2. `claimDueScheduleRows` (and the truncation count) gained the same
   freshness exclusion as trigger-now, written null-safely:
   `outcome IS DISTINCT FROM 'dispatching' OR last_fired_at IS NULL OR
   last_fired_at <= now - 300s`.
3. EN/RU `modal.overlapHint.start_anyway` now states: bypasses the global
   cap (run queues as Pending), a task with an active run is still skipped.

Red-first tests: batch of 4×skip + 1×queue_one against cap 3 (exactly 3
launches, 1 `skipped_cap`, 1 `catchup_queued`); fresh `dispatching` row never
claimed; a deferred-gate slow trigger-now with a concurrent tick (separate
launch spies — tick spy never called, trigger outcome survives).

## Prevention

- **A guard that exists on one entry path must be hunted on every sibling
  path to the same resource.** When adding a staleness/freshness/ownership
  check to one claimer, grep for every other claimer of the same rows and
  apply or explicitly waive it there in the same change.
- **Batch loops that stage side effects must thread accumulated state into
  per-item decisions** (reserved slots, consumed budget). A per-item
  snapshot read is only valid when items are independent.
- **Any SQL predicate of the form `NOT (nullable_col = X AND …)` is a
  three-valued-logic bug magnet** — write `IS DISTINCT FROM` / explicit
  `IS NULL` arms, and always run the EXISTING happy-path suite right after
  touching a claim predicate (the regression showed up instantly there).
- **UI copy for policy enums must be written against the decision matrix in
  the spec, not the enum name.** Review pass: for each policy option hint,
  point at the matrix row that proves it.
