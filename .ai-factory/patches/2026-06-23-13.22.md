# Half-shipped web↔supervisor wire contract: emitter added without the acceptor

**Date:** 2026-06-23 13:22
**Files:** supervisor/src/types.ts, supervisor/src/__tests__/types.test.ts, web/lib/services/hitl.ts, web/lib/assignments/service.ts, web/lib/flows/graph/runner-graph.ts, web/lib/agents/launch.ts, web/lib/runs/__tests__/execution-policy.test.ts, web/lib/services/__tests__/hitl.test.ts, docs/system-analytics/guardrail-hooks.md
**Severity:** high

## Problem

The G4 guardrail-hook-engine Phase 1 added `hooksConfig` to the `POST /sessions`
body in the web tier (`supervisor-client.ts` + two threading sites), but the
supervisor's `StartSessionRequestSchema` ends in `.strict()` and `http-api.ts`
parses with `.parse()` (not `.safeParse`). The supervisor schema had no
`hooksConfig` key. So the moment `resolveHooksConfig` returned anything
non-`undefined` — a `settings.hooks` node, an `unattended`-preset run (auto-seeds),
or an agent declaring `hooks` — the supervisor would throw
`ZodError: Unrecognized key(s) 'hooksConfig'` and the session would never spawn.
The feature would brick the exact unattended/guardrailed runs it exists to protect.

Secondary, dormant fan-out gaps from the same change: the new `hook_trip` HITL
kind was added to the schema enum but NOT to the human-actor-only respond guard
(`hitl.ts`) — a machine/agent token could dismiss its own guardrail trip
(fail-open) — nor to the assignment `actionKind` Extract (a hook_trip assignment
wouldn't typecheck). A domain doc also cited `web.openapi.yaml` as a `hook_trip`
wire home that (correctly) does not exist.

## Root Cause

A **runtime contract was split across two deploy units and only one half landed**.
Two compounding reasons it slipped:

1. **Phase boundary drawn through a contract.** The plan scoped Phase 1 as
   "web-only" and deferred supervisor parsing to Phase 2. But an over-the-wire
   field is a *contract*, not a tier-local concern — the acceptor must land with
   the emitter even if it accepts-and-ignores until enforcement arrives.
2. **The test seam couldn't see it.** Web unit tests mock the supervisor client,
   so no test exercises the real supervisor Zod schema against a real web payload.
   tsc can't catch it either: the web `HooksConfig` type and the supervisor schema
   are structurally unrelated. The gap was invisible to every green gate.

The dormant `hook_trip` gaps share a root cause: **a new enum value was added to
the schema but not fanned out to every consumer of that enum** (the human-only
guard, the assignment Extract). New enum members are silently permissive at the
consumers that don't yet branch on them.

## Solution

- **C1:** Added `hooksConfig` to `StartSessionRequestSchema` as a faithful mirror
  of the OpenAPI contract (`{repetition?:{max:int≥1}, noProgress?:{maxTurns:int≥1},
  pathGuard?:{allowedPaths:string[]}}`, all inner objects `.strict()`, the block
  `.partial().strict().optional()`) — **accept-and-ignore** until Phase 2 wires the
  interceptor. Pulled this one sliver of T2.1 forward so the contract is consistent
  at HEAD. TDD: wrote 4 failing supervisor schema tests first (watched the two
  "accepts" cases fail under `.strict()`), then added the key → green.
- **S1 (fail-open):** Added `hook_trip` to the human-actor-only respond guard in
  `hitl.ts`. Proved the test was non-vacuous by neutralizing the guard line and
  watching the new test go RED (an api_token reached `handleFormHumanResponse`
  instead of being rejected), then restored → green.
- **S2:** Added `hook_trip` to the assignment `actionKind` Extract (verified the
  underlying `assignments.action_kind` enum already carries it, so the Extract is
  a real narrowing, not a silent no-op).
- **C2:** Deleted the false `web.openapi.yaml (hook_trip HITL)` Wire line.
- Added the missing `presetFromSnapshot` unit tests (the fail-safe-to-`supervised`
  invariant — a corrupt policy must never auto-arm guardrails — was untested while
  every sibling `*FromSnapshot` had 7–14 cases) and the plan-required DEBUG log
  (`armed: Object.keys(hooksConfig)`, keys only — never path values) at both
  threading sites, hoisting the resolver call to a `const` to keep the resolver pure.
- Producer-coupled remainders (the hook_trip respond-dispatch branch, the run-detail
  render branch, the migration-snapshot regen vs the parallel Studio `0062`) were
  left to the phases that own their producers (T3.4 / T5.3 / T6.3 — already in the
  plan) rather than pre-implemented untestably.

## Prevention

- **Runtime contract symmetry — never half-ship an over-the-wire field.** When you
  add a field to a cross-process request body (here `POST /sessions`), the
  acceptor schema on the other side MUST change in the SAME commit, even if it
  only accept-and-ignores. "This phase is web-only" is not a license to split a
  contract.
- **Pin cross-tier contracts with a test that runs the REAL acceptor schema.** A
  web payload mocked at the client boundary proves nothing about the supervisor.
  The cheap guard is a supervisor-side unit test that feeds a representative
  resolved payload through `StartSessionRequestSchema` — it would have failed the
  instant the emitter shipped without the acceptor.
- **A new enum value is not done until it is fanned out to EVERY consumer.** Grep
  every site that branches on the enum (auth guards, Extract/narrowing types,
  dispatch switches, read models, i18n) — a consumer that doesn't branch on the
  new value is silently permissive (here: a machine token dismissing a guardrail
  trip). Default a new HITL/escalation kind to the *human-only* side of any
  actor-authorization guard.
- **Mirror an OpenAPI object faithfully**: `additionalProperties:false` → `.strict()`
  on every level; `required:[...]` → non-optional inner fields; per-key optionality
  → `.partial()` on the wrapper. Keep the Zod mirror and the spec in lockstep.
