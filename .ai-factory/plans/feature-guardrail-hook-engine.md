## ⚠ Ground-truth corrections (verified in code on current `main` @ `0e869906`, post-M38)
The brief is structurally accurate; these are the precise seams + number/version corrections confirmed by 4 Explore passes + an aif-improve deep-trace:

- **The generalization target is real and pre-hoc.** `supervisor/src/acp-client.ts` already does inline, hardcoded enforcement in the `requestPermission` SDK callback: L1 `resolveReadOnlySessionDecision()` (lines ~126–154, allow-set `{read,search,fetch,think}`) and L2 `resolveReadOnlyAutoReject()` (lines ~88–103, mutating-set `{edit,write,create,delete,move}`). **Both run BEFORE the tool executes** (the callback `await`s; the SDK does not run the tool until it resolves). This is the single viable pre-hoc blocking point. Attach the new interceptor here, after L1/L2/B1 and before the HITL-deferred path (~line 391). (ADR-090; L3 dirty-watchdog is post-run, `web/lib/agents/dirty-watchdog.ts`.)
- **CRITICAL — only `kind` is standardized at the seam.** `params.toolCall` is typed `unknown`; the current `ToolCallLike` only guarantees `{toolCallId?, title?, kind?}`. The **file path is in the opaque, adapter-specific toolCall body**, NOT standardized. The path-scoped write-guard (rule 2) therefore requires per-adapter parsing of the toolCall body. Owner chose **all-5-adapter** coverage (Phase 0 spike), with a kind-only coarse fallback for any adapter whose shape can't be verified.
- **Per-session state hook.** `SessionRecord` (`supervisor/src/types.ts` ~383–416) is mutable, shared by reference across the `requestPermission`/`sessionUpdate` closures + registry + heartbeat. New counter fields land here (`record.monotonicId` is the only counter today). Counters are in-memory only (lost on supervisor crash → run reconciled `Crashed`; acceptable).
- **`sessionUpdate` is post-hoc** (the tool already ran) — usable for the no-progress watchdog (count turns since last edit/diff-producing tool call), NOT for blocking.
- **Reuse, do not re-implement, escalation:** `onStuck` axis = `z.enum(["escalate","ship_with_warning","notify_only"])` in `web/lib/runs/execution-policy.ts` (`onStuckFromSnapshot`, fail-closed `escalate`); `assign = onStuck !== "notify_only"`. The execution-policy **presets** (`supervised | assisted | unattended`) are the existing axis that classifies "human watching vs not" — the two-tier default keys off `unattended`. `logExecPolicyAction(record)` in `web/lib/runs/exec-policy-audit.ts` is a **pino LOG boundary, not a table**; kind union includes `"escalated"`. `emitRunEscalated` (`runner-graph.ts` ~361) → `emitDomainEvent("run.escalated")` + `emitWebhookEvent("run.escalated")`.

## Locked decisions (owner-approved, 2026-06-22)
1. **Universal mechanism FIRST, claude-native particular SECOND — both in THIS plan** (Q1 + Q4-followup + improve-pass). The supervisor ACP-seam interceptor (P2/P3) is the universal, vendor-neutral mechanism: all 5 adapters, all 3 rules, complete on its own. The owner's correct observation: `settings.local.json` `hooks` is **claude-only** (no universal native-hook file exists). So native is a per-adapter particular delivered THROUGH a clean `NativeHookMaterializer` seam — the **seam interface ships in P1, and the claude PreToolUse implementation ships as P4, AFTER the universal core is proven** (owner: "после основного механизма имплементим и шипим claude code specifics"). P4 is **spike-gated** on the adapter actually honoring settings hooks (graceful degradation otherwise — the universal layer carries enforcement, no dead code).
2. **Trip→HITL: dedicated `hook_trip` kind + migration 0063** (Q2). Follows the `budget_breach` precedent (one kind per escalation cause). Add `hook_trip` to `hitl_requests.kind` AND `assignments.action_kind`.
3. **Path-guard action: deny-and-continue** (Q3). Out-of-set write → denied/cancelled tool outcome (like L1); the agent adapts. Repeated denials feed the breakers, which then halt+escalate.
4. **Adapter path-extraction: all 5 adapters** (Q4) + kind-only coarse fallback for any unverified shape.
5. **Default-ON for `unattended` runs** (Q5). The two liveness breakers (repetition + no-progress) auto-arm with env-default caps **`MAISTER_HOOK_REPETITION_MAX=5` / `MAISTER_HOOK_NO_PROGRESS_TURNS=15`** for runs under the `unattended` execution-policy preset, per-node opt-out. supervised/assisted stay opt-in. Path-guard is always opt-in (needs an explicit writable set).
6. **Agent runs escalate WITH a resume path** (Q6). flow/agent hook-trip → `NeedsInput`, routed through each `run_kind`'s EXISTING resume path (flow → `runFlow`; agent → the agent HITL resume that already powers agent permission HITL; scratch → in-session deny + chat notice, no `NeedsInput`). NOT terminate-only. Verified in Phase 0.
7. **No new `MaisterError` code.** Trip → checkpoint → `NeedsInput` escalation (recoverable), never `Failed`-terminate. `hooks` enforceability = `instructed` in `ENFORCEABILITY_BY_AGENT` for MVP (a `strict` declaration → refused at launch, existing M11c boundary; supervisor enforcement is deterministic but NOT modeled as `enforced` in the static table — avoids reopening ADR-041).
8. **MVP = exactly 3 rules.** Secret-scan / others = fast-follow. **ADR-041 strict-capability flip stays FROZEN.**

## Design decisions resolved (rationale → freeze in Phase 0 spec/ADR)
- **D1 — Trip dispatch is event-driven from the supervisor; the state transition is web-owned.** Supervisor emits `session.hook_trip {rule, lifecycle, disposition: "deny"|"halt", toolCall?}`. `deny` (path-guard) → supervisor returns the reject/cancel option inline (no web round-trip). `halt` (rules 1 & 3) → supervisor returns cancelled + stops issuing work; the **web** consumer performs checkpoint+escalate (the runner owns `NeedsInput`).
- **D2 — Escalate honors `onStuck` + branches on `run_kind`** (decision #5/#6). `escalate`/`ship_with_warning` → assigned HITL; `notify_only` → unassigned. flow + agent → checkpoint + `NeedsInput` + `hook_trip` HITL, resumable via the run-kind's existing resume path; scratch → deny + in-session notice, no `NeedsInput`. (Branch BEFORE routing — the budget non-flow trap.) Absent execution-policy snapshot → treat as non-unattended (fail-safe to opt-in).
- **D3 — `hooksConfig` is the resolved, materialized rule set, delivered via `StartSessionRequest`** beside `readOnlySession` (left intact; ADR-104 documents it as conceptually a special case). It composes with `readOnlySession` (both pre-hoc denies; readOnlySession is the stricter superset — harmless overlap).
- **D4 — Two-tier default resolution** (decision #5). The Phase-1 resolver reads the run's execution-policy preset: if `unattended` and the node did not opt out, seed `repetition` (=5) + `noProgress` (=15) from `MAISTER_HOOK_*`; `pathGuard` only when the node explicitly declares it.
- **D5 — No-progress = "turns since last edit/diff-producing tool call"** counted supervisor-side from `sessionUpdate`. Richer "no new git diff" web cross-check deferred.
- **D6 — Engine floor.** A node/agent declaring `hooks` requires `compat.engine_min ≥ 1.8.0` (load-time gate + compile-time verify). Flows without `hooks` are byte-identical.
- **D7 — `NativeHookMaterializer` seam: interface in P1, claude impl in P4** (decision #1, improve-pass). The per-adapter interface `{ adapter; materialize(hooksConfig, worktreePath): void }` resolves from an adapter→materializer registry; P1 registers a no-op so the universal path is unaffected; **P4 registers the claude materializer** (PreToolUse path-guard into `settings.local.json`), spike-gated. The native hook's `allowedPaths` derive from the SAME resolved `hooksConfig.pathGuard` (one source of truth — the two backends cannot diverge). Native covers **only the path-guard (rule 2)**; rules 1 & 3 stay supervisor-only (cross-turn state). Adding the `hooks` key must respect the M14 ownership-marker/reclaim/cleanup protocol.

## Atomicity & crash-window analysis (skill-context rule)
- **Escalate = ONE transaction (budget's GOOD in-tx pattern):** checkpoint pre-tx (`EXECUTOR_UNAVAILABLE` → bail, retry next signal), write `needs-input.json` pre-tx (unlink on tx failure), then a single `db.transaction` { `runs.status→NeedsInput` CAS-on-`Running`, `markNodeNeedsInput`, `hitl_requests.insert(kind:"hook_trip")`, optional `createHitlAssignmentForRun`, `run.needs_input` + `run.escalated{reason:"hook_trip",rule}` } all in-tx (ADR-086 exactly-once), then post-commit `logExecPolicyAction("escalated", {reason:"hook_trip", rule})`.
- **Crash windows:** (a) supervisor crash mid-trip → in-memory counters lost, no live session → existing reconcile marks `Crashed`. (b) web crash AFTER checkpoint but BEFORE the escalate tx → run still `Running` + valid checkpoint + session gone → existing crash-reconcile sweep handles it; test that a checkpointed-but-not-escalated trip is reconciled, not stranded. (c) Counters are per-session in-memory only — a resume resets them; document that a resumed run starts counts fresh.
- The supervisor counter mutation + deny/halt decision are synchronous within the single `requestPermission` callback (no partial state).
