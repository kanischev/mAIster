## Cross-Cutting Invariants (project rules applied to every task)

These come from `.ai-factory/skill-context/aif-plan/SKILL.md` and the locked ADRs. Each is a standing acceptance gate; `/aif-verify` re-derives them from the diff.

- **R-SDD (front-load spec).** Phase 0 is docs-first and must be COMPLETE + internally consistent before any code phase. No "docs-as-final-sync".
- **R-TRUST (separate trust from execution).** Capability import must order `fetch/install → establish trust → execute`. The `setup.sh`-running function is PHYSICALLY SEPARATE from the install function; execution is gated on a persisted trust state, never on "just installed". Mandatory regression: untrusted source carrying `setup.sh` ⇒ script did NOT run.
- **R-PATH (path-segment safety).** Every capability/import `id` and `version` that can reach a filesystem path or a git operation MUST validate against `SAFE_PATH_SEGMENT` (`web/lib/flow-paths.ts:10`, `/^[A-Za-z0-9._-]+$/`) + `notDotRef` (no `.`/`..`/embedded `..`) BEFORE any path, DB, or route use — mirror the existing `flowIdSchema`/`versionSchema` (flow-paths.ts:18–27) and the `assertFieldSafe`→`MaisterError("FLOW_INSTALL")` guard (flow-paths.ts:77). Validation is enforced twice (defence-in-depth): at the Zod schema AND inside the path builder. Mandatory regression: an import `id` of `../evil`, `..`, or `a/b` is rejected at both layers and never reaches `~/.maister/capabilities/`.
- **R-SYM (config-state symmetry).** Every YAML→DB persist task tests SET / CLEAR / idempotent-reset. Removing a `capability_imports[]` entry disables its `capability_records`.
- **R-ID (identifier trust labels).** Every new/modified HTTP route labels each consumed identifier `url-param | auth-context | server-state | body-controlled`. No `body-controlled` field names a cross-resource locator the handler already has a `server-state` source for.
- **R-2PC (two-phase commit).** Any route whose terminal response depends on a downstream side-effect (git clone, `setup.sh`, supervisor RPC, file write) writes intent BEFORE the side-effect and the idempotency marker (`setupStatus='done'`, `trustStatus`) AFTER. Failure-classification table mandatory.
- **R-DEFER (deferred-release).** Any code that creates a supervisor-side deferred (ACP permission, session) names every release call on every failure path. "Log and continue" is never acceptable. Cleanup-failure ⇒ explicit `MaisterError`, never swallowed.
- **R-ATOMIC (multi-store atomicity + crash windows).** Any transition writing across >1 store (ledger `materialization_plan` + status + on-disk files) folds DB writes into one `db.transaction`; on-disk side-effects stay BEFORE the tx (clean 409, no ledger row) or are recoverable AFTER commit. Enumerate each crash window with a tested recovery path.
- **R-FANOUT (consumer fan-out + allow-list).** A new ledger field / enum / route is grepped into EVERY consumer (read models, scheduler, sweeps, guards, API spec, docs). Guards are allow-lists of exact states, never deny-list complements.
- **R-CONTRACT (spec traceability).** Every external contract surface that changes is traced to its spec file in Phase 0 and the implementing phase (table below).
- **R-TEST (test integrity).** Each promised test names its runner project + include glob; each phase exits on full suite green; assertion migrations are in-scope and enumerated by file.
- **R-DEPLOY (deployment touchpoints).** Every new env var / on-disk cache path / bind mount lands in `.env.example` + the relevant compose service in the same phase.
- **R-I18N.** Every new UI string ships EN + RU (`web/messages/en.json` + `ru.json`).
- **R-SECRET (no secrets in the worktree).** Secret values (env-profile values AND any credential an MCP-server config carries) are NEVER written into the agent worktree and NEVER persisted to the ledger/UI. Secrets reach the adapter ONLY through `adapterLaunch.env` (injected into the child process by `spawn.ts:188–196`, never written to disk in the worktree); worktree config files (`settings.json`, `.mcp.json`) reference secrets by env-var NAME only. Catalog/ledger/logs/SSE keep env redacted to key-names. Mandatory regression: grep the entire materialized `.maister/capabilities/**` tree AND the `materialization_plan` ledger AND any UI payload for every secret value ⇒ absent.
- **R-CONSERVATIVE (enforcement).** A cell flips `instructed→enforced` only after a per-class/per-agent spike PROVES the adapter honors the materialized config. Unverified classes stay `instructed` with a documented rationale. The contract only tightens.

## Architectural Decisions (the non-obvious ones, recorded for the implementer)

- **AD-1 — Materialization plan lives in the ledger, not a new artifact kind.** Store the resolved+materialized plan as a `node_attempts.materialization_plan` jsonb column (mirrors the existing `enforcement_snapshot` column), NOT an `artifact_instances` row. This satisfies AC #4 ("records it in the run ledger") and AC #10 (snapshot resolved revisions) without touching M12's *closed* artifact-kind catalog (no `materialization_plan` kind, no projector/validity-FSM/evidence-graph fan-out). Rejected alternative: new artifact kind — too much fan-out for an internal evidence object.
- **AD-2 — No separate flow-run profile table.** `scratch_capability_profiles` stays scratch-only. Flow runs persist the profile inside `materialization_plan` jsonb (digest + per-capability resolved revisions + materialized file paths). Simpler; no second source of truth.
- **AD-3 — Native provisioning is per-agent, per-session-scope, inside the worktree.** Materialize concrete files into `worktreePath/.maister/capabilities/<runId>/<nodeAttemptId>/` (node-scoped) before `POST /sessions`; pass the dir via `capabilityProfilePath` + concrete adapter flags via `adapterLaunch.preArgs/env`. The supervisor stays dumb (already supports both fields). All adapter-specific knowledge lives in `web/lib/capabilities/agent-map.ts`.
- **AD-4 — Conservative flip.** `ENFORCEABILITY_BY_AGENT` cells flip only after the Phase-5 spike proves enforcement for that exact (agent, class). The `permissionMode` cell requires a live-adapter re-run (M11c spike was unverifiable). Unflippable cells keep `instructed` + a rationale comment. This is the only milestone that may flip cells; flipping activates the `EXECUTOR_UNAVAILABLE` branch.
- **AD-5 — Imports reuse the flow-install pipeline.** `installCapabilityRevision` mirrors `installRevision`: clone-by-tag → `gitRevParseHead` → record SHA/manifest/trust → setup deferred. Cache at `~/.maister/capabilities/<id>@<sha[:12]>/`. Trust via `resolveTrust` + a new `MAISTER_TRUSTED_CAPABILITY_SOURCE_PREFIXES`.
