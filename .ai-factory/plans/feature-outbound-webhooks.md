## Design Decisions (DQ1-8 resolved)

### DQ1 — Enqueue seam: transactional outbox at the transition writepoints
The projector seam is WRONG for this taxonomy: it sees session noise, not lifecycle transitions, and is barred from status semantics. Chosen seam: a tiny **`emitWebhookEvent(tx, …)` INSERT into a `webhook_events` outbox row inside the SAME transaction** as each taxonomy-mapped DB transition (where a transition is a bare CAS UPDATE today, wrap UPDATE+INSERT in one `db.transaction`). Fanout/matching/delivery happen entirely OFF the run path inside the M24 tick.
- ADR-#1-safe: no fs.watch, no new watcher, no polling for state transitions — capture rides the existing write path; the tick draining a DB queue is the sanctioned M24 clock doing queue work (same class as keepalive sweeper), and it drives webhook-delivery state, never run state.
- Run-safety: the only write-path addition is one INSERT with no reads and no network. It can only fail if the surrounding transaction fails anyway (DB down). Matching/subscription reads are NOT done on the write path.
- Rejected alternatives: (a) projector hook — wrong event source, layering violation; (b) jsonl-cursor consumer advanced by tick — sees session events not curated transitions, needs replay-dedupe machinery; (c) inline fanout at transition — puts subscription reads + N inserts into hot transactions for no latency win that matters (delivery is tick-paced anyway).
- Accepted trade-off: delivery latency ≈ one tick interval (cadence 60s + external cron period). Fine for notification/CI semantics; documented.

Deliberately NOT emitted v1 (documented in analytics doc): NeedsInput↔NeedsInputIdle checkpoint/resume, HumanWorking claim/release/return, run creation/Pending, keepalive bumps, gate `pending/running/stale`, `gate.opened`, node_attempts lifecycle, all `session.*` stream events. Adding a type later = taxonomy entry + one emit site + doc row (cheap, additive). `gate.opened` rationale (user-confirmed 2026-06-10): sync gate kinds open-and-decide within one runner pass, so `opened` would be back-to-back noise before `gate.decided`; the genuinely-waiting cases are already signalled by `run.needs_input`+`hitl.requested` (human_review) or belong to the inbound gate-unblock sibling (external_check); additive later via one emit at the `gate_results` insert.

Envelope v1 (HTTP body). Emit sites store only the minimal event record (`type`, `projectId`, `runId`, `occurredAt`, per-type `data`) — NO joins on the write path. The full envelope is built ONCE at FANOUT (the project/run blocks need `projects` + `workspaces.branch` + `tasks` — verified: `runs` has no `branch` column and no unified run+project loader exists) via a single batched join, frozen into `webhook_events.payload` inside the fanout tx, and reused byte-identically by every retry and replay; only `deliveryId`/`attempt` are injected at send:

```json
{
  "apiVersion": 1,
  "id": "<event id>",
  "type": "run.review",
  "occurredAt": "<ISO-8601 UTC>",
  "deliveryId": "<delivery id>",
  "attempt": 2,
  "project": { "id": "…", "slug": "…", "name": "…" },
  "run": { "id": "…", "taskId": "…", "flowId": "…", "branch": "…", "status": "Review" },
  "data": { /* per-type, specified in analytics doc; ids + statuses + titles only — NEVER secrets, env, tokens, or raw agent output */ }
}
```

### DQ3 — Tables (4 new + 1 column; migration 0040)
No existing shape fits (scheduler_jobs is a recurring-job catalog; capability_records is project-only). New, in `web/lib/db/schema.ts`:

- **`webhook_subscriptions`**: `id` text PK (server `crypto.randomUUID()`), `project_id` text FK→projects NULLABLE (**null = platform scope** — single table instead of the two-table MCP split because the row shape is identical and routes/authz differ only by scope; partial indexes per scope), `name` text, `url` text (http/https only, boundary-validated), `method` text enum `POST|PUT` default `POST`, `headers` jsonb `Record<string,string>` default `{}` (values MAY be `env:NAME` refs, resolved server-side at send, returned to clients as the raw ref string), `event_types` jsonb `string[]` (validated against taxonomy; literal `"*"` allowed), `signing_secret_ref` text NOT NULL (`env:NAME`), `secondary_signing_secret_ref` text NULL (rotation overlap), `enabled` boolean default true, timestamps. Index `project_id`.
- **`webhook_events`** (transactional outbox): `id` text PK, `project_id` FK NOT NULL, `run_id` FK NOT NULL, `type` text, `data` jsonb (per-type minimal facts written at emit), `payload` jsonb NULL (**full envelope, built and frozen at FANOUT — null until fanned out**), `occurred_at` timestamptz, `fanout_at` timestamptz NULL (**null = awaiting fanout**; this nullable column IS the fanout cursor — no separate cursor table), `created_at`. Partial index `(created_at) WHERE fanout_at IS NULL`. Retention: zero-delivery events pruned after 7d (T17); events referenced by any delivery are kept forever (replay/audit).
- **`webhook_deliveries`**: `id` text PK, `event_id` FK, `subscription_id` FK ON DELETE CASCADE (vacuous in practice — subscription DELETE is service-guarded, see Risks), `status` text enum `pending|delivered|dead` default `pending`, `attempt_count` int default 0, `next_attempt_at` timestamptz NOT NULL, `lease_expires_at` timestamptz NULL, `idempotency_key` text NOT NULL (= hex sha256(`subscriptionId:eventId`)), `last_http_status` int NULL, `last_error_kind` text NULL, `last_error_message` text NULL (truncated ≤1KB), `delivered_at` NULL, timestamps. **UNIQUE `(subscription_id, event_id)`** (dedupe invariant; fanout inserts `ON CONFLICT DO NOTHING`), partial index `(next_attempt_at) WHERE status='pending'`, index `(subscription_id, created_at DESC)` for the log UI.
- **`webhook_delivery_attempts`** (per-attempt audit, mirrors scheduler_job_runs spirit): `id` PK, `delivery_id` FK ON DELETE CASCADE, `attempt_no` int (UNIQUE with delivery_id), `requested_at`, `duration_ms` int, `http_status` int NULL, `error_kind` text NULL (`timeout|network|http|config`), `error_detail` text NULL ≤1KB, `response_snippet` text NULL ≤1KB.
- **`platform_runtime_settings.webhooks_enabled`** boolean NOT NULL DEFAULT true (global kill-switch).

Dedupe/cursor model: capture is exactly-once because the outbox INSERT shares the transition's transaction (no jsonl replay to dedupe). Fanout is exactly-once because `fanout_at` is set in the SAME tx that inserts the delivery rows, with the unique `(subscription_id,event_id)` as belt-and-braces. Delivery is at-least-once (see DQ6 crash windows).

### DQ4 — Retry policy
- Schedule (delay after failure #n): `1m, 5m, 15m, 1h, 4h, 12h, 24h` → **max 8 attempts total** (initial + 7 retries, ≈ 41.5h span), then terminal `dead` (DLQ state on the delivery row — no separate queue table needed; the row + attempts ARE the dead-letter record). Jitter ±20%. Effective floor = tick cadence (60s).
- HTTP `410 Gone` → `dead` immediately. Any 2xx → `delivered`. Everything else (incl. 3xx — `redirect:"manual"`, signatures must not follow cross-origin redirects) → retry. `error_kind=config` (missing env secret) also retries on the same curve — the operator may export the var; never logs the ref's value.
- Budget: `webhookDelivery: 1` in `budgets.ts` (singleton drainer job — concurrency bound), OUTSIDE the flow cap like `command`. Per-drain volume bounded by `MAISTER_WEBHOOK_DELIVERY_BATCH` (default 20) with fixed in-drain HTTP concurrency 5 and `MAISTER_WEBHOOK_TIMEOUT_MS` (default 10000) per attempt.
- Scheduler-shape decision: **one recurring singleton job** `webhook_delivery.default` (cadence 60s, seeded in `ensureDefaultSchedulerJobs`) whose handler does fanout+drain — NOT one scheduler_job per delivery. Per-delivery jobs would flood the admin jobs catalog, fight `cadenceIntervalSeconds NOT NULL` recurrence semantics, and trip `maxFailures=3` auto-disable against our 8-attempt curve. Retry state lives on `webhook_deliveries.next_attempt_at`; the job's own `consecutiveFailures` only tracks handler crashes.

### DQ5 — Signing
- **HMAC-SHA256** (`node:crypto`), secret resolved server-side at send from `signing_secret_ref` (`env:NAME` → `process.env.NAME`).
- Signature base string: `"${t}.${deliveryId}.${rawBody}"` where `t` = unix-seconds send timestamp (binds time + delivery + exact bytes; consumer rejects stale `t` to kill replay attacks).
- Headers: `X-Maister-Signature: t=<unix>,v1=<hex>[,v1=<hex2>]` · `X-Maister-Event: <type>` · `X-Maister-Event-Id` · `X-Maister-Delivery-Id` · `X-Maister-Idempotency-Key` · `User-Agent: MAIster-Webhooks/1` · `Content-Type: application/json`.
- Rotation: optional `secondary_signing_secret_ref`; when set, a second `v1=` entry is appended (Stripe scheme) — consumer accepts any match, operator overlaps then drops the old ref.

### DQ6 — Delivery semantics: at-least-once, unordered, consumer dedupes
Per-subscription ordering is rejected: it forces serial delivery per endpoint, so one delivery in a 24h backoff dams every later event for that endpoint (head-of-line blocking) — fatal for notification semantics. Events carry `occurredAt` + `run.id` so consumers can order; `X-Maister-Idempotency-Key` (stable across retries AND replays) gives consumers exactly-once effect. Crash windows (enumerated, all converge to duplicate-send which the key absorbs): death between claim-lease and POST → lease expiry → reclaim; death between POST and attempt-record → same. `FOR UPDATE SKIP LOCKED` + lease prevents concurrent double-send; concurrent-tick integration test asserts it.

### DQ7 — UI placement
- **Platform**: new `WebhooksPanel` + `webhook-modal` on `web/app/(app)/settings/page.tsx` (view-table + edit-popup, exactly the `mcp-servers-panel` precedent), incl. the global `webhooks_enabled` switch and per-row: enabled toggle, Ping, Deliveries drawer (attempt history + Replay).
- **Project**: new `webhooks` tab in `VALID_TABS` (`web/app/(app)/projects/[slug]/page.tsx`) rendering `web/components/board/panels/webhooks-panel.tsx`; shared scope-parameterized client components live in `web/components/webhooks/*` and are reused by both surfaces. i18n namespace `webhooks` in `web/messages/en.json` + `ru.json` (+ nav key for the tab).

### DQ8 — Replay
`POST …/deliveries/[deliveryId]/replay`: single tx — `SELECT … FOR UPDATE`; allow-list `status ∈ {delivered, dead}` (pending → 409 `CONFLICT`); set `status='pending', attempt_count=0, next_attempt_at=now(), lease_expires_at=null`. The event row is NOT re-emitted and the idempotency key is unchanged → consumer-side safe. Attempt audit rows keep appending (`attempt_no` continues from the running total). Test ping: synchronous signed POST of a synthetic `ping` envelope from the route (user-initiated, not the run path; 10s timeout), result returned live, NOT persisted — no DB write means no two-phase-commit obligation (documented).

## Security & constraints checklist (locked)
- Delivery NEVER blocks/fails a run: write-path = one same-tx INSERT; all network I/O in the tick handler; handler errors land in `scheduler_job_runs`/attempt rows, never raised into run code.
- One clock: no setInterval/cron besides the existing `/api/cron/tick`; ADR-#1 respected (see DQ1).
- Secrets: `env:`-refs only in DB/UI/API responses (reuse the MCP zod regex), resolved at send, never logged/streamed/clientside. Envelope `data` carries ids/statuses/titles only.
- Errors: reuse `CONFIG` (bad ref/validation), `CONFLICT` (replay state, dup), `PRECONDITION`, `UNAUTHORIZED` — **no new MaisterError code**, `docs/error-taxonomy.md` untouched. Attempt `error_kind` is a local enum, not a MaisterError.
- SSRF hardening: scheme allow-list http/https only; private-address blocking deliberately NOT in v1 (self-hosted, operator-trusted endpoints; M16 precedent: "public-internet webhook hardening beyond token/HMAC" deferred) — documented in the analytics doc + ADR.
- Identifier discipline (all routes): `slug`/`id`/`deliveryId` = url-param; user = auth-context; project↔subscription↔delivery ownership = server-state joins (mismatch → 404); bodies carry config values only — zero body-controlled cross-resource ids; ping/replay take empty bodies.

- **Subscription DELETE is usage-guarded** (user decision 2026-06-10): refused with 409 `CONFLICT` while any delivery history exists — retiring an endpoint = `enabled=false`; hard delete works only for never-delivered subscriptions. Delivery history (`webhook_deliveries`+`webhook_delivery_attempts`) is per-subscription audit (Deliveries drawer, replay targets, DLQ inspection); it has no meaning outside its subscription, but per the project's append-only-ledger DNA it is never silently destroyed. Matches the `platform_mcp_servers` usage-guarded DELETE precedent.
