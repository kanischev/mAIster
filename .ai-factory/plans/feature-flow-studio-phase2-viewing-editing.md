## Decisions (design questions resolved)

**D1 — Installed-content source of truth + viewer route (DQ1).** File bodies and raw flow.yaml text come from DISK at `flow_revisions.installed_path` (the only place they exist); graph topology compiles from the DB `manifest` jsonb (digest-pinned, survives disk issues). The viewer is a **new project-scoped RSC page** `web/app/(app)/projects/[slug]/packages/[flowRefId]/page.tsx` — the URL segment is the HUMAN-READABLE package id `flow_ref_id` (e.g. `/projects/myapp/packages/bugfix`), never a row UUID; unique per project via the `(project_id, flow_ref_id)` flows constraint (owner-confirmed 2026-06-10) — `?rev=<revisionId>` selects a non-enabled revision (default = enabled revision), `?file=<relPath>` selects an artifact. **No new GET content API routes** — server components read disk directly (ADR-066 precedent: the run/project `files/content` routes were deliberately retired in favor of RSC reads; page query params are documented in system-analytics, not OpenAPI). Authz: `requireProjectAction(projectId, "readRepoFiles")` (member — same bar as the workbench repo file browser; brief's "read-only view per project membership"). Degraded mode: if `installed_path` is missing on disk, the page still renders metadata + graph + raw-manifest-derived info from DB, with a typed "bundle not available on disk" state for the files section (no throw).

**D2 — Fork-to-edit seeding (DQ2).** New route `POST /api/projects/[slug]/flow-packages/[flowRefId]/revisions/[revisionId]/fork`, authz `manageCatalog`. Body = `{slug?, title?}` ONLY (names for the NEW resource — not cross-resource locators). Server flow: resolve flow row + revision row from url-params (server-state) → `readAuthoredFlowPackageDirectory(revision.installedPath)` → `createAuthoredCapability({kind:"flow", slug, title, body: packageBody, manifest, sourceFlowRefId: flow.flowRefId})` in ONE transaction. Slug default = `flowRefId`; on `(project_id, kind, slug)` collision probe `-fork`, `-fork-2`, … server-side. `presentation` carries over automatically (it lives inside the seeded `flowYaml` text + manifest). Response `{capId, projectSlug, slug}` → client navigates to the existing editor route `/flows/{projectSlug}/{capId}`. `sourceFlowRefId` is added to `createAuthoredCapabilitySchema` (service type already has it). All reads precede the single DB write — no side-effect-after-response, no idempotency marker needed (a retry just creates a second fork under a probed slug; an explicitly-supplied colliding slug → `CONFLICT` 409). The fork READS `setup.sh` content as draft text and NEVER executes anything (trust/execution separation: no execution path exists in this feature; `exec_trust` is only DISPLAYED).

**D3 — Decoupling the read-only graph (DQ3).** `FlowGraphView` gets an optional `runContext?: { runId; initialStatuses; currentStepId; runStatus }` (existing run callers unchanged). Absent → static mode: no `useRunStream` subscription, no graph-status fetch, no status chips / current-node ring — pure topology + presentation layout. The viewer RSC compiles `compileManifest(revision.manifest)` → `buildGraphTopology` + `presentationLayout` server-side and passes plain DTOs — identical to the run layout's path, honoring authored `presentation` with dagre fallback (already built into `toFlowGraphView`).

**D4 — File model: tree over files[] + kind-by-path (DQ4).** The data model stays `files[{path, content}]` (no schema change); the tree is a derived client view grouped by path segments. **Kind becomes strictly inferred from path** via `classifyPackageFile` — the manual kind `<select>` is removed (it is a correctness fix: install/bridge classify by path only, so a manually-set kind silently diverges at publish). Add = new path (with current folder prefilled); rename and move are ONE operation = editing the full path (validated against the existing `unsafe_path`/`duplicate_path`/`path_conflict` issue codes), kind re-inferred on change. No drag-and-drop.

**D5 — form_schema builder depth (DQ5).** The lighter option: a structured field editor over `formSchemaSchema` (rows: name/label/type/required/options; recursive `fields` for `object`) with a raw-JSON CodeMirror toggle, plus a LIVE preview rendering `HitlDecisionControls` with `formFieldsFromSchema` and no-op callbacks. The same builder serves `output.result` schemas (same grammar per `output-schema.ts`/ADR-063). No full visual drag-builder.

**D6 — Per-kind validation severity (DQ6).** One shared module produces issues `{severity: "block"|"warn", code, path, message}`; the BLOCK set is wired into the server draft-save hard-gate (alongside `assertAuthoredFlowManifestValid`, before the CAS write → `MaisterError("CONFIG")` → 422) and mirrored client-side for inline UX:
- **BLOCK**: everything blocked today (paths/dup/conflict/kind/NUL/UTF-8 + manifest hard-gate); `schemas/*.json` that fails `JSON.parse`; a schema file **referenced by the manifest** (any node `form_schema:` or `output.result.schema:` path) that fails `formSchemaSchema`; `skills/**/SKILL.md` or `agents/*.md` with missing/unparseable frontmatter or missing `name`/`description` (runtime consumes them verbatim — silent dead capability otherwise).
- **WARN** (advisory UI only, never blocks): rule guardrail frontmatter shape (`allowed_paths`/`forbidden_paths`/`allowed_commands`/`require_structured_response` — no web runtime parser exists, so no block), shell heuristic lint findings, `formSchemaSchema` grammar on UNreferenced schema files, unknown frontmatter keys.
- Per-agent frontmatter variants: **keep the flat `agents:["claude","codex"]`** (hardcoded at publish today) — no per-agent variants in this plan.
- Fork compatibility: an installed package whose pre-existing artifacts violate BLOCK rules still forks (fork persists via `createAuthoredCapability`, validation collected not thrown); the first SAVE surfaces blocks — by design (the user is editing anyway).

**D7 — Typed edges: one source of truth (DQ7).** Both the new connect-modal and the existing side-form write through the same `editor-state` `setTransition(manifest, source, outcome, target)` → `applyManifest`. On connect: a small modal collects the outcome (default `success`; suggestions `success|failure|rework|takeover` + free text; if the outcome already exists for the source, the modal warns it will retarget). No second edge store, no double-edit.

**D8 — Position-persistence verdict (DQ8, audited).** (d) drag→serialize and (e) editor↔view projection parity already WORK. Fixes shipped here: **(b)** `addNode` writes the canvas spawn x/y into `presentation` at add time (Phase 2); **(c)** `presentationLayout`/`FlowLayout` carry `width/height/color` through, `toFlowGraphView` applies size + color, node side-form gains the three optional presentation inputs (Phase 2); **(a)** the YAML-tab/canvas state fork is fixed by the Phase-3 live YAML→graph re-seed (single owner of manifest state in `FlowEditorTabs`; debounced parse; parse error keeps last-good graph + inline banner).

**Engine/DB impact: none.** `width/height/color` are already in `flowPresentationSchema`; `source_flow_ref_id`, `version_binding`, `exec_trust` all exist. **No migration, no engine bump, no new `runs.status`, no new `MaisterError` code.**

**Owner sign-off (2026-06-10):** Logging `standard` confirmed. Viewer stays project-scoped with the package name (`flow_ref_id`) as the URL segment — confirmed. Rule-guardrail frontmatter validation is WARN-only — confirmed. Presentation scope = rendering (size/color applied in views) + side-form numeric/color inputs; NO canvas resize handles / color palette — confirmed. Kind-by-path confirmed: the manual kind `<select>` in the package files editor is removed (it is ignored by publish/install anyway — `classifyPackageFile` decides by path); the editor shows a read-only inferred-kind badge (D4/T4.3).

## Cross-cutting compliance (project aif-plan skill-context)

### HTTP identifiers (trust labels)
- **Viewer RSC page** `/projects/[slug]/packages/[flowRefId]` — `slug`, `flowRefId` = `url-param` (access-controlled via `requireProjectAction`); `?rev` = query, validated as id then resolved via DB join `flow_revisions.flow_ref_id = flows.flow_ref_id AND project` (`server-state` cross-check — a revision id from another flow/project → not-found); `?file` = **untrusted** → zod rel-path schema (no NUL/absolute/`..`/leading `-`) BEFORE any fs call, then lexical-prefix + `realpath` double confinement under `installed_path`. Auth check precedes every read.
- **`POST …/revisions/[revisionId]/fork`** — `slug`/`flowRefId`/`revisionId` = `url-param`, revision resolved through the project-scoped flow row (`server-state`; mismatch → 404); body `slug?`/`title?` = `body-controlled` but name only the NEW authored cap — validated by the existing authored slug regex/title bounds; **no body field names a filesystem path or cross-resource locator** (`installedPath` is read from the DB row).
- No other new routes. (No new content GET routes by design — D1.)

### Two-phase commit / multi-store atomicity
Fork = reads (DB lookups, disk directory read — no side effects) → ONE `createAuthoredCapability` transaction → 201. No AFTER-side external call, no idempotency marker. Failure classification: missing dir / unreadable bundle → 422 `CONFIG` (nothing persisted); revision not found in project → 404; explicit slug collision → 409 `CONFLICT`; DB failure → 500, nothing partially persisted (single tx). No deferreds anywhere in this plan.

### Trust / execution separation
This feature adds NO execution of package content: the viewer renders file text; fork copies `setup.sh`/scripts as draft TEXT. The script editor displays an exec/trust warning surfacing the revision's `exec_trust`, and the existing `runRevisionSetup` gate is untouched.

### Fan-out
No new `runs.status`, enum value, or status-bearing route. The fork route creates an authored cap — already consumed by the existing `/flows` + catalog list read models with no query change.

### Config-state symmetry
N/A — no YAML→DB persistence loop is added (fork is a one-shot copy, not a sync).

### Deployment touchpoints
None. No new env var, port, sidecar, or binary. Shell lint is pure-JS heuristics IN the bundle — explicitly NOT a `shellcheck` host binary (that would require container wiring; rejected in D6/T4.5). All new deps: none (CodeMirror langs, `yaml`, lint addon already installed).
