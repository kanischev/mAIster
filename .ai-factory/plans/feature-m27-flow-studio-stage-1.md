## Multiagent Execution Model (how this plan is executed)

Runs through the project's **SDD + agent-driven TDD** workflow (the model M11c/M12/M15/M19/M26 used). `/aif-implement` is the coordinator; `implement-coordinator` + `implement-worker` carry parallel tasks; `/codex:adversarial-review` is the per-phase challenge gate.

**Phase 0 (spec-freeze) is a HARD GATE.** No code task starts until the SDD spec (`.ai-factory/specs/feature-m27-flow-studio-stage-1.md`) + ADR-066..069 + analytics docs are **complete and internally consistent** (allow-lists written exactly as the code will gate, per `docs/CLAUDE.md` R5a and the skill-context "front-load analytics" rule). The SDD is the **single source of truth**; later deviation requires a **spec amendment**, never an ad-hoc code change.

**Per code task — role rotation (RED → GREEN → review):**

| Role | Agent | Responsibility |
|---|---|---|
| **QA-RED** | test author | Writes the failing test(s) FIRST from a specific SDD acceptance criterion; proves **RED for the right reason** before any implementation. |
| **impl-GREEN** | `implement-worker` | Owns an **explicit, disjoint file set**; writes the minimum to turn RED→GREEN; no scope beyond the task. |
| **reviewer** | `review-sidecar` | Bug-risk + convention + skill-context-rule review of the task diff. |
| **security / docs** | `security-sidecar` / `docs-auditor` | Security check (trust gate, env:NAME secret handling, untrusted manifest parsing) + docs-drift check, where the task touches those surfaces. |

**Per-phase checkpoint:** before each commit checkpoint, a `/codex:adversarial-review` pass challenges the phase diff. A phase is **done only when** (a) RED-before-GREEN evidence exists for every TDD task, (b) full suite green (`pnpm --filter maister-web test` = `test:unit && test:integration`; e2e where the phase lists it), (c) reviewer + adversarial findings are resolved, (d) `pnpm validate:docs:all` green if the phase touched `docs/`.

**Parallelism / write scopes:** tasks within a phase touching **disjoint** files MAY run as parallel `implement-worker`s; tasks sharing a file run **sequentially**. Each task names its owned files — two concurrent workers MUST NOT share a write target. **Track C-admin (platform MCP CRUD) is an explicit PARALLEL SUB-TRACK, independent of Phase A** — it can run concurrently with the editor work.

**Spec-to-test trace:** every SDD acceptance criterion maps to ≥1 named test (the spec's spec-to-test matrix). `/aif-verify` re-derives the matrix from the diff at the end.

## Goal

Turn the read-only M22 flow-graph **view** into a flow **editor** (write path) for **any** installed flow, make an edited flow **immediately runnable** on the next launch (the executable bridge), and add **first-class MCP capability management** across platform-instance / project / flow-package scopes. Edits persist as an **M25 authored `flow` draft revision** (reuse `authored_capabilities` draft→publish, ADR-061), layout lives in the `flow.yaml` **presentation section** (ADR-064), and every save passes the `validateGraphManifest` + `compileManifest` **hard-gate** before persistence. At launch the runner **resolves** the effective flow revision (per a `version_binding` model), compiles it, and **snapshots the resolved set** (flow revision + resolved capability/MCP revisions) into the run, so in-flight runs stay immutable.

## Decisions (locked — §9 open questions resolved)

**Owner-confirmed in planning (AskUserQuestion, 2026-06-08):**

1. **Flow lineage precedence = AUTHORED WINS, realized at the `flows`/`flow_revisions` level via the bridge.** Editing installed flow X authors a draft **linked to X's `flow_ref_id`** (T-A5.1); publish→bridge writes the authored revision into **X's own `flows` row + `flow_revisions` lineage** (same `flow_ref_id`), so there is **NO parallel git-vs-authored store to merge** — both are `flow_revisions` rows under one `flows` row, and `enabled_revision_id` + `version_binding` select among them. "Authored wins" = when `version_binding=latest` and both an authored and a git revision exist for the same `flow_ref_id`, the **authored** published revision is the head; git import is the fallback when no authored revision exists. (§9 Q1; refined by the improve pass — the earlier "two separate stores to unify" framing was wrong.)
2. **Trust model = IMPLICIT logic-trust + EXPLICIT executable-trust — TWO axes (owner-confirmed in the improve pass).** Publishing an authored flow implicitly trusts the flow **logic**: the in-app bridge installs `flows.trustStatus = trusted_by_policy` (NOT the CLI bridge's hardcoded `"untrusted"`, `flows.ts:1039`) so launch's trust guard (#9) passes. A **NET-NEW second axis** (`exec_trust`, default untrusted) gates executable side-effects — `setup.sh` AND MCP **stdio `command`** — via an **explicit per-revision flip**, physically separate from publish. **Concretely:** reuse the existing `runRevisionSetup`/`runSetupSh` runner (already separate + sentinel-guarded, `flows.ts:785,273`) but **change its gate from `flows.trustStatus` to the new `exec_trust` axis** — today it runs once `trusted_by_policy`, which must **no longer** suffice for `setup.sh`. Honors "trust → execute, never execute-then-trust" (brief §3.3/§4). (§9 Q3 + improve-pass confirm.)
3. **setup.sh for authored flows = YES, behind the exec-trust gate.** Making an authored flow executable runs its `setup.sh` **once**, only after the explicit executable-trust flip, via a function physically separate from publish/install, **sentinel-guarded** (`.maister-setup-done`), reusing the M4/ADR-043 fetch→trust→execute pipeline shape. (§9 Q5.)
4. **Capability id-collision precedence = PROJECT → PLATFORM → FLOW-PACKAGE (local-first), uniform across ALL capability kinds (mcp, skill, rule, agent_definition, restriction).** This is **consistent with the codebase, not a deviation**: the established **runner-resolution** chain is already local-first — *project* default outranks *platform* default (root `CLAUDE.md` §5, steps 3>4 and 5>6). The brief §3.7 arrow "platform → project → flow-package" is therefore read as a broad→specific **lookup listing**, not a priority order; the **priority is local-first**. A **project**-scoped record overrides a **platform**-scoped one of the same id, which overrides a **flow-package**-declared default — i.e. *operator's project config > operator's instance config > third-party package default*. Same id + **different params** → higher-precedence wins, lower **shadowed (no merge)**. ⚠ **Current code picks NO winner** (`resolver.ts:158-162` returns ALL same-id records → both materialize, a latent duplication); C7 introduces winner-picking, which **both** implements local-first AND fixes that duplication, for **every** capability kind. (§9 Q6.)

**Resolved from the brief / codebase (stated, not re-litigated):**

5. **Write target (§3.1).** Edits (logic + layout) persist as an **M25 authored `flow` DRAFT revision** of `flow.yaml` (reuse `authored_capabilities` + `draft_version` optimistic concurrency, ADR-061). Layout lives in the `flow.yaml` `presentation` section (`flowPresentationSchema`, ADR-064). **Generic for ANY installed flow** — a git-imported flow is edited by creating an authored draft seeded from its pinned manifest, never by mutating the pinned bundle.
6. **Hard gate (§3.2).** `validateGraphManifest` + `compileManifest` run **before** any draft save or publish; an invalid manifest → `MaisterError("CONFIG")` and is **NEVER persisted** (validation precedes the M25 CAS write).

A floated/resolved revision still passes **trust + setup-status + engine-compat** gates before launch. In-flight runs stay IMMUTABLE (ADR-021/§5/§7); an edit changes only what the NEXT launch resolves. 14. **No engine bump.** Presentation is additive/runner-ignored; structural edits stay within the existing `flow.yaml` DSL (`nodes[]`, engine `1.3.0`). MCP transport additions are additive to the capability material shape, not the flow DSL grammar.

### Two-phase commit & deferred-release (skill-context rules)

- **Draft save (A5)** and **version-binding/trust flips (B1/B3)** are pure DB writes (no external side-effect) → single transaction, no two-phase needed; the hard-gate validation runs **before** the CAS write so an invalid manifest never persists.
- **Authored→executable bridge (B2)** has a downstream disk side-effect. **This two-phase + multi-store atomicity is ALREADY implemented** by `installAuthoredFlowPackageBridge`→`installFlowPluginImpl` (`ensureRevisionIntentRow` intent `Installing` BEFORE the disk write → `finalize`/`Installed` AFTER, `web/lib/flows.ts:507,588,811`); the plan **reuses** it (parameterized to `trusted_by_policy`) rather than re-deriving it. The per-phase failure classification already exists; the SDD cites it. **No ACP deferred is created on the editor/bridge path** (no agent session) — the deferred-release rule does not apply here; it DOES apply at launch (B4/C8), reusing the existing `launchRun`→supervisor discipline (M7/M8) unchanged.
- **`setup.sh` execution (B3)** reuses `runRevisionSetup`/`runSetupSh` (`web/lib/flows.ts:785,273`) — already fetch/install→trust→execute, physically separate from install, sentinel-guarded. The only change: its gate moves from `flows.trustStatus` to the **net-new `exec_trust` axis**, so logic-trust (`trusted_by_policy`) alone does NOT run `setup.sh`. Mandatory regression: a logic-trusted (`trusted_by_policy`) authored flow with `setup.sh` → assert the script did NOT run until the explicit `exec_trust` flip.

### Multi-store atomicity / crash windows (skill-context rule)

- **Bridge publish (B2/B3)**: the authored-revision publish (M25 `publishAuthoredCapabilityLocal`, already one tx with `draft_version` CAS) is extended to also create the runnable revision intent row **in the same transaction** (DB-only writes are atomic). The disk materialize + `setup.sh` happen AFTER commit, behind the exec-trust gate, idempotently (re-run safe via sentinel + `Installed` marker). Crash windows: (a) committed authored revision + missing runnable row → next publish/launch re-derives it idempotently; (b) runnable row `Installing` + disk materialize incomplete → launch refuses (`packageStatus≠Installed`, existing guard) and a re-materialize completes it. Each window has a tested recovery path.
- **Launch snapshot (B4/B5)**: the resolved-set write (flow revision + resolved capability/MCP revisions) folds into the **existing single `runs` insert transaction** alongside `flow_revision_id`/`runner_snapshot` (`services/runs.ts:590-601`). No new intermediate committed state.

### Config-state symmetry (skill-context rule)

- **Flow-package MCP declarations → records (C4)**: a flow.yaml that declares required/additional MCPs projects them into the resolution candidate set. SET (declared → resolvable) / CLEAR (declaration removed on next authored revision → no longer required, launch no longer refuses on it) / re-SET (re-added → required again) — all three are mandatory tests, mirroring the M6/ADR-061 `material.origin='authored'` SET/CLEAR discipline.
- **`version_binding` toggle (B1)**: `latest → pinned → latest` round-trip resets resolution behavior each time (toggle is the config; tested both directions).
